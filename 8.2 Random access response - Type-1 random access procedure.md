# 🔖 章节：8.2 Random Access Response - Type-1 random access procedure
---

## 📖 章节导读：Type-1 随机接入的“基站回执” (Msg2)

**背景信息**：
欢迎来到 3GPP 协议非常经典的核心篇章——**8.2 章节：Type-1 随机接入响应 (Random Access Response, RAR)**。

如果说前面的内容是在教手机如何“敲门”（发 PRACH），那么本章节的核心就是教手机如何**“听门里的回应”**。
在传统的 Type-1（四步）随机接入过程 (4-step RACH) 中：
*   **第一步 (Msg1)**：手机发送 PRACH 前导码（敲门）。
*   **第二步 (Msg2，即本章主角 RAR)**：基站听到敲门声后，给手机回一个消息。

**基站的这个“回执 (RAR)”里通常包含三件极其重要的事情：**
1.  **“我听见你了” (RAPID匹配)**：核对你刚才敲门的暗号。
2.  **“你的表走快/慢了” (Timing Advance)**：告诉手机如何微调发射时间，让信号能精准地卡在基站的节拍上。
3.  **“这是给你安排的发言专座” (UL Grant)**：给手机分配一块专属的时频资源，让手机在接下来的第三步 (Msg3) 里发送自己真实的身份和数据。

### 📚 基础术语表 (本章核心)
*   **Type-1 random access procedure**：Type-1 随机接入过程。**一句话解释**：最经典的 4 步走接入流程（Msg1 -> Msg2 -> Msg3 -> Msg4）。
*   **RAR (Random Access Response)**：随机接入响应。**一句话解释**：4步流程里的第 2 步 (Msg2)，是基站在下行信道上发给手机的“入网许可证”和“参数微调指南”。

---

## 📖 协议原文拆解 (一)：Msg2 (RAR) 的“监听窗口”与“专属呼机”

> **协议原文**
> In response to a PRACH transmission, a UE attempts to detect a DCI format 1_0 with CRC scrambled by a corresponding RA-RNTI during a window controlled by higher layers [11, TS 38.321] if the PRACH transmission is not triggered by a PDCCH order that includes a Cell Indicator field with non-zero value; otherwise, the UE does not attempt to detect the DCI format 1_0. The window starts at the first symbol of the earliest CORESET the UE is configured to receive PDCCH for Type1-PDCCH CSS set, as defined in clause 10.1, that is at least one symbol, after the last symbol of the last PRACH occasion corresponding to the PRACH transmission, where the symbol duration corresponds to the SCS for Type1-PDCCH CSS set as defined in clause 10.1. If $N_{TA,adj}^{UE}$ or $N_{TA,adj}^{common}$, as defined in [4, TS 38.211], is not zero, the window starts after an additional $T_{TA}+k_{mac}$ msec where $T_{TA}$ is defined in [4, TS 38.211] and $k_{mac}$ is provided by kmac or $k_{mac}=0$ if kmac is not provided. The length of the window in number of slots, based on the SCS for Type1-PDCCH CSS set, is provided by ra-ResponseWindow.

### 📚 增量术语表
* **DCI format 1_0**：下行控制信息格式 1_0。**一句话解释**：基站发给手机的“调度指令”，告诉手机“接下来在哪个位置接收你的专属数据”。
* **RA-RNTI (Random Access RNTI)**：随机接入无线网络临时标识。**一句话解释**：基站用来加密 RAR 指令的“临时密码”，这个密码是根据手机刚才发 PRACH 时的“时间+频率”算出来的。
* **Type1-PDCCH CSS set**：Type-1 公共搜索空间。**一句话解释**：基站专门用来广播系统消息和随机接入响应（RAR）的“公告栏”区域。
* **ra-ResponseWindow**：RAR 响应窗口。**一句话解释**：手机发完 PRACH 后，竖起耳朵等待基站回执的“倒计时沙漏”（最长等待时间）。

### 🚪 第一关：去哪里听？听什么？(监听条件)
手机刚发完 PRACH（敲完门），接下来就要竖起耳朵听基站的响应（RAR）。
* **听什么？** 手机要在控制信道上疯狂寻找一个特殊的调度指令（`DCI format 1_0`）。
* **怎么确认是给自己的？** 这个指令必须是用 **RA-RNTI** 这个密码加密的。因为 RA-RNTI 是由你刚才敲门的时间和地点决定的，所以如果你能用自己的 RA-RNTI 解开它，说明基站是在回应“刚才在这个时频位置敲门的那个人”。
* **例外（不听的情况）**：如果这次敲门是基站强行命令你去的（PDCCH order），并且明确指了别的基站/小区（Cell Indicator 非零），那你敲完门就**不要**在这里等响应了（因为响应可能从别的地方发给你）。

### 🎯 第二关：什么时候开始听？(窗口起始点)
手机不能发完 PRACH 瞬间就开始听，因为电路从“发信号(TX)”切换到“收信号(RX)”需要时间。
1. **至少喘口气**：必须等 PRACH 结束**至少 1 个符号 (symbol)** 的时间。
2. **寻找公告栏**：喘完这口气后，遇到的**第一个**合法的“公告栏”（Type1-PDCCH CSS CORESET）的第一个符号，就是监听窗口的**正式起点**。
3. **卫星/远距离补丁**：如果在非地面网络（NTN，如卫星通信）或超大覆盖小区，电波跑得太慢（$N_{TA,adj}$ 不为零）。那起始点还得再人为往后推迟 $T_{TA}+k_{mac}$ 毫秒，多给电波一点飞行的绝对时间。

### ⏳ 第三关：听多久？(窗口长度)
窗口一旦开启，倒计时就开始。
* **倒计时长度**：由高层参数 `ra-ResponseWindow` 决定（比如 10 个时隙、20 个时隙）。在这个期限内，手机会一直盯着公告栏。如果倒计时结束还没收到自己的 RAR，就视为本次敲门失败，准备发起重传。

### 🔄 RAR 监听窗口时间轴图解
```text
【时域时间轴】

[发PRACH] ─► [切换缓冲] ─► (卫星额外延迟) ─► [RAR 监听窗口 (ra-ResponseWindow)] ─► (窗口结束)
(敲门结束)    (至少1个符号)   (T_TA + k_mac)       │ 寻找 Type1-PDCCH CSS
                                             │ 试图解密 RA-RNTI
                                             │
                                             ├─► 【成功解密】: 收到 RAR! 准备进行下一步。
                                             │
                                             └─► 【超时未收到】: 宣告失败，准备重传。
```

### 💡 章节硬核总结

**本段协议完整定义了 Type-1 随机接入中 Msg2 (RAR) 接收窗口的时域边界与盲检规则。在控制平面，UE 被要求在特定的公共搜索空间（Type1-PDCCH CSS）内盲检使用 RA-RNTI 加扰的 DCI 1_0。在时域平面，协议给出了窗口开启的绝对数学锚点：即位于 PRACH 传输结束加上一个硬件切换符号（1 symbol）之后的首个合法 CORESET 起始处。同时，为适配 NTN（非地面网络）超远传输距离带来的物理时延，协议引入了基于 $T_{TA}+k_{mac}$ 的窗口起始点额外后退（Backoff）补偿机制。窗口的持续长度则由 RRC 层的 `ra-ResponseWindow` 严格管控。**

## 📖 协议原文拆解 (二)：拆解 Msg2 包裹，寻找“专属暗号”与“入场券”

> **协议原文**
> If the UE detects the DCI format 1_0 with CRC scrambled by the corresponding RA-RNTI and LSBs of a SFN field in the DCI format 1_0, if included and applicable, are same as corresponding LSBs of the SFN where the UE transmitted PRACH, and the UE receives a transport block in a corresponding PDSCH within the window, the UE passes the transport block to higher layers. The higher layers parse the transport block for a random access preamble identity (RAPID) associated with the PRACH transmission. If the higher layers identify the RAPID in RAR message(s) of the transport block, the higher layers indicate an uplink grant to the physical layer. This is referred to as random access response (RAR) UL grant in the physical layer.

### 📚 增量术语表
* **SFN (System Frame Number) & LSBs**：系统帧号与最低有效位。**一句话解释**：基站时间的“年月日”。LSBs 就是时间戳的“尾数”。用来核对这个回执是不是真的是回复你之前那个时间点的。
* **Transport block (TB) / PDSCH**：传输块 / 物理下行共享信道。**一句话解释**：基站真正发来的数据包裹（快递箱），由前面解密的 DCI 指引你去接收。
* **Higher layers**：高层（这里特指 MAC 层）。**一句话解释**：物理层的“上司”。物理层只负责搬运无线电信号，拆解数据包的具体内容得交由 MAC 层来做。
* **RAPID (Random Access Preamble Identity)**：随机接入前导码标识。**一句话解释**：你刚才发送的 PRACH 前导码的“编号”（暗号）。
* **UL grant (Uplink Grant)**：上行授权。**一句话解释**：基站给你批的“发件凭证”，上面写了你下一步（Msg3）可以在哪个时频位置发数据。

### 🚪 第一关：物理层的初步核对与“取件”
手机物理层（底层）如果在窗口期内成功用 `RA-RNTI` 解密了控制信息（DCI），它还会做一个额外校验（通常针对卫星通信等超长时延场景）：
* **核对时间戳尾数 (SFN LSBs)**：看看这个回执上的系统时间尾数，是不是和我发 PRACH 那个时刻的时间尾数对得上？（防止窗口期太长导致时间错乱）。
如果对上了，物理层就会去对应的地方把 **数据大包裹 (Transport Block, 装着 RAR)** 给取回来，然后原封不动地**上报递交给它的领导（高层 MAC 层）**。

### 🎯 第二关：MAC 层的“拆包验身份” (RAPID匹配)
**为什么有了 RA-RNTI 还要查 RAPID？**
因为 `RA-RNTI` 只是由“敲门的时间和频率”决定的。如果有两个手机在**同一个时间、同一个频率**敲门（但用了不同的前导码暗号），它们算出的 RA-RNTI 是一模一样的！
此时，基站会把给这两个手机的回信打包进**同一个大包裹（Transport Block）**里。
* MAC 层拿到大包裹后拆开，逐个检查里面的“回信”。
* 它在找什么？它在找里面有没有一封信上写着自己刚才用的**暗号编号 (RAPID)**。

### 🎫 第三关：喜提“Msg3 入场券” (RAR UL Grant)
如果 MAC 层在包裹里找到了写有自己暗号（RAPID）的回信，那就说明基站真的听到自己了！
这封回信里会附带一张极具价值的“入场券”——**上行授权 (UL grant)**。
MAC 层会把这张入场券重新**下发还给物理层**，说：“暗号对上了，这是基站批给我们的资源，准备在下一步（Msg3）按这个许可发数据吧！”
物理层管这个指令叫做 **RAR UL grant**。

### 🔄 跨层交互与拆包流程图
```text
【物理层 PHY】                                【高层 MAC】
      │                                           │
1. 收到 DCI 1_0 (用 RA-RNTI 解密成功)               │
2. 核对时间戳尾数 (SFN LSBs) 吻合                   │
3. 接收 PDSCH 里的数据大包裹 (Transport Block)      │
      │                                           │
      ├─────── (上交大包裹 Transport Block) ──────►│
      │                                           │ 4. 拆解大包裹，查找里面的多个 RAR 回信
      │                                           │ 5. 核对回信上的暗号 (RAPID) 是否匹配
      │                                           │
      │◄────── (下发入场券 RAR UL Grant) ──────────┤ 6. 【匹配成功!】提取出上行资源许可并下发
      │                                           │
7. 物理层拿到许可，开始准备发送下一步的 Msg3 PUSCH
```

### 💡 章节硬核总结

**本段协议揭示了 RAR 接收过程中跨层（PHY-to-MAC-to-PHY）的精密握手机制，以及解决 RACH 碰撞的“两级寻址”原理。第一级寻址在 L1（物理层），利用关联了时频资源的 RA-RNTI 解扰 DCI，获取承载 MAC PDU（即包含一个或多个 MAC RAR 的 Transport Block）的 PDSCH 资源。此时可能存在基于相同时频资源但不同 Preamble 的碰撞（即多个 UE 共享同一个 RA-RNTI）。第二级寻址在 L2（MAC层），通过解析 MAC RAR 的 subheader，提取 RAPID（码域标识）进行精准身份匹配。一旦 RAPID 匹配成功，MAC 层将抽取出 payload 中的 27-bit `RAR UL grant` 并指示给 L1，完成了从 Msg2 接收到 Msg3 调度的关键业务逻辑闭环。此外，SFN LSBs 的引入是为了解决非地面网络 (NTN) 或极大覆盖场景下，超长响应窗口导致的系统帧号绕回 (Wrapping) 模糊问题。**

## 📖 协议原文拆解 (三)：Msg2 接收失败的“四宗罪”与“重传冷却时间”

> **协议原文**
> If the UE does not detect the DCI format 1_0 with CRC scrambled by the corresponding RA-RNTI within the window, or if the UE detects the DCI format 1_0 with CRC scrambled by the corresponding RA-RNTI within the window and LSBs of a SFN field in the DCI format 1_0, if included and applicable, are not same as corresponding LSBs of the SFN where the UE transmitted PRACH, or if the UE does not correctly receive the transport block in the corresponding PDSCH within the window, or if the higher layers do not identify the RAPID associated with the PRACH transmission from the UE, the higher layers can indicate to the physical layer to transmit a PRACH. If requested by higher layers, the UE shall be ready to transmit a PRACH no later than $N_{T,1}+0.75$ msec after the last symbol of the window, or the last symbol of the PDSCH reception, where $N_{T,1}$ is a time duration of $N_1$ symbols corresponding to a PDSCH processing time for UE processing capability 1 assuming $\mu$ corresponds to the smallest SCS configuration among the SCS configurations for the PDCCH carrying the DCI format 1_0, the corresponding PDSCH when additional PDSCH DM-RS is configured, and the corresponding PRACH. For $\mu=0$, the UE assumes $N_{1,0}=14$ [6, TS 38.214]. For a PRACH transmission using 1.25 kHz or 5 kHz SCS, the UE determines $N_1$ assuming SCS configuration $\mu=0$.

### 📚 增量术语表
* **$N_{T,1}$ (PDSCH processing time)**：PDSCH 处理时间。**一句话解释**：手机芯片把空中收到的无线电波解密、解码成能读懂的数字包裹，所需要的运算时间。
* **UE processing capability 1**：终端处理能力 1。**一句话解释**：3GPP 规定的手机处理数据的“底线基准能力”，确保即使是性能最差的低配手机，也能在这个标准时间内算完数据。

### 🚪 第一关：判定敲门失败的“四种死法”
如果手机在发送 PRACH 后遇到了以下**任何一种**情况，本次接收 RAR 的尝试宣告彻底失败：
1. **没听见通知 (No DCI)**：一直等到倒计时窗口结束，都没收到用自己 RA-RNTI 加密的通知指令。
2. **时间对不上 (SFN LSB mismatch)**：收到了通知，但里面的时间戳尾数，和自己刚才敲门的时间对不上（可能是虚假唤醒或过期消息）。
3. **包裹损坏 (PDSCH decode fail)**：通知和时间都对上了，但是去取包裹（PDSCH 数据）的时候，发现包裹在空中传输时严重损坏（CRC 校验不过），根本解不开。
4. **暗号不匹配 (RAPID mismatch)**：包裹完好地解开了交给了上层（MAC 层），但 MAC 层翻遍了里面的信件，发现里面回应的是别人的暗号，没有自己的那个 RAPID。

### 🎯 第二关：怒摔包裹与“复活冷却时间”
一旦发生上述“四宗罪”之一，高层（MAC 层）就会气愤地通知物理层：“刚才那次敲门不算，准备重新再敲一次（重传 PRACH）！”
**但手机不能毫无限制地立刻狂发重传，协议给出了一个精确的准备时间硬性底线**：
* **计算基准点**：从【窗口期结束的那一刻】（死法1），或者从【收完损坏的/不匹配的 PDSCH 包裹的那一刻】（死法3、4）开始算。
* **冷却时长**：手机必须在最迟 **$N_{T,1} + 0.75$ 毫秒**之后，做好发射下一个 PRACH 的一切准备。其中 0.75 毫秒主要留给射频电路做收发转换和调谐。

### 🧩 第三关：处理时间 $N_{T,1}$ 的“木桶效应”
$N_{T,1}$ 到底有多长？它取决于 $N_1$ 个符号的总时间，而符号时间长短由子载波间隔 ($\mu$) 决定。
为了不逼死低配手机，协议在这里用了“木桶效应（就低不就高）”来给足运算时间：
* 从 PDCCH、PDSCH 和 PRACH 三者各自配置的 $\mu$ 中，挑出一个**最小的 $\mu$**（$\mu$越小，符号越长，算出来的宽限时间就越久）。
* 如果算出来 $\mu=0$ (15kHz) 或者是针对非常极端的物联网超窄带 1.25kHz/5kHz，那就统一按照 $N_1=14$ 个符号的宽裕时间来算。

### 🔄 失败与重传时间轴图解

```text
(情况3/4：收到包裹但解算失败/暗号不对)
[接收 PDSCH 结束] ─────────► [高层解包，发现没有我的暗号] ──► [下令重传]
       │
       ├─────────────────────────────────► 时间流逝
       │◄─── 处理时间 N_{T,1} ───►│◄─ 硬件缓冲 0.75ms ─►│

```

### 💡 章节硬核总结

**本段协议完整定义了 Msg2 (RAR) 接收链路的 L1/L2 异常处理机制（Exception Handling）及相应的 L1 重传时序硬约束（Timing Constraint）。协议严谨地罗列了导致 RAR 失败的四个充要条件：DCI 1_0 漏检、SFN LSB 校验失败、PDSCH 解码失败以及 MAC PDU 中 RAPID 缺失。当任一条件触发时，MAC 层将启动 PRACH 重传机制。为了确保基站能够可预测地调度后续资源并给 UE 基带/射频预留足够的 turnaround time，协议强制规定了 UE 的最大准备时延：以最后一个相关的 RX 符号为锚点，经过 $N_{T,1} + 0.75$ ms 的时间上限后，UE 必须处于 PRACH Ready-to-transmit 状态。其中 $N_{T,1}$ 的计算基于 UE Capability 1 并遵循最小 SCS（最宽容时间）原则，这极大体现了 3GPP 在异常恢复流程中“保底兼容性”的设计思维。**

## 📖 协议原文拆解 (四)：Msg2 接收的“波束绑定”与“QCL 假设”

> **协议原文**
> If the UE detects a DCI format 1_0 with CRC scrambled by the corresponding RA-RNTI and LSBs of a SFN field in the DCI format 1_0, if included and applicable, are same as corresponding LSBs of the SFN where the UE transmitted the PRACH, and the UE receives a transport block in a corresponding PDSCH, the UE may assume same DM-RS antenna port quasi co-location properties, as described in [6, TS 38.214], as for a SS/PBCH block or a CSI-RS resource the UE used for PRACH association, as described in clause 8.1, regardless of whether or not the UE is provided TCI-State for the CORESET where the UE receives the PDCCH with the DCI format 1_0.

### 📚 增量术语表
* **Quasi co-location (QCL)**：准共址。**一句话解释**：在多天线/多波束系统中，如果两个信号是 QCL 的，就意味着它们经历了极为相似的空气信道传播路径。对于手机来说，这就意味着可以用**同一个方向的接收波束（同一把伞）**去接收这两个信号，而不需要重新调整天线。
* **CSI-RS (Channel State Information Reference Signal)**：信道状态信息参考信号。**一句话解释**：除了 SSB (公共广播波束) 之外，基站还可以发这种专属的测量波束让手机去测信号质量，也可以用来作为 PRACH 关联的参考锚点。
* **TCI-State (Transmission Configuration Indicator State)**：传输配置指示状态。**一句话解释**：基站显式下发的一组高层配置，用来精确告诉手机“现在该用几号波束来接收信号”。

### 🚪 第一关：收信时的“手电筒对准”问题
前面我们讨论了如何解密通知 (DCI)、如何处理失败，但忽略了一个物理层最棘手的问题：
在 5G 毫米波或多波束场景下，基站的信号是像手电筒光束一样定向打过来的。手机接收 Msg2 (RAR) 时，**该把自己的接收天线对准哪个方向去听呢？**

### 🎯 第二关：不用别人教，抄自己发 PRACH 的作业
协议给出了一个极其霸道且直接的法则：**空间关系（波束）直接绑定。**
只要手机确认这个 DCI 是回执（RA-RNTI 和时间戳对上了），在接收后续的数据包裹（PDSCH）时，手机**完全可以假设（may assume）**这个包裹的无线电信道特征（QCL 属性），和它最初发 PRACH 时所挂靠的那个参考信号（SSB 或者 CSI-RS）是一模一样的。

**生活类比**：
你在音乐节上看到一束追光灯（SSB 波束）照亮了你，你借着这束光的方向朝天上射了一发红色信号弹（发送 PRACH）。
那么，当舞台总监回复你（发送 Msg2 RAR）的时候，他一定会沿着你刚才信号弹的轨迹，用同样方向的追光灯给你打回馈。
因此，你不需要再四处张望，只要死死盯住你最初看到那束光的方向，就一定能最好地接收到包裹。

### ⛔ 第三关：绝对优先级的“霸道条款” (无论高层有无 TCI 配置)
协议最后加了一个非常硬核的补充：`regardless of whether or not the UE is provided TCI-State for the CORESET`。
意思是说：在正常业务阶段，手机接收 DCI 是要看高层（RRC）给分配的波束状态（TCI-State）的。但在随机接入这种极为原始的阶段，基站可能已经下了 TCI 配置，也可能还没来得及下。
**统统不管！** 无论高层有没有给你发 TCI 配置，在收这个特定阶段的 DCI 和 PDSCH 时，**波束一律强制采用 PRACH 关联的锚点波束**。底层物理直觉大于高层配置。

### 💡 章节硬核总结

**本段协议确立了 Msg2 (RAR) 接收阶段至关重要的空间域信道假设（Spatial Domain QCL assumption）。在多波束（Multi-beam）的 5G 网络中，Msg2 的接收发生在 RRC 连接完全建立之前，此时精准的闭环波束管理尚未成型。协议通过强制指定 Msg2（包含 DCI 1_0 及其调度的 PDSCH）与终端发起 PRACH 时所锚定的 Downlink Reference Signal (SSB 或 CSI-RS) 具备相同的 DM-RS antenna port quasi co-location properties，构建了一个闭环的“波束对应（Beam Correspondence）”逻辑。尤为关键的是，该 QCL 假设拥有 L1 级别的最高优先级，直接覆盖或无视任何针对于该 CORESET 潜在配置的高层 TCI-State 参数。这确保了在极早期接入阶段波束选择的确定性，避免了由于 TCI 配置模糊导致的严重失锁（Mis-alignment）。**

## 📖 协议原文拆解 (五)：LTM 极速切换场景下的 Msg2“波束继承”

> **协议原文**
> For the CFRA procedure triggered by LTM Cell Switch Command MAC CE, if the UE detects a DCI format 1_0 with CRC scrambled by the corresponding RA-RNTI and the UE receives a transport block in a corresponding PDSCH, the UE may assume same DM-RS antenna port quasi co-location properties, as described in [6, TS 38.214], as for a SS/PBCH block the UE used for PRACH association, as described in clause 8.1.

### 📚 增量术语表
* **CFRA (Contention-Free Random Access)**：无竞争随机接入。**一句话解释**：基站提前给你发了“VIP专属房卡”（专用的 PRACH 前导码），你敲门时绝对不会和别人撞车，直接进入免排队通道。
* **LTM (L1/L2 Triggered Mobility)**：L1/L2 触发的移动性切换。**一句话解释**：这是 5G Rel-18 引入的“极速换挡”技术。以前手机跨基站切换要走沉重的高层 (RRC) 流程；现在直接通过底层信令瞬间完成目标基站的切换，极大地降低了切换掉线率和时延。
* **LTM Cell Switch Command MAC CE**：LTM 小区切换命令 MAC CE。**一句话解释**：就是触发上述“极速换挡”的底层发车信号（一条 MAC 层控制指令）。

### 🚪 第一关：新技术带来的新触发条件
这段协议看似和上一段很像，但它的前置条件变了。
上一段讲的是普通的随机接入，而这一段专门针对 **5G 最新的极速切换技术 (LTM)**。
在 LTM 流程中，基站会通过一条底层的 `MAC CE` 指令，瞬间触发手机向目标基站发起一次无竞争的随机接入（CFRA），以此来快速建立连接。

### 🎯 第二关：“波束继承”法则在新场景下的延展
既然是新技术、新触发方式，底层的物理层该怎么接收目标基站回传的 Msg2 (RAR) 呢？
协议的回答是：**规矩不变，继续抄作业！**
即使你是被新潮的 LTM MAC CE 指令触发去发起免排队敲门（CFRA）的，只要你听到了对应的 RA-RNTI 通知并去收数据包裹（PDSCH），你依然可以**大胆假设（may assume）**：
这个包裹的波束方向（QCL 属性），和你刚才发 PRACH 时瞄准的那个目标基站的同步信号波束（SS/PBCH block）是完全一致的。

**生活类比**：
（普通接入）：你在黑暗中找光，冲着光喊了一声，然后顺着光的方向听回音。（上一段的场景）
（LTM极速切换）：指挥中心（原基站）直接给你塞了一个VIP对讲机，并用对讲机下令“立刻向东北方那束光报到！”（LTM MAC CE 触发 CFRA）。你冲着东北方的光喊了一声，接下来你该朝哪听回音？依然是顺着东北方那束光听。（本段场景，证明物理层法则的普适性）。

### 🔄 LTM 极速切换下的波束绑定流程
```text
【原基站】 ──(下发 LTM Cell Switch Command MAC CE)──► 【手机】
                                                      │
                                                      ├─► 1. 瞬间触发 CFRA 流程
                                                      │
【目标基站】 ◄──(向目标基站的特定 SSB 方向发送 VIP PRACH) ────┤
    │                                                 │
    │  (下发 Msg2 RAR)                                 │
    └─────────────────────────────────────────────────► 2. 接收 Msg2
                                                          【天线该朝哪？】
                                                          协议指明：与刚才瞄准的那个 SSB
                                                          保持绝对相同的 QCL 波束方向！
```

### 💡 章节硬核总结

**本段协议将 Msg2 的空间域 QCL 假设（Spatial QCL Assumption）明确延展至 Rel-18 引入的 LTM (L1/L2 Triggered Mobility) 特性中。在 LTM 架构下，终端的切换动作由较低开销的 MAC CE 直接触发并伴随 CFRA 过程。为了保证在目标小区 (Target Cell) 极早期下行链路的健壮接收，协议强制规定即使在这种由 MAC 层控制单元直接触发的特殊极速接入场景下，L1 物理层仍须遵循“波束对应”原则。即 Msg2 的 PDSCH DMRS 端口必须与其关联的 PRACH 传输所锚定的 SS/PBCH block 共享相同的 QCL-TypeD 属性。这一明文规定消除了新特性在底层实现时的空间滤波器配置歧义。**

## 📖 协议原文拆解 (六)：被基站“点名”强制接入时的波束该听谁的？

> **协议原文**
> If the UE attempts to detect the DCI format 1_0 with CRC scrambled by the corresponding RA-RNTI in response to a PRACH transmission initiated by a PDCCH order that triggers a contention-free random access procedure for the SpCell [11, TS 38.321], the UE may assume that the PDCCH that includes the DCI format 1_0 and the PDCCH order have same DM-RS antenna port quasi co-location properties. If the UE attempts to detect the DCI format 1_0 with CRC scrambled by the corresponding RA-RNTI in response to a PRACH transmission initiated by a PDCCH order that triggers a contention-free random access procedure for a secondary cell, or if the UE is configured with tag2-Id for the SpCell and the CORESET where the UE receives the PDCCH order that triggers a contention-free random access procedure for the SpCell is not associated with the physical cell ID for the serving cell, the UE may assume the DM-RS antenna port quasi co-location properties of the CORESET associated with the Type1-PDCCH CSS set for receiving the PDCCH that includes the DCI format 1_0 and the PDSCH scheduled by the DCI format 1_0.

### 📚 增量术语表
* **PDCCH order**：PDCCH 命令。**一句话解释**：基站发出的“强制征召令”，物理层直接下令让手机立刻发起一次随机接入（不需要高层磨叽）。
* **SpCell (Special Cell)**：特殊小区。**一句话解释**：手机的“大本营”，包括主小区 (PCell) 或双连接下的主辅小区 (PSCell)。
* **Secondary Cell (SCell)**：辅小区。**一句话解释**：基站给手机额外开通的“扩容车道”，用来增加网速。
* **tag2-Id / physical cell ID 不匹配**：多发射点 / 跨小区复杂场景。**一句话解释**：代表网络环境很复杂，基站用了好几个不同位置的天线塔（mTRP）在同时服务你。

### 🚪 第一关：大本营里的“从哪来，回哪去” (SpCell 场景)
以前我们学过，自己主动去敲门，就按自己敲门时对准的方向去听回执。
但这回情况变了：是基站主动发了一个 **PDCCH 命令**，强迫你去敲门（这叫 CFRA 无竞争接入）。敲完门后，收回执（Msg2）的波束该怎么选？
* **场景 A**：如果基站是叫你 在自己的“大本营” (SpCell) 里重新敲门。
* **波束法则**：协议说，此时你可以直接假设，接收回执（DCI 1_0）的波束方向，和**当初给你下达“PDCCH命令”的那个波束**是一模一样的！
* **生活类比**：大本营的老板在办公室（用波束 A）用对讲机喊你：“立刻给我打个卡（发 PRACH）！”你打完卡后，直接留在办公室听老板的后续指示就行了，肯定还是从老板那个方向传来的。

### 🎯 第二关：被派去外地时的“寻找公共大喇叭” (SCell 或复杂场景)
如果情况变得复杂，你不能再“死脑筋”地抄原命令的波束了：
* **场景 B**：基站下令让你去“辅小区” (SCell，外地分公司) 敲门。
* **场景 C**：虽然让你在大本营敲门，但下达命令的天线塔，和真正服务你的天线塔不是同一个（`physical cell ID` 不匹配）。
* **波束法则**：你刚才收到命令的那个波束已经“失效”了（因为你被派去别的物理节点了）。此时，你必须去目标小区寻找最稳妥的“公共大喇叭”——也就是**关联了 Type1-PDCCH CSS（公共搜索空间）的那个默认波束**。不仅收通知（DCI）要用这个默认波束，收数据包裹（PDSCH）也要用它。
* **生活类比**：老板在总公司对讲机里喊：“立刻去外地分公司（SCell）打卡！”你跑到分公司打完卡后，你还能盯着总公司老板的方向听回执吗？肯定不行。你必须抬头看分公司大堂里的**公共广播喇叭（Type1-PDCCH CSS）**，顺着喇叭的方向去听分公司给你的回执。

### 🔄 PDCCH Order 触发接入的波束决策树
```text
【手机收到 PDCCH Order，强迫发起随机接入】
        │
        ├─► 去哪里接入？
        │
        ├──► 【去大本营 (SpCell) 且环境简单】
        │       └─► 波束继承法：直接使用【接收 PDCCH Order 时】的波束来接收 Msg2。 ✅
        │
        └──► 【去辅小区 (SCell)】 或 【网络拓扑极其复杂 (跨天线塔 mTRP)】
                └─► 公共兜底法：刚才的波束不能用了！
                    强制切换到目标小区的【Type1-PDCCH CSS (公共大喇叭)】的波束来接收 Msg2。 ✅
```

### 💡 章节硬核总结

**本段协议精细化了由 PDCCH Order 触发的无竞争随机接入 (CFRA) 在 Msg2 接收时的下行空间 QCL 假设。这里体现了 3GPP “空间信道相关性”设计的严密逻辑：对于同小区（SpCell）、无 mTRP 断层的自调度场景，下达触发指令的 PDCCH 与后续响应的 PDCCH 物理空间路径高度一致，因此 UE 可直接复用指令 PDCCH 的 QCL-TypeD 假设。然而，对于跨载波调度（触发 SCell RACH）或多发送接收节点（mTRP，如 `tag2-Id` 和不同 PCI 的跨点调度）场景，指令的下达链路与最终的响应链路在物理空间上发生了解耦。为防止波束失锁，协议强制 UE 回退到最具鲁棒性的公共空间锚点——即目标小区的 Type1-PDCCH CSS 所在 CORESET 的 QCL 配置，从而保障了异地或异构物理节点响应接收的可靠性。**

## 📖 协议原文拆解 (七)：拿到 Msg3 的“发车许可证” (RAR UL grant)

> **协议原文**
> A RAR UL grant schedules a PUSCH transmission from the UE. The contents of the RAR UL grant, starting with the MSB and ending with the LSB, are given in Table 8.2-1.

### 📚 增量术语表
* **RAR UL grant**：RAR 上行授权。**一句话解释**：附带在基站回执（Msg2）里的一张“发件许可单”，上面写明了手机下一步发送真实数据（Msg3 PUSCH）时该用哪个频率、什么时间、多大功率。
* **MSB / LSB (Most/Least Significant Bit)**：最高/最低有效位。**一句话解释**：规定了一串二进制数据（比如 `101100...`）到底是从左往右读，还是从右往左切分的排版顺序规则。

### 🚪 第一关：从“盲目呼喊”到“持证上岗”
在随机接入的第一步（Msg1），手机是“盲目”地占用公共的 PRACH 资源去敲门的，很容易和别人撞车。
一旦手机成功收到了属于自己的 RAR 回执，情况就变了。这段协议正式宣布：**手机接下来要发的数据（PUSCH，也就是传统流程里的 Msg3），将不再是盲发的，而是被“RAR UL grant”这个调度器给严格安排好的。**
有了这张“许可证”，手机就正式从“排队挂号”阶段进入了“拿号看诊”阶段。

### 🎯 第二关：解剖“许可证”的排版 (Table 8.2-1 的预告)
基站发来的这张“许可证”本质上是一串连续的二进制比特流（比如总共 27 个比特）。
协议严谨地规定了拆解这串比特的“阅读顺序”：**从最高有效位（MSB，通常是最左边或第一位）开始，一直切分到最低有效位（LSB，最右边或最后一位）。**
虽然这里没有直接画出 Table 8.2-1 表格，但我们可以提前剧透一下，这张表里从头到尾通常包含了：跳频标志、频域资源位置、时域资源位置、调制编码策略（MCS）以及发射功率控制（TPC）等核心指令。

### 🔄 RAR 带来的流程跃迁图
```text
【Type-1 RACH 流程状态转换】

 [Msg1: 敲门]   (公共资源，可能撞车，无需调度)
      │
      ▼
 [Msg2: 回执]   (基站验证暗号，下发 RAR UL grant)
      │
      ├─► 获得 RAR UL grant 许可证 (按 MSB -> LSB 排版解析)
      │      1. 拿到频域车道
      │      2. 拿到时域时间
      │      3. 拿到语速 (MCS) 和 嗓门大小 (TPC)
      ▼
 [Msg3: 递交]   (独占资源，凭证上岗，绝对服从基站调度发 PUSCH)
```

### 💡 章节硬核总结

**本段协议标志着 Type-1 随机接入过程中，上行传输控制权从“基于竞争的开环选择”向“基于 DCI 调度的闭环指示”的根本性跨越。`RAR UL grant` 作为封装在 MAC RAR payload 中的 27-bit 物理层调度指令（其作用等同于常规业务阶段的 DCI format 0_0），确立了 Msg3 PUSCH 传输的全部 L1 参数基线。协议明确了该 grant field 必须严格遵循 MSB 到 LSB 的 Big-endian（大端序）比特流映射解析规则，确保 UE 的物理层能够确定性地切分并提取出频域资源分配（FDRA）、时域资源分配（TDRA）、MCS 及 TPC 等字段。**

## 📖 协议原文拆解 (八)：许可证里的“身法指令” (跳频开关)

> **协议原文**
> If the value of the frequency hopping flag is 0, the UE transmits the PUSCH without frequency hopping; otherwise, the UE transmits the PUSCH with frequency hopping.

### 📚 增量术语表
* **Frequency hopping flag**：跳频标志位。**一句话解释**：`RAR UL grant`（许可证）里的一个只有 1 比特大小的开关按钮（0或1），用来控制手机在发 Msg3 时要不要在空中“反复横跳”。

### 🚪 第一关：简单粗暴的“走位”控制
手机在解析 `RAR UL grant` 这张许可证时，最先读取到的指令之一就是这个**跳频标志位**。
* **读到 0**：基站说“老老实实呆在原地发”（without frequency hopping）。手机在整个发送 Msg3 的过程中，都会死死占据分配给它的那个固定频率车道，不挪窝。
* **读到 1**：基站说“开启闪避身法”（with frequency hopping）。手机在发送 Msg3 时，发一半数据后，必须瞬间跳跃到另一个频率车道去发剩下的一半。这能有效避免原来那个车道上恰好遇到短暂的无线电干扰（利用了频率分集效应保命）。

**生活类比**：
交警（基站）给你的车发了一张特别通行证。
如果证上盖的是“蓝戳（0）”，你就沿着最右边的车道一直开到底。
如果证上盖的是“红戳（1）”，你就必须开到一半路程时，强行变道到最左边车道去开完剩下的路，防止被路上的坑坑洼洼颠坏。

### 🔄 PUSCH 发送状态分支
```text
解析 RAR UL grant 中的 【跳频标志位】
        │
        ├─► 值为 0
        │    └─► 【固定频率发送】
        │         时域 ──► [    数据包前半段 + 后半段    ]
        │         频域 ──► [        同一段频率         ]
        │
        └─► 值为 1
             └─► 【跳频发送】
                  时域 ──► [ 数据包前半段 ] (瞬间切频) [ 数据包后半段 ]
                  频域 ──► [   频率 A   ]            [   频率 B   ]
```

### 💡 章节硬核总结

**本段明确了 Msg3 PUSCH 传输中频率分集（Frequency Diversity）功能的使能机制。作为开环阶段极少数可以动态指示的 L1 行为，是否开启时隙内跳频（Intra-slot frequency hopping）完全由 RAR UL grant 载荷中的 1-bit `frequency hopping flag` 显式控制。这一极简的设计赋予了基站调度器在面对复杂无线衰落信道时，为 Msg3 提供灵活抗突发干扰手段（提升鲁棒性）的能力。**

## 📖 协议原文拆解 (九)：许可证里的“语速档位”限制 (Msg3 MCS)

> **协议原文**
> The UE determines the MCS of the PUSCH transmission from the first sixteen indexes of the applicable MCS index table for PUSCH as described in [6, TS 38.214].

### 📚 增量术语表
* **MCS (Modulation and Coding Scheme)**：调制与编码策略。**一句话解释**：代表了手机发信号时的“语速快慢”和“抗干扰保护层的厚度”。
* **MCS index table**：MCS 索引表。**一句话解释**：3GPP 协议里写死的一张“语速档位表”，一共通常有 32 个档位（0-31）。档位越低（如 0, 1），说话越慢越清晰；档位越高（如 28, 29），说话越像连珠炮，虽然传得快但极容易听错。

### 🚪 第一关：查表定“语速”
在 `RAR UL grant` 许可证里，有几个比特（通常是 4 个比特）是专门用来告诉手机用什么语速（MCS）发 Msg3 的。
手机拿到这几个比特算出一个数字后，就要去翻开 3GPP 38.214 协议里的那张“语速档位表（MCS index table）”对号入座，确定具体的调制方式（QPSK 或 16QAM）和冗余码率。

### 🎯 第二关：强制被锁死在“新手低速档” (前 16 个索引)
这是本段协议的精髓：**Msg3 的 MCS 被强制限制在了整张表的前半段（The first sixteen indexes）。**
普通的业务表一共有 32 个档位（0~31）。但 `RAR UL grant` 留给 MCS 的比特位只有 4 bit（$2^4 = 16$ 个状态）。
这意味着，基站在指挥你发 Msg3 时，**最多只能让你从 0 到 15 号档位里选**，绝对不允许你挂到 16 档以上的高速挡位去。

**为什么要锁死在低速档？（物理意义）**
在发 Msg3 的时候，手机和基站才刚接上头，信道质量（CSI）基站完全摸不透，闭环功率控制也还没建好。在这个极其脆弱的“蒙眼走路”阶段，追求“极速飙车（高 MCS）”就是找死。
强行把选项限制在前 16 个极其稳妥、抗干扰能力极强的低速/中速档位里，是为了保底，确保极其关键的入网身份文件（Msg3）能被基站一次性成功解调。

**生活类比**：
驾校教练（基站）给刚摸方向盘的新手（发 Msg3 的 UE）发指令。
虽然这辆车最高能挂 6 挡（32个 MCS 级别），但教练在教练车的变速箱上加了物理锁，你**最多只能挂到 3 挡**（前 16 个 index）。宁可开得慢一点（速率低），也绝对不能在不熟悉的路况下飙车翻沟里（解码失败掉线）。

### 🔄 Msg3 MCS 查表机制
```text
【提取 RAR UL grant 中的 4-bit MCS 指令】
        │
        ▼ (假设取值为二进制 1001，即十进制 9)
        │
【翻开 38.214 协议的通用 PUSCH MCS 表 (0~31档)】
        │
        ├─► [索引 0 ~ 15]: ✅ 允许 Msg3 使用的绝对安全区 (低/中阶调制，高冗余保护)
        │   └─► 找到 9 号档位 ──► 确定调制方式为 QPSK，目标码率 378/1024。
        │
        └─► [索引 16 ~ 31]: ❌ 危险区，高速档，Msg3 阶段被硬性封印，不可选。
```

### 💡 章节硬核总结

**本段明确了 Msg3 PUSCH 链路自适应（Link Adaptation）的极简查表逻辑及边界约束。由于 `RAR UL grant` 的载荷容量极度受限（仅预留 4 bits 用于 MCS 指示），其寻址空间被硬性截断限制在了标准 MCS table 的下半区（Index 0~15）。这种截断不仅是物理层下行控制信令 payload size 的现实妥协，更深层体现了 L1 协议在初始接入阶段“Robustness prior to Spectral Efficiency (鲁棒性优先于频谱效率)”的保守设计准则，确保 Msg3 的初传强制处于低阶调制（如 QPSK/16QAM）与重度纠错编码的安全保护伞之下。**

## 📖 协议原文拆解 (十)：许可证里的“音量旋钮” (TPC 指令)

> **协议原文**
> The TPC command value $\delta_{msg2,b,f,c}$ is used for setting the power of the PUSCH transmission, as described in clause 7.1.1, and is interpreted according to Table 8.2-2.

### 📚 增量术语表
* **TPC (Transmit Power Control)**：发射功率控制。**一句话解释**：基站用来遥控手机“大点声”还是“小点声”的音量调节旋钮。
* **$\delta_{msg2,b,f,c}$**：Msg2 传达的功率微调偏移量。**一句话解释**：一个带正负号的数字（比如 +2dB，-2dB），精确告诉手机在发 Msg3 时，需要把初始计算好的基础嗓门（功率）调大多少或调小多少。

### 🚪 第一关：基站的第一次“声控干预”
在发第一步（PRACH）敲门时，手机是自己凭感觉（开环计算）瞎猜一个嗓门去喊的。
基站听到敲门声后，如果觉得手机喊得实在太轻了（信号弱），或者喊得太大声震耳朵了（产生干扰），基站就会在第二步回执（RAR UL grant）里，塞进一个关键的 **TPC 命令**。
这是基站**第一次**对手机进行闭环的音量干预。手机在准备发 Msg3 PUSCH 时，必须乖乖听话，把这个微调量 ($\delta_{msg2}$) 叠加上去。

### 🎯 第二关：查表解读“加减音量” (Table 8.2-2 预告)
RAR 里的这个 TPC 命令通常只占 3 个比特（代表 8 种状态）。
手机拿到这 3 个比特后，不是直接当成加减多少，而是要去查协议里规定的 `Table 8.2-2` 来“解码”。
虽然这里没画出表格，但查表的逻辑是绝对映射的：
比如，读到 `000` 可能代表“压低音量 -6dB”；读到 `100` 可能代表“音量正好，不变 0dB”；读到 `111` 可能代表“大声喊叫 +8dB”。
手机查完表，拿到这个以 dB（分贝）为单位的修正值，再送入 7.1.1 章节庞大的功率计算公式中去算出最终的发射功率。

**生活类比**：
你在黑屋子里冲着导演（基站）试音喊了一声（PRACH）。
导演给你扔了一张纸条（RAR），上面写着密码“档位7”（TPC）。
你翻开工作手册（Table 8.2-2），发现“档位7”的意思是“比刚才试音时再大声 6 个分贝”。
于是你深吸一口气，加足马力，用精确调大后的音量开始唱正歌（发 Msg3）。

### 🔄 Msg3 的功率闭环修正流程
```text
【开环试音】 手机凭感觉估算功率 P_PRACH ──► 发射 Msg1 敲门
                                            │
                                            ▼
【基站判决】 觉得声音偏小，决定让你再大点声。生成 TPC 码字送入 RAR。
                                            │
                                            ▼
【闭环修正】 手机解析 RAR UL grant，提取出 TPC 码字。
               │
               ├─► 查表 (Table 8.2-2) ──► 解码出物理微调量 δ_msg2 (比如 +6dB)
               │
               └─► 送入终极公式 ──► P_Msg3 = 基础计算功率 + δ_msg2
                                    (带着修正后的完美音量发射 Msg3)
```

### 💡 章节硬核总结

**本段协议在 L1 层面启动了上行链路从纯开环功率控制（Open-loop Power Control）向混合闭环功率控制（Closed-loop Power Control）演进的极早期“第一脚油门”。它明确了 RAR UL grant 中的 3-bit 载荷将被映射为物理层关键参量 $\delta_{msg2,b,f,c}$。这绝不是一个绝对功率值，而是一个基于 Table 8.2-2 的查表映射离散差分修正量（Offset）。该修正在 clause 7.1.1 的 PUSCH 功率控制主方程中作为加性项，用于补偿基站在捕获 Msg1 时测量到的信道衰落残差或干扰水平波动，从而精确校准 Msg3 的到达端信噪比（Target SINR），保障后续 RRC 建链流程的生命线。**

## 📖 协议原文拆解 (十一)：被“封印”的信道探测预留位

> **协议原文**
> The CSI request field is reserved.

### 📚 增量术语表
* **CSI request (Channel State Information request)**：信道状态信息请求。**一句话解释**：基站发出的命令，要求手机立刻测一下周围环境的信号质量，然后把一份详细的“路况体检报告”（CSI）发给基站。
* **Reserved**：保留的。**一句话解释**：在数据包（这里指 RAR UL grant）里留了这个位置，但规定当前阶段谁也不准用它，当作空气或者填死默认值（通常是全 0），留给 5G 未来演进升级时备用。

### 🚪 第一关：为什么有这个空位却不让用？
普通的业务阶段，基站经常会在发给手机的调度指令（DCI）里带上一个 `CSI request`，意思是：“顺便把你测好的网络质量报告（CSI）一起给我传回来。”
但是在目前的随机接入（Type-1 RACH 的 RAR）阶段，虽然 `RAR UL grant` 许可证的结构看起来很像一个普通的调度指令，协议却硬性规定：**这部分用来请求体检报告的比特位，必须被封印（Reserved），手机和基站都不准用。**

**为什么？（物理意义）**
因为此时手机和基站还在极其早期的“互相对暗号确认身份”的阶段。
1. **身份都没确认**：基站连你是谁都没确认（Msg3还没收上来），根本没必要也没心思看你的体检报告。
2. **连接极度脆弱**：体检报告（CSI payload）通常数据量很大，如果在极其重要、需要力保成功的 Msg3 身份文件里再硬塞一大段体检报告，会导致包裹变得非常臃肿，大幅增加 Msg3 传输失败、丢包重传的风险。

**生活类比**：
你（手机）去大公司（基站）大堂递交你的**入职申请表（Msg3）**。
前台给你批了一张“**通行证（RAR UL grant）**”。这张通行证用的是公司的通用版式，上面有一栏写着“**附带提交年度体检表（CSI request）**”。
但前台明确告诉你：“这一栏是给老员工用的，你一个新来的、身份还没核实的，这一栏**必须留空（Reserved）**，绝对不要现在把你的体检表塞在申请表里一起递上来，丢了算谁的！”

### 🔄 RAR 结构与预留字段示意
```text
【RAR UL grant 比特流排版 (27 bit)】

 ├─ [跳频开关 (1 bit)] ──► 控制 Msg3 要不要走位
 ├─ [频域位置 (14 bit)] ─► 控制 Msg3 停哪个车位
 ├─ [时域位置 (4 bit)] ──► 控制 Msg3 什么时候发
 ├─ [MCS 语速 (4 bit)] ──► 控制 Msg3 的编码档位
 ├─ [TPC 音量 (3 bit)] ──► 控制 Msg3 的发射功率
 │
 └─► [CSI Request (1 bit)] ──► 🔒 【封印 (Reserved)】
                               协议规定此位不生效，强行填空。
                               禁止在 Msg3 夹带私货（信道状态报告）。
```

### 💡 章节硬核总结

**本段协议虽仅有一句话，但通过将 `CSI request field` 设定为 `Reserved`，确立了 Msg3 传输的纯粹性与高鲁棒性基调。在 L1/L2 状态机尚未越过 Contention Resolution（竞争解决，即 Msg4 阶段）之前，任何非必要的物理层上行反馈（如 Aperiodic CSI 报告）都被视作对初传可靠性的无谓负担。协议通过预留该字段，既在物理信令结构上为 DCI 格式的演进或向后兼容保留了位宽冗余（Padding bit），又在业务逻辑上绝对掐断了基站在此极早期阶段触发大量上行状态反馈的合法性。**

## 📖 协议原文拆解 (十二)：免费频段上的“抢麦指南”与“填缝占座”

> **协议原文**
> The ChannelAccess-CPext field indicates a channel access type and CP extension for operation with shared spectrum channel access [15, TS 37.213] in FR1 as defined in Table 7.3.1.1.1-4 in [5, TS 38.212] or Table 7.3.1.1.1-4A in [5, TS 38.212] if channelAccessMode = "semiStatic" is provided. The ChannelAccess-CPext field indicates a channel access type for operation with shared spectrum channel access [15, TS 37.213] in FR2-2 as defined in Table 7.3.1.1.1-4B in [5, TS 38.212] if ChannelAccessMode2-r17 is provided.

### 📚 增量术语表
* **Shared spectrum (NR-U)**：共享频谱。**一句话解释**：不需要花几百亿买牌照的公共频段（比如家里 Wi-Fi 用的 5GHz/6GHz 频段），谁都可以用，规矩是“先听后说（LBT）”。
* **ChannelAccess-CPext**：信道接入与CP扩展字段。**一句话解释**：在免费频段上发 Msg3 时特有的一个指令，指导手机用什么姿势抢麦（Channel Access Type），以及如何拉长信号占座位（CP extension）。
* **FR1 / FR2-2**：频段范围1 / 频段范围2-2。**一句话解释**：FR1 指 7GHz 以下的低中频段（拥挤）；FR2-2 是 5G R17 版本新加入的 60GHz 左右的极高频毫米波免授权频段。
* **CP extension (Cyclic Prefix Extension)**：循环前缀扩展。**一句话解释**：在真正发数据前，故意制造一段无意义的冗长前奏，用来把时间空隙填满，防止被别的设备（如 Wi-Fi）趁虚而入。

### 🚪 第一关：去公共自习室发 Msg3 的特殊规矩
如果基站和手机是在花钱买的独占频段里通信，发 Msg3 时根本不需要看这个字段（因为没人跟你抢）。
但如果你们是在免费的**共享频段（Shared spectrum）**里通信，环境就像一个闹哄哄的公共自习室。手机在发 Msg3 之前，必须执行 LBT（Listen-Before-Talk，先听后说）。
基站会在 RAR 的许可证里附带这个 `ChannelAccess-CPext` 字段，给手机下发具体的“抢麦策略”：
1. **Channel access type (信道接入类型)**：告诉手机是“听 9 微秒没人就直接说”（Type 2），还是“要随机摇个号，等好几十微秒没人才能说”（Type 1）。
2. **CP extension (CP 扩展)**：如果手机抢到了麦，但距离真正该发 Msg3 的那个法定时间点还有一丝丝微秒级的空隙，手机必须拉长自己的前缀把这个空隙**填死**。否则旁边的 Wi-Fi 路由器一看这几微秒没声音，就会瞬间把信道抢走。

### 🎯 第二关：不同频段，查不同的执法表
由于低频和极高频的物理特性不同，协议规定了在不同频段下查不同的密码表：
* **在 FR1 频段 (日常拥挤的频段)**：查 38.212 的 Table -4（动态抢麦）或 -4A（半静态抢麦）。这里不仅规定了抢麦姿势，还严格规定了 CP 要拉长多少去填空隙。
* **在 FR2-2 频段 (60GHz 毫米波频段)**：这是 R17 新引入的。毫米波是像激光一样极细的光束，互相干扰的概率低很多。所以手机查的是专属的 Table -4B，通常只指示信道接入类型，规则相对会有所不同。

**生活类比**：
你在免费的公共会议室（共享频谱）准备宣读你的入职报告（Msg3）。
老板（基站）在给你的入场券（RAR）上贴了一张小纸条（`ChannelAccess-CPext`）：
* **在 1 楼大会议室 (FR1)**：纸条上写着“去查 Table-4：先在门外偷听 9 秒钟（接入类型）。一旦发现没人说话，里面哪怕还有 3 秒钟才到你的发言时间，你也得立刻清嗓子发出‘呃——’的声音把时间占满（CP扩展），千万别让隔壁部门的 Wi-Fi 团队抢了先！”
* **在 60 楼高级包间 (FR2-2)**：纸条上写着“去查 Table-4B：因为大家都是用传声筒私聊，干扰少，你只需要简单听一下没人就行了。”

### 🔄 NR-U 场景下 Msg3 发送时序防抢占图解
```text
【共享频段的时间轴】

[其他设备的干扰]   [ LBT 监听阶段 ] [   空隙   ] [ 正式规定的 Msg3 发送时间点 ]
 〰〰〰〰〰〰〰 │◄── 听没人 ──►│◄─ 危险期 ─►│
               │ (按接入类型查表)│            │
               │               ├─────────┼──────────────────────────►
               │               │ CP 扩展 │ Msg3 真实数据 (PUSCH)
               │               └─────────┴──────────────────────────►
               │                  ▲
               │      (如果不发 CP 扩展填死这里，
               │       Wi-Fi 就会趁机插入抢走信道！)
```

### 💡 章节硬核总结

**本段协议是 5G NR-U (Unlicensed) 在初始接入阶段应对全球未授权频段管制法规（如 ETSI 欧洲电标）的物理层具象化表达。它阐明了当小区运行在共享频谱时，RAR UL grant 会重载或利用预留比特生成 `ChannelAccess-CPext` 字段。该字段是 PHY 层的 LBT (Listen-Before-Talk) 触发器：一方面指示 Channel Access Category（如 Cat 2 快速接入或 Cat 4 基于退避的随机接入），确保 Msg3 发射的合法性；另一方面指示 CP Extension，通过在符号前增加冗余采样点来物理占有（Occupy）信道，消除由于 LBT 结束点与 PUSCH 符号起始边界不对齐（Misalignment）而产生的间隙，完美防御了 Inter-RAT (如 Wi-Fi CSMA/CA) 的信道抢占。针对 Rel-17 引入的 60GHz 频段 (FR2-2)，由于空间隔离度高，协议分配了独立的查找表以适配更宽松的定向 LBT 规则。**

## 📖 协议原文拆解 (十三)：没定规矩前，照旧跟基站原样接收 (SCS 继承原则)

> **协议原文**
> Unless the UE is configured a SCS, the UE receives subsequent PDSCH using same SCS as for the PDSCH reception providing the RAR message.

### 📚 增量术语表
* **SCS (Subcarrier Spacing)**：子载波间隔。**一句话解释**：决定了无线电信号中每一个“音符（符号）”的宽度和持续时间。比如 15kHz、30kHz 或 120kHz。SCS 越大，符号时间越短，传输速度越快，但抗多径干扰能力可能变弱。
* **Subsequent PDSCH**：后续的物理下行共享信道。**一句话解释**：在收完当前的随机接入回执（RAR Msg2）之后，基站接下来发给手机的其他数据包裹（比如 Msg4 冲突解决消息，或更早期的一些数据）。

### 🚪 第一关：收发信号的“节拍器”问题
在 5G 这样高度灵活的系统中，基站和手机通信的“节拍（SCS）”是可以随时变来变去的。
但是，在随机接入的极早期（比如刚收完 Msg2 准备收 Msg4 的时候），手机可能还没和基站建立起完整的 RRC (高层) 连接，基站还没来得及下发一份完整的《通讯节拍配置手册》。
此时，如果基站突然又砸过来一个下行数据包（Subsequent PDSCH），手机该用什么节拍（SCS）去解码呢？

### 🎯 第二关：无配置下的“惯性继承法则”
协议给出了一个极其简单的物理层惯性原则：
**如果没人专门给你配置新的节拍（Unless configured），那就按照刚才接收 RAR (Msg2) 时用的那个节拍（SCS）继续收后续的数据包！**

**生活类比**：
你在黑屋子里和基站对暗号。
基站用 **1.5倍速（比如 30kHz SCS）** 给你读了一段长长的回执录音（RAR）。
在没有收到新的通知告诉你该怎么听之前，如果基站接着又发来一段录音，你闭着眼睛直接继续用 **1.5倍速** 的播放器去听，肯定没错。基站在正式通知你调速之前，绝对不会偷偷改变语速。

### 🔄 L1 初期的 SCS (子载波间隔) 状态机
```text
【随机接入阶段的下行数据接收节拍 (SCS) 】

[接收 PDSCH 承载的 Msg2 (RAR)] ──► 假设此时使用的是 SCS_A
               │
               ▼
[准备接收后续的下行数据包裹 (Subsequent PDSCH)]
               │
               ├─► 【检查高层】: 给我下发了明确的专属 SCS 配置吗？
               │
               ├──► (有配置) ──► 抛弃旧规矩，严格按照新配置的专属 SCS 去解码。
               │
               └──► (无配置) ──► 【惯性继承】: 继续沿用 SCS_A 去解码这个包裹。✅
```

### 💡 章节硬核总结

**本段协议在 L1 层面定义了初始接入（Initial Access）状态下下行信道接收的 SCS（子载波间隔）缺省继承逻辑（Default Fallback/Inheritance Logic）。由于 Msg2（RAR）至 Msg4（Contention Resolution）阶段，UE 的 UE-Specific RRC configuration 可能尚未实例化或未完全生效，此时对 `BWP-Downlink` 的参数掌握存在真空期。为了保证下行数据链路的连续性，协议强制规定：在缺乏显式 SCS 配置的盲区内，所有后续的 PDSCH 接收，其 Numerology (参数集，这里特指 SCS) 必须严格继承承载 RAR message 的那个 PDSCH 所使用的 SCS。这种状态保持（State keeping）机制是保障基站调度自由度与终端盲检能力对称性的关键。**

## 📖 协议原文拆解 (十四)：接收失败后的“跨层甩锅”机制 (交由 MAC 定夺)

> **协议原文**
> If the UE does not detect the DCI format 1_0 with CRC scrambled by the corresponding RA-RNTI within the window, or if the UE detects the DCI format 1_0 with CRC scrambled by the corresponding RA-RNTI within the window and the LSBs of a SFN field in the DCI format 1_0, if included and applicable, are not same as corresponding LSBs of the SFN where the UE transmitted the PRACH, or the UE does not correctly receive a corresponding transport block within the window, the UE procedure is as described in [11, TS 38.321].

### 📚 增量术语表
* **[11, TS 38.321]**：3GPP MAC 层 (媒体接入控制层) 协议标准。**一句话解释**：物理层（L1）的直接“上司”。物理层只负责搬砖（收发信号），而像“重试几次、等多久再试、要不要加大功率”这种复杂的决策逻辑，全都写在这个 MAC 层的“法律条文”里。

### 🚪 第一关：物理层确认“接件失败”的三大死状
在前面的拆解中，我们提到过类似的情况，这里协议再次严谨地列出了物理层 (PHY) 认定 RAR 接收彻底失败的三个核心条件：
1. **没通知 (No DCI)**：窗口期过了，都没收到自己 RA-RNTI 的调度指令。
2. **时间错 (SFN Mismatch)**：指令收到了，但时间戳尾数对不上。
3. **件损坏 (PDSCH Decode Failure)**：数据包裹拿到了，但破损严重（CRC报错），打不开。
*(注：这里没提 RAPID 暗号对不上的情况，因为 RAPID 是包裹打开后由 MAC 层去看的，物理层不知道包裹里面的事。)*

### 🎯 第二关：底层搬砖人的智慧——“有事找领导”
当物理层确认发生了以上三种物理层面的失败后，它自己能决定接下来怎么办吗？能自己决定 1 毫秒后重发吗？
**绝对不行！**
这段协议的核心精神就是四个字：**跨层甩锅（层间分工）**。
物理层（本协议 38.213）明确声明：一旦发生这三种失败，物理层的使命就此暂停，接下来的所有行动（UE procedure），**请完全遵照 [TS 38.321] 这个上层主管单位的命令行事。**

**生活类比**：
快递员（物理层）在网点等回执。
等了一天没拿到（死状1）、拿到一张过期回执（死状2）、拿到一个碎掉的包裹（死状3）。
快递员绝不会自己决定“那我明天再去跑一趟”。他会立刻给站点经理（MAC 层，38.321）打电话汇报：“件没取到！”
经理（MAC层）会根据规章制度决定：是等 10 分钟再去（退避时间 Backoff），还是换个大嗓门去喊（功率抬升 Power Ramping），或者是重试次数太多直接放弃。

### 🔄 失败处理的跨层状态机
```text
【物理层 PHY (38.213)】                  【MAC层 (38.321)】
        │                                     │
   监听 RAR 窗口结束                             │
        │                                     │
 发生 DCI漏检 / SFN错 / PDSCH破损              │
        │                                     │
        ├───────── 宣告失败，上报状态 ────────►│
        │                                     │ 启动 RACH 异常恢复程序：
        │                                     │ 1. 更新重传计数器 (PREAMBLE_TRANSMISSION_COUNTER)
        │                                     │ 2. 决定是否触发后退延时 (Backoff)
        │                                     │ 3. 决定增加多少发射功率 (Power Ramping)
        │◄──────── 下达新的 PRACH 发射命令 ────┤
        │                                     │
 在硬件底线时间 (N_T,1 + 0.75ms) 后准备好再发
```

### 💡 章节硬核总结

**本段协议看似重复，实则在协议架构上划清了 L1 (物理层) 与 L2 (MAC层) 在 Exception Handling（异常处理）上的楚河汉界。3GPP 遵循严格的层级解耦设计 (Layered Architecture)：物理层 (38.213) 仅作为状态检测器 (State Detector)，负责评估物理信道的信噪比、解码 CRC 及核对底层同步帧号 (SFN)。一旦判定物理链路层面的 Msg2 获取失败，L1 不会自作主张执行任何业务重试，而是触发原语 (Primitive) 向上方 MAC 层 (38.321) 报告。后续所有的竞争解决退避算法 (Backoff Indicator)、重传功率攀升 (Power Ramping Step) 以及前导码重选逻辑，均由 MAC 层的状态机全权接管。**

## 🏆 章节全局硬核总结：8.2 Random Access Response (Type-1 RACH)

**8.2 章节是 5G NR 随机接入过程中“承上启下”的核心枢纽。它的系统级使命是：完成上行链路从“完全开环、基于竞争的盲发 (Msg1)”向“基于 DCI 调度、受控闭环的有序传输 (Msg3)”的根本性跨越。本章节在 L1 物理层构建了一套极其严密的时间、空间与参数交接逻辑，确保在 RRC 尚未建链的脆弱时期，基站与终端能完成绝对可靠的首轮握手。**

总结为以下五大核心架构设计：

### 1. 跨层双重鉴权机制 (Cross-layer Authentication)
解决“如何确认基站是在叫我”的核心难题：
*   **L1 初筛 (基于时频资源)**：物理层使用 `RA-RNTI` 解扰 DCI 1_0（隐含了 PRACH 的发送时间/频率），并辅助 `SFN LSBs` 校验防错。
*   **L2 终验 (基于码域特征)**：MAC 层接管 Transport Block，通过核对 `RAPID` 提取出专属于自己的 `RAR UL grant`。
这种 PHY-MAC 协同的双级寻址，完美解决了多 UE 共用相同时频资源敲门导致的碰撞识别问题。

### 2. 空间波束的强制“锚定” (Spatial QCL Anchoring)
在闭环波束管理（Beam Management）生效前，协议通过霸道的 L1 假设保障了空间链路的连续性：
*   **常规与 LTM 切换**：接收 Msg2 的 PDSCH/DCI 强制继承关联 PRACH 的 SSB/CSI-RS 波束，**直接无视**高层任何现有的 TCI-State 配置。
*   **PDCCH Order 触发**：若在同基站自调度，继承触发命令的波束；若跨节点（SCell, mTRP 等复杂场景），强制回退到最具鲁棒性的“公共大喇叭”——目标小区 Type1-PDCCH CSS 的 QCL 配置。

### 3. Msg3 的保守派调度哲学 (Conservative L1 Baseline)
`RAR UL grant` (27-bit) 为即将发送的 Msg3 制定了极其保守的物理层基线，一切以“保底存活率”为最高优先级：
*   **语速封印 (MCS)**：强制将 PUSCH MCS 寻址空间截断在索引表的前 16 位（低阶调制），保障高冗余抗干扰。
*   **负载减负 (CSI Request)**：强制预留（Reserved）CSI 请求位，绝不允许在脆弱的 Msg3 夹带臃肿的信道测量报告。
*   **闭环微调 (TPC)**：下发物理层第一条相对功率控制指令 ($\delta_{msg2}$)，修正 Msg1 开环计算的误差。
*   **分集与合规**：允许使能时隙内跳频获取频率分集；针对 NR-U 共享频谱，精准下发 LBT 类型与 CP 扩展（`ChannelAccess-CPext`）以合规“填缝占座”。

### 4. 严苛的时序与异常恢复 (Timing & Exception Recovery)
*   **监听窗口**：定义了物理绝对锚点（至少 1 symbol 间隔后的首个合法 CORESET）及 NTN 超远距离的延时补偿（$T_{TA} + k_{mac}$）。
*   **重传底线**：规定 UE 在遇到漏检、CRC 失败或暗号不匹配时，其重新准备发送 PRACH 的 L1 硬件处理时间上限严格界定为 $N_{T,1} + 0.75$ 毫秒。
*   **跨层甩锅**：L1 只负责上报失败状态，具体的重试策略、功率攀升、退避等待，全权移交上层主管协议 [TS 38.321 (MAC)] 执行，体现了完美的协议栈解耦。

### 5. 缺省状态继承 (Fallback Inheritance)
在 RRC 专属参数缺失的盲区，对于未配 SCS 的 Subsequent PDSCH，物理层采取“惯性继承”法则，继续使用接收 RAR PDSCH 时的 SCS 节拍，确保了下行数据流的无缝接力。

**总结而言，8.2 章节就像是 5G 入网的“新兵接待大厅”。它用一套不依赖任何专属配置、极其鲁棒（Robust）的强制物理法则，稳稳地接住了盲目撞进来的 Msg1，并为其颁发了精准合规的 Msg3 发射执照。**
