# 8.4 PDSCH with UE contention resolution identity (携带 UE 冲突解决标识的下行数据传输)

---

## 📖 协议原文拆解 (一)：Msg4 冲突解决与终极签收确认 (ACK 发射法则)

> **协议原文**
> In response to a PUSCH transmission scheduled by a RAR UL grant when a UE has not been provided a C-RNTI, the UE attempts to detect a DCI format 1_0 with CRC scrambled by a corresponding TC-RNTI scheduling a PDSCH that includes a UE contention resolution identity [11, TS 38.321]. In response to the PDSCH reception with the UE contention resolution identity, the UE transmits HARQ-ACK information in a PUCCH. The PUCCH transmission is within a same active UL BWP as the PUSCH transmission. A minimum time between the last symbol of the PDSCH reception and the first symbol of the corresponding PUCCH transmission with the HARQ-ACK information is equal to $N_{T,1}+0.5$ msec.

### 📚 术语表
* **UE contention resolution identity**：UE 冲突解决标识。**一句话解释**：基站在 Msg4 里发回来的“终极防伪码”。基站会把你刚才在 Msg3 里报上的身份证号原封不动地抄在里面，你核对无误后，才能证明这个 Msg4 确实是发给你的（踢开那些和你用了同一个暗号的冒牌货）。
* **HARQ-ACK**：混合自动重传请求确认。**一句话解释**：终端发给基站的“好评签收单”。

### 🚪 第一关：锁定临时访客牌 (TC-RNTI 监听)
* **场景倒带**：手机是个刚开机的黑户（没有分配正式的 C-RNTI）。它刚刚发完了自己的核心身份信息（Msg3 PUSCH），现在只能眼巴巴地盯着基站的方向。
* **监听目标**：手机兜里现在只有一张 Msg2 时给的“临时访客牌（TC-RNTI）”。所以，它死死盯着大屏幕，试图截获一张用 **TC-RNTI 加密**的小纸条（DCI format 1_0）。如果解密成功，就顺着指引去拿最终的大包裹（Msg4 PDSCH）。

### ⚔️ 第二关：终极真假美猴王之战 (冲突解决)
* **核心痛点**：为什么要有这一步？因为在第一步发暗号时，可能有两个人都用了“8号”暗号。他们都会以为临时访客牌（TC-RNTI）是发给自己的，所以两个人都会发 Msg3（这就撞车了）。
* **解决机制**：基站虽然听到了两个 Msg3 叠在一起的声音，但它的超级算法解开了其中信号较强的那个人的包裹，发现里面写着“我是张三”。于是基站在 Msg4 包裹里贴上了一个 **“冲突解决标识（UE contention resolution identity） = 张三”**。
* **宣判结果**：两个手机都拆开了 Msg4，一看防伪码。张三狂喜：“是我！”；李四一看：“卧槽，基站听清的是张三不是我。” 李四默默退下（冲突解决失败，重新排号）。

### 🚀 第三关：赢家的最后狂欢 (发射终极 ACK)
张三既然赢了，就必须给基站发个确认信（ACK），告诉基站：“感谢老大，正式确立关系！” 但发这个确认信有两大铁律：
1. **跑道铁律（同频段绑定）**：你这封签收单（PUCCH），必须和你刚才发射 Msg3 数据（PUSCH）走**同一条激活的上行跑道（same active UL BWP）**！坚决杜绝收信和回信不在同一个车道上导致的混乱。
2. **时间铁律（绝对硬件红线）**：从收到 Msg4 结束，到发出这封 ACK，中间的间隔时间**绝对不能小于 $N_{T,1} + 0.5$ 毫秒**！
   * 这又是一个我们极度熟悉的“防剥削条款”。其中 $N_{T,1}$ 是手机大脑拆解 Msg4 包裹并核对防伪码的时间，0.5 毫秒是射频天线大转身预热的时间。少一毫秒手机都得炸毛罢工。

### 🔄 Msg4 冲突解决与终极握手流水线
```text
 手机 (UE，此时是无正式身份的黑户)
    │
    ├─ 刚才发出了 Msg3，兜里揣着临时牌 (TC-RNTI) 焦急等待...
    │
    ├─ 拦截到 TC-RNTI 解密的 DCI，并拿到 Msg4 数据包裹 (PDSCH)
    │
    ├─ 动作 1: 大脑开始高速运转 (消耗 NT,1 时间)，拆解包裹寻找防伪码！
    │
    ├─ 问自己：里面写的名字 (Contention Resolution Identity) 是我吗？
    │
    ├──► 【不是我 (我是冒牌货)】 ──► 认输退场，重新发起随机接入！
    │
    └──► 【是我！(我是正主)】
              │
              ├─ 动作 2: 射频天线开始预热 (消耗 0.5ms)
              │
              ├─ 动作 3: 找准刚才发 Msg3 用的那条老跑道 (Active UL BWP)
              │
              └─► 在满足时间红线 ( > NT,1 + 0.5ms) 后，准时轰出终极 ACK 签收单！
                  (接入流程圆满结束，临时牌升级为正式 VIP 卡！)
```

## 📖 协议原文拆解 (二)：监听 Msg4 时的波束锁定法则 (看哪里的问题)

> **协议原文**
> When detecting a DCI format in response to a PUSCH transmission scheduled by a RAR UL grant... or corresponding PUSCH retransmission scheduled by a DCI format 0_0 with CRC scrambled by a TC-RNTI..., the UE may assume the PDCCH carrying the DCI format has the same DM-RS antenna port quasi co-location properties, as described in [6, TS 38.214], as for a SS/PBCH block the UE used for PRACH association, as described in clause 8.1, regardless of whether or not the UE is provided TCI-State for the CORESET where the UE receives the PDCCH with the DCI format.

### 📚 增量术语表
* **DCI format in response to a PUSCH transmission**：回应 PUSCH 的控制小纸条。**一句话解释**：手机发完 Msg3 后，苦苦等待的那张用来通知它“去哪里拿 Msg4 最终审判大包裹”的取件单。
* **QCL (Quasi Co-Location)**：准共址。**一句话解释**：代表接收天线可以保持同样的姿势去接信号。
* **SS/PBCH block (SSB)**：同步信号块。**一句话解释**：基站的基础广播探照灯。

### 🚪 第一关：等待最后审判时的动作
* **场景倒带**：手机刚刚发送完 Msg3 数据（或者因为第一次失败，用 TC-RNTI 解密的 DCI 0_0 刚刚发完了重传的 Msg3）。现在，手机正盯着基站，等待最后的回信（Msg4）。
* **物理难题**：5G 的波束极窄，手机现在该把天线转到哪个方向去等这封信？

### 📡 第二关：万法归宗，认准老祖宗 (SSB)
* **一锤定音**：协议再次祭出了极其霸道的“原路返回”法则：不管你是第一次等回信，还是重传后等回信，你在接收这封信（PDCCH DCI）时，**天线波束方向（QCL）必须且只能锁定在你最初发送暗号（PRACH）时选中的那个灯塔（SSB）上！**
* **霸道豁免 (regardless of)**：即使基站平时给这个区域（CORESET）配置了专门的波束指南（TCI-State），手机在等 Msg4 的这一刻也**一律无视它**。
* **现实意义**：在整个随机接入流程彻底完成之前（拿到 Msg4 之前），手机和基站之间的“信任链”非常脆弱。唯一坚如磐石的，就是第一步建立联系时那束最亮的光（SSB）。所以，不到最后一刻，绝不换路！

### 🔄 监听 Msg4 时的波束继承流水线
```text
 手机 (UE) 的天线动作                                       基站 (gNB)
      │                                                        │
      ├─ 回忆：我最初发 PRACH 时，认准的是哪座灯塔？ (比如 SSB #1) │
      │                                                        │
      ├──► 【发送 Msg3 初传】 ────────────────────────────────►│ 
      │    (如果在 SSB #1 的方向发)                               │
      │                                                        │
      ├──► 【或者：发送 Msg3 重传】 ───────────────────────────►│
      │                                                        │
 【死死锁住方向】: 接收天线继续死死盯住 SSB #1 的方向 (同源 QCL)     │
      │                                                        │
      │◄────── 接住基站发来的终极审判纸条 (DCI 响应 Msg4) ────────┤
      │                                                        │
  (彻底无视常规 TCI-State 指南，直到接入大圆满为止！)
```

## 🏆 终极硬核总结：8.4 章节 (Msg4 冲突解决) 的生死审判法则

8.4 章节是整个随机接入流程（Type-1 4步接入）的最终乐章。在前面的章节中，手机一直在“瞎子摸象”般地和基站交互，而在这一章，系统通过两道极其严苛的物理层铁律，彻底解决了“我是谁”和“信到底给谁”的终极问题，宣告了接入的圆满成功或彻底失败。

### 1. 唯一信任锚点：TC-RNTI 加解密
在尚未分配正式 VIP 卡号（C-RNTI）的黑户期，手机能够从混沌的空中截获属于自己 Msg4 的唯一凭证，就是 Msg2 下发的 **临时访客牌（TC-RNTI）**。通过 TC-RNTI 对 DCI 1_0 进行加扰监听，协议巧妙地实现了身份的初步过滤，确保手机不会去拆别人的包裹。

### 2. 真假美猴王之战：载荷级防伪核对
即使 DCI 解密成功，物理层递交上去的包裹依然面临多终端碰撞的可能。直到高层在 PDSCH 载荷中读出 **UE 冲突解决标识 (Contention Resolution Identity)**，并确认里面抄写的就是自己之前上报的身份证号时，才算真正击败了其他“撞车”的冒牌货。赢家留下，输家退场，网络秩序得以肃清。

### 3. 通关确认信 (ACK) 的物理双铁律
当正牌赢家准备向基站拍下最后一封“收货确认单（HARQ-ACK）”时，物理层绝不容许任何差池：
* **频域强绑定**：确认单（PUCCH）必须发射在刚才发送身份数据（Msg3 PUSCH）的同一条激活跑道（Active UL BWP）上，实现上下文的频域连贯。
* **绝对时间防线**：从接收完 Msg4 的最后一刻起，到发射这封 ACK 之前，必须留足 **$N_{T,1} + 0.5$ 毫秒**的不可侵犯时间。这不仅保证了终端芯片（$N_1$ 符号处理时长）有足够的时间去解密拆包，也确保了射频硬件（0.5ms）能从容地完成收发转换。

### 4. 空间波束的“至死不渝” (QCL 继承)
在 5G 极窄的波束丛林中，手机在等这封终极判决书（Msg4）时，到底该看向哪里？协议祭出了整个随机接入流程中最坚如磐石的法则：**无论基站给当前区域配置了怎样高级的 TCI-State，全部无视！必须死死盯住最初发起请求时认准的那座灯塔（SSB）！** 这种从 Msg1 到 Msg4 一以贯之的“同源波束硬性继承”，用最原始、最粗暴的方式兜住了高频信道断链的下限。

**结语：8.4 章节是随机接入逻辑闭环的最后一环。它用 TC-RNTI 实现了控制面的寻址，用载荷防伪码解决了数据面的冲突，用 $N_{T,1}+0.5$ms 守住了硬件处理的底线，更用“无视 TCI 强锁 SSB”的霸道法则守住了空间波束的生命线。当手机遵守这些规矩，成功把最后一封 ACK 轰向基站的瞬间，它终于摆脱了混沌，化身为 5G 网络中一个拥有合法坐标的正式节点！**
