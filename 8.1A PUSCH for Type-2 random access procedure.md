# 🔖 章节：8.1A PUSCH for Type-2 random access procedure

---

## 📖 协议原文拆解 (一)：Type-2 RACH 中 PRACH 与 PUSCH 的“黄金距离”

> **协议原文**
> For a Type-2 random access procedure, a UE transmits a PUSCH, when applicable, after transmitting a PRACH. The UE encodes a transport block provided for the PUSCH transmission using redundancy version number 0. The PUSCH transmission is after the PRACH transmission by at least $N$ symbols where $N=2$ for $\mu=0$ or $\mu=1$, $N=4$ for $\mu=2$ or $\mu=3$, $N=16$ for $\mu=5$, $N=32$ for $\mu=6$, and $\mu$ is the SCS configuration for the active UL BWP.

### 📚 增量术语表
* **Type-2 random access procedure**：2步随机接入过程 (2-step RACH)。**一句话解释**：手机为了更快接入基站，把传统的4步流程浓缩成2步，在第一步（MsgA）就同时把接入暗号（PRACH）和自己的详细信息（PUSCH）打包发给基站。
* **Redundancy version number 0 (RV0)**：冗余版本 0。**一句话解释**：数据编码发送时的“初始原版”数据，包含了解码所需的最多、最完整的信息（相对后续重传的补丁数据而言）。
* **$\mu$ (Mu)**：子载波间隔 (SCS) 的配置序号。**一句话解释**：决定了频域上“车道”的宽度和时域上每个符号 (Symbol) 的时间长度。$\mu$ 越大，时域上每个符号的时间就越短。

### 🚪 第一关：两步走的“连招”与“初版数据”
* **连招动作**：在 2-step RACH 中，手机的第一步叫 MsgA。MsgA 实际上是由两部分组成的：先发 **PRACH**（前导码，相当于敲门砖），然后再发 **PUSCH**（共享信道，里面装着真正的身份和请求数据）。
* **数据打包要求**：协议规定，这个 PUSCH 里的数据块 (Transport Block) 必须使用 **RV0 (冗余版本0)** 来编码。因为这是手机第一次发送这些数据，必须把最完整、自解码能力最强的初始版本发给基站，确保基站一次就能大概率解出来。

### 🎯 第二关：动作之间的“喘息时间” ($N$ 个符号)
* **物理限制**：手机刚用一种波形或资源发完了 PRACH，立刻就要接着发 PUSCH。手机的射频器件 (RF) 和基带芯片需要一点时间来切换状态、调整功率或重置参数。
* **动态的间隔**：协议规定，在 PRACH 结束到 PUSCH 开始之间，必须留出**至少 $N$ 个符号 (Symbols)** 的间隔。
* **水涨船高的 $N$ 值**：因为当 $\mu$ (子载波间隔配置) 变大时，每个符号的绝对物理时间 (毫秒级) 会变短。为了保证手机有足够的绝对时间来完成硬件状态切换，符号数 $N$ 必须随着 $\mu$ 的增大而增加：
    * $\mu=0, 1$ (15/30 kHz)：$N=2$ 个符号
    * $\mu=2, 3$ (60/120 kHz)：$N=4$ 个符号
    * $\mu=5$ (480 kHz)：$N=16$ 个符号
    * $\mu=6$ (960 kHz)：$N=32$ 个符号

### 🔄 Type-2 RACH 发送流水线
```text
 手机发起 Type-2 RACH (MsgA)
        │
        ├─► 动作1：发送 PRACH (敲门砖)
        │
        ├─► ⏳ 强制等待阶段 (硬件切换缓冲期)：
        │    查看当前的上行带宽配置 (active UL BWP) 的 SCS 是多少 ($\mu$)
        │    - 如果是 30kHz ($\mu=1$) ──► 至少等 2 个符号
        │    - 如果是 120kHz ($\mu=3$) ─► 至少等 4 个符号
        │
        ▼
        ├─► 动作2：准备 PUSCH 数据
        │    强制要求：必须使用 RV=0 (最完整的信息版本) 进行编码
        │
        └─► 动作3：发送 PUSCH (递交真实身份和请求)
```

### 💡 章节硬核总结

**这段协议明确了 8.1A 章节下 2-step RACH 中 MsgA 内部的时序与编码硬性约束。首先，MsgA 的 PUSCH payload 必须采用重传冗余版本 RV0 进行初传，以最大化基站的初传解码成功率。其次，规定了从 PRACH 传输结束到 PUSCH 传输开始之间的最小时间 Gap。该 Gap 以正交频分复用（OFDM）符号数 $N$ 为单位，且 $N$ 的取值随上行活动部分带宽（active UL BWP）的子载波间隔（SCS，即 $\mu$）成比例缩放。其物理本质是为了在不同的系统参数配置下，为终端的射频前端和基带处理保留一段恒定的、足够完成信道切换与调谐的绝对时间。**

## 📖 协议原文拆解 (二)：MsgA 中 PUSCH 与 PRACH 的“强绑定”与“弱解绑”规则

> **协议原文**
> A UE does not transmit a PUSCH in a PUSCH occasion if the PUSCH occasion associated with a DMRS resource is not mapped to a preamble of valid PRACH occasions or if the associated PRACH preamble is not transmitted as described in clause 7.5 or clause 11.1 or clause 15 or clause 17.2. A UE can transmit a PRACH preamble in a valid PRACH occasion if the PRACH preamble is not mapped to a valid PUSCH occasion.

### 📚 增量术语表
* **PUSCH occasion (PO)**：PUSCH 发送时刻/机会。**一句话解释**：网络给手机预留的，专门用来发送 PUSCH 数据的时间-频率“停车位”。
* **DMRS resource**：解调参考信号资源。**一句话解释**：伴随 PUSCH 一起发送的导频信号，基站靠它来估算信道状态，从而正确解出 PUSCH 里的数据。
* **Mapped (映射/绑定)**：**一句话解释**：在 2-step RACH 中，特定的 PRACH 前导码（敲门砖）会和特定的 PUSCH 车位在系统配置里提前“绑定”好，形成一一对应关系。

### 🚪 第一关：PUSCH 的“禁发”原则（无前导码，不发数据）
在这段规则中，协议明确了在哪些情况下，手机**绝对不能**发送 MsgA 里的 PUSCH 数据：
1. **找不到对应的敲门砖（未映射）**：如果当前 PUSCH 所在的资源（以及它的 DMRS）在配置上并没有绑定到任何“有效的 PRACH 机会里的前导码”，那就不能发。
2. **敲门砖没扔出去（未发送）**：即使 PUSCH 资源有绑定的 PRACH 前导码，但如果因为某些冲突（比如 clauses 7.5/11.1 等规定的特殊情况，如半静态信道被占用、TDD 上下行冲突等）导致手机**实际上并没有把那个 PRACH 前导码发出去**，那么后续的 PUSCH 也必须取消发送。

**生活类比**：去寄快递（2-step RACH）。PRACH 是“快递单”，PUSCH 是“包裹”。
如果你的包裹没有贴有效的快递单（未映射），或者你虽然填了单子但单子被风吹跑了没给到快递员（未发送），那么你的包裹（PUSCH）绝对不能单独寄出去。

### 🎯 第二关：PRACH 的“可发”原则（单发敲门砖）
协议的后半句开了一个口子：
如果一个 PRACH 前导码在一个有效的 PRACH 机会里，**即使它没有绑定到任何有效的 PUSCH 机会**，手机**仍然可以**把这个 PRACH 前导码发出去。

**生活类比**：还是寄快递。如果你今天只拿了快递单（PRACH）去网点，但是发现没有空余的纸箱或者车位来装你的包裹（没有有效的 PUSCH 机会），你依然可以把单子先递给快递员。此时，基站收到单子后，可能会把它当成传统的 4 步 RACH 来处理（基站后续会给你单独分配发包裹的车位）。

### 🔄 MsgA 的发送逻辑判断树
```text
 准备发送 2-step RACH (MsgA)
        │
        ├─► 检查 PRACH 前导码与 PUSCH occasion 是否有效映射？
        │
        ├──► 【情况 A】没有有效的 PUSCH occasion 与之映射！
        │       └─► 依然可以单发 PRACH 前导码 ✅ (此时不发 PUSCH ❌)
        │
        └──► 【情况 B】有映射！
                │
                └─► 检查 PRACH 前导码是否成功发送出去了？
                      │
                      ├──► 因为资源冲突等原因，PRACH 没发出去 ❌
                      │       └─► 必须取消发送 PUSCH ❌ (同生共死)
                      │
                      └──► PRACH 成功发送 ✅
                              └─► 允许在映射的 PUSCH occasion 发送 PUSCH ✅
```

### 💡 章节硬核总结

**本段明确了 Type-2 随机接入中 MsgA 两部分（PRACH 与 PUSCH）的依赖关系。PUSCH 的发送严格依赖于 PRACH：只有在 PUSCH occasion 及其 DMRS 被合法映射到 valid PRACH occasion，且该 PRACH 前导码被实际物理发送的前提下，UE 才被允许发送 PUSCH。反之则不然，UE 可以在没有有效 PUSCH occasion 映射的情况下，单独发送 PRACH 前导码，这种机制为系统资源受限或配置不完整时提供了向后兼容（fallback）到类似 Type-1 处理流程的可能。**

## 📖 协议原文拆解 (三)：PRACH 与 PUSCH 映射的“红娘”参数

> **协议原文**
> A mapping between one or multiple PRACH preambles and a PUSCH occasion associated with a DMRS resource is per PUSCH configuration provided by MsgA-PUSCH-Resource.

### 📚 增量术语表
* **MsgA-PUSCH-Resource**：MsgA PUSCH 资源配置。**一句话解释**：这是基站通过高层信令（RRC）发给手机的一份“资源分配表”，里面详细规定了 2-step RACH 中 PUSCH 的时间、频率资源位置，以及它和前导码的绑定规则。

### 🚪 第一关：谁来牵线搭桥？（配置来源）
在前面的解析中我们知道，Type-2 RACH 的核心是发送 PRACH 前导码后，紧接着在一个绑定的 PUSCH occasion (PO) 上发数据。
但手机怎么知道哪个 PRACH 对应哪个 PUSCH 车位呢？协议在这里给出了答案：**一切规则都由 `MsgA-PUSCH-Resource` 这个配置参数说了算**。它就像是系统里的“红娘”或者“分配表”，手机必须严格按照它提供的配置来找座位。

### 🎯 第二关：映射关系的“多对一”与“一对一”
这段话指出，映射关系可以是：
* **One PRACH preamble -> A PUSCH occasion + DMRS** (一对一)
* **Multiple PRACH preambles -> A PUSCH occasion + DMRS** (多对一)

**生活类比**：
假设 **PRACH 前导码**是进入会场的“门票”，**PUSCH occasion** 是会场里的“特定会议室”，而 **DMRS 资源** 是会议室里的“特定麦克风”。`MsgA-PUSCH-Resource` 就是大堂的“会务指示牌”。
* **一对一**：指示牌规定，拿 1 号门票的人，去 A 会议室用 1 号麦克风。这是独享资源，不会和别人撞车。
* **多对一**：指示牌规定，拿 2 号、3 号、4 号门票的人，都可以去 B 会议室用 2 号麦克风。如果两个手机碰巧同时选了 2号和 3号门票，它们就会在同一个会议室（同一个时频资源）用同一个麦克风发声音，这就产生了冲突（碰撞），基站后续需要通过竞争解决机制来处理。

### 🔄 映射关系图解
```text
【基站下发的配置表：MsgA-PUSCH-Resource】
        │
        ├─► 规则 1 (一对一映射)
        │     [PRACH Preamble #0] ──────► [PUSCH Occasion #0 + DMRS #0]
        │
        └─► 规则 2 (多对一映射)
              [PRACH Preamble #1] ──┐
              [PRACH Preamble #2] ──┼───► [PUSCH Occasion #1 + DMRS #1]
              [PRACH Preamble #3] ──┘
```

### 💡 章节硬核总结

**本段明确了 Type-2 随机接入中 PRACH 与 PUSCH 资源之间物理映射关系的配置信源。这种映射关系（包括具体的 PUSCH 时频机会和 DMRS 解调参考信号资源）是基于高层 RRC 参数 `MsgA-PUSCH-Resource` 提供的每个 PUSCH 配置来定义的。协议允许 N:1（多对一）或 1:1（一对一）的映射维度，这决定了系统在 MsgA PUSCH 阶段的资源复用度和潜在的碰撞概率（Contention probability）。**

## 📖 协议原文拆解 (四)：去哪儿找 PUSCH 的“时刻表”和“地图”？

> **协议原文**
> A UE determines time resources and frequency resources for PUSCH occasions in an active UL BWP from msgA-PUSCH-Config or separateMsgA-PUSCH-Config for the active UL BWP. If the active UL BWP is not the initial UL BWP and msgA-PUSCH-Config or separateMsgA-PUSCH-Config is not provided for the active UL BWP, the UE uses the msgA-PUSCH-Config or separateMsgA-PUSCH-Config provided for the initial UL BWP.

### 📚 增量术语表
* **msgA-PUSCH-Config / separateMsgA-PUSCH-Config**：MsgA PUSCH 配置 / 独立 MsgA PUSCH 配置。**一句话解释**：这是基站下发的两个高层参数配置。它们就像是包含 PUSCH “时刻表”（时域）和“地图”（频域）的说明书。前者通常和 Type-1 RACH 资源共享配置，后者则是为 Type-2 独立配置的资源。
* **active UL BWP**：激活的上行部分带宽。**一句话解释**：手机当前真正在使用的上行传输频段范围（手机不需要在整个大载波上工作，省电）。
* **initial UL BWP**：初始上行部分带宽。**一句话解释**：手机刚开机、还没和基站混熟时，系统默认分配给大家进行初始接入的公共频段范围。

### 🚪 第一关：直接在当前工作的“车道”上找地图
当手机准备发 MsgA 里的 PUSCH 时，它需要知道具体的时频资源（也就是车位到底在哪）。
协议规定，手机首先应该在**当前正在使用的频段 (active UL BWP)** 里寻找基站发来的“说明书”，这两个说明书名叫 `msgA-PUSCH-Config` 或 `separateMsgA-PUSCH-Config`。只要找到了，就按照这本说明书上的时频资源去发 PUSCH。

### 🎯 第二关：如果当前“车道”没贴说明书怎么办？（回退机制）
手机在业务运行中，可能会被基站切到一个很宽的或专属的 active UL BWP（它不再是最开始的初始 BWP 了）。
此时手机想发起 2-step RACH，却发现基站**并没有**在这个专属 BWP 里下发 `msgA-PUSCH-Config` 这个配置，怎么办？能罢工不发吗？
**不行！**协议给了一个“兜底（Fallback）”方案：
如果当前 BWP 不是初始 BWP，且没有找到配置，手机必须**回头去找“新手村”的配置**。也就是直接借用**初始 BWP (initial UL BWP)** 里的那个 `msgA-PUSCH-Config` 来决定车位的位置。

**生活类比**：
假设 **active UL BWP** 是你现在常驻的“VIP 办公室”，**initial UL BWP** 是楼下的“公共接待大厅”。
你要填表寄快递（发送 PUSCH）。
* 正常情况：你在 VIP 办公室的桌上找“快递指南（msgA-PUSCH-Config）”，按指南操作。
* 特殊情况：如果 VIP 办公室桌上没放这本指南，你不能不寄，你必须跑到一楼大厅，去看那里贴着的、每个人都能用的那份“快递指南”。

### 🔄 PUSCH 配置查找决策树
```text
手机寻找 MsgA PUSCH 资源配置 (时刻表与地图)
        │
        ├─► 第一步：当前在哪个 BWP？(active UL BWP)
        │
        ├─► 第二步：在这个 active UL BWP 下，有没有 msgA-PUSCH-Config？
        │
        ├──► 【有配置】 ✅
        │       └─► 完美！直接使用当前 BWP 的配置获取 PUSCH 资源。
        │
        └──► 【没有配置】 ❌
                │
                └─► 第三步：检查当前 BWP 是不是 初始 BWP (initial UL BWP)？
                      │
                      ├──► 【是初始 BWP】
                      │       └─► 报错/异常（连新手村都没有说明书，没法玩了）。
                      │
                      └──► 【不是初始 BWP】 (比如在专用的业务 BWP)
                              └─► 触发兜底机制：去借用初始 BWP 里的配置来用！ ✅
```

### 💡 章节硬核总结

**本段明确了 MsgA PUSCH 时频资源配置的 BWP 级查找逻辑。UE 的首选规则是直接解析其当前 active UL BWP 下的 `msgA-PUSCH-Config`（或独立的 `separateMsgA-PUSCH-Config`）IE（Information Element）。为了防止高层配置开销过大或配置遗漏，协议引入了跨 BWP 的缺省回退机制 (Fallback mechanism)：若 UE 当前处于非初始 UL BWP 且未收到专属于该 BWP 的 MsgA PUSCH 资源配置，UE 必须继承并使用 Initial UL BWP 中广播或配置的资源参数进行 PUSCH 传输。这一设计兼顾了配置灵活性与接入鲁棒性。**

## 📖 协议原文拆解 (五)：MsgA PUSCH 在频域上的“画车位”规则

> **协议原文**
> A UE determines a first interlace or first RB for a first PUSCH occasion in an active UL BWP respectively from interlaceIndexFirstPO-MsgA-PUSCH or from frequencyStartMsgA-PUSCH that provides an offset, in number of RBs in the active UL BWP, from a first RB of the active UL BWP. A PUSCH occasion includes a number of interlaces or a number of RBs provided by nrofInterlacesPerMsgA-PO or by nrofPRBs-perMsgA-PO, respectively. Consecutive PUSCH occasions in the frequency domain of an UL BWP are separated by a number of RBs provided by guardBandMsgA-PUSCH. A number $N_f$ of PUSCH occasions in the frequency domain of an UL BWP is provided by nrofMsgA-PO-FDM.

### 📚 增量术语表
* **RB (Resource Block)**：资源块。**一句话解释**：5G里分配频率频宽的基本单位，就像地上画好的一格一格的格子，连续的几格构成了一个发送通道。
* **Interlace**：交织结构。**一句话解释**：一种在频域上像梳子齿一样跳跃散布的资源分配方式（主要用于 5G 运行在类似 Wi-Fi 这种非授权/共享频段时，为了满足法规要求的占用带宽）。
* **Guard Band**：保护带。**一句话解释**：两个相邻信号频段之间人为留出的“空白隔离地带”，防止信号靠得太近互相干扰（刮蹭）。
* **FDM (Frequency Division Multiplexing)**：频分复用。**一句话解释**：在同一个时间点，把总频率切成好几条并行的“车道”，让不同的手机同时发送数据，互不干扰。

### 🚪 第一关：找准第一个车位的“起跑线”和“宽度”
基站要在上行带宽（BWP）里划出一片区域，专门给手机发 MsgA PUSCH（我们可以想象成建一个**专属停车场**）。
* **起点在哪？** 协议说，第一个车位（First PUSCH occasion）的位置，是由一个偏移量参数 `frequencyStartMsgA-PUSCH` 决定的。也就是从当前 BWP 的最边缘（第0号 RB）开始，往后数一段距离，这就是第 1 个车位的起点。
* **车位多宽？** 每个 PUSCH 车位的宽度（包含了多少个 RB），由参数 `nrofPRBs-perMsgA-PO` 决定。宽度决定了你能一次性塞多少数据包进去。
*(注：如果用的是非授权频段，这两个参数则替换为看 Interlace 的参数。)*

### 🎯 第二关：车位排排坐（数量与安全车距）
一个车位肯定不够用，基站会在频域上并排画好几个车位：
* **画多少个？** 同一时间在频域上并排的 PUSCH 车位总数量（$N_f$），由参数 `nrofMsgA-PO-FDM` 规定。
* **安全车距：** 为了防止两辆车（两个不同手机发出的信号）靠得太近发生无线电干扰，相邻两个 PUSCH 车位之间必须隔开一段距离。这个“保护带”的宽度（包含了几个空 RB），由参数 `guardBandMsgA-PUSCH` 给出。

│(frequencyStart)│(nrofPRBs...) │(guardBand)│(nrofPRBs...) │(guardBand)
│              │                │         │                │
├──────────────┼────────────────┼─────────┼────────────────┼─────────...
│  正常的业务RB │ PUSCH Occasion │ 空白RB  │ PUSCH Occasion │ 空白RB  ...

### 💡 章节硬核总结

**本段协议完整定义了 Type-2 RACH MsgA 中 PUSCH Occasion (PO) 的频域（Frequency Domain）资源分配拓扑结构。整个频域映射逻辑是一个确定性的线性方程：首个 PO 在 BWP 内的绝对起点由偏移量 `frequencyStart` 锚定；每个 PO 的频域跨度为 `nrofPRBs` 个 PRB；为了减轻频域能量泄漏（OOBE）带来的邻信道干扰，相邻 PO 间插入了 `guardBand` 个 PRB 的保护带；最终在频域上通过 FDM 方式复用了 $N_f$ 个独立的 PUSCH 机会。此外，协议还严谨地覆盖了 Shared Spectrum（如 NR-U）场景，将连续的 PRB 映射平移替换为了基于梳状 Interlace 的映射规则。**

## 📖 协议原文拆解 (六)：共享频谱上的“地盘”限制 (RB Set 约束)

> **协议原文**
> For operation with shared spectrum channel access, if the PUSCH occasion is provided by higher layer parameters frequencyStartMsgA-PUSCH and nrofPRBs-perMsgA-PO, the UE expects a PUSCH occasion to be confined within the same RB set as the corresponding PRACH transmission.

### 📚 增量术语表
* **Shared spectrum channel access**：共享频谱信道接入。**一句话解释**：5G 跑在不需要买牌照的公共免费频段上（类似 Wi-Fi 用的 5GHz/6GHz 频段），大家都要“听一下没人说话，再张嘴”（LBT 机制）。通常也叫 NR-U (NR Unlicensed)。
* **RB set**：资源块集合。**一句话解释**：在很宽的共享频段里，为了方便管理和做“听跳测试(LBT)”，人为把整个大频段切成了一段一段相对独立的“小区隔”，每一个小区隔就是一个 RB set。

### 🚪 第一关：非授权频段的“抢麦”规则
在普通的运营商买下牌照的独占频段里，想怎么发就怎么发。但在**共享频谱（免费公用频段）**里，规矩变了。
手机在发送信号前，必须先在指定的频率范围里竖起耳朵听一听：“现在有人在用这个频率吗？”（这叫 LBT, Listen-Before-Talk）。只有确定没人用，手机才能抢到这个“麦克风”发信号。
通常，这个“听”和“抢”的动作，是按 **RB set（资源块集合，也就是人为划分的地盘）** 为单位来进行的。

### 🎯 第二关：PRACH 与 PUSCH 必须在同一个“地盘”里
这句协议约束了一个非常硬核的场景：如果基站在共享频谱上，使用传统的连续 RB 分配方式（即通过 `frequencyStart` 和 `nrofPRBs` 画车位，而不是用梳子状的 Interlace）来分配 MsgA PUSCH，那么：
**手机绑定的这个 PUSCH occasion，它的物理频率范围，必须完全被限制在（confined within）刚才发送 PRACH 前导码所在的那个同一个 RB set 里！**

**为什么要这样限制？（物理意义）**
如果 PRACH 在“地盘 A”发出去了，说明手机刚在“地盘 A”做过 LBT 听跳测试，确认那里是空闲的，抢到了“地盘 A”的使用权。紧接着它要发 PUSCH，如果 PUSCH 跨到了“地盘 B”，那手机是不是又得重新去“地盘 B”听一听有没有人？这就大大增加了发送失败的概率和时延。
所以，强制它们在同一个“地盘（RB set）”里，可以保证手机只要抢到一次说话的机会，就能一口气把 PRACH 和 PUSCH（即完整的 MsgA）全发出去（这叫 Channel Occupancy Time 共享）。

**生活类比**：
共享频谱就像是“公共自习室的多个大圆桌”（RB sets）。你要发言（MsgA），先要看看“A桌”有没有人说话（LBT），没人你就可以清清嗓子（发 PRACH）。
协议规定：既然你是在“A桌”清的嗓子，你接下来长篇大论的正文（发 PUSCH），也必须老老实实坐在“A桌”讲。绝不允许你跑到“B桌”去占座位讲正文。

### 🔄 RB Set 约束示意图
```text
【共享频谱的整个上行带宽】
        │
        ├─ [RB Set 0] (圆桌A) ─── 手机在此进行 LBT，成功占用。
        │     │
        │     ├──► 发送 PRACH (清嗓子)
        │     │
        │     └──► 发送 PUSCH (讲正文)  <-- 必须在这个框内！✅
        │
        ├─ [RB Set 1] (圆桌B)
        │     │
        │     └──► 如果 PUSCH 分配跨到了这里，报错！协议不允许跨桌。❌
        ...
```

### 💡 章节硬核总结

**本段协议在 Shared Spectrum (NR-U) 场景下引入了一项强一致性 L1 约束。当网络通过基于连续 PRB 分配（而非基于交织 Interlace）的方式调度 MsgA PUSCH 时，终端（UE）预期该 PUSCH 所在的频域资源必须完全内含于其对应 PRACH 传输所在的同一个 RB Set 中。这一约束的设计初衷是为了保证信道接入机制（Channel Access Procedure, LBT）的连续性和有效性。使得 MsgA 的 PRACH 和 PUSCH 能够复用同一次成功执行的 LBT 结果（共享相同的 COT，Channel Occupancy Time），避免因跨 RB Set 传输导致的额外 LBT 开销、时延不确定性以及违反未授权频段的管制法规。**

## 📖 协议原文拆解 (七)：共享频谱上“梳子形”车位的“地盘”与“边界”约束

> **协议原文**
> For operation with shared spectrum channel access, if the PUSCH occasion is provided by higher layer parameters interlaceIndexFirstPO-MsgA-PUSCH and nrofInterlacesPerMsgA-PO, the RB set for the PUSCH occasion in the active UL BWP is the same RB set as the corresponding PRACH transmission. The UE assumes that the RB set is defined as when the UE is not provided intraCellGuardBandsPerSCS for an UL carrier as described in clause 7 of [6, TS 38.214].

### 📚 增量术语表
* **Interlace**：交织结构。**一句话解释**：在频域上像梳子齿一样“隔几个格子抽一根”的离散资源分配方式。在免授权频段（如 Wi-Fi 频段）发信号时，为了满足法规要求的“信号必须均匀占满一定带宽宽度”，5G 会采用这种梳子状结构。
* **intraCellGuardBandsPerSCS**：小区内保护带配置。**一句话解释**：基站人为在一个大带宽内部划分多个 RB set（地盘）时，在中间留出的空白隔离带。

### 🚪 第一关：梳子状资源，同样不许“跨桌”（RB Set 一致性）
上一段协议讲的是连续分配的车位（排排坐），这段讲的是用 `Interlace`（交织）方式分配的车位（梳子状分布）。
核心原则**依然不变**：
不管你的 PUSCH 资源在频域上是连续的一整块，还是像梳子齿一样散开的，只要是在共享频谱（非授权频段）上，**这个梳子所在的“地盘”（RB Set），必须和刚才发 PRACH 的“地盘”是同一个。**
原因依然是为了复用那一次来之不易的 LBT（听跳测试）结果，一抢到麦就赶紧把两段话（MsgA 的两部分）都说完。

### 🎯 第二关：地盘的“边界”按默认图纸算
协议的后半句解决了一个“对齐”问题：手机怎么知道这个 RB set 的物理边界到底在哪？
协议强制规定：在算这个 RB set 边界的时候，手机必须**“假装”没有收到基站配置的 `intraCellGuardBandsPerSCS`（小区内保护带）参数**。
也就是要求手机直接使用协议 [38.214] 中规定的一套**默认、标准的静态算法**来划定 RB set 的边界，不要去考虑那些可能后来动态添加的隔离带。

**为什么要“假装”？（物理意义）**
在随机接入这种“初来乍到”或者“快速恢复”的阶段，基站和手机之间的信息同步可能还不完美。如果手机把保护带算进去，基站没算，两边对这把“梳子”具体插在哪些频率格子上就会产生几十 kHz 的偏差，导致基站根本解不出 PUSCH。强制使用无保护带的“原始默认图纸”，消除了双方在资源网格映射上的任何歧义。

### 🔄 梳子形资源与默认边界示意图
```text
【共享频谱的某个大带宽】
采用默认公式计算的 RB Set 边界 (假装没有配置额外的隔离带)
├─────────────────────────────────────────────────┤
│                  RB Set A (圆桌A)                │
│                                                 │
│  [PRACH] (抢到麦，发敲门砖)                       │
│                                                 │
│  [PUSCH 梳子齿1]  [PUSCH 梳子齿2]  [PUSCH 梳子齿3] │<-- Interlace PUSCH
│    │               │               │            │
│    └─► 所有的梳子齿都必须落在这个默认计算好的 RB Set A 内！
└─────────────────────────────────────────────────┘
```

### 💡 章节硬核总结

**本段协议将共享频谱（NR-U）下的 MsgA PUSCH 频率范围一致性约束扩展到了基于 Interlace（交织）调度的场景。它重申了 LBT COT（信道占用时间）复用的核心原则：Interlace PUSCH 必须与关联的 PRACH 位于同一 RB Set。更为关键的是，它通过硬编码物理层假设（即强制 UE 假设未配置 `intraCellGuardBandsPerSCS`），统一了基站与 UE 对 RB Set 物理边界的计算基准。这一防错设计（Failsafe design）确保了在复杂的 NR-U 频域网格划分中，MsgA MsgA 的交织物理资源块（PRB）级映射能够达成绝对的确定性与一致性。**

## 📖 协议原文拆解 (八)：MsgA PUSCH 在时间上的“排片表” (时域资源分配)

> **协议原文**
> If a UE does not have dedicated RRC configuration, or has an initial UL BWP as an active UL BWP, or is not provided startSymbolAndLengthMsgA-PO, msgA-PUSCH-timeDomainAllocation provides a SLIV and a PUSCH mapping type for a PUSCH transmission by indicating
> - one of the first maxNrofUL-Allocations values from PUSCH-TimeDomainResourceAllocationList, if PUSCH-TimeDomainResourceAllocationList is provided in PUSCH-ConfigCommon
> - one of the entries from table 6.1.2.1.1-2 or table 6.1.2.1.1-3 in [6, TS 38.214], if PUSCH-TimeDomainResourceAllocationList is not provided in PUSCH-ConfigCommon
> else, the UE is provided a SLIV by startSymbolAndLengthMsgA-PO, and a PUSCH mapping type by mappingTypeMsgA-PUSCH for a PUSCH transmission.

### 📚 增量术语表
* **SLIV (Start and Length Indicator Value)**：起点和长度指示值。**一句话解释**：一个数字密码，手机解开它后，就能同时知道这段数据从第几个符号（Symbol）开始发，以及总共要发几个符号。
* **PUSCH mapping type**：PUSCH 映射类型（Type A 或 Type B）。**一句话解释**：决定了用来解调的导频信号（DMRS）是固定在时隙的最前面，还是紧挨着数据的起始位置发送。
* **PUSCH-TimeDomainResourceAllocationList**：时域资源分配列表。**一句话解释**：基站下发的一张“时域套餐表”，里面列了好几种不同的 (SLIV + 映射类型) 组合供挑选。
* **msgA-PUSCH-timeDomainAllocation**：MsgA PUSCH 时域分配指示。**一句话解释**：一个像点菜时的“编号”，用来告诉手机去上面的“套餐表”里挑第几个套餐。
* **startSymbolAndLengthMsgA-PO**：MsgA PUSCH 专用起始与长度参数。**一句话解释**：不走查表套餐，直接硬编码给出的 SLIV 绝对值。

### 🚪 第一关：新手/公共状态下的“查表点菜”模式
如果手机处于比较基础的状态（**符合以下任一条件**）：
1. 还没建立专属的 RRC 连接（刚开机/还没转正）。
2. 目前正在“新手村”（使用初始 BWP）。
3. 基站没有单独给它下发 `startSymbolAndLengthMsgA-PO` 这个具体参数。

此时，手机决定 PUSCH 发送时间的策略是“**按编号点套餐**”。参数 `msgA-PUSCH-timeDomainAllocation` 就是套餐编号。
* **去哪查表？**
  * **优先看大堂黑板**：如果基站在公共配置（`PUSCH-ConfigCommon`）里下发了“时域套餐表”（`PUSCH-TimeDomainResourceAllocationList`），手机就按编号去这个表里找对应的 SLIV 和映射类型。
  * **翻出厂说明书**：如果基站连公共套餐表都没给，手机就只能翻开 3GPP 协议的 38.214 标准（Table 6.1.2.1.1-2/3），使用协议里写死的最基础的出厂默认表格来进行查找。

### 🎯 第二关：VIP状态下的“私人订制”模式
也就是原文最后的 `else` 分支。
如果手机已经转正（有专属 RRC 配置），不在新手村（在专属 BWP），并且基站明确给下发了专属字段：
那么，手机就**不再需要去查套餐表**了。基站会通过 `startSymbolAndLengthMsgA-PO` 直接把具体的 SLIV 密码塞给手机，并且通过 `mappingTypeMsgA-PUSCH` 直接告诉手机用什么映射类型。
这是一种极度明确、无需引用的“私人订制”分配方式。

### 🔄 MsgA PUSCH 时域分配查找决策树
```text
 手机确定 MsgA PUSCH 发送的起始时间和长度 (SLIV + Mapping Type)
        │
        ├─► 检查前提：无专属RRC？或在初始BWP？或没给专属SLIV字段？
        │
        ├──► 【是】(触发公共查表模式)
        │    │    使用参数 msgA-PUSCH-timeDomainAllocation 作为“行号”去查表：
        │    │
        │    ├─► 基站给了公共配置表 (PUSCH-TimeDomainResourceAllocationList) 吗？
        │    │    ├──► 【给了】 ──> 去这个广播表里提取对应的 SLIV 和 Type
        │    │    └──► 【没给】 ──> 翻开 3GPP 协议，查默认的硬编码出厂表
        │
        └──► 【否】(触发 VIP 专属模式)
                  说明处于深度连接阶段且参数完备：
                  └─► 抛弃查表！
                      直接读取 startSymbolAndLengthMsgA-PO (获得 SLIV)
                      直接读取 mappingTypeMsgA-PUSCH (获得 Type)
```

### 💡 章节硬核总结

**本段协议完整定义了 MsgA PUSCH 时域资源（Start Symbol, Length 和 Mapping Type）的动态解析层级。为了兼顾初始接入阶段的公共广播开销与连接态阶段的调度灵活性，协议设计了“索引查表（Index-based）”与“直接指示（Direct-value）”两种模式。在 RRC 建立前、Initial BWP 阶段或缺乏专属配置时，UE 提取一个整型索引去查询系统广播的 `PUSCH-TimeDomainResourceAllocationList`（或回退至 38.214 的 Default Tables）。而在专属 BWP 配置成熟后，网络可以直接通过 RRC 专属信令将 SLIV 和 Mapping Type 作为离散参数下发给 UE，从而越过公共表的限制。这种 fallback to default 的层级设计是 3GPP 保证在任何信令完备度下物理层都能成功调度的核心思想。**
