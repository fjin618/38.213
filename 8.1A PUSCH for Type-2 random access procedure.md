# 🔖 章节：8.1A PUSCH for Type-2 random access procedure

---

## 📖 协议原文拆解 (一)：Type-2 RACH 中 PRACH 与 PUSCH 的“黄金距离”

> **协议原文**
> For a Type-2 random access procedure, a UE transmits a PUSCH, when applicable, after transmitting a PRACH. The UE encodes a transport block provided for the PUSCH transmission using redundancy version number 0. The PUSCH transmission is after the PRACH transmission by at least $N$ symbols where $N=2$ for $\mu=0$ or $\mu=1$, $N=4$ for $\mu=2$ or $\mu=3$, $N=16$ for $\mu=5$, $N=32$ for $\mu=6$, and $\mu$ is the SCS configuration for the active UL BWP.

### 📚 增量术语表
* **Type-2 random access procedure**：2步随机接入过程 (2-step RACH)。**一句话解释**：手机为了更快接入基站，把传统的4步流程浓缩成2步，在第一步（MsgA）就同时把接入暗号（PRACH）和自己的详细信息（PUSCH）打包发给基站。
* **Redundancy version number 0 (RV0)**：冗余版本 0。**一句话解释**：数据编码发送时的“初始原版”数据，包含了解码所需的最多、最完整的信息（相对后续重传的补丁数据而言）。
* **$\mu$ (Mu)**：子载波间隔 (SCS) 的配置序号。**一句话解释**：决定了频域上“车道”的宽度和时域上每个符号 (Symbol) 的时间长度。$\mu$ 越大，时域上每个符号的时间就越短。

### 🚪 第一关：两步走的“连招”与“初版数据”
* **连招动作**：在 2-step RACH 中，手机的第一步叫 MsgA。MsgA 实际上是由两部分组成的：先发 **PRACH**（前导码，相当于敲门砖），然后再发 **PUSCH**（共享信道，里面装着真正的身份和请求数据）。
* **数据打包要求**：协议规定，这个 PUSCH 里的数据块 (Transport Block) 必须使用 **RV0 (冗余版本0)** 来编码。因为这是手机第一次发送这些数据，必须把最完整、自解码能力最强的初始版本发给基站，确保基站一次就能大概率解出来。

### 🎯 第二关：动作之间的“喘息时间” ($N$ 个符号)
* **物理限制**：手机刚用一种波形或资源发完了 PRACH，立刻就要接着发 PUSCH。手机的射频器件 (RF) 和基带芯片需要一点时间来切换状态、调整功率或重置参数。
* **动态的间隔**：协议规定，在 PRACH 结束到 PUSCH 开始之间，必须留出**至少 $N$ 个符号 (Symbols)** 的间隔。
* **水涨船高的 $N$ 值**：因为当 $\mu$ (子载波间隔配置) 变大时，每个符号的绝对物理时间 (毫秒级) 会变短。为了保证手机有足够的绝对时间来完成硬件状态切换，符号数 $N$ 必须随着 $\mu$ 的增大而增加：
    * $\mu=0, 1$ (15/30 kHz)：$N=2$ 个符号
    * $\mu=2, 3$ (60/120 kHz)：$N=4$ 个符号
    * $\mu=5$ (480 kHz)：$N=16$ 个符号
    * $\mu=6$ (960 kHz)：$N=32$ 个符号

### 🔄 Type-2 RACH 发送流水线
```text
 手机发起 Type-2 RACH (MsgA)
        │
        ├─► 动作1：发送 PRACH (敲门砖)
        │
        ├─► ⏳ 强制等待阶段 (硬件切换缓冲期)：
        │    查看当前的上行带宽配置 (active UL BWP) 的 SCS 是多少 ($\mu$)
        │    - 如果是 30kHz ($\mu=1$) ──► 至少等 2 个符号
        │    - 如果是 120kHz ($\mu=3$) ─► 至少等 4 个符号
        │
        ▼
        ├─► 动作2：准备 PUSCH 数据
        │    强制要求：必须使用 RV=0 (最完整的信息版本) 进行编码
        │
        └─► 动作3：发送 PUSCH (递交真实身份和请求)
```

### 💡 章节硬核总结

**这段协议明确了 8.1A 章节下 2-step RACH 中 MsgA 内部的时序与编码硬性约束。首先，MsgA 的 PUSCH payload 必须采用重传冗余版本 RV0 进行初传，以最大化基站的初传解码成功率。其次，规定了从 PRACH 传输结束到 PUSCH 传输开始之间的最小时间 Gap。该 Gap 以正交频分复用（OFDM）符号数 $N$ 为单位，且 $N$ 的取值随上行活动部分带宽（active UL BWP）的子载波间隔（SCS，即 $\mu$）成比例缩放。其物理本质是为了在不同的系统参数配置下，为终端的射频前端和基带处理保留一段恒定的、足够完成信道切换与调谐的绝对时间。**
