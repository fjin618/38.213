# 8.1 Random Access Preamble (随机接入前导码的发射)

本章节聚焦于随机接入流程的最起点——**Msg1（或 Type-2 中的 MsgA Preamble）的发射**。这是终端在 5G 网络中发出的第一声呼喊。物理层将在这里接收高层的最高指令，并加载一系列极其精密的时空与能量参数，确保这声暗号能够精准地轰进基站的接收天线中。

---

## 📖 协议原文拆解 (一)：开火的触发源与“弹药库”清单

> **协议原文**
> Physical random access procedure for a UE is triggered upon request of a PRACH transmission by higher layers or by a PDCCH order or LTM Cell Switch Command MAC CE in clause 6.1.3.75 [11, TS 38.321] for a cell. A configuration by higher layers for a PRACH transmission includes the following:
> - A configuration for PRACH transmission on the cell [4, TS 38.211].
> - A preamble index, a preamble SCS, $P_{PRACH,target}$, a corresponding RA-RNTI when applicable [11, TS 38.321], and a PRACH resource for the cell.
> - A number of $N_{preamble}^{rep} > 1$ preamble repetitions for the PRACH transmission if the UE would transmit the PRACH with repetitions.

### 📚 增量术语表
* **PRACH (Physical Random Access Channel)**：物理随机接入信道。**一句话解释**：手机用来向基站发射“暗号”的专属发射通道。
* **Preamble index**：前导码索引。**一句话解释**：你要喊的具体是哪一句暗号（0 到 63 里的哪一个数字）。
* **$P_{PRACH,target}$**：PRACH 目标接收功率。**一句话解释**：基站对你下达的音量指标：“小子，不管你离我多远，你发出的声音到达我耳朵里时，必须要有这么大声！”
* **$N_{preamble}^{rep}$**：前导码重复次数。**一句话解释**：如果在极度弱网环境，暗号喊一遍基站听不见，高层就会命令你连续喊 N 遍。

### 🚪 第一关：谁扣动了扳机？(三大触发源)
物理层自己是个“哑巴”，绝对不会主动去发暗号。它的每一次发射，必然是由以下三大“扳机”之一触发的：
1. **高层主动请求 (request by higher layers)**：最常见的场景。比如你刚打开手机要发微信，手机大脑（MAC/RRC层）主动命令物理层去连网。
2. **基站大喇叭强制命令 (PDCCH order)**：你在上网时突然数据断流或者失步了，基站直接通过底层控制信道（PDCCH）拿枪指着你：“立刻给我发个 PRACH 报位置！”
3. **极速换区加急信令 (LTM MAC CE)**：高铁上移动太快，旧基站扛不住了，直接甩给你一张 MAC 层加急指令：“马上用这张 VIP 免排队暗号，去隔壁新基站报到！”

### 📦 第二关：高层下发的“弹药库清单”
一旦扳机扣动，高层会立刻给物理层的“弹药库”里装填三类核心弹药（参数）：

* **基础枪管配置 (Configuration on the cell)**
  * 高层指明了在这个基站下发射的基础波形参数（指向了 38.211 协议里的底层生成公式）。
* **核心开火参数 (The Big Five)**
  * **Preamble index**：弹药型号（你要发哪个暗号）。
  * **Preamble SCS**：车道宽度（你在多宽的频段里发）。
  * **$P_{PRACH,target}$**：开火力度（计算发射功率的基准，保证基站能听清）。
  * **RA-RNTI**：接头暗号（发完之后，用来监听基站回执 Msg2 的专用防伪码）。
  * **PRACH resource**：开火坐标（精确到哪一个时隙、哪一个频率格子去发）。
* **火力覆盖增强 (Repetitions 连发机制)**
  * 如果你在地下室，高层还会塞给你一个连发数字（$N_{preamble}^{rep} > 1$），命令你：“别只发一枪，给我对着这个坐标连开 N 枪（重复发送）！”

### 🔄 PRACH 点火触发与参数装载流水线
```text
 【三大触发源】
  ├─ 手机大脑主动想上网 (Higher layers)
  ├─ 基站强制点名 (PDCCH Order)
  └─ 高铁极速切区指令 (LTM MAC CE)
         │
         ▼ (扣动扳机)
 高层 (MAC) 收到触发，开始为物理层 (PHY) 准备弹药
         │
         ├─ 装填 1: 基础波形生成法则
         ├─ 装填 2: 暗号数字、频率宽度、发射目标功率、回执防伪码、发射坐标
         └─ 装填 3: (如果在弱网) 连发开火次数
         │
         ▼
 物理层 (PHY) 接收全部参数，“弹药”上膛，进入点火倒计时！
```

## 📖 协议原文拆解 (二)：真正的“开火”与波束方向锁定 (PRACH 发射动作)

> **协议原文**
> A UE transmits a PRACH on a cell using the selected PRACH format with transmission power $P_{PRACH,b,f,c}(i)$, as described in clause 7.4, on the indicated PRACH resource or on a determined set of $N_{preamble}^{rep}$ resources using a same spatial filter in case of $N_{preamble}^{rep}$ preamble repetitions.

### 📚 增量术语表
* **$P_{PRACH,b,f,c}(i)$**：PRACH 实际发射功率。**一句话解释**：手机真正开火轰出暗号时的音量大小（它是在高层给的“目标音量”基础上，结合当前自己估算出的空中路径损耗，最终推算出来的绝对发射功率）。公式下角标代表了具体的 BWP、载波和物理小区。
* **$N_{preamble}^{rep}$**：前导码重复次数。**一句话解释**：为了增强信号覆盖，连续喊暗号的遍数。
* **Spatial filter**：空间滤波器（发送波束）。**一句话解释**：手机发射信号时的“天线朝向/喇叭指向”。

### 🚪 核心机制：点火发射与“不许转头”的连发死令
* **场景倒带**：在前面的第 (三) 关中，物理层已经从高层那里拿到了所有发射参数，包括在哪发、发什么、如果信号差要连发几次。现在，时辰已到！
* **执行单发（常规开火）**：
  * 物理层在指定的时间和频率格子上（indicated PRACH resource），按照指定的长短波形（PRACH format），将计算好的能量（$P_{PRACH,b,f,c}(i)$）注入天线，轰出这一声暗号。
* **执行连发（覆盖增强的绝对红线）**：
  * 如果手机身处地下室等弱覆盖区域，触发了“重复发送”的被动技能（比如要连发 4 遍）。协议在这里下达了一道极其严厉的死命令：**在这 $N_{preamble}^{rep}$ 次连续发射的过程中，手机必须使用“完全相同的空间滤波器（same spatial filter）”！**
  * **现实意义**：什么意思？就是“**不许边转头边喊**”！基站之所以让你连发 4 遍，是为了在它那头把这 4 次微弱的信号收集起来，像叠罗汉一样叠加在一起（能量合并），才能勉强听清。如果你第一遍冲着东边喊，第二遍冲着西边喊（中途改变了波束方向），基站收到的信号相位就会彻底乱套，根本没法叠加。所以，连发期间，你的天线姿势必须像石头一样死死锁住。

### 🔄 PRACH 点火与波束锁定流水线
```text
 物理层 (Layer 1) 等到了预定的发射时间 (PRACH resource)
    │
    ├─ 动作 1: 将暗号调制成指定的波形 (PRACH format)
    ├─ 动作 2: 注入计算好的发射能量 (P_PRACH)
    │
    ├─ 问自己：我这是一次性发射，还是连续重复发射 (Repetitions)？
    │
    ├──► 【单次发射】 
    │         └─► 动作: 对准刚才锁定的灯塔方向，开火！(1次)
    │
    └──► 【连续重复发射 (比如 N = 4 次)】
              │
              └─► 动作: 【锁定空间滤波器 (Spatial Filter)】天线姿势彻底冻结！
                        ├── 第 1 次开火 🚀
                        ├── 第 2 次开火 🚀 (方向保持绝对一致！)
                        ├── 第 3 次开火 🚀 (方向保持绝对一致！)
                        └── 第 4 次开火 🚀 (方向保持绝对一致！)
```

## 📖 协议原文拆解 (三)：灯塔与暗号的“空间结对” (SSB 到 PRACH 的映射法则)

> **协议原文**
> For Type-1 random access procedure, a UE is provided a number $N$ of SS/PBCH block indexes associated with one PRACH occasion and a number $R$ of contention based preambles per SS/PBCH block index per valid PRACH occasion by ssb-perRACH-OccasionAndCB-PreamblesPerSSB.

### 📚 增量术语表
* **PRACH occasion (RO)**：物理随机接入时机。**一句话解释**：基站划定的具体“发车位”。它在时域和频域上是一个确定的方格，在这个格子里发出来的信号才算合法。
* **Contention based preambles (CB preambles)**：基于竞争的前导码。**一句话解释**：手机可以随便盲抽的“公共暗号”。因为是盲抽，所以容易和别人撞车（竞争）。
* **ssb-perRACH-OccasionAndCB-PreamblesPerSSB**：SSB到RO及前导码的映射参数。**一句话解释**：这是系统广播（SIB1）里的一条“超级绑定规则”，用来规定基站的灯塔（SSB）和手机发暗号的车位（RO）到底是怎么结对子的。
* **$N$ (Number of SSB per RO)**：N 值。**一句话解释**：几个灯塔（SSB）共用一个发车位（RO）。
* **$R$ (Number of CB preambles per SSB)**：R 值。**一句话解释**：在这个发车位里，每个灯塔能分到多少个公共暗号（Preamble）。

### 🚪 核心机制：波束映射的“结对子”艺术
* **场景倒带**：在 5G 毫米波里，基站可能朝四面八方发射了 8 个甚至 64 个极窄的灯塔波束（SSB）。手机在地上测了一圈，挑了一个最亮的（比如 SSB#2）准备发暗号。
* **基站的痛点**：手机是在天上发暗号的，基站怎么知道手机挑了哪个灯塔（也就是手机现在处于哪个波束的覆盖范围内）？如果基站不知道，它回信（Msg2）的时候就不知道该往哪个方向打波束。
* **协议的破局之法（隐式绑定）**：协议非常聪明，它不让手机显式地报告“我选了 2 号灯塔”，而是让基站提前定下规矩（通过参数 `ssb-perRACH-OccasionAndCB-PreamblesPerSSB` 规定 $N$ 和 $R$）：
  * **如果 $N = 1$**：代表 1 个灯塔独占 1 个发车位（RO）。那么手机只要在 2 号发车位里发暗号，基站一听，秒懂：“这小子绝对在 2 号灯塔下！”
  * **如果 $N = 2$**：代表 2 个灯塔（比如 SSB#0 和 SSB#1）挤在同一个发车位里。这怎么区分？这就靠 $R$ 值了！基站规定每个车位里有 64 个暗号，那么 SSB#0 分走前 32 个，SSB#1 分走后 32 个（$R=32$）。手机如果选了 SSB#1，它就必须从后 32 个暗号里抽。基站在这个车位里一听到后半段的暗号，再次秒懂：“原来是在 SSB#1 的方向！”

### 🔄 SSB 与 PRACH 资源的映射流水线
```text
 基站 (gNB) 广播映射规则：参数 ssb-perRACH-OccasionAndCB-PreamblesPerSSB
    │
    ├─ 规定了 N (几个灯塔挤一个车位)
    ├─ 规定了 R (每个灯塔分几个暗号)
    │
    ▼
 手机 (UE) 接收规则，开始找车位发暗号
    │
    ├─ 动作 1: 测出最亮的灯塔 (比如认定 SSB#3 最好)
    │
    ├─ 动作 2: 去查基站给的映射表，找出 SSB#3 对应的发车位 (RO) 在哪里
    │
    ├─ 动作 3: 在这个车位里，抽出只属于 SSB#3 的专属暗号池 (R 个暗号)
    │
    ├─ 动作 4: 闭上眼睛，从这 R 个暗号里随机盲抽 1 个
    │
    └─► 动作 5: 在对应的时间、对应的频率，把抽到的暗号发射出去！🚀
              │
              ▼
    (基站听到这个特定的暗号，立刻知道了手机对应的空间方位！)
```

## 📖 协议原文拆解 (四)：2 步与 4 步接入的“拼车与分家” (共享 RO 映射法则)

> **协议原文**
> For Type-2 random access procedure with common configuration of PRACH occasions with Type-1 random access procedure, a UE is provided a number $N$ of SS/PBCH block indexes associated with one PRACH occasion by ssb-perRACH-OccasionAndCB-PreamblesPerSSB and a number $Q$ of contention based preambles per SS/PBCH block index per valid PRACH occasion by msgA-CB-PreamblesPerSSB-PerSharedRO. The PRACH transmission can be on a subset of PRACH occasions associated with a same SS/PBCH block index within an SSB-RO mapping cycle for a UE provided with a PRACH mask index by msgA-SSB-SharedRO-MaskIndex according to [11, TS 38.321].

### 📚 增量术语表
* **Common configuration of PRACH occasions**：RO 发车位共用。**一句话解释**：2步极速接入（Type-2）和 4步传统接入（Type-1）的手机，在同一个时间、同一个频率格子（RO）里发暗号。俗称“拼车”。
* **$Q$ (msgA-CB-PreamblesPerSSB-PerSharedRO)**：Q 值。**一句话解释**：在拼车的情况下，每个灯塔（SSB）专门划拨给 2步接入（MsgA）使用的专属暗号数量。
* **PRACH mask index (msgA-SSB-SharedRO-MaskIndex)**：掩码索引。**一句话解释**：基站下发的一张“限行/过滤表”。即使你算出来有几个车位可以发车，也必须用这个过滤网筛一下，只能在基站指定的那几个子车位里发。

### 🚪 第一关：物理层的“同车厢，不同座” (码域隔离)
* **场景痛点**：在网络里，同时存在走 4 步的老手机和走 2 步的 VIP 新手机。如果为了它们分别建两条 PRACH 物理车道，太浪费极其昂贵的频谱资源了。
* **拼车机制**：所以协议允许它们“拼车（Common configuration）”，都在同一个时频格子里发暗号。$N$（几个灯塔挤一个车位）的规则完全照抄上一段（Type-1）。
* **防止撞车（Q值分家）**：大家在同一个格子里发，基站怎么知道谁是走 4 步、谁是走 2 步的？
  * 协议采用了极其聪明的“暗号池切分”。在这个格子里，基站规定：前 $R$ 个暗号只准走 4 步的人用，后 $Q$ 个暗号只准走 2 步的人（MsgA）用！
  * 这样，虽然大家在同一个时间、频率点火，但基站只要听出暗号的波形，就能瞬间判定：“嗯，这是个 MsgA 暗号，我要准备收它的数据包并回 MsgB 盲盒了！”

### 🛡️ 第二关：VIP 通道的限行令 (Mask Index)
* **痛点**：有时候一个灯塔（SSB）对应了好几个发车位（RO）。走 2 步的人（MsgA）通常包袱很重（后面跟着庞大的数据 PUSCH）。如果他们随意在这些车位里乱发，容易引起大规模的连环撞车。
* **限行过滤**：协议引入了 `msgA-SSB-SharedRO-MaskIndex`（掩码）。高层会通过这个掩码告诉物理层：“虽然从映射表上看，这 4 个车位你都能用，但是为了分流，我命令你只能在其中的第 1 个和第 3 个车位（Subset）里发射 MsgA！”

### 🔄 Type-2 共享车道资源选择流水线
```text
 走极速 2步 (Type-2) 的手机准备发 MsgA 暗号
    │
    ├─ 问高层：我们自己有专用的独立车道吗？
    │
    └──► 【高层回答: 没有，为了省资源，你要和走 4 步的人“拼车”！】
              │
              ├─ 动作 1 (找车位): 照抄 4 步的 N 值法则，算出属于自己 SSB 的那一批车位。
              │
              ├─ 动作 2 (掩码限行): 拿着 [MaskIndex]，把不让发的车位过滤掉，锁定最终车位。
              │
              ├─ 动作 3 (暗号分家): 坚决不去碰 4步的暗号池！去查属于 2步的 Q 值参数，
              │                    在专门划拨给 MsgA 的暗号池里，盲抽一个暗号！
              │
              ▼
    在限行后的合法车位里，轰出这发专属的 MsgA 暗号！🚀
```

## 📖 协议原文拆解 (五)：2步极速通道的“专属车道” (独立 RO 映射法则)

> **协议原文**
> For Type-2 random access procedure with separate configuration of PRACH occasions with Type-1 random access procedure, a UE is provided a number $N$ of SS/PBCH block indexes associated with one PRACH occasion and a number $R$ of contention based preambles per SS/PBCH block index per valid PRACH occasion by msgA-SSB-PerRACH-OccasionAndCB-PreamblesPerSSB when provided; otherwise, by ssb-perRACH-OccasionAndCB-PreamblesPerSSB.

### 📚 增量术语表
* **Separate configuration of PRACH occasions**：RO 发车位独立配置。**一句话解释**：2步极速接入（Type-2）不再和 4步老手机“拼车”了。基站专门为 Type-2 开辟了一条完全独立的物理发车通道。
* **msgA-SSB-PerRACH-OccasionAndCB-PreamblesPerSSB**：MsgA 的专属映射参数。**一句话解释**：专门针对 2步接入（MsgA）下发的灯塔到车位的映射表，定义了专属的 $N$（几个灯塔挤一个车位）和 $R$（每个灯塔分几个暗号）。

### 🚪 第一关：VIP 独立车道的映射法则
* **场景倒带**：在拆解 (四) 里，因为是“拼车”，Type-2 只能憋屈地和 Type-1 共用 N 值的算法，然后再用 Q 值把暗号劈开。
* **如今的阔气**：现在的网络配置变了，基站为 Type-2 单独配置了 PRACH 资源（Separate configuration）。这意味着走 2 步流程的手机，拥有了完全独立的时频格子（RO）。
* **独立映射（$N$ 和 $R$）**：既然车道都是独立的了，那么“几个灯塔（SSB）挤一个车位（RO）”以及“每个灯塔分几个暗号”的规矩，当然也可以独立制定了。基站会下发一个全新的专属参数 `msgA-SSB-PerRACH-OccasionAndCB-PreamblesPerSSB`，让手机去查阅专属于 Type-2 的 $N$ 值和 $R$ 值。

### 🛡️ 第二关：协议的严谨“兜底”法则
* **痛点**：万一基站说好了给你“独立车道”，但是在发广播信令时，忘了把这本专属的“参数映射表（msgA-SSB...）”发下来怎么办？手机总不能在这个独立车道上瞎停吧？
* **抄作业（Otherwise...）**：协议在最后半句给出了极具画面感的兜底法则：“`otherwise, by ssb-perRACH-OccasionAndCB-PreamblesPerSSB`”。意思是：**如果基站没给你发专属的表，那你就直接去抄隔壁 4步接入（Type-1）的那张老表！** 就算是在独立车道上，也按 4步接入的规矩（N和R）来找发车位。

### 🔄 Type-2 专属独立车道的选位流水线
```text
 走极速 2步 (Type-2) 的手机准备发 MsgA 暗号
    │
    ├─ 问高层：我们自己有专用的独立车道吗？
    │
    └──► 【高层回答: 有！我们不跟 4 步拼车了，走独立配置 (Separate configuration)！】
              │
              ├─ 问自己：基站给我发专属于我的映射表 (msgA-SSB-...) 了吗？
              │
              ├──► 【收到了专属表】 
              │         └─► 动作: 用专属表里的 N 和 R，去独立的 RO 车道上找车位发暗号！
              │
              └──► 【没收到专属表 (基站漏配了)】
                        └─► 动作 (兜底): 直接照抄隔壁 4步接入的映射表 (ssb-perRACH-...)，
                                        拿着老规矩，去新车道上找车位发暗号！
```

## 📖 协议原文拆解 (六)：特殊业务的“VIP 专属暗号池” (Feature Combination 映射法则)

> **协议原文**
> For a random access procedure associated with a feature combination indicated by FeatureCombinationPreambles, a UE is provided a number $N$ of SS/PBCH block indexes associated with one PRACH occasion by ssb-perRACH-OccasionAndCB-PreamblesPerSSB or msgA-SSB-PerRACH-OccasionAndCB-PreamblesPerSSB when provided and a number $S$ of contention based preambles per SS/PBCH block index per valid PRACH occasion by startPreambleForThisPartition and numberOfPreamblesPerSSB-ForThisPartition. The PRACH transmission can be on a subset of PRACH occasions associated with a same SS/PBCH block index within an SSB-RO mapping cycle for a UE provided with a PRACH mask index by ssb-SharedRO-MaskIndex according to [11, TS 38.321].

### 📚 增量术语表
* **Feature combination**：特性组合。**一句话解释**：手机拥有某种“特殊身份”或“特殊业务需求”，比如它是省电的 RedCap（轻量化 5G）终端，或者是在做切片业务（Slicing），或者是小包数据传输（SDT）。
* **FeatureCombinationPreambles**：特性组合前导码。**一句话解释**：基站专门为这批有“特殊身份”的手机，单独圈出来的一批暗号。
* **$S$ (numberOfPreamblesPerSSB-ForThisPartition)**：S 值。**一句话解释**：在这个被切分出来的“VIP 包厢（Partition）”里，每个灯塔（SSB）能分到多少个专属暗号。
* **startPreambleForThisPartition**：VIP 包厢的起点。**一句话解释**：在这个发车位的所有暗号里，从第几个暗号开始算作是你们这个特殊业务的 VIP 专属暗号。
* **ssb-SharedRO-MaskIndex**：共享发车位掩码。**一句话解释**：即使是特殊业务，也不能随便占用所有车位。基站通过这个掩码过滤网，限制你只能在指定的那几个车位里发车。

### 🚪 第一关：物理层的“大包厢切分术” (Partition)
* **场景倒带**：在之前的基础映射中，我们知道一个发车位（RO）里的 64 个暗号，可能被普通手机（Type-1）和极速手机（Type-2）瓜分。
* **新贵入场**：现在，随着 5G 的发展，各种五花八门的新手机进网了。比如来了一批极其弱小的 RedCap（手表/手环）设备。如果让它们和普通大手机去抢同一个暗号池，它们永远抢不赢。
* **隔离策略**：基站决定给它们搞个“VIP 隔断（Partition）”。$N$ 值（几个灯塔挤一个车位）还是照旧（无论是复用老规矩还是用专属规矩）。但暗号的分配彻底变了！
  * 基站下发指令：“RedCap 手机听令，你们的暗号池起点是第 40 号（`startPreambleForThisPartition`），每个灯塔给你们分了 10 个暗号（$S$ 值）！”
  * 于是，拥有特殊身份的手机，就能完美避开普通手机的暗号池，去抽取专属于自己的那 $S$ 个暗号。

### 🛡️ 第二关：特殊业务的限行令 (Shared RO Mask Index)
* **痛点**：因为这种特殊业务往往是和普通业务“共享车道（Shared RO）”的，如果任由这些特殊手机在所有合法的车位里狂轰滥炸，也会影响交通。
* **掩码限行**：协议故技重施，再次引入了掩码 `ssb-SharedRO-MaskIndex`。高层会告诉物理层：“你虽然属于特殊业务群，但你只能在我给你的掩码允许的子集（subset）车位里发射！”

### 🔄 特殊业务的资源选择流水线
```text
 特殊身份手机 (如 RedCap) 准备发射 PRACH 暗号
    │
    ├─ 问自己：我有什么“特殊特性 (Feature combination)”吗？
    │
    └──► 【高层确认: 有！我们需要使用隔离包厢 (Partition) 的参数】
              │
              ├─ 动作 1 (找车位): 用 N 值算出属于自己 SSB 的那一批车位。
              │
              ├─ 动作 2 (限行过滤): 拿着 [SharedRO-MaskIndex]，过滤掉不让发的车位。
              │
              ├─ 动作 3 (定位包厢): 找到专属起跑线 (startPreamble)
              │
              ├─ 动作 4 (盲抽暗号): 在专属的包厢容量 (S 个暗号) 内，抽取一个暗号！
              │
              ▼
    在限行后的合法车位里，轰出带有“特殊身份”标志的专属暗号！🚀
```

## 📖 协议原文拆解 (七)：发车位里暗号分配的数学公式 (Preamble Index 映射)

> **协议原文**
> For Type-1 random access procedure, or for Type-2 random access procedure with separate configuration of PRACH occasions from Type 1 random access procedure, if $N<1$, one SS/PBCH block index is mapped to $1/N$ consecutive valid PRACH occasions and $R$ contention based preambles with consecutive indexes associated with the SS/PBCH block index per valid PRACH occasion start from preamble index 0. If $N \ge 1$, $R$ contention based preambles with consecutive indexes associated with SS/PBCH block index $n$, $0 \le n \le N-1$, per valid PRACH occasion start from preamble index $n \cdot N_{preamble}^{total}/N$ where $N_{preamble}^{total}$ is provided by totalNumberOfRA-Preambles... and is an integer multiple of $N$.

### 📚 增量术语表
* **$N$**：几个灯塔（SSB）挤一个发车位（RO）。如果 $N<1$（比如 $N=1/2$），说明灯塔太少车位太多，一个灯塔能分到 2 个完整的发车位。如果 $N \ge 1$（比如 $N=2$），说明两个灯塔得挤在同一个发车位里。
* **$R$**：每个灯塔分到的暗号数量。
* **$N_{preamble}^{total}$**：这个发车位里总共提供的暗号数量（最多64个）。
* **Preamble index 0**：暗号的编号从 0 开始。

### 🚪 第一关：土豪模式（$N < 1$，一个灯塔霸占多个车位）
* **场景倒带**：当 $N = 1/2$ 时，意味着基站的灯塔（波束）很少，但是配置的 PRACH 发车位却很多。
* **分配法则**：这就像一个土豪（灯塔）直接包下了 2 个连着的停车场（$1/N$ 个 consecutive ROs）。
* **暗号起编规则**：既然整个停车场都是你的，那你在这个停车场里分配暗号时，当然是极其霸道地**直接从 0 号开始编（start from preamble index 0）**！手机只要在这个停车场里抽，就算抽到了 0 号暗号，基站也知道这绝对是属于这个土豪灯塔的。

### ✂️ 第二关：挤车厢模式（$N \ge 1$，多个灯塔挤一个车位）
* **场景倒带**：最常见的情况。比如 $N=2$，两个灯塔（SSB#0 和 SSB#1）必须共用同一个发车位（RO）。
* **切香肠法则**：既然大家挤在同一个车位里，暗号就不能都从 0 开始编了，否则基站根本分不清你是哪个灯塔派来的。协议给出了一个精确的**“切香肠公式”：起点 = $n \cdot (N_{preamble}^{total} / N)$**。
  * **公式带入算一算**：假设这个车位里一共有 64 个暗号（$N_{preamble}^{total}=64$）。
  * 灯塔 SSB#0（$n=0$）：起点 = $0 \cdot (64 / 2) = 0$。它分走了 **第 0 号 ~ 第 31 号** 暗号。
  * 灯塔 SSB#1（$n=1$）：起点 = $1 \cdot (64 / 2) = 32$。它分走了 **第 32 号 ~ 第 63 号** 暗号。
* **物理防错**：协议最后特别严谨地补了一句：“$N_{preamble}^{total}$ 必须是 $N$ 的整数倍”。这意味着香肠必须能被平分，绝不允许出现半个暗号分不平的情况。

### 🔄 暗号起点编号分配流水线
```text
 物理层算出自己该去哪个发车位 (RO) 发暗号了
    │
    ├─ 问高层：我们这个车位，是大家挤一挤的，还是我们灯塔独占的？(看 N 值)
    │
    ├──► 【N < 1 (比如 1/2，独占两个车位)】
    │         │
    │         └─► 动作: 土豪包场！在这个车位里，属于我的暗号直接从【 0 号】开始算！
    │
    └──► 【N >= 1 (比如 N=2，两个灯塔挤一个车位)】
              │
              └─► 动作: 切香肠分摊！
                  利用公式起点： n * (总暗号数 / N)
                  我是第几个灯塔 (n)，我就去领属于我的那一截暗号池！
                  (比如 0号灯塔领 0~31，1号灯塔领 32~63)
```

## 📖 协议原文拆解 (八)：拼车模式下暗号分配的“避让公式” (Type-2 共用 RO 映射)

> **协议原文**
> For Type-2 random access procedure with common configuration of PRACH occasions with Type-1 random access procedure, if $N<1$, one SS/PBCH block index is mapped to $1/N$ consecutive valid PRACH occasions and $Q$ contention based preambles with consecutive indexes associated with the SS/PBCH block index per valid PRACH occasion start from preamble index $R$. If $N \ge 1$, $Q$ contention based preambles with consecutive indexes associated with SS/PBCH block index $n$, $0 \le n \le N-1$, per valid PRACH occasion start from preamble index $n \cdot N_{preamble}^{total}/N + R$, where $N_{preamble}^{total}$ is provided by totalNumberOfRA-Preambles for Type-1 random access procedure.

### 📚 增量术语表
* **Common configuration**：拼车模式。**一句话解释**：2 步（Type-2）和 4 步（Type-1）手机在同一个发车位里发暗号。
* **$R$**：4 步老手机分走的暗号数量。
* **$Q$**：2 步极速手机专属的暗号数量。
* **Start from preamble index $R$**：从第 $R$ 号暗号开始算起。**一句话解释**：这是拼车机制的精髓——“先来后到，绝不抢座”。

### 🚪 第一关：土豪拼车模式下的“避让起点” ($N < 1$)
* **场景倒带**：当 $N = 1/2$ 时，一个灯塔（SSB）霸占了两个连着的车位。
* **暗号分配冲突**：既然是“拼车”，这两个车位里不仅有 2 步的手机，也有 4 步的手机。我们在拆解（七）里学过，4 步老手机在这个车位里，是极其霸道地从 `0 号`开始挑暗号的（拿走 $R$ 个，也就是 0 到 $R-1$ 号）。
* **极速手机的避让**：现在，轮到走 2 步极速通道的手机（MsgA）来选暗号了。它能从 0 号开始挑吗？绝对不行，会撞车！
  * 协议给出的破局公式：**既然 0 到 $R-1$ 已经被 4 步老手机拿走了，那你走 2 步的，就乖乖从第 $R$ 号暗号开始起编（start from preamble index $R$）！** 往后数 $Q$ 个暗号，这才是你们的独立包厢。

### ✂️ 第二关：挤车厢拼车模式下的“复合偏移公式” ($N \ge 1$)
* **场景倒带**：当 $N=2$ 时，两个灯塔（SSB#0 和 SSB#1）挤在同一个发车位里，并且这车位里还混着 2 步和 4 步两拨人。情况极度混乱。
* **终极复合切割法则**：
  * **第一刀（按灯塔切分）**：先别管你是走几步的，首先必须把 64 个暗号按灯塔均分（等分香肠）。这就是公式的前半段 $n \cdot (N_{preamble}^{total} / N)$。
  * **第二刀（按步数切分）**：在分给某个灯塔的这一截香肠里，4 步手机总是抢在前面，拿走前 $R$ 块肉。所以，2 步手机必须**在灯塔起点的基础上，再往后退让 $R$ 个身位**！
  * **合体公式**：**起点 = $n \cdot (N_{preamble}^{total}/N) + R$**。
* **实战带入**：假设一个车位 64 个暗号，两个灯塔挤一挤（$N=2$）。在分给灯塔的 32 个暗号里，4 步手机分走前 20 个（$R=20$），2步手机分走后 12 个（$Q=12$）。
  * 灯塔 SSB#0（$n=0$）的 2 步手机：起点 = $0 \cdot 32 + 20 = 20$。专属暗号池：**第 20 ~ 31 号**。
  * 灯塔 SSB#1（$n=1$）的 2 步手机：起点 = $1 \cdot 32 + 20 = 52$。专属暗号池：**第 52 ~ 63 号**。

### 🔄 拼车模式下的暗号切割流水线 (Type-2 视角)
```text
 走极速 2步 (Type-2) 的物理层算出了该去的车位
    │
    ├─ 问高层：我们这车厢是“拼车”的吗？
    │
    └──► 【高层确认: 是拼车！里面有 4 步的老手机在抢资源。】
              │
              ├─ 问自己：我们的灯塔是独占车位 (N<1) 还是跟别的灯塔挤 (N>=1)？
              │
              ├──► 【灯塔独占 (N<1)】
              │         └─► 动作: 避让前 R 个暗号，直接从【第 R 号】开始挑选我们的 Q 个暗号！
              │
              └──► 【灯塔挤压 (N>=1)】
                        └─► 动作: 祭出终极复合公式！
                            起点 = (基准灯塔切分点) + (避让老手机的 R 个身位)
                            也就是： n * (总暗号/N) + R
                            从这个起点开始往后数 Q 个，这就是我们在大乱斗里的安全区！
```

### 💡 章节硬核总结

**这段协议堪称 5G 物理层资源复用的“教科书级排他算法”。当系统的时空资源（RO）达到极限压缩——不仅有多个空间波束（SSB）挤在同一个时频位置，甚至连两种截然不同的架构体系（Type-1 和 Type-2）都必须在此“拼车”时，协议利用码域空间打出了一套完美的“组合拳”。在原有的波束均分基准面 $n \cdot (N_{preamble}^{total}/N)$ 上，协议强行叠加了一个基于业务架构的排他偏移量（$+R$）。这相当于在每一个已经被切分好的波束子空间内，再次划出了一条刚性的防撞护栏，将 Type-2 的 Preamble 索引池死死地推挤到了 Type-1 占用池的后方，从而在极其混沌的混合争用环境中，开辟出了一条绝对正交的零碰撞通道。**

## 📖 协议原文拆解 (九)：极速救场与VIP通道的“专属映射” (特殊场景的 SSB-to-RO)

> **协议原文**
> For link recovery, a UE is provided $N$ SS/PBCH block indexes associated with one PRACH occasion by ssb-perRACH-Occasion in BeamFailureRecoveryConfig. For a dedicated RACH configuration provided by RACH-ConfigDedicated, if cfra is provided, a UE is provided $N$ SS/PBCH block indexes associated with one PRACH occasion by ssb-perRACH-Occasion in occasions. For the PRACH transmission on a candidate cell [38.213, clause 21], a UE is provided $N$ SS/PBCH block indexes associated with one PRACH occasion by ssb-PerRACH-Occasion-r18 in EarlyUL-SyncConfig. If $N<1$, one SS/PBCH block index is mapped to $1/N$ consecutive valid PRACH occasions. If $N \ge 1$, all consecutive $N$ SS/PBCH block indexes are associated with one PRACH occasion.

### 📚 增量术语表
* **Link recovery (Beam Failure Recovery, BFR)**：链路恢复（波束失败恢复）。**一句话解释**：手机连着连着网，突然一扭头，基站的波束照不到自己了（失联了）。这时候手机必须赶紧找到一束新光，并发起紧急呼救。
* **Dedicated RACH / CFRA**：专属随机接入 / 无竞争随机接入。**一句话解释**：基站给你提前塞了一张 VIP 专属暗号，你发暗号时绝不会跟别人撞车。
* **Candidate cell / EarlyUL-SyncConfig**：候选小区 / 早期上行同步配置。**一句话解释**：这是 R18 极其前沿的技术。手机为了切基站切得更快，在还没真正连上新基站时，就“偷偷摸摸”提前和新基站对表发暗号。

### 🚪 第一关：三大“救场/VIP”场景的专属参数
* **痛点倒带**：在普通的 4 步或 2 步接入里，手机查阅的都是全网广播的公共参数表（比如 `ssb-perRACH-OccasionAndCB-PreamblesPerSSB`）。但在上述三种极其危急或极其高级的场景下，手机还能用公共大喇叭里的参数吗？
* **各自为营的专属参数**：协议规定，在这些特殊场景下，决定“几个灯塔挤一个发车位（$N$ 值）”的参数，全部改为由高层下发的**私密/专属配置文件**来提供：
  1. **波束救命时 (BFR)**：去查阅 `BeamFailureRecoveryConfig` 里的 $N$ 值。
  2. **拿着 VIP 暗号时 (CFRA)**：去查阅 `RACH-ConfigDedicated` 里的 $N$ 值。
  3. **提前“偷跑”对表时 (Early UL Sync)**：去查阅 `EarlyUL-SyncConfig` 里的 $N$ 值。

### ✂️ 第二关：万法归宗的“切香肠法则”
* **场景倒带**：虽然获取 $N$ 值的“文件路径”各不相同，但拿到了 $N$ 值之后，底层的物理映射法则会变吗？
* **铁律不变**：完全不变！协议的最后两句话和常规接入一模一样：
  * **土豪独占 ($N < 1$)**：一个灯塔（SSB）直接霸占 $1/N$ 个连续的车位（RO）。
  * **结伴挤车厢 ($N \ge 1$)**：连续的 $N$ 个灯塔，必须老老实实挤进同一个车位（RO）里。
* **细节差异**：你会发现这里**只提了 $N$ 值，没有提 $R$ 值（每个灯塔分几个暗号）**。为什么？因为在 CFRA（无竞争接入）或专属恢复场景下，手机用的暗号是基站提前通过信令**一对一私下发给它的**（比如直接告诉它“你用第 45 号暗号”），根本不需要手机自己去公共暗号池里“切香肠盲抽”，所以自然就不需要 $R$ 值法则了！

### 🔄 特殊场景下的 SSB 到 RO 映射流水线
```text
 手机遭遇特殊情况：波束断联 / 拿到VIP票 / 提前切区
    │
    ├─ 问自己：我现在要去发暗号了，N 值 (灯塔到车位的拥挤度) 去哪查？
    │
    ├──► 【波束断联呼救】 ──► 去翻救命锦囊 (BeamFailureRecoveryConfig)
    ├──► 【VIP无竞争接入】──► 去翻专属票根 (RACH-ConfigDedicated)
    └──► 【提前切区对表】 ──► 去翻极速配置 (EarlyUL-SyncConfig)
             │
             ▼
      (查出特定的 N 值，比如 N=2)
             │
             └─► 动作: 物理法则归一！按 N=2 的规矩，把两个灯塔塞进一个车位！
                 带着基站私下给我的绝对不撞车的专属暗号，在车位里点火发射！🚀
```

### 💡 章节硬核总结

**这段协议完美展现了 3GPP 物理层设计的“高层解耦与底层复用”原则。针对波束失败恢复（BFR）、无竞争接入（CFRA）以及 R18 引入的早期上行同步（Early UL Sync）等对时延和可靠性要求极其苛刻的场景，协议在 RRC 信令面上将 $N$ 值的配置通道进行了彻底的分离（Dedicated Config），防止特殊流程受制于拥堵的公共配置。然而，一旦信令穿透到物理层（L1），其 SSB-to-RO 的映射逻辑（即 $N<1$ 的展开映射与 $N \ge 1$ 的聚合映射）又立刻收敛回归到最基础的公共法则上。同时，由于专属接入规避了冲突，协议在此处绝口不提码域切分参数（如 $R$ 或 $Q$），从侧面印证了“无竞争机制下 Preamble 索引由高层直接强指派”的设计哲学。**

## 📖 协议原文拆解 (十)：资源分配的“四重循环法则” (SSB 到 PRACH 的映射顺序)

> **协议原文**
> SS/PBCH block indexes provided by ssb-PositionsInBurst in SIB1 or in ServingCellConfigCommon or in SSB-MTC-AdditionalPCI or in LTM-SSB-Config are mapped to valid PRACH occasions in the following order where the parameters are described in [4, TS 38.211].
> - First, in increasing order of preamble indexes within a single PRACH occasion
> - Second, in increasing order of frequency resource indexes for frequency multiplexed PRACH occasions
> - Third, in increasing order of time resource indexes for time multiplexed PRACH occasions within a PRACH slot
> - Fourth, in increasing order of indexes for PRACH slots

### 📚 增量术语表
* **ssb-PositionsInBurst**：灯塔出勤表。**一句话解释**：基站广播的一张位图（Bitmap），告诉你全网 64 个可能的灯塔（SSB）中，哪几个今天是真正上班打卡（发射）的。
* **Mapped to valid PRACH occasions**：映射到合法的发车位。**一句话解释**：把这些来上班的灯塔，一个个按顺序塞进地上的发车位（RO）和暗号池（Preamble）里。
* **Multiplexed (frequency / time)**：复用。**一句话解释**：在同一个时间点有几条并行的跑道，叫频域复用；在一条跑道上按先后顺序排着几个车位，叫时域复用。

### 🚪 核心痛点：谁先谁后的“铺床法则”
* **场景倒带**：假设现在有 4 个灯塔（SSB #0, #1, #2, #3）来上班了。地上有一个巨大的停车场（PRACH 资源池），里面有时隙（Slot）、有频段（Frequency）、有时隙里的先后车位（Time within a slot）、还有每个车位里的暗号（Preamble）。
* **物理层的困境**：如果基站说“灯塔 0 分配给第一批资源，灯塔 1 分配给第二批资源”，那么**“什么叫第一批？什么叫第二批？”** 是时间在前面的叫第一批，还是频率在前面的叫第一批？如果两边理解不一致，手机对着东边喊，基站却去西边听，彻底完蛋。

### 📏 破解法则：极其严苛的 4 重 `for` 循环
为了绝对统一思想，协议给出了这套极其出名的**“码域 -> 频域 -> 时域（槽内）-> 时域（跨槽）”**的四重填坑顺序（也就是程序员眼里的 4 层嵌套 `for` 循环）：

1. **第一层（最内层，变得最快）：前导码索引 (Preamble indexes)**
   * **动作**：先把一个车位（RO）里的暗号（0 到 63）按顺序分给灯塔。
   * **比喻**：先填满同一个房间里的所有座位。
2. **第二层：频域车位 (Frequency resource indexes)**
   * **动作**：如果一个房间（车位）的暗号发完了灯塔还没分完，就去找**同一时间下、旁边那条频段上的车位**继续分。
   * **比喻**：一个房间坐满了，去隔壁房间（频分复用）接着坐，但时间不变。
3. **第三层：槽内时域车位 (Time resource indexes within a slot)**
   * **动作**：如果同一时间下所有频段的车位都分满了，就顺延到**当前时隙（Slot）内，时间靠后的下一个车位**继续分。
   * **比喻**：同一层楼的房间都坐满了，去本楼层后面的候场区（时分复用）接着坐。
4. **第四层（最外层，变得最慢）：跨槽时域车位 (PRACH slots)**
   * **动作**：如果当前时隙（Slot）里的所有车位全被榨干了，才会跨越到**下一个合法的时隙（下一个 Slot）**去继续映射剩下的灯塔。
   * **比喻**：今天彻底没位子了，明天（下一个 Slot）再来接着坐！

### 🔄 SSB 到 RO 映射的 4 层 `for` 循环代码图
从程序员的视角，基站和手机的大脑里同时运行着这样一段绝对不能出错的伪代码：

```python
for (slot = 0; slot < 总时隙数; slot++) {                     // 步骤 4：跨槽
    for (time_res = 0; time_res < 槽内车位数; time_res++) {     // 步骤 3：槽内时分
        for (freq_res = 0; freq_res < 频段数; freq_res++) {     // 步骤 2：频分
            for (preamble = 0; preamble < 64; preamble++) {     // 步骤 1：码分 (最内层循环)
                
                // 把当前的 [时隙, 槽内时间, 频段, 暗号] 
                // 按照 N 和 R 的规矩，按顺序分发给正在排队等待映射的 SSB 灯塔！
                
            }
        }
    }
}
```

### 💡 章节硬核总结

**这段 3GPP 协议是随机接入中用来消除空间歧义的“底层拓扑基石”。它定义了实际发射的同步信号块（SSB）向 PRACH 物理资源（包含时间、频率、码域）映射的四维空间张量法则。其核心思想是：“先耗尽码域（Preamble），再耗尽频域（FDM），再耗尽微观时域（TDM within slot），最后耗尽宏观时域（Slot）”。这种由微观到宏观、由极快变化到极慢变化的嵌套映射序列，确保了无论基站的 PRACH 时频资源池配置得多么复杂碎片化，全网所有的手机都能和基站保持着绝对一致的“铺床顺序”，从而实现了 SSB 索引与物理发射资源的无缝、零歧义双向解算。**

## 📖 协议原文拆解 (十一)：确保每个灯塔都有车位的“宏观周期表” (Association Period)

> **协议原文**
> An association period, starting from frame 0, for mapping SS/PBCH block indexes to PRACH occasions is the smallest integer number in the set determined by the PRACH configuration period according Table 8.1-1 such that $N_{Tx}^{SSB}$ SS/PBCH block indexes are mapped at least once to the PRACH occasions within the association period, where a UE obtains $N_{Tx}^{SSB}$ from the value of ssb-PositionsInBurst... If after an integer number of SS/PBCH block indexes to PRACH occasions mapping cycles within the association period there is a set of PRACH occasions or PRACH preambles that are not mapped to $N_{Tx}^{SSB}$ SS/PBCH block indexes, no SS/PBCH block indexes are mapped to the set of PRACH occasions or PRACH preambles. An association pattern period includes one or more association periods and is determined so that a pattern between PRACH occasions and SS/PBCH block indexes repeats at most every 160 msec. PRACH occasions not associated with SS/PBCH block indexes after an integer number of association periods, if any, are not used for PRACH transmissions.

### 📚 增量术语表
* **$N_{Tx}^{SSB}$**：实际发送的 SSB 数量。**一句话解释**：基站真正打卡上班的“灯塔”总数。手机通过广播里的 `ssb-PositionsInBurst` 查到这个数字（比如今天有 8 个灯塔上班）。
* **Association period**：映射关联周期。**一句话解释**：基站向手机保证的一段最基本的时间跨度。在这个跨度内，**所有上班的灯塔（$N_{Tx}^{SSB}$）都必须至少被分配到一个合法的发车位（RO）**，确保没人掉队。
* **Mapping cycle**：映射循环。**一句话解释**：就是我们上一段学过的“四层 `for` 循环”完整跑完一遍的时间。
* **Association pattern period**：大循环模式周期。**一句话解释**：比上面那个周期更大的一个时间跨度（包含一个或多个关联周期），它的硬性指标是“每隔最多 160 毫秒，所有的映射规律必须重播一次”，方便手机算时间。

### 🚪 第一关：逼出来的“最小整数周期”
* **场景倒带**：基站今天有 8 个灯塔上班（$N_{Tx}^{SSB}=8$）。基站每 10 毫秒（一个 PRACH 配置周期）提供 2 个发车位。
* **痛点**：跑完一次 10 毫秒的小循环（mapping cycle），只塞得下 2 个灯塔。剩下的 6 个灯塔怎么办？
* **破局之法（拉长战线）**：协议规定，必须把周期拉长！拉多长？拉到**“刚刚好能让这 8 个灯塔都塞进车位为止”**！
  * 所以在这个例子里，需要 4 个小周期（$4 \times 2 = 8$），也就是 40 毫秒。这 40 毫秒，就被称为一个**“映射关联周期 (Association period)”**。在这个周期内，8 个灯塔“至少一次（at least once）”拿到了发车位。

### ✂️ 第二关：多余的资源？直接扔掉！(无效资源的废弃)
* **痛点**：假设 8 个灯塔上班，但基站很大方，一个小周期里给了 3 个发车位。
  * 第 1 个小周期：塞了 3 个灯塔。
  * 第 2 个小周期：又塞了 3 个灯塔（共 6 个）。
  * 第 3 个小周期：还剩 2 个灯塔没塞，放进前 2 个车位里。此时，**第 3 个小周期里的第 3 个车位，空下来了！**
* **铁血废弃法则**：协议极度强迫症地规定：如果在一个“关联周期”的末尾，灯塔已经全部分配完了，但还剩下了多余的车位（或者多余的暗号），**这些多余的资源绝对不准分给下一轮的灯塔！直接全部作废（not mapped / not used）！**
* **现实意义**：为什么要这么浪费？为了**“整齐划一”**。只有把多余的尾巴砍掉，下一个关联周期才能清清爽爽地从“1 号灯塔”重新开始映射，绝不让时序关系产生错位偏移。

### 📏 第三关：160毫秒的宇宙重播法则
* **大循环兜底**：为了让手机在长期睡眠唤醒后能快速找到规律，协议在“关联周期”之上，又盖了一个宏观的“大循环模式周期（Association pattern period）”。
* **强制重播**：无论你的基础映射配置多复杂，协议强制规定：**这种映射模式（pattern），最多每隔 160 毫秒，必须像录像带一样从头重播一次！** 因为 160ms 也是很多系统级广播参数更新的公倍数周期，这让手机永远能在 160ms 内找到确定的时空坐标。

### 🔄 映射周期的推演与废弃流水线
```text
 基站声明有 8 个灯塔 (SSB) 上班 ──► 目标：必须让这 8 个灯塔都有车位！
    │
    ├─ 开始跑四层 `for` 循环，把灯塔往发车位 (RO) 里塞
    │
    ├─ 循环 1 次不够？那就循环 2 次、3 次... 
    │  直到找到一个“最小的整数周期”，刚好能把 8 个灯塔全塞进去！
    │  (这个周期就叫 Association period)
    │
    ├─ 问自己：在这个周期的末尾，灯塔分完了，发车位还剩多余的吗？
    │
    ├──► 【没有多余】 ──► 完美！直接进入下一个周期。
    │
    └──► 【有多余车位】 ──► 动作: 挥刀斩断！多余的车位直接锁死作废，
                            严禁下一轮灯塔提前抢跑！
                            (确保所有规律每隔最多 160ms 完美循环重播一次)
```

### 💡 章节硬核总结

**这段协议构建了 5G 随机接入资源映射在时间轴上的“宏观周期体系”。当单个配置周期（PRACH configuration period）提供的时频资源不足以覆盖全部发送的波束（$N_{Tx}^{SSB}$）时，协议通过定义“映射关联周期（Association period）”，从数学上确保了网络在有限的时间窗口内能够实现空间波束的全覆盖。而本段最精妙的工程设计在于“尾部资源截断机制”：通过强制废弃周期末尾未被映射的冗余 PRACH Occasions 或 Preambles，协议强行切断了映射指针的跨周期滑动。这种看似浪费资源的“对齐操作”，换来的是整个大循环周期（Pattern period，至多 160ms）内映射关系的绝对稳态与周期性，极大降低了终端解析波束时空映射表的计算复杂度。**

## 📖 协议原文拆解 (十二)：强制唤醒下的发车位锁定与时域红线 (PDCCH Order 触发的 PRACH)

> **协议原文**
> For a PRACH transmission by a UE triggered by a PDCCH order or an LTM cell switch command MAC CE, the PRACH mask index field, if the value of the random access preamble index field is not zero, indicates the PRACH occasion for the PRACH transmission where the PRACH occasions are associated with the SS/PBCH block index indicated by the SS/PBCH block index field of the PDCCH order or the LTM cell switch command MAC CE and, if any, a cell indicator field in PDCCH order [5, TS 38.212] or a Target Configuration ID field in LTM cell switch command MAC CE [11, TS 38.321] indicates a cell for the PRACH transmission. If the UE is provided $K_{cell,offset}$ by cellSpecificKoffset, the PRACH occasion is after slot $n+2^\mu \cdot K_{cell,offset}$ where $n$ is the slot of the UL BWP for the PRACH transmission that overlaps with the end of the PDCCH order reception assuming $T_{TA}=0$, and $\mu$ is the SCS configuration for the PRACH transmission. If the PDCCH reception for the PDCCH order includes two PDCCH candidates from two linked search space sets based on searchSpaceLinkingId, as described in clause 10.1, the last symbol of the PDCCH reception is the last symbol of the PDCCH candidate that ends later.

### 📚 增量术语表
* **PRACH mask index field**：发车位掩码字段。**一句话解释**：强制命令里自带的一张“限行表”。它告诉你：“虽然属于这个灯塔的车位有很多，但你只能去掩码指定的那个车位发暗号！”
* **SS/PBCH block index field**：灯塔指示字段。**一句话解释**：基站强制唤醒你时，直接帮你选好了灯塔（比如：“你现在就在 SSB#3 的方向，给我对着它发！”），省去了手机自己去空中测光的步骤。
* **Cell indicator / Target Configuration ID**：目标小区指示。**一句话解释**：命令你“跨区发暗号”时，告诉你去哪个基站的地盘发。
* **PDCCH candidate ends later**：结束较晚的控制信道候选。**一句话解释**：如果你为了增强信号，在两个不同的地方重复收到了同一条唤醒命令，倒计时必须从**较晚收完的那一条**开始算起。

### 🚪 第一关：锁定三维坐标 (基站、灯塔、车位)
* **场景倒带**：手机正睡着觉，突然收到基站发来的“强制唤醒命令（PDCCH Order / LTM MAC CE）”。
* **基站的“包办婚姻”**：这可不是普通的接入，基站在命令里把所有坐标都给你安排得明明白白！
  1. **去哪发（小区）**：提取命令里的 `Cell indicator` 或 `Target ID`，锁定你要去哪个基站发暗号（有可能是跨基站切区）。
  2. **朝哪看（灯塔）**：提取命令里的 `SS/PBCH block index field`。基站直接替你决定了你要锚定哪个空间波束（灯塔），不准你自己瞎测瞎选。
  3. **停哪个车位（RO）**：虽然属于这个灯塔的车位有一大把，但如果命令里的暗号不是 0（也就是基站给你指派了专用暗号），那你必须提取 `PRACH mask index`。这个掩码会像狙击枪准星一样，从众多车位中，极其精准地锁定**唯一合法的那一个（或几个）车位**。

### ⏱️ 第二关：发车时间的物理红线 (倒计时公式)
* **痛点**：手机收到强制唤醒命令后，能立马在下一秒就点火发暗号吗？不行！解析命令、切换频道、准备发车都需要时间。
* **物理红线公式**：`slot n + 2^μ * K_cell,offset`
  * **$n$ (基准零点)**：手机接收完强制命令（PDCCH Order）最后一刻所对应的上行时隙。
  * **$K_{cell,offset}$ (小区校准)**：又是我们熟悉的那个为了大覆盖小区准备的“全网校准偏移值”。
  * **发车铁律**：你找到的那个发车位，它的时间点**必须**在这个公式算出来的红线之后！如果你算出来的车位在红线之前，说明时间太赶了，手机芯片根本处理不过来，你只能放弃这个车位，去等下一轮的合法车位。
* **多路接收的防剥削补丁**：为了防止漏听，基站有时会在两个地方发同一条命令（Linked search space）。此时，时隙 $n$ 必须以**“最后才收完的那条命令（ends later）”**为准来计时，绝不让手机吃“时间差”的亏。

### 🔄 强制唤醒 (PDCCH Order) 点火流水线
```text
 手机 (UE) 突然收到强制唤醒信令 (PDCCH Order 或 LTM MAC CE)
    │
    ├─ 动作 1 (记录时间): 掐下秒表！记住收完信令的最后一个符号，推算出基准时隙 n。
    │
    ├─ 动作 2 (三维定位):
    │      ├─ 提取小区 ID ──► 知道去哪个地盘发。
    │      ├─ 提取 SSB 索引 ──► 知道该朝哪个波束方向发。
    │      └─ 提取 Mask 掩码 ──► 知道该停在哪个具体的车位 (RO) 发。
    │
    ├─ 动作 3 (算红线): 结合小区校准值 K_cell,offset，画出物理底线 (n + 偏移)。
    │
    ├─ 问自己：我找到的车位，在这个时间红线之后吗？
    │
    ├──► 【在红线之前】 ──► 时间太仓促，身体反应不过来！放弃这个车位，顺延等下一个！
    │
    └──► 【在红线之后】 ──► 完美！准备就绪，准时在这个车位点火发射 PRACH！🚀
```

### 💡 章节硬核总结

**这段协议展示了 5G 物理层在处理“网络强制分配接入资源（Network-Triggered PRACH）”时的绝对集权与底层保护机制。当通过 PDCCH Order 或 LTM MAC CE 触发接入时，终端彻底丧失了自主选择小区、自主测量下行波束（SSB）、以及自主挑选接入时机（RO）的权利。取而代之的，是严格解析信令中显式携带的 Target Cell、SSB Index 和 Mask Index，完成三维时空坐标的硬性锚定。与此同时，协议也划定了严苛的物理时序底线：合法发车位的时域起点，必须大于指令接收完毕的时隙叠加宏观小区偏移量（$K_{cell,offset}$）。哪怕是面临双候选控制信道的冗余接收场景，也必须以“最晚结束”的一路为基准点，在极致的调度集权中，为终端的硬件处理留下了不可剥夺的喘息空间。**

## 📖 协议原文拆解 (十三)：高层限制下的“狙击选位” (主动触发下的 Mask Index)

> **协议原文**
> For a PRACH transmission triggered by higher layers, if ssb-ResourceList is provided, the PRACH mask index is indicated by ra-ssb-OccasionMaskIndex which indicates the PRACH occasions for the PRACH transmission where the PRACH occasions are associated with the selected SS/PBCH block index.

### 📚 增量术语表
* **Triggered by higher layers**：高层触发。**一句话解释**：这是手机主动想上网（比如你要发微信），而不是被基站大喇叭（PDCCH Order）强迫叫醒的。
* **ssb-ResourceList**：SSB 资源白名单。**一句话解释**：高层（MAC/RRC）在下发发车指令时，顺带给的一张“白名单”。意思是：“虽然基站有很多灯塔，但你这次发暗号，只能从这张名单里挑灯塔！”
* **ra-ssb-OccasionMaskIndex**：随机接入专属掩码索引。**一句话解释**：这和我们上一关讲的“限行表”一模一样，但它的名字变了，是专门针对这种主动触发且带白名单情况下的“狙击枪准星”。

### 🚪 第一关：受限环境下的自主选择 (白名单里的最亮灯塔)
* **场景倒带**：手机大脑（高层）觉得该上网了，于是触发物理层去发 PRACH。但是，高层这次不是让物理层随便在空中测，而是给了它一张白名单（`ssb-ResourceList`）。
* **物理层的动作**：物理层拿着这张白名单，只去测量名单上的那几个灯塔（SSB）。测完之后，它在白名单里挑了一个最亮的（selected SS/PBCH block index）。

### 🎯 第二关：狙击准星锁定车位 (Mask Index 过滤)
* **痛点**：物理层挑好了白名单里的最亮灯塔。但根据映射表，这个灯塔名下可能分配了好几个合法的发车位（PRACH occasions）。手机能随便挑一个发吗？
* **掩码限行**：绝不可以！如果高层给了你白名单，它一定也会同步塞给你一张“限行过滤网”，这张网在这里被命名为 **`ra-ssb-OccasionMaskIndex`**。
* **一锤定音的坐标**：
  * **第一步**：找出与你挑中的那个灯塔（selected SSB）关联的所有合法车位。
  * **第二步**：把 `ra-ssb-OccasionMaskIndex` 这个掩码盖上去。掩码上挖了洞的地方，就是你最终真正能把 PRACH 发射出去的唯一合法车位！

### 🔄 高层受限触发模式下的选位流水线
```text
 高层 (MAC) 主动想上网，触发物理层发送 PRACH
    │
    ├─ 问高层：老大，有什么特殊限制吗？
    │
    ├──► 【没有限制】 ──► 物理层自由去空中测最亮的灯塔，按常规找车位。
    │
    └──► 【有限制！给了一张白名单 (ssb-ResourceList)】
              │
              ├─ 动作 1: 物理层乖乖在白名单里测光，挑出一个最亮的灯塔 (selected SSB)。
              │
              ├─ 动作 2: 去查映射表，把这个灯塔名下的所有车位 (ROs) 全列出来。
              │
              └─► 动作 3: 拿出高层给的狙击准星【ra-ssb-OccasionMaskIndex】！
                  对准刚才列出的一排车位，掩码指哪，就去哪个车位点火发射！🚀
```

### 💡 章节硬核总结

**这段协议是对随机接入触发源与资源选择逻辑的又一块重要拼图。它处理的是“终端自主触发（Triggered by higher layers）但在频域/空域受限（配置了 ssb-ResourceList 白名单）”的半自由场景。在这种场景下，终端保留了对白名单内下行波束（SSB）的自主测量和选择权，但彻底丧失了对具体发车时机（PRACH Occasion）的自主随机权。协议通过引入专用的 RRC 层掩码参数 `ra-ssb-OccasionMaskIndex`，在物理层执行了一次强制的“AND（与）”逻辑运算：将自主选定的 SSB 所映射的 RO 集合，与 Mask Index 允许的 RO 集合求交集，从而在赋予终端一定自主性的同时，依然保持了网络侧对微观接入时机的绝对管控。**
