# 8.1 Random Access Preamble (随机接入前导码的发射)

本章节聚焦于随机接入流程的最起点——**Msg1（或 Type-2 中的 MsgA Preamble）的发射**。这是终端在 5G 网络中发出的第一声呼喊。物理层将在这里接收高层的最高指令，并加载一系列极其精密的时空与能量参数，确保这声暗号能够精准地轰进基站的接收天线中。

---

## 📖 协议原文拆解 (一)：开火的触发源与“弹药库”清单

> **协议原文**
> Physical random access procedure for a UE is triggered upon request of a PRACH transmission by higher layers or by a PDCCH order or LTM Cell Switch Command MAC CE in clause 6.1.3.75 [11, TS 38.321] for a cell. A configuration by higher layers for a PRACH transmission includes the following:
> - A configuration for PRACH transmission on the cell [4, TS 38.211].
> - A preamble index, a preamble SCS, $P_{PRACH,target}$, a corresponding RA-RNTI when applicable [11, TS 38.321], and a PRACH resource for the cell.
> - A number of $N_{preamble}^{rep} > 1$ preamble repetitions for the PRACH transmission if the UE would transmit the PRACH with repetitions.

### 📚 增量术语表
* **PRACH (Physical Random Access Channel)**：物理随机接入信道。**一句话解释**：手机用来向基站发射“暗号”的专属发射通道。
* **Preamble index**：前导码索引。**一句话解释**：你要喊的具体是哪一句暗号（0 到 63 里的哪一个数字）。
* **$P_{PRACH,target}$**：PRACH 目标接收功率。**一句话解释**：基站对你下达的音量指标：“小子，不管你离我多远，你发出的声音到达我耳朵里时，必须要有这么大声！”
* **$N_{preamble}^{rep}$**：前导码重复次数。**一句话解释**：如果在极度弱网环境，暗号喊一遍基站听不见，高层就会命令你连续喊 N 遍。

### 🚪 第一关：谁扣动了扳机？(三大触发源)
物理层自己是个“哑巴”，绝对不会主动去发暗号。它的每一次发射，必然是由以下三大“扳机”之一触发的：
1. **高层主动请求 (request by higher layers)**：最常见的场景。比如你刚打开手机要发微信，手机大脑（MAC/RRC层）主动命令物理层去连网。
2. **基站大喇叭强制命令 (PDCCH order)**：你在上网时突然数据断流或者失步了，基站直接通过底层控制信道（PDCCH）拿枪指着你：“立刻给我发个 PRACH 报位置！”
3. **极速换区加急信令 (LTM MAC CE)**：高铁上移动太快，旧基站扛不住了，直接甩给你一张 MAC 层加急指令：“马上用这张 VIP 免排队暗号，去隔壁新基站报到！”

### 📦 第二关：高层下发的“弹药库清单”
一旦扳机扣动，高层会立刻给物理层的“弹药库”里装填三类核心弹药（参数）：

* **基础枪管配置 (Configuration on the cell)**
  * 高层指明了在这个基站下发射的基础波形参数（指向了 38.211 协议里的底层生成公式）。
* **核心开火参数 (The Big Five)**
  * **Preamble index**：弹药型号（你要发哪个暗号）。
  * **Preamble SCS**：车道宽度（你在多宽的频段里发）。
  * **$P_{PRACH,target}$**：开火力度（计算发射功率的基准，保证基站能听清）。
  * **RA-RNTI**：接头暗号（发完之后，用来监听基站回执 Msg2 的专用防伪码）。
  * **PRACH resource**：开火坐标（精确到哪一个时隙、哪一个频率格子去发）。
* **火力覆盖增强 (Repetitions 连发机制)**
  * 如果你在地下室，高层还会塞给你一个连发数字（$N_{preamble}^{rep} > 1$），命令你：“别只发一枪，给我对着这个坐标连开 N 枪（重复发送）！”

### 🔄 PRACH 点火触发与参数装载流水线
```text
 【三大触发源】
  ├─ 手机大脑主动想上网 (Higher layers)
  ├─ 基站强制点名 (PDCCH Order)
  └─ 高铁极速切区指令 (LTM MAC CE)
         │
         ▼ (扣动扳机)
 高层 (MAC) 收到触发，开始为物理层 (PHY) 准备弹药
         │
         ├─ 装填 1: 基础波形生成法则
         ├─ 装填 2: 暗号数字、频率宽度、发射目标功率、回执防伪码、发射坐标
         └─ 装填 3: (如果在弱网) 连发开火次数
         │
         ▼
 物理层 (PHY) 接收全部参数，“弹药”上膛，进入点火倒计时！
```

## 📖 协议原文拆解 (二)：真正的“开火”与波束方向锁定 (PRACH 发射动作)

> **协议原文**
> A UE transmits a PRACH on a cell using the selected PRACH format with transmission power $P_{PRACH,b,f,c}(i)$, as described in clause 7.4, on the indicated PRACH resource or on a determined set of $N_{preamble}^{rep}$ resources using a same spatial filter in case of $N_{preamble}^{rep}$ preamble repetitions.

### 📚 增量术语表
* **$P_{PRACH,b,f,c}(i)$**：PRACH 实际发射功率。**一句话解释**：手机真正开火轰出暗号时的音量大小（它是在高层给的“目标音量”基础上，结合当前自己估算出的空中路径损耗，最终推算出来的绝对发射功率）。公式下角标代表了具体的 BWP、载波和物理小区。
* **$N_{preamble}^{rep}$**：前导码重复次数。**一句话解释**：为了增强信号覆盖，连续喊暗号的遍数。
* **Spatial filter**：空间滤波器（发送波束）。**一句话解释**：手机发射信号时的“天线朝向/喇叭指向”。

### 🚪 核心机制：点火发射与“不许转头”的连发死令
* **场景倒带**：在前面的第 (三) 关中，物理层已经从高层那里拿到了所有发射参数，包括在哪发、发什么、如果信号差要连发几次。现在，时辰已到！
* **执行单发（常规开火）**：
  * 物理层在指定的时间和频率格子上（indicated PRACH resource），按照指定的长短波形（PRACH format），将计算好的能量（$P_{PRACH,b,f,c}(i)$）注入天线，轰出这一声暗号。
* **执行连发（覆盖增强的绝对红线）**：
  * 如果手机身处地下室等弱覆盖区域，触发了“重复发送”的被动技能（比如要连发 4 遍）。协议在这里下达了一道极其严厉的死命令：**在这 $N_{preamble}^{rep}$ 次连续发射的过程中，手机必须使用“完全相同的空间滤波器（same spatial filter）”！**
  * **现实意义**：什么意思？就是“**不许边转头边喊**”！基站之所以让你连发 4 遍，是为了在它那头把这 4 次微弱的信号收集起来，像叠罗汉一样叠加在一起（能量合并），才能勉强听清。如果你第一遍冲着东边喊，第二遍冲着西边喊（中途改变了波束方向），基站收到的信号相位就会彻底乱套，根本没法叠加。所以，连发期间，你的天线姿势必须像石头一样死死锁住。

### 🔄 PRACH 点火与波束锁定流水线
```text
 物理层 (Layer 1) 等到了预定的发射时间 (PRACH resource)
    │
    ├─ 动作 1: 将暗号调制成指定的波形 (PRACH format)
    ├─ 动作 2: 注入计算好的发射能量 (P_PRACH)
    │
    ├─ 问自己：我这是一次性发射，还是连续重复发射 (Repetitions)？
    │
    ├──► 【单次发射】 
    │         └─► 动作: 对准刚才锁定的灯塔方向，开火！(1次)
    │
    └──► 【连续重复发射 (比如 N = 4 次)】
              │
              └─► 动作: 【锁定空间滤波器 (Spatial Filter)】天线姿势彻底冻结！
                        ├── 第 1 次开火 🚀
                        ├── 第 2 次开火 🚀 (方向保持绝对一致！)
                        ├── 第 3 次开火 🚀 (方向保持绝对一致！)
                        └── 第 4 次开火 🚀 (方向保持绝对一致！)
```

## 📖 协议原文拆解 (三)：灯塔与暗号的“空间结对” (SSB 到 PRACH 的映射法则)

> **协议原文**
> For Type-1 random access procedure, a UE is provided a number $N$ of SS/PBCH block indexes associated with one PRACH occasion and a number $R$ of contention based preambles per SS/PBCH block index per valid PRACH occasion by ssb-perRACH-OccasionAndCB-PreamblesPerSSB.

### 📚 增量术语表
* **PRACH occasion (RO)**：物理随机接入时机。**一句话解释**：基站划定的具体“发车位”。它在时域和频域上是一个确定的方格，在这个格子里发出来的信号才算合法。
* **Contention based preambles (CB preambles)**：基于竞争的前导码。**一句话解释**：手机可以随便盲抽的“公共暗号”。因为是盲抽，所以容易和别人撞车（竞争）。
* **ssb-perRACH-OccasionAndCB-PreamblesPerSSB**：SSB到RO及前导码的映射参数。**一句话解释**：这是系统广播（SIB1）里的一条“超级绑定规则”，用来规定基站的灯塔（SSB）和手机发暗号的车位（RO）到底是怎么结对子的。
* **$N$ (Number of SSB per RO)**：N 值。**一句话解释**：几个灯塔（SSB）共用一个发车位（RO）。
* **$R$ (Number of CB preambles per SSB)**：R 值。**一句话解释**：在这个发车位里，每个灯塔能分到多少个公共暗号（Preamble）。

### 🚪 核心机制：波束映射的“结对子”艺术
* **场景倒带**：在 5G 毫米波里，基站可能朝四面八方发射了 8 个甚至 64 个极窄的灯塔波束（SSB）。手机在地上测了一圈，挑了一个最亮的（比如 SSB#2）准备发暗号。
* **基站的痛点**：手机是在天上发暗号的，基站怎么知道手机挑了哪个灯塔（也就是手机现在处于哪个波束的覆盖范围内）？如果基站不知道，它回信（Msg2）的时候就不知道该往哪个方向打波束。
* **协议的破局之法（隐式绑定）**：协议非常聪明，它不让手机显式地报告“我选了 2 号灯塔”，而是让基站提前定下规矩（通过参数 `ssb-perRACH-OccasionAndCB-PreamblesPerSSB` 规定 $N$ 和 $R$）：
  * **如果 $N = 1$**：代表 1 个灯塔独占 1 个发车位（RO）。那么手机只要在 2 号发车位里发暗号，基站一听，秒懂：“这小子绝对在 2 号灯塔下！”
  * **如果 $N = 2$**：代表 2 个灯塔（比如 SSB#0 和 SSB#1）挤在同一个发车位里。这怎么区分？这就靠 $R$ 值了！基站规定每个车位里有 64 个暗号，那么 SSB#0 分走前 32 个，SSB#1 分走后 32 个（$R=32$）。手机如果选了 SSB#1，它就必须从后 32 个暗号里抽。基站在这个车位里一听到后半段的暗号，再次秒懂：“原来是在 SSB#1 的方向！”

### 🔄 SSB 与 PRACH 资源的映射流水线
```text
 基站 (gNB) 广播映射规则：参数 ssb-perRACH-OccasionAndCB-PreamblesPerSSB
    │
    ├─ 规定了 N (几个灯塔挤一个车位)
    ├─ 规定了 R (每个灯塔分几个暗号)
    │
    ▼
 手机 (UE) 接收规则，开始找车位发暗号
    │
    ├─ 动作 1: 测出最亮的灯塔 (比如认定 SSB#3 最好)
    │
    ├─ 动作 2: 去查基站给的映射表，找出 SSB#3 对应的发车位 (RO) 在哪里
    │
    ├─ 动作 3: 在这个车位里，抽出只属于 SSB#3 的专属暗号池 (R 个暗号)
    │
    ├─ 动作 4: 闭上眼睛，从这 R 个暗号里随机盲抽 1 个
    │
    └─► 动作 5: 在对应的时间、对应的频率，把抽到的暗号发射出去！🚀
              │
              ▼
    (基站听到这个特定的暗号，立刻知道了手机对应的空间方位！)
```

## 📖 协议原文拆解 (四)：2 步与 4 步接入的“拼车与分家” (共享 RO 映射法则)

> **协议原文**
> For Type-2 random access procedure with common configuration of PRACH occasions with Type-1 random access procedure, a UE is provided a number $N$ of SS/PBCH block indexes associated with one PRACH occasion by ssb-perRACH-OccasionAndCB-PreamblesPerSSB and a number $Q$ of contention based preambles per SS/PBCH block index per valid PRACH occasion by msgA-CB-PreamblesPerSSB-PerSharedRO. The PRACH transmission can be on a subset of PRACH occasions associated with a same SS/PBCH block index within an SSB-RO mapping cycle for a UE provided with a PRACH mask index by msgA-SSB-SharedRO-MaskIndex according to [11, TS 38.321].

### 📚 增量术语表
* **Common configuration of PRACH occasions**：RO 发车位共用。**一句话解释**：2步极速接入（Type-2）和 4步传统接入（Type-1）的手机，在同一个时间、同一个频率格子（RO）里发暗号。俗称“拼车”。
* **$Q$ (msgA-CB-PreamblesPerSSB-PerSharedRO)**：Q 值。**一句话解释**：在拼车的情况下，每个灯塔（SSB）专门划拨给 2步接入（MsgA）使用的专属暗号数量。
* **PRACH mask index (msgA-SSB-SharedRO-MaskIndex)**：掩码索引。**一句话解释**：基站下发的一张“限行/过滤表”。即使你算出来有几个车位可以发车，也必须用这个过滤网筛一下，只能在基站指定的那几个子车位里发。

### 🚪 第一关：物理层的“同车厢，不同座” (码域隔离)
* **场景痛点**：在网络里，同时存在走 4 步的老手机和走 2 步的 VIP 新手机。如果为了它们分别建两条 PRACH 物理车道，太浪费极其昂贵的频谱资源了。
* **拼车机制**：所以协议允许它们“拼车（Common configuration）”，都在同一个时频格子里发暗号。$N$（几个灯塔挤一个车位）的规则完全照抄上一段（Type-1）。
* **防止撞车（Q值分家）**：大家在同一个格子里发，基站怎么知道谁是走 4 步、谁是走 2 步的？
  * 协议采用了极其聪明的“暗号池切分”。在这个格子里，基站规定：前 $R$ 个暗号只准走 4 步的人用，后 $Q$ 个暗号只准走 2 步的人（MsgA）用！
  * 这样，虽然大家在同一个时间、频率点火，但基站只要听出暗号的波形，就能瞬间判定：“嗯，这是个 MsgA 暗号，我要准备收它的数据包并回 MsgB 盲盒了！”

### 🛡️ 第二关：VIP 通道的限行令 (Mask Index)
* **痛点**：有时候一个灯塔（SSB）对应了好几个发车位（RO）。走 2 步的人（MsgA）通常包袱很重（后面跟着庞大的数据 PUSCH）。如果他们随意在这些车位里乱发，容易引起大规模的连环撞车。
* **限行过滤**：协议引入了 `msgA-SSB-SharedRO-MaskIndex`（掩码）。高层会通过这个掩码告诉物理层：“虽然从映射表上看，这 4 个车位你都能用，但是为了分流，我命令你只能在其中的第 1 个和第 3 个车位（Subset）里发射 MsgA！”

### 🔄 Type-2 共享车道资源选择流水线
```text
 走极速 2步 (Type-2) 的手机准备发 MsgA 暗号
    │
    ├─ 问高层：我们自己有专用的独立车道吗？
    │
    └──► 【高层回答: 没有，为了省资源，你要和走 4 步的人“拼车”！】
              │
              ├─ 动作 1 (找车位): 照抄 4 步的 N 值法则，算出属于自己 SSB 的那一批车位。
              │
              ├─ 动作 2 (掩码限行): 拿着 [MaskIndex]，把不让发的车位过滤掉，锁定最终车位。
              │
              ├─ 动作 3 (暗号分家): 坚决不去碰 4步的暗号池！去查属于 2步的 Q 值参数，
              │                    在专门划拨给 MsgA 的暗号池里，盲抽一个暗号！
              │
              ▼
    在限行后的合法车位里，轰出这发专属的 MsgA 暗号！🚀
```

## 📖 协议原文拆解 (五)：2步极速通道的“专属车道” (独立 RO 映射法则)

> **协议原文**
> For Type-2 random access procedure with separate configuration of PRACH occasions with Type-1 random access procedure, a UE is provided a number $N$ of SS/PBCH block indexes associated with one PRACH occasion and a number $R$ of contention based preambles per SS/PBCH block index per valid PRACH occasion by msgA-SSB-PerRACH-OccasionAndCB-PreamblesPerSSB when provided; otherwise, by ssb-perRACH-OccasionAndCB-PreamblesPerSSB.

### 📚 增量术语表
* **Separate configuration of PRACH occasions**：RO 发车位独立配置。**一句话解释**：2步极速接入（Type-2）不再和 4步老手机“拼车”了。基站专门为 Type-2 开辟了一条完全独立的物理发车通道。
* **msgA-SSB-PerRACH-OccasionAndCB-PreamblesPerSSB**：MsgA 的专属映射参数。**一句话解释**：专门针对 2步接入（MsgA）下发的灯塔到车位的映射表，定义了专属的 $N$（几个灯塔挤一个车位）和 $R$（每个灯塔分几个暗号）。

### 🚪 第一关：VIP 独立车道的映射法则
* **场景倒带**：在拆解 (四) 里，因为是“拼车”，Type-2 只能憋屈地和 Type-1 共用 N 值的算法，然后再用 Q 值把暗号劈开。
* **如今的阔气**：现在的网络配置变了，基站为 Type-2 单独配置了 PRACH 资源（Separate configuration）。这意味着走 2 步流程的手机，拥有了完全独立的时频格子（RO）。
* **独立映射（$N$ 和 $R$）**：既然车道都是独立的了，那么“几个灯塔（SSB）挤一个车位（RO）”以及“每个灯塔分几个暗号”的规矩，当然也可以独立制定了。基站会下发一个全新的专属参数 `msgA-SSB-PerRACH-OccasionAndCB-PreamblesPerSSB`，让手机去查阅专属于 Type-2 的 $N$ 值和 $R$ 值。

### 🛡️ 第二关：协议的严谨“兜底”法则
* **痛点**：万一基站说好了给你“独立车道”，但是在发广播信令时，忘了把这本专属的“参数映射表（msgA-SSB...）”发下来怎么办？手机总不能在这个独立车道上瞎停吧？
* **抄作业（Otherwise...）**：协议在最后半句给出了极具画面感的兜底法则：“`otherwise, by ssb-perRACH-OccasionAndCB-PreamblesPerSSB`”。意思是：**如果基站没给你发专属的表，那你就直接去抄隔壁 4步接入（Type-1）的那张老表！** 就算是在独立车道上，也按 4步接入的规矩（N和R）来找发车位。

### 🔄 Type-2 专属独立车道的选位流水线
```text
 走极速 2步 (Type-2) 的手机准备发 MsgA 暗号
    │
    ├─ 问高层：我们自己有专用的独立车道吗？
    │
    └──► 【高层回答: 有！我们不跟 4 步拼车了，走独立配置 (Separate configuration)！】
              │
              ├─ 问自己：基站给我发专属于我的映射表 (msgA-SSB-...) 了吗？
              │
              ├──► 【收到了专属表】 
              │         └─► 动作: 用专属表里的 N 和 R，去独立的 RO 车道上找车位发暗号！
              │
              └──► 【没收到专属表 (基站漏配了)】
                        └─► 动作 (兜底): 直接照抄隔壁 4步接入的映射表 (ssb-perRACH-...)，
                                        拿着老规矩，去新车道上找车位发暗号！
```

## 📖 协议原文拆解 (六)：特殊业务的“VIP 专属暗号池” (Feature Combination 映射法则)

> **协议原文**
> For a random access procedure associated with a feature combination indicated by FeatureCombinationPreambles, a UE is provided a number $N$ of SS/PBCH block indexes associated with one PRACH occasion by ssb-perRACH-OccasionAndCB-PreamblesPerSSB or msgA-SSB-PerRACH-OccasionAndCB-PreamblesPerSSB when provided and a number $S$ of contention based preambles per SS/PBCH block index per valid PRACH occasion by startPreambleForThisPartition and numberOfPreamblesPerSSB-ForThisPartition. The PRACH transmission can be on a subset of PRACH occasions associated with a same SS/PBCH block index within an SSB-RO mapping cycle for a UE provided with a PRACH mask index by ssb-SharedRO-MaskIndex according to [11, TS 38.321].

### 📚 增量术语表
* **Feature combination**：特性组合。**一句话解释**：手机拥有某种“特殊身份”或“特殊业务需求”，比如它是省电的 RedCap（轻量化 5G）终端，或者是在做切片业务（Slicing），或者是小包数据传输（SDT）。
* **FeatureCombinationPreambles**：特性组合前导码。**一句话解释**：基站专门为这批有“特殊身份”的手机，单独圈出来的一批暗号。
* **$S$ (numberOfPreamblesPerSSB-ForThisPartition)**：S 值。**一句话解释**：在这个被切分出来的“VIP 包厢（Partition）”里，每个灯塔（SSB）能分到多少个专属暗号。
* **startPreambleForThisPartition**：VIP 包厢的起点。**一句话解释**：在这个发车位的所有暗号里，从第几个暗号开始算作是你们这个特殊业务的 VIP 专属暗号。
* **ssb-SharedRO-MaskIndex**：共享发车位掩码。**一句话解释**：即使是特殊业务，也不能随便占用所有车位。基站通过这个掩码过滤网，限制你只能在指定的那几个车位里发车。

### 🚪 第一关：物理层的“大包厢切分术” (Partition)
* **场景倒带**：在之前的基础映射中，我们知道一个发车位（RO）里的 64 个暗号，可能被普通手机（Type-1）和极速手机（Type-2）瓜分。
* **新贵入场**：现在，随着 5G 的发展，各种五花八门的新手机进网了。比如来了一批极其弱小的 RedCap（手表/手环）设备。如果让它们和普通大手机去抢同一个暗号池，它们永远抢不赢。
* **隔离策略**：基站决定给它们搞个“VIP 隔断（Partition）”。$N$ 值（几个灯塔挤一个车位）还是照旧（无论是复用老规矩还是用专属规矩）。但暗号的分配彻底变了！
  * 基站下发指令：“RedCap 手机听令，你们的暗号池起点是第 40 号（`startPreambleForThisPartition`），每个灯塔给你们分了 10 个暗号（$S$ 值）！”
  * 于是，拥有特殊身份的手机，就能完美避开普通手机的暗号池，去抽取专属于自己的那 $S$ 个暗号。

### 🛡️ 第二关：特殊业务的限行令 (Shared RO Mask Index)
* **痛点**：因为这种特殊业务往往是和普通业务“共享车道（Shared RO）”的，如果任由这些特殊手机在所有合法的车位里狂轰滥炸，也会影响交通。
* **掩码限行**：协议故技重施，再次引入了掩码 `ssb-SharedRO-MaskIndex`。高层会告诉物理层：“你虽然属于特殊业务群，但你只能在我给你的掩码允许的子集（subset）车位里发射！”

### 🔄 特殊业务的资源选择流水线
```text
 特殊身份手机 (如 RedCap) 准备发射 PRACH 暗号
    │
    ├─ 问自己：我有什么“特殊特性 (Feature combination)”吗？
    │
    └──► 【高层确认: 有！我们需要使用隔离包厢 (Partition) 的参数】
              │
              ├─ 动作 1 (找车位): 用 N 值算出属于自己 SSB 的那一批车位。
              │
              ├─ 动作 2 (限行过滤): 拿着 [SharedRO-MaskIndex]，过滤掉不让发的车位。
              │
              ├─ 动作 3 (定位包厢): 找到专属起跑线 (startPreamble)
              │
              ├─ 动作 4 (盲抽暗号): 在专属的包厢容量 (S 个暗号) 内，抽取一个暗号！
              │
              ▼
    在限行后的合法车位里，轰出带有“特殊身份”标志的专属暗号！🚀
```

## 📖 协议原文拆解 (七)：发车位里暗号分配的数学公式 (Preamble Index 映射)

> **协议原文**
> For Type-1 random access procedure, or for Type-2 random access procedure with separate configuration of PRACH occasions from Type 1 random access procedure, if $N<1$, one SS/PBCH block index is mapped to $1/N$ consecutive valid PRACH occasions and $R$ contention based preambles with consecutive indexes associated with the SS/PBCH block index per valid PRACH occasion start from preamble index 0. If $N \ge 1$, $R$ contention based preambles with consecutive indexes associated with SS/PBCH block index $n$, $0 \le n \le N-1$, per valid PRACH occasion start from preamble index $n \cdot N_{preamble}^{total}/N$ where $N_{preamble}^{total}$ is provided by totalNumberOfRA-Preambles... and is an integer multiple of $N$.

### 📚 增量术语表
* **$N$**：几个灯塔（SSB）挤一个发车位（RO）。如果 $N<1$（比如 $N=1/2$），说明灯塔太少车位太多，一个灯塔能分到 2 个完整的发车位。如果 $N \ge 1$（比如 $N=2$），说明两个灯塔得挤在同一个发车位里。
* **$R$**：每个灯塔分到的暗号数量。
* **$N_{preamble}^{total}$**：这个发车位里总共提供的暗号数量（最多64个）。
* **Preamble index 0**：暗号的编号从 0 开始。

### 🚪 第一关：土豪模式（$N < 1$，一个灯塔霸占多个车位）
* **场景倒带**：当 $N = 1/2$ 时，意味着基站的灯塔（波束）很少，但是配置的 PRACH 发车位却很多。
* **分配法则**：这就像一个土豪（灯塔）直接包下了 2 个连着的停车场（$1/N$ 个 consecutive ROs）。
* **暗号起编规则**：既然整个停车场都是你的，那你在这个停车场里分配暗号时，当然是极其霸道地**直接从 0 号开始编（start from preamble index 0）**！手机只要在这个停车场里抽，就算抽到了 0 号暗号，基站也知道这绝对是属于这个土豪灯塔的。

### ✂️ 第二关：挤车厢模式（$N \ge 1$，多个灯塔挤一个车位）
* **场景倒带**：最常见的情况。比如 $N=2$，两个灯塔（SSB#0 和 SSB#1）必须共用同一个发车位（RO）。
* **切香肠法则**：既然大家挤在同一个车位里，暗号就不能都从 0 开始编了，否则基站根本分不清你是哪个灯塔派来的。协议给出了一个精确的**“切香肠公式”：起点 = $n \cdot (N_{preamble}^{total} / N)$**。
  * **公式带入算一算**：假设这个车位里一共有 64 个暗号（$N_{preamble}^{total}=64$）。
  * 灯塔 SSB#0（$n=0$）：起点 = $0 \cdot (64 / 2) = 0$。它分走了 **第 0 号 ~ 第 31 号** 暗号。
  * 灯塔 SSB#1（$n=1$）：起点 = $1 \cdot (64 / 2) = 32$。它分走了 **第 32 号 ~ 第 63 号** 暗号。
* **物理防错**：协议最后特别严谨地补了一句：“$N_{preamble}^{total}$ 必须是 $N$ 的整数倍”。这意味着香肠必须能被平分，绝不允许出现半个暗号分不平的情况。

### 🔄 暗号起点编号分配流水线
```text
 物理层算出自己该去哪个发车位 (RO) 发暗号了
    │
    ├─ 问高层：我们这个车位，是大家挤一挤的，还是我们灯塔独占的？(看 N 值)
    │
    ├──► 【N < 1 (比如 1/2，独占两个车位)】
    │         │
    │         └─► 动作: 土豪包场！在这个车位里，属于我的暗号直接从【 0 号】开始算！
    │
    └──► 【N >= 1 (比如 N=2，两个灯塔挤一个车位)】
              │
              └─► 动作: 切香肠分摊！
                  利用公式起点： n * (总暗号数 / N)
                  我是第几个灯塔 (n)，我就去领属于我的那一截暗号池！
                  (比如 0号灯塔领 0~31，1号灯塔领 32~63)
```

## 📖 协议原文拆解 (八)：拼车模式下暗号分配的“避让公式” (Type-2 共用 RO 映射)

> **协议原文**
> For Type-2 random access procedure with common configuration of PRACH occasions with Type-1 random access procedure, if $N<1$, one SS/PBCH block index is mapped to $1/N$ consecutive valid PRACH occasions and $Q$ contention based preambles with consecutive indexes associated with the SS/PBCH block index per valid PRACH occasion start from preamble index $R$. If $N \ge 1$, $Q$ contention based preambles with consecutive indexes associated with SS/PBCH block index $n$, $0 \le n \le N-1$, per valid PRACH occasion start from preamble index $n \cdot N_{preamble}^{total}/N + R$, where $N_{preamble}^{total}$ is provided by totalNumberOfRA-Preambles for Type-1 random access procedure.

### 📚 增量术语表
* **Common configuration**：拼车模式。**一句话解释**：2 步（Type-2）和 4 步（Type-1）手机在同一个发车位里发暗号。
* **$R$**：4 步老手机分走的暗号数量。
* **$Q$**：2 步极速手机专属的暗号数量。
* **Start from preamble index $R$**：从第 $R$ 号暗号开始算起。**一句话解释**：这是拼车机制的精髓——“先来后到，绝不抢座”。

### 🚪 第一关：土豪拼车模式下的“避让起点” ($N < 1$)
* **场景倒带**：当 $N = 1/2$ 时，一个灯塔（SSB）霸占了两个连着的车位。
* **暗号分配冲突**：既然是“拼车”，这两个车位里不仅有 2 步的手机，也有 4 步的手机。我们在拆解（七）里学过，4 步老手机在这个车位里，是极其霸道地从 `0 号`开始挑暗号的（拿走 $R$ 个，也就是 0 到 $R-1$ 号）。
* **极速手机的避让**：现在，轮到走 2 步极速通道的手机（MsgA）来选暗号了。它能从 0 号开始挑吗？绝对不行，会撞车！
  * 协议给出的破局公式：**既然 0 到 $R-1$ 已经被 4 步老手机拿走了，那你走 2 步的，就乖乖从第 $R$ 号暗号开始起编（start from preamble index $R$）！** 往后数 $Q$ 个暗号，这才是你们的独立包厢。

### ✂️ 第二关：挤车厢拼车模式下的“复合偏移公式” ($N \ge 1$)
* **场景倒带**：当 $N=2$ 时，两个灯塔（SSB#0 和 SSB#1）挤在同一个发车位里，并且这车位里还混着 2 步和 4 步两拨人。情况极度混乱。
* **终极复合切割法则**：
  * **第一刀（按灯塔切分）**：先别管你是走几步的，首先必须把 64 个暗号按灯塔均分（等分香肠）。这就是公式的前半段 $n \cdot (N_{preamble}^{total} / N)$。
  * **第二刀（按步数切分）**：在分给某个灯塔的这一截香肠里，4 步手机总是抢在前面，拿走前 $R$ 块肉。所以，2 步手机必须**在灯塔起点的基础上，再往后退让 $R$ 个身位**！
  * **合体公式**：**起点 = $n \cdot (N_{preamble}^{total}/N) + R$**。
* **实战带入**：假设一个车位 64 个暗号，两个灯塔挤一挤（$N=2$）。在分给灯塔的 32 个暗号里，4 步手机分走前 20 个（$R=20$），2步手机分走后 12 个（$Q=12$）。
  * 灯塔 SSB#0（$n=0$）的 2 步手机：起点 = $0 \cdot 32 + 20 = 20$。专属暗号池：**第 20 ~ 31 号**。
  * 灯塔 SSB#1（$n=1$）的 2 步手机：起点 = $1 \cdot 32 + 20 = 52$。专属暗号池：**第 52 ~ 63 号**。

### 🔄 拼车模式下的暗号切割流水线 (Type-2 视角)
```text
 走极速 2步 (Type-2) 的物理层算出了该去的车位
    │
    ├─ 问高层：我们这车厢是“拼车”的吗？
    │
    └──► 【高层确认: 是拼车！里面有 4 步的老手机在抢资源。】
              │
              ├─ 问自己：我们的灯塔是独占车位 (N<1) 还是跟别的灯塔挤 (N>=1)？
              │
              ├──► 【灯塔独占 (N<1)】
              │         └─► 动作: 避让前 R 个暗号，直接从【第 R 号】开始挑选我们的 Q 个暗号！
              │
              └──► 【灯塔挤压 (N>=1)】
                        └─► 动作: 祭出终极复合公式！
                            起点 = (基准灯塔切分点) + (避让老手机的 R 个身位)
                            也就是： n * (总暗号/N) + R
                            从这个起点开始往后数 Q 个，这就是我们在大乱斗里的安全区！
```

### 💡 章节硬核总结

**这段协议堪称 5G 物理层资源复用的“教科书级排他算法”。当系统的时空资源（RO）达到极限压缩——不仅有多个空间波束（SSB）挤在同一个时频位置，甚至连两种截然不同的架构体系（Type-1 和 Type-2）都必须在此“拼车”时，协议利用码域空间打出了一套完美的“组合拳”。在原有的波束均分基准面 $n \cdot (N_{preamble}^{total}/N)$ 上，协议强行叠加了一个基于业务架构的排他偏移量（$+R$）。这相当于在每一个已经被切分好的波束子空间内，再次划出了一条刚性的防撞护栏，将 Type-2 的 Preamble 索引池死死地推挤到了 Type-1 占用池的后方，从而在极其混沌的混合争用环境中，开辟出了一条绝对正交的零碰撞通道。**

## 📖 协议原文拆解 (九)：极速救场与VIP通道的“专属映射” (特殊场景的 SSB-to-RO)

> **协议原文**
> For link recovery, a UE is provided $N$ SS/PBCH block indexes associated with one PRACH occasion by ssb-perRACH-Occasion in BeamFailureRecoveryConfig. For a dedicated RACH configuration provided by RACH-ConfigDedicated, if cfra is provided, a UE is provided $N$ SS/PBCH block indexes associated with one PRACH occasion by ssb-perRACH-Occasion in occasions. For the PRACH transmission on a candidate cell [38.213, clause 21], a UE is provided $N$ SS/PBCH block indexes associated with one PRACH occasion by ssb-PerRACH-Occasion-r18 in EarlyUL-SyncConfig. If $N<1$, one SS/PBCH block index is mapped to $1/N$ consecutive valid PRACH occasions. If $N \ge 1$, all consecutive $N$ SS/PBCH block indexes are associated with one PRACH occasion.

### 📚 增量术语表
* **Link recovery (Beam Failure Recovery, BFR)**：链路恢复（波束失败恢复）。**一句话解释**：手机连着连着网，突然一扭头，基站的波束照不到自己了（失联了）。这时候手机必须赶紧找到一束新光，并发起紧急呼救。
* **Dedicated RACH / CFRA**：专属随机接入 / 无竞争随机接入。**一句话解释**：基站给你提前塞了一张 VIP 专属暗号，你发暗号时绝不会跟别人撞车。
* **Candidate cell / EarlyUL-SyncConfig**：候选小区 / 早期上行同步配置。**一句话解释**：这是 R18 极其前沿的技术。手机为了切基站切得更快，在还没真正连上新基站时，就“偷偷摸摸”提前和新基站对表发暗号。

### 🚪 第一关：三大“救场/VIP”场景的专属参数
* **痛点倒带**：在普通的 4 步或 2 步接入里，手机查阅的都是全网广播的公共参数表（比如 `ssb-perRACH-OccasionAndCB-PreamblesPerSSB`）。但在上述三种极其危急或极其高级的场景下，手机还能用公共大喇叭里的参数吗？
* **各自为营的专属参数**：协议规定，在这些特殊场景下，决定“几个灯塔挤一个发车位（$N$ 值）”的参数，全部改为由高层下发的**私密/专属配置文件**来提供：
  1. **波束救命时 (BFR)**：去查阅 `BeamFailureRecoveryConfig` 里的 $N$ 值。
  2. **拿着 VIP 暗号时 (CFRA)**：去查阅 `RACH-ConfigDedicated` 里的 $N$ 值。
  3. **提前“偷跑”对表时 (Early UL Sync)**：去查阅 `EarlyUL-SyncConfig` 里的 $N$ 值。

### ✂️ 第二关：万法归宗的“切香肠法则”
* **场景倒带**：虽然获取 $N$ 值的“文件路径”各不相同，但拿到了 $N$ 值之后，底层的物理映射法则会变吗？
* **铁律不变**：完全不变！协议的最后两句话和常规接入一模一样：
  * **土豪独占 ($N < 1$)**：一个灯塔（SSB）直接霸占 $1/N$ 个连续的车位（RO）。
  * **结伴挤车厢 ($N \ge 1$)**：连续的 $N$ 个灯塔，必须老老实实挤进同一个车位（RO）里。
* **细节差异**：你会发现这里**只提了 $N$ 值，没有提 $R$ 值（每个灯塔分几个暗号）**。为什么？因为在 CFRA（无竞争接入）或专属恢复场景下，手机用的暗号是基站提前通过信令**一对一私下发给它的**（比如直接告诉它“你用第 45 号暗号”），根本不需要手机自己去公共暗号池里“切香肠盲抽”，所以自然就不需要 $R$ 值法则了！

### 🔄 特殊场景下的 SSB 到 RO 映射流水线
```text
 手机遭遇特殊情况：波束断联 / 拿到VIP票 / 提前切区
    │
    ├─ 问自己：我现在要去发暗号了，N 值 (灯塔到车位的拥挤度) 去哪查？
    │
    ├──► 【波束断联呼救】 ──► 去翻救命锦囊 (BeamFailureRecoveryConfig)
    ├──► 【VIP无竞争接入】──► 去翻专属票根 (RACH-ConfigDedicated)
    └──► 【提前切区对表】 ──► 去翻极速配置 (EarlyUL-SyncConfig)
             │
             ▼
      (查出特定的 N 值，比如 N=2)
             │
             └─► 动作: 物理法则归一！按 N=2 的规矩，把两个灯塔塞进一个车位！
                 带着基站私下给我的绝对不撞车的专属暗号，在车位里点火发射！🚀
```

### 💡 章节硬核总结

**这段协议完美展现了 3GPP 物理层设计的“高层解耦与底层复用”原则。针对波束失败恢复（BFR）、无竞争接入（CFRA）以及 R18 引入的早期上行同步（Early UL Sync）等对时延和可靠性要求极其苛刻的场景，协议在 RRC 信令面上将 $N$ 值的配置通道进行了彻底的分离（Dedicated Config），防止特殊流程受制于拥堵的公共配置。然而，一旦信令穿透到物理层（L1），其 SSB-to-RO 的映射逻辑（即 $N<1$ 的展开映射与 $N \ge 1$ 的聚合映射）又立刻收敛回归到最基础的公共法则上。同时，由于专属接入规避了冲突，协议在此处绝口不提码域切分参数（如 $R$ 或 $Q$），从侧面印证了“无竞争机制下 Preamble 索引由高层直接强指派”的设计哲学。**

## 📖 协议原文拆解 (十)：资源分配的“四重循环法则” (SSB 到 PRACH 的映射顺序)

> **协议原文**
> SS/PBCH block indexes provided by ssb-PositionsInBurst in SIB1 or in ServingCellConfigCommon or in SSB-MTC-AdditionalPCI or in LTM-SSB-Config are mapped to valid PRACH occasions in the following order where the parameters are described in [4, TS 38.211].
> - First, in increasing order of preamble indexes within a single PRACH occasion
> - Second, in increasing order of frequency resource indexes for frequency multiplexed PRACH occasions
> - Third, in increasing order of time resource indexes for time multiplexed PRACH occasions within a PRACH slot
> - Fourth, in increasing order of indexes for PRACH slots

### 📚 增量术语表
* **ssb-PositionsInBurst**：灯塔出勤表。**一句话解释**：基站广播的一张位图（Bitmap），告诉你全网 64 个可能的灯塔（SSB）中，哪几个今天是真正上班打卡（发射）的。
* **Mapped to valid PRACH occasions**：映射到合法的发车位。**一句话解释**：把这些来上班的灯塔，一个个按顺序塞进地上的发车位（RO）和暗号池（Preamble）里。
* **Multiplexed (frequency / time)**：复用。**一句话解释**：在同一个时间点有几条并行的跑道，叫频域复用；在一条跑道上按先后顺序排着几个车位，叫时域复用。

### 🚪 核心痛点：谁先谁后的“铺床法则”
* **场景倒带**：假设现在有 4 个灯塔（SSB #0, #1, #2, #3）来上班了。地上有一个巨大的停车场（PRACH 资源池），里面有时隙（Slot）、有频段（Frequency）、有时隙里的先后车位（Time within a slot）、还有每个车位里的暗号（Preamble）。
* **物理层的困境**：如果基站说“灯塔 0 分配给第一批资源，灯塔 1 分配给第二批资源”，那么**“什么叫第一批？什么叫第二批？”** 是时间在前面的叫第一批，还是频率在前面的叫第一批？如果两边理解不一致，手机对着东边喊，基站却去西边听，彻底完蛋。

### 📏 破解法则：极其严苛的 4 重 `for` 循环
为了绝对统一思想，协议给出了这套极其出名的**“码域 -> 频域 -> 时域（槽内）-> 时域（跨槽）”**的四重填坑顺序（也就是程序员眼里的 4 层嵌套 `for` 循环）：

1. **第一层（最内层，变得最快）：前导码索引 (Preamble indexes)**
   * **动作**：先把一个车位（RO）里的暗号（0 到 63）按顺序分给灯塔。
   * **比喻**：先填满同一个房间里的所有座位。
2. **第二层：频域车位 (Frequency resource indexes)**
   * **动作**：如果一个房间（车位）的暗号发完了灯塔还没分完，就去找**同一时间下、旁边那条频段上的车位**继续分。
   * **比喻**：一个房间坐满了，去隔壁房间（频分复用）接着坐，但时间不变。
3. **第三层：槽内时域车位 (Time resource indexes within a slot)**
   * **动作**：如果同一时间下所有频段的车位都分满了，就顺延到**当前时隙（Slot）内，时间靠后的下一个车位**继续分。
   * **比喻**：同一层楼的房间都坐满了，去本楼层后面的候场区（时分复用）接着坐。
4. **第四层（最外层，变得最慢）：跨槽时域车位 (PRACH slots)**
   * **动作**：如果当前时隙（Slot）里的所有车位全被榨干了，才会跨越到**下一个合法的时隙（下一个 Slot）**去继续映射剩下的灯塔。
   * **比喻**：今天彻底没位子了，明天（下一个 Slot）再来接着坐！

### 🔄 SSB 到 RO 映射的 4 层 `for` 循环代码图
从程序员的视角，基站和手机的大脑里同时运行着这样一段绝对不能出错的伪代码：

```python
for (slot = 0; slot < 总时隙数; slot++) {                     // 步骤 4：跨槽
    for (time_res = 0; time_res < 槽内车位数; time_res++) {     // 步骤 3：槽内时分
        for (freq_res = 0; freq_res < 频段数; freq_res++) {     // 步骤 2：频分
            for (preamble = 0; preamble < 64; preamble++) {     // 步骤 1：码分 (最内层循环)
                
                // 把当前的 [时隙, 槽内时间, 频段, 暗号] 
                // 按照 N 和 R 的规矩，按顺序分发给正在排队等待映射的 SSB 灯塔！
                
            }
        }
    }
}
```

### 💡 章节硬核总结

**这段 3GPP 协议是随机接入中用来消除空间歧义的“底层拓扑基石”。它定义了实际发射的同步信号块（SSB）向 PRACH 物理资源（包含时间、频率、码域）映射的四维空间张量法则。其核心思想是：“先耗尽码域（Preamble），再耗尽频域（FDM），再耗尽微观时域（TDM within slot），最后耗尽宏观时域（Slot）”。这种由微观到宏观、由极快变化到极慢变化的嵌套映射序列，确保了无论基站的 PRACH 时频资源池配置得多么复杂碎片化，全网所有的手机都能和基站保持着绝对一致的“铺床顺序”，从而实现了 SSB 索引与物理发射资源的无缝、零歧义双向解算。**

## 📖 协议原文拆解 (十一)：确保每个灯塔都有车位的“宏观周期表” (Association Period)

> **协议原文**
> An association period, starting from frame 0, for mapping SS/PBCH block indexes to PRACH occasions is the smallest integer number in the set determined by the PRACH configuration period according Table 8.1-1 such that $N_{Tx}^{SSB}$ SS/PBCH block indexes are mapped at least once to the PRACH occasions within the association period, where a UE obtains $N_{Tx}^{SSB}$ from the value of ssb-PositionsInBurst... If after an integer number of SS/PBCH block indexes to PRACH occasions mapping cycles within the association period there is a set of PRACH occasions or PRACH preambles that are not mapped to $N_{Tx}^{SSB}$ SS/PBCH block indexes, no SS/PBCH block indexes are mapped to the set of PRACH occasions or PRACH preambles. An association pattern period includes one or more association periods and is determined so that a pattern between PRACH occasions and SS/PBCH block indexes repeats at most every 160 msec. PRACH occasions not associated with SS/PBCH block indexes after an integer number of association periods, if any, are not used for PRACH transmissions.

### 📚 增量术语表
* **$N_{Tx}^{SSB}$**：实际发送的 SSB 数量。**一句话解释**：基站真正打卡上班的“灯塔”总数。手机通过广播里的 `ssb-PositionsInBurst` 查到这个数字（比如今天有 8 个灯塔上班）。
* **Association period**：映射关联周期。**一句话解释**：基站向手机保证的一段最基本的时间跨度。在这个跨度内，**所有上班的灯塔（$N_{Tx}^{SSB}$）都必须至少被分配到一个合法的发车位（RO）**，确保没人掉队。
* **Mapping cycle**：映射循环。**一句话解释**：就是我们上一段学过的“四层 `for` 循环”完整跑完一遍的时间。
* **Association pattern period**：大循环模式周期。**一句话解释**：比上面那个周期更大的一个时间跨度（包含一个或多个关联周期），它的硬性指标是“每隔最多 160 毫秒，所有的映射规律必须重播一次”，方便手机算时间。

### 🚪 第一关：逼出来的“最小整数周期”
* **场景倒带**：基站今天有 8 个灯塔上班（$N_{Tx}^{SSB}=8$）。基站每 10 毫秒（一个 PRACH 配置周期）提供 2 个发车位。
* **痛点**：跑完一次 10 毫秒的小循环（mapping cycle），只塞得下 2 个灯塔。剩下的 6 个灯塔怎么办？
* **破局之法（拉长战线）**：协议规定，必须把周期拉长！拉多长？拉到**“刚刚好能让这 8 个灯塔都塞进车位为止”**！
  * 所以在这个例子里，需要 4 个小周期（$4 \times 2 = 8$），也就是 40 毫秒。这 40 毫秒，就被称为一个**“映射关联周期 (Association period)”**。在这个周期内，8 个灯塔“至少一次（at least once）”拿到了发车位。

### ✂️ 第二关：多余的资源？直接扔掉！(无效资源的废弃)
* **痛点**：假设 8 个灯塔上班，但基站很大方，一个小周期里给了 3 个发车位。
  * 第 1 个小周期：塞了 3 个灯塔。
  * 第 2 个小周期：又塞了 3 个灯塔（共 6 个）。
  * 第 3 个小周期：还剩 2 个灯塔没塞，放进前 2 个车位里。此时，**第 3 个小周期里的第 3 个车位，空下来了！**
* **铁血废弃法则**：协议极度强迫症地规定：如果在一个“关联周期”的末尾，灯塔已经全部分配完了，但还剩下了多余的车位（或者多余的暗号），**这些多余的资源绝对不准分给下一轮的灯塔！直接全部作废（not mapped / not used）！**
* **现实意义**：为什么要这么浪费？为了**“整齐划一”**。只有把多余的尾巴砍掉，下一个关联周期才能清清爽爽地从“1 号灯塔”重新开始映射，绝不让时序关系产生错位偏移。

### 📏 第三关：160毫秒的宇宙重播法则
* **大循环兜底**：为了让手机在长期睡眠唤醒后能快速找到规律，协议在“关联周期”之上，又盖了一个宏观的“大循环模式周期（Association pattern period）”。
* **强制重播**：无论你的基础映射配置多复杂，协议强制规定：**这种映射模式（pattern），最多每隔 160 毫秒，必须像录像带一样从头重播一次！** 因为 160ms 也是很多系统级广播参数更新的公倍数周期，这让手机永远能在 160ms 内找到确定的时空坐标。

### 🔄 映射周期的推演与废弃流水线
```text
 基站声明有 8 个灯塔 (SSB) 上班 ──► 目标：必须让这 8 个灯塔都有车位！
    │
    ├─ 开始跑四层 `for` 循环，把灯塔往发车位 (RO) 里塞
    │
    ├─ 循环 1 次不够？那就循环 2 次、3 次... 
    │  直到找到一个“最小的整数周期”，刚好能把 8 个灯塔全塞进去！
    │  (这个周期就叫 Association period)
    │
    ├─ 问自己：在这个周期的末尾，灯塔分完了，发车位还剩多余的吗？
    │
    ├──► 【没有多余】 ──► 完美！直接进入下一个周期。
    │
    └──► 【有多余车位】 ──► 动作: 挥刀斩断！多余的车位直接锁死作废，
                            严禁下一轮灯塔提前抢跑！
                            (确保所有规律每隔最多 160ms 完美循环重播一次)
```

### 💡 章节硬核总结

**这段协议构建了 5G 随机接入资源映射在时间轴上的“宏观周期体系”。当单个配置周期（PRACH configuration period）提供的时频资源不足以覆盖全部发送的波束（$N_{Tx}^{SSB}$）时，协议通过定义“映射关联周期（Association period）”，从数学上确保了网络在有限的时间窗口内能够实现空间波束的全覆盖。而本段最精妙的工程设计在于“尾部资源截断机制”：通过强制废弃周期末尾未被映射的冗余 PRACH Occasions 或 Preambles，协议强行切断了映射指针的跨周期滑动。这种看似浪费资源的“对齐操作”，换来的是整个大循环周期（Pattern period，至多 160ms）内映射关系的绝对稳态与周期性，极大降低了终端解析波束时空映射表的计算复杂度。**

## 📖 协议原文拆解 (十二)：强制唤醒下的发车位锁定与时域红线 (PDCCH Order 触发的 PRACH)

> **协议原文**
> For a PRACH transmission by a UE triggered by a PDCCH order or an LTM cell switch command MAC CE, the PRACH mask index field, if the value of the random access preamble index field is not zero, indicates the PRACH occasion for the PRACH transmission where the PRACH occasions are associated with the SS/PBCH block index indicated by the SS/PBCH block index field of the PDCCH order or the LTM cell switch command MAC CE and, if any, a cell indicator field in PDCCH order [5, TS 38.212] or a Target Configuration ID field in LTM cell switch command MAC CE [11, TS 38.321] indicates a cell for the PRACH transmission. If the UE is provided $K_{cell,offset}$ by cellSpecificKoffset, the PRACH occasion is after slot $n+2^\mu \cdot K_{cell,offset}$ where $n$ is the slot of the UL BWP for the PRACH transmission that overlaps with the end of the PDCCH order reception assuming $T_{TA}=0$, and $\mu$ is the SCS configuration for the PRACH transmission. If the PDCCH reception for the PDCCH order includes two PDCCH candidates from two linked search space sets based on searchSpaceLinkingId, as described in clause 10.1, the last symbol of the PDCCH reception is the last symbol of the PDCCH candidate that ends later.

### 📚 增量术语表
* **PRACH mask index field**：发车位掩码字段。**一句话解释**：强制命令里自带的一张“限行表”。它告诉你：“虽然属于这个灯塔的车位有很多，但你只能去掩码指定的那个车位发暗号！”
* **SS/PBCH block index field**：灯塔指示字段。**一句话解释**：基站强制唤醒你时，直接帮你选好了灯塔（比如：“你现在就在 SSB#3 的方向，给我对着它发！”），省去了手机自己去空中测光的步骤。
* **Cell indicator / Target Configuration ID**：目标小区指示。**一句话解释**：命令你“跨区发暗号”时，告诉你去哪个基站的地盘发。
* **PDCCH candidate ends later**：结束较晚的控制信道候选。**一句话解释**：如果你为了增强信号，在两个不同的地方重复收到了同一条唤醒命令，倒计时必须从**较晚收完的那一条**开始算起。

### 🚪 第一关：锁定三维坐标 (基站、灯塔、车位)
* **场景倒带**：手机正睡着觉，突然收到基站发来的“强制唤醒命令（PDCCH Order / LTM MAC CE）”。
* **基站的“包办婚姻”**：这可不是普通的接入，基站在命令里把所有坐标都给你安排得明明白白！
  1. **去哪发（小区）**：提取命令里的 `Cell indicator` 或 `Target ID`，锁定你要去哪个基站发暗号（有可能是跨基站切区）。
  2. **朝哪看（灯塔）**：提取命令里的 `SS/PBCH block index field`。基站直接替你决定了你要锚定哪个空间波束（灯塔），不准你自己瞎测瞎选。
  3. **停哪个车位（RO）**：虽然属于这个灯塔的车位有一大把，但如果命令里的暗号不是 0（也就是基站给你指派了专用暗号），那你必须提取 `PRACH mask index`。这个掩码会像狙击枪准星一样，从众多车位中，极其精准地锁定**唯一合法的那一个（或几个）车位**。

### ⏱️ 第二关：发车时间的物理红线 (倒计时公式)
* **痛点**：手机收到强制唤醒命令后，能立马在下一秒就点火发暗号吗？不行！解析命令、切换频道、准备发车都需要时间。
* **物理红线公式**：`slot n + 2^μ * K_cell,offset`
  * **$n$ (基准零点)**：手机接收完强制命令（PDCCH Order）最后一刻所对应的上行时隙。
  * **$K_{cell,offset}$ (小区校准)**：又是我们熟悉的那个为了大覆盖小区准备的“全网校准偏移值”。
  * **发车铁律**：你找到的那个发车位，它的时间点**必须**在这个公式算出来的红线之后！如果你算出来的车位在红线之前，说明时间太赶了，手机芯片根本处理不过来，你只能放弃这个车位，去等下一轮的合法车位。
* **多路接收的防剥削补丁**：为了防止漏听，基站有时会在两个地方发同一条命令（Linked search space）。此时，时隙 $n$ 必须以**“最后才收完的那条命令（ends later）”**为准来计时，绝不让手机吃“时间差”的亏。

### 🔄 强制唤醒 (PDCCH Order) 点火流水线
```text
 手机 (UE) 突然收到强制唤醒信令 (PDCCH Order 或 LTM MAC CE)
    │
    ├─ 动作 1 (记录时间): 掐下秒表！记住收完信令的最后一个符号，推算出基准时隙 n。
    │
    ├─ 动作 2 (三维定位):
    │      ├─ 提取小区 ID ──► 知道去哪个地盘发。
    │      ├─ 提取 SSB 索引 ──► 知道该朝哪个波束方向发。
    │      └─ 提取 Mask 掩码 ──► 知道该停在哪个具体的车位 (RO) 发。
    │
    ├─ 动作 3 (算红线): 结合小区校准值 K_cell,offset，画出物理底线 (n + 偏移)。
    │
    ├─ 问自己：我找到的车位，在这个时间红线之后吗？
    │
    ├──► 【在红线之前】 ──► 时间太仓促，身体反应不过来！放弃这个车位，顺延等下一个！
    │
    └──► 【在红线之后】 ──► 完美！准备就绪，准时在这个车位点火发射 PRACH！🚀
```

### 💡 章节硬核总结

**这段协议展示了 5G 物理层在处理“网络强制分配接入资源（Network-Triggered PRACH）”时的绝对集权与底层保护机制。当通过 PDCCH Order 或 LTM MAC CE 触发接入时，终端彻底丧失了自主选择小区、自主测量下行波束（SSB）、以及自主挑选接入时机（RO）的权利。取而代之的，是严格解析信令中显式携带的 Target Cell、SSB Index 和 Mask Index，完成三维时空坐标的硬性锚定。与此同时，协议也划定了严苛的物理时序底线：合法发车位的时域起点，必须大于指令接收完毕的时隙叠加宏观小区偏移量（$K_{cell,offset}$）。哪怕是面临双候选控制信道的冗余接收场景，也必须以“最晚结束”的一路为基准点，在极致的调度集权中，为终端的硬件处理留下了不可剥夺的喘息空间。**

## 📖 协议原文拆解 (十三)：高层限制下的“狙击选位” (主动触发下的 Mask Index)

> **协议原文**
> For a PRACH transmission triggered by higher layers, if ssb-ResourceList is provided, the PRACH mask index is indicated by ra-ssb-OccasionMaskIndex which indicates the PRACH occasions for the PRACH transmission where the PRACH occasions are associated with the selected SS/PBCH block index.

### 📚 增量术语表
* **Triggered by higher layers**：高层触发。**一句话解释**：这是手机主动想上网（比如你要发微信），而不是被基站大喇叭（PDCCH Order）强迫叫醒的。
* **ssb-ResourceList**：SSB 资源白名单。**一句话解释**：高层（MAC/RRC）在下发发车指令时，顺带给的一张“白名单”。意思是：“虽然基站有很多灯塔，但你这次发暗号，只能从这张名单里挑灯塔！”
* **ra-ssb-OccasionMaskIndex**：随机接入专属掩码索引。**一句话解释**：这和我们上一关讲的“限行表”一模一样，但它的名字变了，是专门针对这种主动触发且带白名单情况下的“狙击枪准星”。

### 🚪 第一关：受限环境下的自主选择 (白名单里的最亮灯塔)
* **场景倒带**：手机大脑（高层）觉得该上网了，于是触发物理层去发 PRACH。但是，高层这次不是让物理层随便在空中测，而是给了它一张白名单（`ssb-ResourceList`）。
* **物理层的动作**：物理层拿着这张白名单，只去测量名单上的那几个灯塔（SSB）。测完之后，它在白名单里挑了一个最亮的（selected SS/PBCH block index）。

### 🎯 第二关：狙击准星锁定车位 (Mask Index 过滤)
* **痛点**：物理层挑好了白名单里的最亮灯塔。但根据映射表，这个灯塔名下可能分配了好几个合法的发车位（PRACH occasions）。手机能随便挑一个发吗？
* **掩码限行**：绝不可以！如果高层给了你白名单，它一定也会同步塞给你一张“限行过滤网”，这张网在这里被命名为 **`ra-ssb-OccasionMaskIndex`**。
* **一锤定音的坐标**：
  * **第一步**：找出与你挑中的那个灯塔（selected SSB）关联的所有合法车位。
  * **第二步**：把 `ra-ssb-OccasionMaskIndex` 这个掩码盖上去。掩码上挖了洞的地方，就是你最终真正能把 PRACH 发射出去的唯一合法车位！

### 🔄 高层受限触发模式下的选位流水线
```text
 高层 (MAC) 主动想上网，触发物理层发送 PRACH
    │
    ├─ 问高层：老大，有什么特殊限制吗？
    │
    ├──► 【没有限制】 ──► 物理层自由去空中测最亮的灯塔，按常规找车位。
    │
    └──► 【有限制！给了一张白名单 (ssb-ResourceList)】
              │
              ├─ 动作 1: 物理层乖乖在白名单里测光，挑出一个最亮的灯塔 (selected SSB)。
              │
              ├─ 动作 2: 去查映射表，把这个灯塔名下的所有车位 (ROs) 全列出来。
              │
              └─► 动作 3: 拿出高层给的狙击准星【ra-ssb-OccasionMaskIndex】！
                  对准刚才列出的一排车位，掩码指哪，就去哪个车位点火发射！🚀
```

**这段协议是对随机接入触发源与资源选择逻辑的又一块重要拼图。它处理的是“终端自主触发（Triggered by higher layers）但在频域/空域受限（配置了 ssb-ResourceList 白名单）”的半自由场景。在这种场景下，终端保留了对白名单内下行波束（SSB）的自主测量和选择权，但彻底丧失了对具体发车时机（PRACH Occasion）的自主随机权。协议通过引入专用的 RRC 层掩码参数 `ra-ssb-OccasionMaskIndex`，在物理层执行了一次强制的“AND（与）”逻辑运算：将自主选定的 SSB 所映射的 RO 集合，与 Mask Index 允许的 RO 集合求交集，从而在赋予终端一定自主性的同时，依然保持了网络侧对微观接入时机的绝对管控。**

## 📖 协议原文拆解 (十四)：发车位的“物理重置法则” (PRACH Occasion 编号逻辑)

> **协议原文**
> The PRACH occasions are mapped consecutively per corresponding SS/PBCH block index. The indexing of the PRACH occasion indicated by the mask index value is reset per mapping cycle of consecutive PRACH occasions per SS/PBCH block index. The UE selects for a PRACH transmission the PRACH occasion indicated by PRACH mask index value for the indicated SS/PBCH block index in the first available mapping cycle.

### 📚 增量术语表
* **Consecutively**：连续地。**一句话解释**：属于同一个灯塔（SSB）的车位，哪怕跨越了频率或者时间，在逻辑编号上也是挨个紧挨着的（1号、2号、3号...）。
* **Mask index value**：掩码索引值。**一句话解释**：高层发给你的“狙击准星”（比如“去第 3 号车位发”）。
* **Mapping cycle**：映射循环。**一句话解释**：把所有上班的灯塔全部轮完一遍所需的时间（就是我们之前拆解过的四层 `for` 循环跑完一遍）。
* **Reset**：重置/清零。**一句话解释**：每一个新循环开始时，车位的编号必须重新从 1 开始数。

### 🚪 第一关：发车位的“串糖葫芦”排序法
* **场景倒带**：我们在上一段拆解中知道，如果高层给了你一张“掩码限行网（Mask Index）”，它会告诉你：“去属于你这个灯塔的**第 2 号**车位发暗号！”
* **物理层难题**：但是，属于我这个灯塔的车位可能散落在不同的时隙、不同的频段上，我怎么知道哪个是“第 2 号”？
* **排序法则**：协议规定，所有属于同一个灯塔（SSB）的车位，必须**严格按照我们在第 (十) 关里讲的“四重循环法则”依次排队编号**。就像把散落各地的山楂串成一串糖葫芦，挨个贴上 1号、2号、3号的标签（mapped consecutively）。

### ⏱️ 第二关：时空轮回的“编号清零机制” (Reset per mapping cycle)
* **核心痛点**：时间是无限延伸的，映射循环（Mapping cycle）也会一轮接一轮地跑下去。如果车位编号一直累加（比如涨到第 8900 号车位），高层发掩码时根本没法指挥。
* **物理层重置机制**：协议定下死规矩，**每当一个完整的映射循环（Mapping cycle）跑完，下一个循环重新开始时，属于这个灯塔的车位编号必须立刻“清零重置（Reset）”！**
* **现实意义**：这就意味着，无论你身处宇宙的哪一秒，只要进入了一个新的循环周期，属于你灯塔的第一个车位永远叫“1 号”，第二个永远叫“2 号”。掩码（Mask Index）只需要极其简单的低位数（比如“第 2 号”），就能指挥手机在任意一个循环中准确找到车位。

### 🚀 第三关：只争朝夕的发车原则 (First available mapping cycle)
* **痛点**：既然每个循环都有“第 2 号”车位，那我到底是等明天的第 2 号车位，还是后天的第 2 号车位呢？
* **最高指示**：协议要求手机必须是个“急性子”！**你必须在当下能赶得上的“第一个可用循环（first available mapping cycle）”里，锁定那个掩码指向的车位，立刻开火！** 绝对不准拖延到下一个循环！

### 🔄 掩码寻址与编号重置流水线
```text
 时间轴推进... 循环 1 开始 ──► 循环 1 结束 ──► 循环 2 开始 ──► 循环 2 结束...
                           │
    高层下达死命令：        │
   “去第 3 号车位发暗号！”  │
                           ▼
            【进入第一个可用的循环 (First available mapping cycle)】
                           │
                           ├─ 动作 1 (编号清零): 刚进循环，车位编号立刻 Reset。
                           │
                           ├─ 动作 2 (排序找位): 把散落在本循环里的本灯塔车位，
                           │   按“码->频->槽内->跨槽”的顺序，贴上 1,2,3...的标签。
                           │
                           └─► 动作 3 (锁定开火): 找到刚刚贴上【第 3 号】标签的那个车位，
                               毫不犹豫，点火发射！🚀
```

### 💡 章节硬核总结

**这段协议完美解决了“宏观时间无限延伸”与“微观掩码索引位宽受限”之间的底层矛盾。通过在每一个映射周期（Mapping Cycle）的边界强行引入“编号重置（Reset）”机制，协议将无限的时间轴切分成了一个个相互独立的、坐标完全一致的“平行时空”。配合基于四重映射规则生成的连续逻辑索引（Consecutive Mapping），终端的高层（MAC）只需下发极小位宽的掩码索引（Mask Index），物理层就能在当前“最早可达（First Available）”的时间切片内，实现对复杂物理资源（RO）的绝对精确定位与零延迟接入。这就是 5G 物理层极简却极具鲁棒性的寻址哲学。**

## 📖 协议原文拆解 (十五)：指定暗号下的“三重排序法则” (PRACH Occasion 的降维排序)

> **协议原文**
> For the indicated preamble index, the ordering of the PRACH occasions is
> - First, in increasing order of frequency resource indexes for frequency multiplexed PRACH occasions
> - Second, in increasing order of time resource indexes for time multiplexed PRACH occasions within a PRACH slot
> - Third, in increasing order of indexes for PRACH slots

### 📚 增量术语表
* **Indicated preamble index**：指定的/已明确的前导码索引。**一句话解释**：手机不再需要自己去暗号池里“切香肠盲抽”了。这通常发生在高层（或基站大喇叭 PDCCH Order）直接发话：“你就给我用第 35 号暗号！”
* **Ordering of the PRACH occasions**：发车位的排序。**一句话解释**：既然暗号定死了，手机接下来要找的，就是把所有合法的发车位（RO）像串糖葫芦一样排好队，依次去试。

### 🚪 核心痛点：已知暗号，怎么选车位？
* **场景倒带**：在拆解(十)中，我们学到了把 SSB 映射到 RO 和 Preamble 的 4 层 `for` 循环（顺序是：码域 $\to$ 频域 $\to$ 槽内时域 $\to$ 跨槽时域）。
* **降维打击**：但在这个特定场景下，手机已经带着一个“指定的暗号（Indicated preamble index）”了。这意味着手机根本不需要在“第一层循环（Preamble index）”里挑挑拣拣了！
* **三重填坑顺序**：既然“码域”已经被抽离，原本的 4 层循环立刻降维成了 **3 层 `for` 循环**。手机寻找合法发车位（RO）的扫描顺序变成了：
  1. **第一层（最内层）：频分发车位 (Frequency resource indexes)**
     * **动作**：先在同一时刻，去找隔壁频段上的发车位。
  2. **第二层：槽内时分发车位 (Time resource indexes within a slot)**
     * **动作**：如果同一时刻的所有频段都找完了，就顺延到当前时隙（Slot）内，时间靠后的下一个发车位。
  3. **第三层（最外层）：跨槽时分发车位 (PRACH slots)**
     * **动作**：如果当前时隙（Slot）全找完了，才会跨越到明天的下一个时隙（Slot）去继续找。

### 🔄 指定暗号下的 3 层 `for` 循环代码图
从物理层的视角，一旦暗号（Preamble）锁定，寻找车位（RO）的代码逻辑瞬间变成了这样：

```python
// 提前锁定：preamble = 指定的暗号 (比如 35号)

for (slot = 0; slot < 总时隙数; slot++) {                     // 步骤 3：跨槽
    for (time_res = 0; time_res < 槽内车位数; time_res++) {     // 步骤 2：槽内时分
        for (freq_res = 0; freq_res < 频段数; freq_res++) {     // 步骤 1：频分 (最内层)
            
            // 拿着这个指定的暗号 (35号)
            // 在当前的 [时隙, 槽内时间, 频段] 这个车位里开火！
            
        }
    }
}
```

### 💡 章节硬核总结

**这段协议是对之前经典的“四重映射法则（SSB-to-RO/Preamble mapping）”在特定条件下的“降维重载”。当终端执行无竞争随机接入（CFRA），或受限于 PDCCH Order、高层强制指派（Indicated preamble index）时，码域维度的自由度被收敛为常数。物理层剥离了对 Preamble Index 的最内层遍历，直接将时频资源（PRACH Occasion）的张量顺序拉平为：频域复用优先 $\to$ 槽内时域次之 $\to$ 跨槽时域最后。这种绝对确定的资源扫描顺序，保证了终端在持有特定前导码时，始终能以极度确定、毫无歧义的物理时序依次占配 RO 资源，将多终端碰撞的概率降到了理论最低。**

## 📖 协议原文拆解 (十六)：极限弱网下的“连发车位绑定法则” (PRACH Repetitions 的 RO 集合)

> **协议原文**
> For a PRACH transmission with $N_{preamble}^{rep}$ preamble repetitions, a set consists of $N_{preamble}^{rep}$ valid PRACH occasions that are consecutive in time, use same frequency resources, and are associated with same one or more SS/PBCH block index(es), and each SS/PBCH block index is associated with same preamble indexes in all valid PRACH occasions within the set.

### 📚 增量术语表
* **Preamble repetitions**：前导码重复发送。**一句话解释**：身处弱覆盖（比如地下室）的手机，为了让基站能听见，把同一个暗号连续发好几遍。
* **$N_{preamble}^{rep}$**：重复发送次数。**一句话解释**：具体连发几遍（比如发 4 遍）。
* **A set (of PRACH occasions)**：一个发车位集合。**一句话解释**：连发 4 遍，就需要 4 个发车位（RO）。这 4 个车位被死死捆绑在一起，形成一个不可分割的“连发套餐（Set）”。

### 🚪 核心痛点：连发期间能“左右横跳”吗？
* **场景倒带**：高层命令手机：“你信号太差了，这发暗号（Msg1）你必须给我连发 4 遍（$N_{preamble}^{rep}=4$）！”
* **物理层的困惑**：那我这 4 遍怎么发？我是先在这个频段发一次，再去那个频段发一次？还是对着东边喊一次，再对着西边喊一次？换个暗号喊行不行？

### ⛓️ 破解法则：全维度的“四重死锁”
为了让基站能够完美地把这 4 遍微弱的信号“相干叠加（合并能量）”，协议对这个包含 4 个车位的“连发套餐（Set）”下达了极其变态的四重铁血禁令：

1. **时间死锁 (consecutive in time)**
   * **法则**：这 4 个发车位必须在时间轴上是**连续挨着的**！绝不允许发完第 1 遍，睡一觉，明天再发第 2 遍。必须一口气连着发完！
2. **频率死锁 (use same frequency resources)**
   * **法则**：这 4 遍发射必须在**完全相同的频率格子**里！绝不允许开启跳频，也不允许中途换跑到其他频段。
3. **空间波束死锁 (associated with same SS/PBCH block index)**
   * **法则**：我们在拆解（八）里讲过连发时不许转头（Same spatial filter）。这里进一步强调，不仅你不能转头，这 4 个发车位在系统广播里，也必须是**死死绑定在同一个（或同一组）灯塔波束（SSB）上**的。
4. **码域死锁 (same preamble indexes)**
   * **法则**：最后，也是最严苛的一点。在这 4 个车位里，手机选中的那个**暗号（Preamble Index）绝不能变**！第一遍喊的是“天王盖地虎（比如 12 号暗号）”，后面 3 遍必须也是一模一样的“天王盖地虎”。如果车位映射导致 12 号暗号在第 3 个车位里变了意思，那这个车位就不能用来连发！

### 🔄 连发模式 (Repetitions) 的时空码频流水线
```text
 物理层接到了连发 4 遍暗号的死命令 (N_rep = 4)
    │
    ├─ 动作 1: 开始在系统中寻找一个“连发套餐 (Set)”，必须包含 4 个车位。
    │
    ├─ 校验这 4 个车位是否满足“四重死锁”：
    │    ├── 校验时间：这 4 个车位是连续挨着的吗？ ──► (必须是)
    │    ├── 校验频率：都在同一条车道上吗？ ──► (必须是)
    │    ├── 校验波束：都对应着刚才那个最亮的灯塔吗？ ──► (必须是)
    │    └── 校验暗号：在这个 4 个车位里，我能一直用同一个编号的暗号吗？ ──► (必须能)
    │
    └─► 全部满足！锁定这个“连发套餐”！
             │
             ▼
        【天线姿势冻结】【频率死锁】【暗号死锁】
         连续轰出 4 枪！确保基站能把这 4 枪的微弱能量完美合成！🚀
```

**这段协议是 5G 覆盖增强特性（Coverage Enhancement）在物理随机接入信道上的终极映射限制。为了实现 PRACH 信号在极低信噪比（SNR）下的基站侧软合并（Soft Combining），协议彻底剥夺了重复发送（Repetitions）期间的任何时频空码自由度。它通过强制定义一个“同频、同波束、同前导码、时域连续”的刚性 PRACH Occasion 集合（Set），确保了信道脉冲响应（CIR）在这几次重复传输期间保持最大程度的静态一致性。这是牺牲了系统调度灵活性和频率分集增益，来换取纯粹的能量相干积累，是 5G 物理层向极端恶劣无线电环境妥协与抗争的极致体现。**

## 📖 协议原文拆解 (十七)：连发套餐的“宏观排班表” (Repetitions 的映射大周期)

> **协议原文**
> For a PRACH transmission with preamble repetitions, a time period, starting from frame 0, is the smallest integer number of association pattern periods such that at least one set of valid PRACH occasions for each of the $N_{Tx}^{SSB}$ SS/PBCH block indexes can be determined within the time period for all configured number of preamble repetitions for each supported feature combination provided in each RACH-ConfigCommon. The set(s) of valid PRACH occasions for each configured number of preamble repetitions repeats every time period.

### 📚 增量术语表
* **Preamble repetitions**：前导码重复发送。**一句话解释**：弱覆盖下手机把暗号连发好几遍。
* **Association pattern periods**：大循环模式周期。**一句话解释**：我们在拆解（十一）中讲过，这是普通接入流程中包含所有映射规律的超级大循环（最多160毫秒）。
* **A time period**：一个新定义的时间周期（专为连发设计）。**一句话解释**：因为连发极其占地方（一个人要占好几个车位），普通的大循环可能装不下，所以协议专门为连发定义了一个**超级加长版的宏观排班表**。

### 🚪 第一关：填鸭式排班的困境
* **场景倒带**：在拆解（十一）的常规流程中，我们说过基站会寻找一个最小的周期，保证每个上班的灯塔（$N_{Tx}^{SSB}$）都至少能分到一个发车位（RO）。
* **连发带来的灾难**：现在问题来了，因为有了“连发（Repetitions）”，那些信号差的手机不仅要车位，它要的是**一整个连续的车位套餐（A set of valid PRACH occasions）**！如果基站规定可以连发 2 遍、4 遍、甚至 8 遍，那么这 8 个连续的车位，在原来的普通周期里根本塞不下。

### 📏 第二关：被迫拉长的“超级排班表” (A time period)
* **协议的破局之法**：既然原有的周期（association pattern periods）塞不下，那就把它当成积木一块块往上叠！
* **计算法则**：这个专属于连发的“超级周期（A time period）”，等于**最小整数个**的“普通大循环周期”。
  * 协议要求基站一直往上叠加周期，**直到把所有上班的灯塔（$N_{Tx}^{SSB}$），以及它们对应的每一种连发组合（发 2 遍的套餐、发 4 遍的套餐等），都至少成功塞进去一次为止！**
  * 这个塞满了所有可能的连发套餐的超长跨度，就是基站为弱网手机精心准备的终极排班表。

### 🔄 第三关：超级排班表的无限轮回
* **铁律重申**：`The set(s) of valid PRACH occasions... repeats every time period.`
* **现实意义**：一旦这个超级排班表（A time period）的长度被计算出来了（比如算出是 320 毫秒），那么在这 320 毫秒的内部，所有的连发车位分布得错落有致。当这 320 毫秒结束，下一个 320 毫秒到来时，**所有的发车位套餐必须一模一样地从头重播一次**！这保证了无论手机什么时候从深层睡眠中醒来，都能找到确定的连发车位去呼救。

### 🔄 PRACH 连发套餐的宏观周期推演
```text
 基站准备给弱网手机安排“连发套餐 (Set of ROs)”
    │
    ├─ 问自己：我这 1 个普通的循环周期 (比如 160ms) 装得下这么多 8 连发的套餐吗？
    │
    ├──► 【装得下】 ──► 完美！这个 time period 就等于 1 个普通周期 (160ms)。
    │
    └──► 【装不下！】
              │
              └─► 动作 1: 拉长战线！1 个不够就拼 2 个，2 个不够就拼 3 个...
                  (寻找最小整数倍的普通周期)
                  │
                  ├─ 动作 2: 直到所有的灯塔、所有的连发组合，都至少能被塞进一次为止！
                  │  (假设算出拼 2 个刚好装下，那么 time period = 320ms)
                  │
                  └─► 动作 3: 把这 320ms 刻入系统 DNA。以后每隔 320ms，
                              这套庞大而复杂的连发车道阵列，就完美轮回重播一次！
```

### 💡 章节硬核总结

**这段协议是 5G 物理层为了解决 Coverage Enhancement (覆盖增强) 带来的“资源极度膨胀”而打出的时域补丁。当引入 PRACH Repetitions 后，终端所需的资源从孤立的时频点（Single RO）爆炸式增长为连续的资源集合（A Set of ROs）。原有的 Association Pattern Period 极有可能无法容纳这些庞大的资源块。为此，协议在最外层重新定义了一个更宏大的时间张量（A time period）。它以原 Pattern Period 为步长进行离散化拉伸，直到其物理容量足以至少一次性吞吐所有发送波束（$N_{Tx}^{SSB}$）和所有特性组合（Feature Combination）下的重复发送集合。通过这种“整数倍扩展与全局重置”的设计，系统在满足极端弱网终端接入需求的同时，依然极其优雅地维护了物理层底层映射关系在时间轴上的绝对周期性与可预测性。**

## 📖 协议原文拆解 (十八)：连发套餐的“排兵布阵” (Repetition Sets 的时频排列法则)

> **协议原文**
> Within a time period, for set(s) of $N_{preamble}^{rep}$ valid PRACH occasions for a PRACH transmission with $N_{preamble}^{rep}$ preamble repetitions
> - the first valid PRACH occasion of the first set is the first valid PRACH occasion
> - the first valid PRACH occasion of subsequent sets, if any, is determined according to an ordering of valid PRACH occasions
>   - first, in increasing order of frequency resource indexes for frequency multiplexed PRACH occasions
>   - second, in increasing order of time resource indexes for time multiplexed PRACH occasions
> where, for each frequency resource index for frequency multiplexed PRACH occasions
> - the first valid PRACH occasion of the first set is the first valid PRACH occasion
> - the first valid PRACH occasion of subsequent sets, if any,
>   - is after msg1-RepetitionTimeOffsetROGroup consecutive valid PRACH occasions in time from the first valid PRACH occasion of the previous set, where each PRACH occasion is associated with same SS/PBCH block index(es) and each SS/PBCH block index is associated with same preambles, if msg1-RepetitionTimeOffsetROGroup is provided
>   - is after the PRACH occasions for the previous set, if msg1-RepetitionTimeOffsetROGroup is not provided

### 📚 增量术语表
* **Set of valid PRACH occasions**：合法发车位套餐。**一句话解释**：连发 4 遍就需要 4 个连续的车位组成的一个“套餐（Set）”。
* **Subsequent sets**：后续的套餐。**一句话解释**：在这个极其漫长的宏观周期（time period）里，基站不止给你安排了一个连发套餐，可能安排了第 2 个、第 3 个套餐供你挑选或重传。
* **msg1-RepetitionTimeOffsetROGroup**：套餐间时间偏移步长。**一句话解释**：决定了“第二个套餐”距离“第一个套餐”到底隔了多久的标尺。

### 🚪 第一关：套餐的宏观寻址方向 (频域优先)
* **痛点**：在一个大周期里，散落着几十个车位。如何把这些车位组合成一个个“4连发套餐”？第一个套餐放哪？第二个放哪？
* **第一套餐**：极其简单，占据最前面的资源，`the first valid PRACH occasion` 就是它的起点。
* **后续套餐的扫描顺序**：为了安放第 2 个、第 3 个套餐，协议定下了大方向：**频分优先，时分在后**。
  * 也就是说，排完第一个套餐后，先去**同一个时间点的其他频率车道**上看看能不能放下第二个套餐；如果所有频率车道都放满了，再顺延到**时间轴后面的车位**去放。

### 📏 第二关：同一频段内，套餐间的“安全距离”
最复杂的情况发生在上一步的“顺延时间轴”时。如果在同一条频率车道上，我要排两个套餐，它们之间是紧挨着，还是隔开一段距离？
* **默认模式（背靠背排队）**：
  * 如果基站没下发那个特殊的步长参数（`msg1-RepetitionTimeOffsetROGroup is not provided`）。
  * **动作**：极其粗暴，后一个套餐直接紧贴着前一个套餐！比如第一个套餐占了第 1 到 4 号车位，第二个套餐就**紧接着（after the PRACH occasions for the previous set）**从第 5 号车位开始排。
* **步长模式（拉开安全距离）**：
  * 如果基站下发了步长参数（比如 `msg1-RepetitionTimeOffsetROGroup = 8`）。
  * **动作**：两个套餐之间必须强行拉开距离！第二个套餐的起点，必须距离第一个套餐的起点，**足足跨越 8 个合法的发车位（consecutive valid PRACH occasions in time）**！这意味着中间空出来的车位可以留给别人用，实现了连发套餐在时间轴上的交织与错位，极大降低了瞬间拥堵。
  * **铁律防错**：协议还不忘补一句，这跨越的 8 个车位，在映射关系上必须还是属于同一个灯塔（SSB）和同一组暗号（Preambles），绝对不能数错刻度。

### 🔄 连发套餐 (Repetition Sets) 时频排布流水线
```text
 基站开始在一个大周期内规划多个“4连发套餐”的版图：
    │
    ├─ 放置【第 1 个套餐】: 占据最左上角的最早车位。
    │
    ├─ 准备放置【第 2 个套餐】: 
    │    ├─ 优先去同一时间的“隔壁频率车道”放 (频分优先)
    │    │
    │    └─ 如果频率排满了，只能往时间轴后面排：
    │         │
    │         ├─ 问：基站配了“套餐间的时间步长 (Offset)”吗？
    │         │
    │         ├──► 【没配】 ──► 动作: 背靠背！第 2 个套餐紧贴着第 1 个套餐的屁股放！
    │         │
    │         └──► 【配了 (如步长=8)】 ──► 动作: 拉开距离！第 2 个套餐的起点，
    │                                     必须距离第 1 个起点往后数满 8 个同类车位！
```

### 💡 章节硬核总结

**这段协议完美解决了多重复发送集合（Multiple Repetition Sets）在时频二维资源池中的拓扑排列问题。当网络为覆盖增强终端提供冗余的 PRACH 传输机会时，资源集合的排列顺序严格遵循“频域复用为主、时域复用为辅”的降维扫池法则。在纯时域维度上，为了防止连续多个长周期 Repetition Set 引发的时域长期霸占与拥塞，协议引入了 `msg1-RepetitionTimeOffsetROGroup` 组间偏移步长。通过这个宏观步长，网络能够将紧密耦合的连续重复传输块，在更宽广的时间轴上进行梳子状交织拉伸（Interleaving），从而在微观的连续能量累积（满足相干合并）与宏观的系统抗拥塞调度之间，找到了极其优雅的数学平衡。**

## 📖 协议原文拆解 (十九)：VIP 连发的“参数跨表借用” (CFRA 下的套餐偏移步长)

> **协议原文**
> For a PRACH transmission with preamble repetitions in CFRA procedure, msg1-RepetitionTimeOffsetROGroup is determined by the FeatureCombinationPreambles indicating msg1-Repetitions with same value as msg1-RepetitionNum provided by RACH-ConfigDedicated.

### 📚 增量术语表
* **CFRA (Contention-Free Random Access)**：无竞争随机接入。**一句话解释**：VIP 通道，基站提前私下塞给你一个专属暗号，你发暗号时绝不会和别人撞车（竞争）。
* **msg1-RepetitionTimeOffsetROGroup**：连发套餐间的时间偏移步长。**一句话解释**：我们在拆解（十八）里讲过，这是决定第一个连发套餐和第二个连发套餐之间“隔多远”的标尺。
* **RACH-ConfigDedicated**：专属随机接入配置。**一句话解释**：基站给你开 VIP 权限时，单独发给你的“小灶”配置表，里面只写了你个人的专属参数。

### 🚪 第一关：VIP 手机的参数缺失危机
* **场景倒带**：一个身处地下室的 VIP 手机准备发暗号了。基站在 `RACH-ConfigDedicated`（私发的小灶配置表）里告诉它：“你是 VIP（走 CFRA），并且因为你信号差，我特批你连发 4 遍（`msg1-RepetitionNum = 4`）！”
* **物理层的困境**：连发没问题，但是如果第一套连发套餐失败了，我找第二套连发套餐时，它们之间该隔多远？物理层去翻这本 `RACH-ConfigDedicated` 小灶配置表，尴尬地发现：**表里竟然没写 `msg1-RepetitionTimeOffsetROGroup` 这个决定距离的步长参数！**

### 🔗 第二关：跨表查询的“同名借用”法则
* **痛点**：基站在发 VIP 配置时，为了省信令开销，故意把这个步长参数给省略了。那物理层该去哪里找这个丢掉的步长参数？
* **破局之法 (跨表匹配)**：协议极其精妙地给出了“去公共大厅抄作业”的法则！
  1. 手机拿着自己手里的“连发 4 遍”这个需求（`msg1-RepetitionNum = 4`），跑去基站广播的**公共配置大屏（FeatureCombinationPreambles）**前。
  2. 手机在公共大屏上扫视，寻找哪里也写了“连发 4 遍（`msg1-Repetitions = 4`）”的公共规则。
  3. 一旦找到（with same value），手机就直接把公共规则里配好的那个步长参数（`msg1-RepetitionTimeOffsetROGroup`）**原封不动地“借”过来**，用到自己的 VIP 流程里！

### 🔄 VIP 连发步长参数的跨表提取流水线
```text
 物理层拿着 VIP 配置 (CFRA)，准备排布多个连发套餐
    │
    ├─ 问自己：VIP 专属配置表里，写了套餐间的距离 (步长) 吗？
    │
    └─ 发现没写！触发协议的“跨表借用”法则：
         │
         ├─ 动作 1: 提取自己当前的连发次数 (比如连发 4 遍)。
         │
         ├─ 动作 2: 跑去全网广播的公共大屏 (FeatureCombinationPreambles)。
         │
         ├─ 动作 3: 在公共大屏上搜索“连发 4 遍”对应的步长配置。
         │
         └─► 动作 4: 找到匹配的步长！直接拿回来，
                     用这个步长去拉开自己的 VIP 连发套餐之间的距离！
```

### 💡 章节硬核总结

**这段协议展现了 5G RRC 信令配置在物理层解析时极其经典的“Dedicated（专属）借用 Common（公共）”的设计美学。在无竞争随机接入（CFRA）场景下，网络侧为了极度压缩 `RACH-ConfigDedicated` 的信令开销，故意省略了用于时域资源排布的 `msg1-RepetitionTimeOffsetROGroup` 参数。为了填补这个物理层的参数真空，协议硬性建立了一个跨配置栈的“外键映射”：终端必须提取自身的专属重复次数（msg1-RepetitionNum），去公共特性组合列表（FeatureCombinationPreambles）中寻找具有相同重复次数（msg1-Repetitions）的条目，从而将该条目下的时间偏移参数强行继承到 CFRA 流程中。这既保证了时频网格排布的闭环，又完美避免了网络侧在下发 VIP 信令时的参数冗余。**

## 📖 协议原文拆解 (二十)：精细化探照灯下的车位锁定 (基于 CSI-RS 的 PRACH Occasion)

> **协议原文**
> For a PRACH transmission triggered upon request by higher layers, a value of ra-OccasionList [12, TS 38.331], if csirs-ResourceList is provided, indicates a list of PRACH occasions for the PRACH transmission where the PRACH occasions are associated with the selected CSI-RS index indicated by csi-RS. The indexing of the PRACH occasions indicated by ra-OccasionList is reset per association pattern period.

### 📚 增量术语表
* **CSI-RS (Channel State Information Reference Signal)**：信道状态信息参考信号。**一句话解释**：如果说 SSB 是基站用来扫射全网的“粗犷大灯塔”，那 CSI-RS 就是用来追踪特定手机的“精细小探照灯”。
* **csirs-ResourceList**：CSI-RS 资源白名单。**一句话解释**：高层给物理层的一张新测光表，命令它别去看 SSB 灯塔了，去测这几个更精细的 CSI-RS 探照灯。
* **ra-OccasionList**：发车位列表。**一句话解释**：一张直接告诉你“去哪几个发车位（RO）发射”的具体清单。
* **Association pattern period**：大循环模式周期。**一句话解释**：我们在拆解（十一）中学过的，那个最多 160 毫秒的超级大轮回。

### 🚪 第一关：测光目标的“鸟枪换炮” (从 SSB 切换到 CSI-RS)
* **场景倒带**：手机高层主动发起随机接入（比如为了切换基站）。通常情况下，物理层会去测 SSB（粗犷大灯塔）。
* **高级模式开启**：但这回，高层给物理层塞了一张高级白名单（`csirs-ResourceList`）。
* **物理层的动作**：物理层立刻心领神会，不再去管全网广播的 SSB，而是开始极其精细地测量白名单里的那几束 CSI-RS 探照灯，并挑出最亮的一束（selected CSI-RS index）。

### 🎯 第二关：顺藤摸瓜找车位 (解析 ra-OccasionList)
* **痛点**：挑好了最亮的 CSI-RS，我该去哪个车位（PRACH Occasion）发暗号呢？CSI-RS 可没有 SSB 那种复杂的“四重 `for` 循环”映射公式啊！
* **破局之法（直接发名单）**：既然没有通用的底层映射公式，高层就直接“喂饭到嘴边”！高层会同时下发一张 `ra-OccasionList`（车位清单）。物理层拿着自己刚才挑中的那个最亮探照灯（csi-RS），去这张清单里一查，立刻就能得到一组专门为这束探照灯绑定的合法发车位（RO）。

### ⏱️ 第三关：车位编号的“大轮回清零” (Reset 法则)
* **痛点**：`ra-OccasionList` 里的车位可能是写着“去第 5 号车位发”。但随着时间无尽流逝，车位总数会涨到成千上万，怎么定位“第 5 号”？
* **周期重置**：协议再次祭出我们熟悉的“清零法则”。但请注意，这里重置的边界不是普通的小循环（Mapping Cycle），而是那个**宏大的“大循环模式周期（Association pattern period，最多160ms）”**！
* **现实意义**：每当一个超级大轮回跑完，所有基于 CSI-RS 的车位编号立刻从 0 开始重新数。这保证了高层下发的 `ra-OccasionList` 永远能用极小的数字，精准指向大周期里的特定车位。

### 🔄 基于 CSI-RS 的车位锁定流水线
```text
 高层 (MAC) 主动发起随机接入，并下发高级探测名单 (csirs-ResourceList)
    │
    ├─ 动作 1: 物理层放弃测 SSB，转而去精细测量 CSI-RS 探照灯。
    │
    ├─ 动作 2: 挑出最亮的一束探照灯 (Selected CSI-RS)。
    │
    ├─ 动作 3: 翻开高层给的车位清单 (ra-OccasionList)，
    │          查出这个探照灯对应的专属发车位编号 (比如第 5 号车位)。
    │
    ├─ 动作 4: 监控时间轴，一旦进入新的大轮回周期 (Association pattern period)，
    │          立刻将车位编号清零！
    │
    └─► 动作 5: 在当前大周期内，精准数到【第 5 号】车位，点火发射 PRACH！🚀
```

### 💡 章节硬核总结

**这段协议补充了随机接入中“高精度波束管理”场景下的资源映射逻辑。当终端处于连接态（Connected Mode）且进行波束恢复或无竞争切换时，基于 SSB 的宽波束测距往往不够精确，网络会转而配置 `csirs-ResourceList` 引导终端测量更精细的 CSI-RS。由于 CSI-RS 不具备 SSB 那种底层的、全网硬编码的 4-Step 时频映射张量，协议通过 RRC 层参数 `ra-OccasionList` 执行了一次“显式资源绑定（Explicit Binding）”。为了防止无限时间轴导致的显式索引溢出，协议将该 List 的物理寻址重置周期，死死锚定在了宏观的 Association pattern period 上。这一设计在极速高级的精准波束切换与底层物理周期的稳定性之间，找到了完美的契合点。**

## 📖 协议原文拆解 (二十一)：FDD/SUL 的“上行专属特权” (Valid PRACH Occasion 判定)

> **协议原文**
> For paired spectrum or supplementary uplink band all PRACH occasions are valid.

### 📚 增量术语表
* **Paired spectrum**：成对频谱，通常指 FDD（频分双工）。**一句话解释**：上行和下行各占一条专属的物理马路，井水不犯河水。
* **Supplementary uplink band (SUL)**：补充上行频段。**一句话解释**：为了弥补 5G 高频段上行覆盖不足，基站给手机额外开辟的一条低频段“专属上行慢车道”。
* **Valid PRACH occasion**：合法（有效）的随机接入时机。**一句话解释**：手机真正可以用来发射接入信号的“安全合法车位”。

### 🚪 第一关：频谱模式的“先天优势” (FDD 与 SUL)
* **背景痛点（TDD的烦恼）**：在 5G 最常用的 TDD（时分双工）模式下，上行（手机发）和下行（基站发）是**共享同一条物理频段**的。这意味着手机在发射 PRACH 之前，必须小心翼翼地检查：*“这个车位（RO）会不会正好撞上基站要下发数据的下行时间（DL symbol）？”* 如果撞了，这个车位就是**无效的（Invalid）**，绝对不能用。
* **FDD / SUL 的降维打击**：但是，如果手机运行在**成对频谱 (Paired spectrum)** 或者专属的**补充上行频段 (SUL)**，情况就完全不同了。在这两种模式下，上行传输有一条绝对独立的专属物理车道，永远不可能和下行信号发生物理维度的“相撞”。

### 🎯 第二关：直接“全绿灯”放行 (All are valid)
* **判定逻辑**：因为没有了“上下行同频相撞”的顾虑，物理层在做车位有效性判断时极其爽快。
* **一刀切法则**：只要网络通过高层参数（比如 `RACH-ConfigCommon`）分配给你的那些初始 PRACH Occasions（发车位），**不用做任何交叉比对和冲突检测，100% 全部视为合法有效（Valid）**。

### 🔄 频谱判定车位有效性流水线
```text
 物理层拿到了一张初始的 PRACH Occasion (RO) 配置表
    │
    ├─ 灵魂拷问：我们现在跑在什么频谱上？
    │
    ├─ 情况 A：跑在 TDD (Unpaired spectrum) 上？
    │    └──► (需启动复杂的碰撞检测法则，踢出和下行冲突的 RO... 详见后续协议)
    │
    └─► 情况 B：跑在 FDD (Paired) 或 SUL 上？
         │
         └──► 动作：不用查了！直接盖章认定：
              ✅ 配置表里的【所有车位】通通合法可用 (Valid)！
```

### 💡 章节硬核总结

**这段协议极其简短，但一语道破了 FDD（频分双工）和 SUL（补充上行）架构在随机接入时的时序判定特权。在 TDD 架构下，由于上下行时分复用同一频段，物理层必须执行严格的有效性剔除算法，防止终端的上行盲发干扰基站的下行广播。而 FDD 和 SUL 天然具备物理隔离的独立上行频带，因此从根源上消除了半双工带来的时域冲撞风险。协议在这里做了一个“极简豁免”：在 FDD/SUL 下，高层配置的所有初始 PRACH 时机（RO）天然全部合法（Valid），直接省去了复杂的时分冲突校验。**

## 📖 协议原文拆解 (二十二)：TDD 模式下的“车位防撞指南” (Unpaired Spectrum Valid RO 判定)

> **协议原文**
> For unpaired spectrum,
> - if a UE is not provided tdd-UL-DL-ConfigurationCommon for a cell, a PRACH occasion for the cell in a PRACH slot is valid if it does not precede a SS/PBCH block in the PRACH slot and starts at least $N_{gap}$ symbols after a last SS/PBCH block reception symbol, where $N_{gap}$ is provided in Table 8.1-2 and, if channelAccessMode = "semiStatic" is provided, does not overlap with a set of consecutive symbols before the start of a next channel occupancy time where the UE does not transmit [15, TS 37.213]
> - the candidate SS/PBCH block index of the SS/PBCH block corresponds to the SS/PBCH block index provided by ssb-PositionsInBurst in SIB1 or in ServingCellConfigCommon or in SSB-MTC-AdditionalPCI corresponding to the cell, as described in clause 4.1
> - for each of the candidate cells configured by LTM-Config, if a UE is not provided ltm-TDD-UL-DL-ConfigurationCommon, a PRACH occasion in a PRACH slot for a candidate cell is valid if it does not precede a SS/PBCH block in the PRACH slot and starts at least $N_{gap}$ symbols after a last SS/PBCH block reception symbol, where $N_{gap}$ is provided in Table 8.1-2
> - the SS/PBCH block index of the SS/PBCH block corresponds to the SS/PBCH block index provided by ssb-PositionsInBurst in LTM-SSB-Config corresponding to the candidate cell
> - if a UE is provided tdd-UL-DL-ConfigurationCommon for a cell, a PRACH occasion for the cell in a PRACH slot is valid if
> - it is within UL symbols, or
> - it does not precede a SS/PBCH block in the PRACH slot and starts at least $N_{gap}$ symbols after a last downlink symbol and at least $N_{gap}$ symbols after a last SS/PBCH block symbol, where $N_{gap}$ is provided in Table 8.1-2, and if channelAccessMode = "semiStatic" is provided, does not overlap with a set of consecutive symbols before the start of a next channel occupancy time where there shall not be any transmissions, as described in [15, TS 37.213]
> - the candidate SS/PBCH block index of the SS/PBCH block corresponds to the SS/PBCH block index provided by ssb-PositionsInBurst in SIB1 or in ServingCellConfigCommon or in SSB-MTC-AdditionalPCI corresponding to the cell, as described in clause 4.1
> - for each of the candidate cells configured by LTM-config, if a UE is provided ltm-tdd-UL-DL-ConfigurationCommon, a PRACH occasion in a PRACH slot for a candidate cell is valid if
> - it is within UL symbols, or
> - it does not precede a SS/PBCH block in the PRACH slot and starts at least $N_{gap}$ symbols after a last downlink symbol and at least $N_{gap}$ symbols after a last SS/PBCH block symbol, where $N_{gap}$ is provided in Table 8.1-2
> - the SS/PBCH block index of the SS/PBCH block corresponds to the SS/PBCH block index provided by ssb-PositionsInBurst in LTM-SSB-Config corresponding to the candidate cell.

### 📚 增量术语表
* **Unpaired spectrum**：非成对频谱，通常指 TDD（时分双工）。**一句话解释**：手机和基站共用同一条马路，只能通过“错峰出行”（分时）来避免相撞。
* **tdd-UL-DL-ConfigurationCommon**：TDD 上下行公共配置。**一句话解释**：基站广播的一张“作息表”，明确规定了哪些时间是下行（基站发）、哪些是上行（手机发）、哪些是灵活时间。
* **$N_{gap}$**：保护间隔符号数。**一句话解释**：在接收下行信号（如 SSB）和发送上行信号（如 PRACH）之间，必须留出一段“喘息时间”，让手机的射频硬件完成收发状态的切换。
* **LTM-Config / ltm-TDD-UL-DL-ConfigurationCommon**：LTM 专属配置。**一句话解释**：针对 5G-Advanced 引入的 LTM（极速切换）特性，给目标备选基站（Candidate cell）准备的“专属作息表”。

### 🚪 背景痛点：TDD 模式下的“幽灵撞车”
* **冲突危机**：在 TDD（非成对频谱）下，基站给手机分配的初始 PRACH 车位（RO）在物理时间上是固定的。但是，基站可能也会在同样的时间下发 SSB（同步信号块）或者纯下行数据。
* **物理防撞法则**：为了防止手机“逆行”发射 PRACH 导致整个网络受到干扰，物理层必须执行严格的**“查表排雷（碰撞检测）”**。只有安全通过排雷程序的 RO，才能被称为合法有效的发车位（**Valid PRACH occasion**）。

---

### 🛡️ 查表排雷第一关：UE 没有拿到“作息表”怎么办？
*(对应：if a UE is not provided tdd-UL-DL-ConfigurationCommon)*

有时候手机刚开机，还没来得及读懂基站完整的“TDD 作息表”。此时，手机只能靠“眼睛看（盲猜）”：
* **避开大灯塔（SSB）**：手机知道基站的 SSB 是必定会发的（位置由 `ssb-PositionsInBurst` 指示）。
* **合法条件**：
  1. 这个 RO **绝对不能排在 SSB 的前面**（为了保护基站下发后续信号的连续性）。
  2. 这个 RO **必须在 SSB 结束之后，再向后退至少 $N_{gap}$ 个符号**（给硬件留出收发转换的喘息时间）。

---

### 🛡️ 查表排雷第二关：UE 拿到了完整的“作息表”
*(对应：if a UE is provided tdd-UL-DL-ConfigurationCommon)*

这是最常见的情况。手机手里有了一张清晰的 `tdd-UL-DL-ConfigurationCommon`，上面清清楚楚写着每个时间槽是 U（上行）、D（下行）还是 F（灵活）。
手机会用极其严格的标准来审视这个车位（满足以下**任意一条**即算 Valid）：
* **条件 A（完美躺平）**：这个 RO **完完全全落在了被明确标记为 U（上行）的符号里**。此时绝对安全，直接 Valid。
* **条件 B（夹缝求生）**：如果 RO 落在了 F（灵活符号）里，那必须同时满足以下极度苛刻的防撞距离：
  1. 不能排在 SSB 前面。
  2. 距离上一个明确的**下行符号 (last downlink symbol)**，必须至少隔开 $N_{gap}$。
  3. 距离上一个 **SSB 符号 (last SS/PBCH block symbol)**，也必须至少隔开 $N_{gap}$。

---

### 🚀 关于 LTM (极速切换) 备胎基站的“照猫画虎”
协议中花了将近一半的篇幅在描述 `for each of the candidate cells configured by LTM-Config...`。
* **核心思想**：一模一样！没有任何新逻辑。
* **唯一区别**：原来查的是服务小区的表（`tdd-UL-DL-ConfigurationCommon`），现在查的是目标备选小区的专属表（`ltm-TDD-UL-DL-ConfigurationCommon`）。原来查服务小区的 SSB 位置，现在查备选小区的 SSB 位置（`LTM-SSB-Config`）。仅此而已。

### 🔄 TDD 车位有效性 (Valid RO) 排雷流水线
```text
 物理层拿到一个待判定的 PRACH Occasion (RO)
    │
    ├─ 灵魂拷问：高层给我发 TDD 作息表 (tdd-UL-DL-Config) 了吗？
    │
    ├─► 没发 (Not provided)！
    │      │
    │      └──► 这个 RO 躲开 SSB 了吗？且距离 SSB 结束 >= Ngap 吗？
    │             ├─ 是 ──► ✅ Valid (合法)
    │             └─ 否 ──► ❌ Invalid (作废)
    │
    └─► 发了 (Provided)！
           │
           ├─ 这个 RO 全都在 U (上行) 符号里吗？
           │    └──► 是 ──► ✅ Valid (合法)
           │
           └─ 如果在 F (灵活) 符号里：
                └──► 距离【上一个下行符号】和【上一个 SSB】都 >= Ngap 吗？且不排在 SSB 前面吗？
                       ├─ 满足所有 ──► ✅ Valid (合法)
                       └─ 有一不满足 ──► ❌ Invalid (作废)
```

### 💡 章节硬核总结

**这段协议定义了在 TDD（时分双工 / Unpaired Spectrum）架构下，最核心的物理层“半双工冲突解决机制”。因为 TDD 上下行共用频带，终端必须在发送上行接入前导码（PRACH）前，执行严格的“Valid RO”过滤算法。协议以终端是否配置了静态 TDD 上下行配比（`tdd-UL-DL-ConfigurationCommon`）为分水岭：如果没有配置，终端退化为“基于 SSB 的避让策略”，强行在 SSB 之后留出 $N_{gap}$ 收发转换余量；如果有配置，终端首选被静态划定为全上行（UL）的符号，若落在灵活符号（Flexible）内，则必须与任何已知的下行符号及 SSB 强行拉开至少 $N_{gap}$ 的安全距离。该机制同样被无缝平移到了 5G-A 最新的 LTM（层 1/2 触发移动性）场景中，供终端提前探明候选目标小区（Candidate cell）的安全发射时机。**

## 📖 协议原文拆解 (二十三)：极限操作下的“零间隙”特权 (Preamble Format B4 的 $N_{gap}$ 豁免)

> **协议原文**
> For preamble format B4 [4, TS 38.211], $N_{gap}=0$.

### 📚 增量术语表
* **Preamble format B4**：随机接入前导码格式 B4。**一句话解释**：5G 武器库里定义的一种特殊“暗号款式”，它是一款为了极限压榨时间而设计的极短款前导码。

### 🚪 第一关：常规防撞法则的“漏网之鱼”
* **场景倒带**：在上一段我们学到，在 TDD 模式下，手机从接收下行（比如看基站的 SSB）切换到发射上行（发 PRACH 暗号）时，中间必须强行留出一个保护间隔（$N_{gap}$），给手机内部的射频天线一点“喘息和切换”的时间。
* **物理极限挑战**：通常情况下，$N_{gap}$ 都是大于 0 的。但是 5G 协议对有一种特殊的接入前导码网开一面，那就是 **Format B4**。

### 🎯 第二关：Format B4 为什么能享用“零间隙”特权？
* **内卷的物理层设计**：如果你去查阅定义物理层波形的 TS 38.211 规范，你会发现 Format B4 本身的设计极其微小和特殊。它通常被安排在时隙的最后，或者其波形结构内部已经精妙地消化了收发转换的开销。
* **结果**：对于这款专门设计的短前导码，协议直接大手一挥：**不需要额外预留 $N_{gap}$！** （即 $N_{gap} = 0$）。手机在判定车位（RO）是否有效时，只要它不和 SSB 重叠，哪怕紧紧贴在下行符号的屁股后面，这个车位也是合法的（Valid）。

### 🔄 现实类比
* **常规前导码**：就像开着一辆普通汽车，从前进挡（接收）切到倒车挡（发送）时，必须彻底踩死刹车停稳（$N_{gap} > 0$），否则会毁坏变速箱。
* **Format B4**：就像一辆经过特殊改装的“特技赛车”。在使用这辆车时，驾驶员拥有了无需缓冲、瞬间切挡的特权（$N_{gap} = 0$），极限压榨了赛道的时间效率。

### 💡 章节硬核总结

**这是 TDD 半双工碰撞检测法则中的一个极度微小但硬核的特例。常规的 PRACH Occasion 有效性检测必须在下行符号后强制扣除 $N_{gap}$ 个符号的时域余量，以吸收 UE 的 Rx-to-Tx 射频切换时间。但协议规定，当采用 Preamble Format B4 这款特殊的极短前导码时，$N_{gap}$ 被强制覆写为 0。这意味着物理层在进行 Valid RO 查表排雷时，对于 B4 格式给予了时序上的最大宽容度，允许其以极限的“零间隙”紧贴在下行符号之后发射。**

## 📖 协议原文拆解 (二十四)：基站催接指令下的“备战时间公式” (PDCCH order to PRACH timing)

> **协议原文**
> If a random access procedure is initiated by a PDCCH order, the UE, if requested by higher layers, transmits a PRACH in the selected PRACH occasion, as described in [11, TS 38.321], for which a time between the last symbol of the PDCCH order reception and the first symbol of the PRACH transmission is larger than or equal to $N_{T,2} + \Delta_{BWPSwitching} + \Delta_{Delay} + T_{switch} + T_{SSB} + \Delta_{RF/BB preparation}$ msec, where
> - $N_{T,2}$ is a time duration of $N_2$ symbols corresponding to a PUSCH preparation time for UE processing capability 1 [6, TS 38.214] assuming $\mu$ corresponds to the smallest SCS configuration between the SCS configuration of the PDCCH order and the SCS configuration of the corresponding PRACH transmission
> - $\Delta_{BWPSwitching}=0$ if the active UL BWP does not change, or if a cell indicator field in the PDCCH order indicates a non-serving cell [5, TS 38.212], and $\Delta_{BWPSwitching}$ is a time duration of $T_{BWPswitchDelay}$ defined in [10, TS 38.133] otherwise
> - $\Delta_{Delay}=0.5$ msec for FR1 and $\Delta_{Delay}=0.25$ msec for FR2
> - $T_{switch}$ is a switching gap duration as defined in [6, TS 38.214]
> - $T_{SSB}=0$ if a cell indicator field in the PDCCH order indicates a serving cell or if cell indicator field is not present, and $T_{SSB}$ is defined in [10, TS 38.133] otherwise
> - $\Delta_{RF/BB preparation}=0$ if a cell indicator field in the PDCCH order indicates a serving cell or if cell indicator field is not present, and $\Delta_{RF/BB preparation}$ is defined in [10, TS 38.133] otherwise

### 📚 增量术语表
* **PDCCH order**：基站强制下发的“催接指令”。**一句话解释**：基站发现下行有数据要发给手机，但手机失联了（失去上行同步），于是基站发一条特定的控制消息，命令手机立刻发起随机接入。
* **$N_{T,2}$**：基带处理时间。**一句话解释**：手机芯片用来“思考、解包 PDCCH、打包生成 PRACH”所需的纯算力时间。
* **$\Delta_{BWPSwitching}$**：BWP 切换延时。**一句话解释**：如果基站命令你在另一个频率带宽（BWP）上发暗号，手机重新调谐收音机频段所需的时间。
* **$\Delta_{Delay}$**：兜底缓冲余量。**一句话解释**：硬件信号在模拟链路中传输的固有延迟（低频 FR1 给 0.5ms，高频 FR2 给 0.25ms）。
* **$T_{switch}$**：上下行切换间隙。**一句话解释**：前面提到的从“听（Rx）”切换到“说（Tx）”所需的硬件电路切换时间。
* **$T_{SSB}$ & $\Delta_{RF/BB preparation}$**：异频/异小区同步时间。**一句话解释**：如果基站是命令你去“别人家（非当前服务小区）”发暗号，手机需要额外花时间去寻找并同步别人家的 SSB 信号。

### 🚪 第一关：逼急了的基站与手忙脚乱的手机
* **场景倒带**：正常情况下，随机接入是手机主动发起的（比如我想上网）。但还有一种情况：手机很久没发上行数据，基站认为它走丢了。此时基站突然有下行数据要给它，基站就会通过 PDCCH 下发一道**强制命令 (PDCCH order)**：“别睡了，立刻给我回一个 PRACH 证明你还在！”
* **核心痛点**：手机上一秒刚收到这道圣旨，下一秒就能把 PRACH 射出去吗？绝对不可能。手机需要解码指令、可能要换挡（换频段）、还要重新对准波束。

### 🎯 第二关：物理极限的“最低备战时间”公式
为了保证手机不会因为反应太慢而错过基站的期望，协议在此祭出了一个极其严谨的**延迟下限公式**。
手机在挑选最近的 PRACH 车位（RO）时，必须保证：
**(选中的车位起点) - (收到圣旨的终点) $\ge$ 备战时间总和**。

这个“备战时间总和”就是由 6 个组件拼凑而成的六边形战士：
1. **脑力消耗 ($N_{T,2}$)**：解算指令、生成前导码的基础耗时。
2. **频率切换 ($\Delta_{BWPSwitching}$)**：如果在同一个车道（BWP不变），耗时为 0；如果被要求变道，就得加上调频延时。
3. **硬件电路固有延迟 ($\Delta_{Delay}$)**：死规定，FR1 0.5ms，FR2 0.25ms。
4. **收发转换 ($T_{switch}$)**：嘴巴和耳朵切换的时间。
5. **找新基站的代价 ($T_{SSB} + \Delta_{RF/BB preparation}$)**：如果在原基站发，耗时为 0；如果圣旨要求去另一个新基站（Non-serving cell）发，必须加上寻找新基站 SSB 并准备对应基带的时间。

### 🔄 备战时间计算流水线
```text
 手机收到 PDCCH order (圣旨)
    │
    ├─► 基础准备：加上算力时间 (N_T,2) + 固有电路延迟 (Δ_Delay) + 收发切换 (T_switch)
    │
    ├─► 灵魂拷问 1：需要换 BWP 频段吗？
    │      ├─ 是 ──► 加上换频段延时 (Δ_BWPSwitching)
    │      └─ 否 ──► 加 0
    │
    ├─► 灵魂拷问 2：是让我去另一个基站发吗？
    │      ├─ 是 ──► 加上找新基站 SSB 的时间 (T_SSB) + 异小区基带准备时间 (Δ_RF/BB)
    │      └─ 否 ──► 加 0
    │
    ▼
 计算出总耗时 T_total！
 物理层去查时刻表：只有那些距离圣旨结束 > T_total 的发车位 (RO)，我才来得及赶上！
```

### 💡 章节硬核总结

**这段协议精确量化了网络触发（Network-triggered）随机接入场景下的“UE 最小反应延迟（Processing Timeline）”。当终端收到 PDCCH order 强制唤醒时，由于其原本可能处于休眠、窄带或失步状态，物理层不能立即在下一个时隙盲目发射 PRACH。协议要求终端将基带处理耗时（$N_{T,2}$）、BWP 切换开销、硬件固有延迟（$\Delta_{Delay}$）、半双工收发切换（$T_{switch}$），以及（若是跨小区调度）目标小区的重同步与射频调谐开销（$T_{SSB}$, $\Delta_{RF/BB preparation}$）进行全量累加。只有时间起点晚于这个“综合缓冲延迟”的 PRACH Occasion，才是终端在物理极限上能够安全抵达并合法使用的有效发车位。**

## 📖 协议原文拆解 (二十五)：窄带长序列的“时间换算降级法则” (1.25/5kHz SCS 的 $N_2$ 映射)

> **协议原文**
> For a PRACH transmission using 1.25 kHz or 5 kHz SCS, the UE determines $N_2$ assuming SCS configuration $\mu=0$.

### 📚 增量术语表
* **1.25 kHz or 5 kHz SCS**：极窄子载波间隔。**一句话解释**：这是 5G 专门为了极远距离覆盖（比如海面、大草原）而保留的“长前导码”专用的子载波间隔，它的频带极窄，但在时间上拉得很长。
* **SCS configuration $\mu=0$**：子载波配置参数 $\mu=0$。**一句话解释**：5G 里最基础的 15 kHz 子载波间隔的代号。

### 🚪 第一关：长前导码的“参数真空”
* **场景倒带**：在刚刚学完的那个长长的“备战时间公式”里，计算基带算力耗时 $N_{T,2}$ 是基于符号数 $N_2$ 的。而 $N_2$ 的查表是需要输入 SCS 配置参数 $\mu$ 的。
* **物理痛点**：5G 核心数据信道的 $\mu$ 通常是 0 (15kHz), 1 (30kHz), 2 (60kHz) 等等。但是，如果手机要发射的是 1.25 kHz 或 5 kHz 这种**非主流的窄带长序列 PRACH**，它们根本就没有对应的 $\mu$ 值！手机拿着 1.25kHz 去查 $N_2$ 的表，会发现表里根本没有这一项，导致算不出备战时间。

### 🎯 第二关：强行对齐的“降级换算”
* **协议的打补丁**：为了解决这个参数真空，协议在这里给了一个非常简单粗暴的兜底规则。
* **对齐法则**：如果基站逼着你去发 1.25 kHz 或 5 kHz 这种慢吞吞的窄带长暗号，你别管什么找不到表了，**直接假装它是 15 kHz（即假设 $\mu=0$）去查表**！
* **结果**：通过强行套用 $\mu=0$ 的查表结果，手机就能顺利算出对应的符号数 $N_2$，从而完美闭环计算出基带处理所需的最小备战时间（$N_{T,2}$）。

### 💡 章节硬核总结

**这段话是前文网络触发（PDCCH order）随机接入延迟公式中的一个必不可少的补丁。当终端采用传统的长序列前导码（Formats 0/1/2/3），其对应的子载波间隔（SCS）为 1.25 kHz 或 5 kHz，这超出了 5G 标准数据信道定义的 $\mu \in \{0,1,2,3,4\}$ 框架。为了复用现有的 PUSCH 准备时间（$N_2$）查找表，协议规定：一旦遇到这两种超窄带 SCS，终端在计算基带处理延迟时，必须统一降级并映射到 $\mu=0$（即 15 kHz SCS）的参数基准上进行换算。这避免了协议中为极低 SCS 重新制定冗余时序表格的开销。**

## 📖 协议原文拆解 (二十六)：单射频链路下的“同频传输互斥法则” (Simultaneous Tx Constraints)

> **协议原文**
> For single cell operation or for operation with contiguous carrier aggregation in a same frequency band or for operation with non-contiguous carrier aggregation in a same frequency band if the UE is not provided with intraBandNC-PRACH-simulTx-r17, a UE
> - does not transmit PRACH and PUSCH/PUCCH/SRS in a same slot with respect to the smallest SCS configuration between the SCS configuration for the UL BWP with the PRACH and the SCS configuration for the UL BWP with the PUSCH/PUCCH/SRS transmissions
> - does not transmit PRACH and PUSCH/PUCCH/SRS when a first or last symbol of a PRACH transmission in a first slot is separated by less than $N$ symbols from the last or first symbol, respectively, of a PUSCH/PUCCH/SRS transmission in a second slot; for a PRACH transmission with $N_{preamble}^{rep}>1$ preamble repetitions, this applies to each preamble repetition
> - for a PRACH transmission with $N_{preamble}^{rep}>1$ preamble repetitions, if the UE does not indicate prach-Repetition, the UE does not transmit a first repetition of the PRACH and a second repetition of the PRACH when a first or last symbol of the first repetition of the PRACH in a first slot is separated by less than $N$ symbols from the last or first symbol, respectively, of the second repetition of the PRACH in a second slot; otherwise, the UE transmits the first repetition of the PRACH and the second repetition of the PRACH
> where $N=2$ for $\mu=0$ or $\mu=1$, $N=4$ for $\mu=2$ or $\mu=3$, $N=16$ for $\mu=5$, $N=32$ for $\mu=6$, and $\mu$ is the smallest SCS configuration between the SCS configuration for the UL BWP with the PRACH and the SCS configuration for the UL BWP with the PUSCH/PUCCH/SRS transmissions. For a PUSCH transmission with repetition Type B, this applies to each actual repetition for PUSCH transmission [6, TS 38.214].

### 📚 增量术语表
* **Contiguous / non-contiguous carrier aggregation**：连续 / 非连续载波聚合。**一句话解释**：手机同时连着好几个载波（马路）上网。如果是同频段内的聚合，对手机射频前端硬件的压力很大。
* **intraBandNC-PRACH-simulTx-r17**：同频非连续并发发射能力。**一句话解释**：手机拍胸脯向基站保证：“我的硬件很牛，可以在同一个频段的不同载波上，一边发 PRACH 一边发别的数据！”（如果没有这个能力，就得乖乖遵守本节的互斥法则）。
* **PUSCH/PUCCH/SRS**：常规上行信号三剑客。**一句话解释**：分别是发数据的（PUSCH）、发控制信息的（PUCCH）和发探测信号的（SRS）。
* **$N_{preamble}^{rep}$**：PRACH 盲发重复次数。**一句话解释**：为了增加被基站听到的概率，手机把同一个暗号连着喊好几遍。

### 🚪 第一关：一嘴不能两用 (场景定义)
* **硬件瓶颈**：如果你只有一个小区（单天线），或者在同一个频段内做载波聚合，且**没有特殊的高级并发能力（未配置 simulTx-r17）**。那么，你的手机射频前端在物理上就是个“单通道的喉咙”。
* **法则前提**：既然只有一个喉咙，你就绝不能在同一时刻，既唱美声（发业务信号 PUSCH/PUCCH/SRS）又唱摇滚（发接入暗号 PRACH），两者必须严格错开。

### 🎯 第二关：同一时隙的绝对禁区 (Slot-level Restriction)
* **规则 1**：手机**绝对不允许**在同一个 Slot（时隙）内，同时发射 PRACH 和 常规上行信号（PUSCH/PUCCH/SRS）。
* **对齐标准**：如果两者的子载波间隔（SCS）不一样导致 Slot 长度不一样怎么办？协议规定：以**最小的那个 SCS**（即时间跨度最长的那个大 Slot）为准，在这个大 Slot 范围内，两者有我没他。

### 🛡️ 第三关：跨时隙的安全隔离带 ($N$ symbol gap)
* **痛点**：那如果 PRACH 和业务信号分别在两个相邻的 Slot 发，不冲突了吧？不行！因为射频功放（PA）在切换不同类型的信号时，需要时间去改变功率和滤波器状态。
* **规则 2**：即使跨了 Slot，PRACH 的边缘（开头或结尾）与 常规上行信号的边缘之间，**必须强行拉开至少 $N$ 个符号的安全隔离带**。
* **规则 3（连发 PRACH 的自我隔离）**：如果手机要连着发好几遍 PRACH（$N_{preamble}^{rep}>1$），两次 PRACH 之间也得留出 $N$ 个符号的隔离带！除非……手机提前上报了 `prach-Repetition` 能力（意思是“我的喉咙可以连续吼，不需要喝水喘息”），这才可以无缝连发。
* **隔离带大小 ($N$) 的动态计算**：
  - $\mu=0, 1$ (低频)：$N = 2$ 个符号
  - $\mu=2, 3$ (中频)：$N = 4$ 个符号
  - $\mu=5, 6$ (极高频毫米波)：$N = 16$ 或 $32$ 个符号。
  频率越高，符号时间越短，所以需要的绝对符号数量 $N$ 就要成倍增加，以保证物理时间上的绝对安全。

### 🔄 射频互斥与隔离带流水线
```text
 准备发射 PRACH，但发现旁边还有别的上行信号 (PUSCH/PUCCH/SRS)
    │
    ├─ 硬件查户口：是同频段单通道吗？有并发特权 (simulTx) 吗？
    │      └─ 没有特权！触发互斥法则！
    │
    ├─ 查时隙：两者在同一个大 Slot 里吗？
    │      └─ 是 ──► ❌ 绝对禁止！直接丢弃其中一个。
    │
    └─ 查边缘距离：两者跨 Slot 了，但首尾距离 < N 个符号吗？
           ├─ 是 (贴得太近) ──► ❌ 禁止发射！
           └─ 否 (拉开了距离) ──► ✅ 安全，可以先后发射。
```

### 💡 章节硬核总结

**这段协议详述了在受限的射频前端架构下（单小区或同频带 CA 且缺乏并发发射能力），物理层如何处理 PRACH 与其它常规上行物理信道（PUSCH/PUCCH/SRS）的资源冲突。核心逻辑是彻底的“时分复用（TDM）与隔离”。首先是在最粗的粒度上，禁止在同一个（基于最小 SCS 评估的）时隙内发生任何重叠。其次是在符号粒度上，强制要求 PRACH 和其它信道之间，甚至在连续重复发射的 PRACH 自身之间，必须插入取决于 SCS 的 $N$ 个符号作为“射频过渡带（RF Transition Gap）”。这确保了 UE 的功率放大器（PA）有足够的时间在不同的功率电平和波形要求之间安全切换，防止带外泄漏和信号畸变。**

## 📖 协议原文拆解 (二十七)：异小区测量的“收发互斥” (SSB-MTC-AdditionalPCI 约束)

> **协议原文**
> For a UE configured with SSB-MTC-AdditionalPCI, it does not transmit PRACH and receive SS/PBCH associated with different cells in a same slot.

### 📚 增量术语表
* **SSB-MTC-AdditionalPCI**：附加物理小区 ID 的 SSB 测量时间配置。**一句话解释**：基站给手机下发的一项任务：“除了盯着我（服务小区），你还要在特定的时间去测量一下隔壁老王（其他 PCI 的邻区）的 SSB 信号强度。”
* **SS/PBCH**：同步信号和物理广播信道块（即 SSB）。**一句话解释**：基站的心跳和身份广播，手机靠它来识别基站并与之同步。

### 🚪 第一关：一心不可二用的“跨小区操作”
* **场景倒带**：手机正连着当前的服务小区，突然服务小区说：“你去测一下隔壁基站的信号，看看是不是比我强。”于是手机领了任务（配置了 `SSB-MTC-AdditionalPCI`），准备把接收机（耳朵）调谐到隔壁基站的频率/方向去听 SSB。
* **冲突爆发**：就在同一个时间槽（Slot）内，手机的发射机（嘴巴）又接到了任务：“快向当前服务小区发射接入暗号（PRACH）！”
* **物理矛盾**：让一部普通的单天线/单收发链路的手机，在极短的一个 Slot 内，耳朵去听 A 小区，嘴巴去朝 B 小区喊话。这要求手机同时锁定两个不同小区的时钟、频率和波束方向，硬件上根本做不到。

### 🎯 第二关：简单粗暴的“互斥禁令”
* **一刀切法则**：为了保护手机单薄的射频器件不被这种“精神分裂”的要求搞崩溃，协议直接下达了禁令。
* **规则**：**绝对禁止！** 手机不允许在**同一个 Slot 内**，既发射 PRACH，又接收另一个小区的 SSB。
* **隐含动作**：遇到这种冲突时，手机必须做出取舍（通常是根据高层优先级的规定，放弃或者推迟其中一个动作），总之绝不能强行同时执行。

### 🔄 跨小区收发互斥流水线
```text
 手机进入一个新的 Slot (时隙)
    │
    ├─► 任务列表检查：
    │    1. 嘴巴有任务：需要发 PRACH 吗？
    │    2. 耳朵有任务：需要测隔壁小区的 SSB (SSB-MTC-AdditionalPCI) 吗？
    │
    ├─► 灵魂拷问：这两个任务是针对同一个基站吗？
    │    └─ 如果一个是针对基站 A，一个是针对基站 B (Different cells)
    │
    ▼
 🚨 触发保护机制 🚨
 协议断言：这种“听A喊B”的动作无法在同一个 Slot 内完成！
 强制阻止这两者同时发生 (Does not transmit PRACH and receive SS/PBCH)。
```

### 💡 章节硬核总结

**这段协议补充了 UE 在执行异小区测量（Inter-cell measurement）时的半双工与射频调谐约束。当终端被配置了 `SSB-MTC-AdditionalPCI` 以监测邻区 SSB 时，其接收机需要对准邻区的波束、频率与时序。此时，若终端同时被调度或触发向服务小区发射 PRACH，将导致严重的射频前端冲突（RX 锁相邻区 vs TX 锁相服务小区）。为规避这一硬件不可能完成的任务，协议在 Slot 级别强制规定了物理互斥（Mutually Exclusive）：绝对禁止在同一时隙内，跨小区并发执行 PRACH 发送与 SSB 接收。**
