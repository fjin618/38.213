# 8.3 PUSCH scheduled by RAR UL grant (由随机接入回执调度的上行数据传输)

当我们成功收到基站的响应（RAR），并从中抠出了那张极其珍贵的“上行通行证（RAR UL Grant）”后，手机终于获得了向基站正式开口说话的权利。本章节详细规定了：手机该如何破解这张通行证上的物理参数，并在正确的“跑道（PUSCH）”上，把包含自己真实身份的 Msg3 数据精准无误地发射出去。

---

## 📖 协议原文拆解 (一)：发射 Msg3 的跑道锁定（Active UL BWP 配置）

> **协议原文**
> An active UL BWP with SCS configuration $\mu$, as described in clause 12 and in [4, TS 38.211], for a PUSCH transmission scheduled by a RAR UL grant is indicated by higher layers.

### 📚 术语表
* **UL BWP (Uplink Bandwidth Part)**：上行带宽部分。**一句话解释**：5G 频谱就像一条极宽的超级高速公路（比如 100MHz 宽），但手机能力有限跑不了全宽，于是基站给它划出了一条专用的“子干道”（比如 20MHz 宽），这就叫 BWP。
* **Active UL BWP**：激活的上行带宽部分。**一句话解释**：手机虽然可能被分配了好几条备用跑道，但在任意时刻，它**只能且必须**在唯一一条处于“激活态”的跑道上发数据。
* **$\mu$ (Mu)**：SCS 的配置参数。**一句话解释**：这条跑道里划分的“单根车道宽度”（比如 $\mu=1$ 代表 30kHz 的车道宽度）。

### 🚪 核心机制：物理层不背锅，听高层指挥
* **场景倒带**：手机物理层刚刚成功解析了基站发来的 RAR UL Grant（通行证），准备开足马力发射 Msg3（也就是 PUSCH 传输）。
* **物理层的困惑**：通行证里写了要在“第 X 个频率格子”上发射。但问题是，这个“格子”是相对于哪条跑道（BWP）而言的呢？而且这条跑道的车道有多宽（SCS）？物理层自己是不知道的。
* **高层一锤定音**：协议在此处给出了最高指示——**“由高层（higher layers）来指示！”** 
* **现实意义**：物理层就是一个纯粹的“打工人”，不要自己瞎猜。发射 Msg3 时，到底用哪条“激活的上行跑道（Active UL BWP）”，以及用什么“车道宽度（SCS $\mu$）”，统统由手机的 MAC/RRC 层（高层）查阅系统配置后，直接下达给物理层执行。

### 🔄 Msg3 跑道配置流水线
```text
 高层 (MAC / RRC)
    │
    ├─ 1. 查询系统广播或专属配置，确定当前处于激活状态的跑道 (Active UL BWP)
    ├─ 2. 确认这条跑道的车道宽度 (SCS configuration μ)
    │
    ├─► 下达硬性指令给物理层：“就在这条跑道上，按这个宽度给我发！”
    │
 物理层 (PHY)
    │
    ├─ 接收高层指令，锁定频段和宽度
    ├─ 叠加 RAR UL Grant 里的微调资源位置
    │
    ▼
 在精确划定的 Active UL BWP 上，点火发射 Msg3 (PUSCH)！🚀
```
