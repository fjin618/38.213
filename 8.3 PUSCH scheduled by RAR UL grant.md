# 8.3 PUSCH scheduled by RAR UL grant (由随机接入回执调度的上行数据传输)

当我们成功收到基站的响应（RAR），并从中抠出了那张极其珍贵的“上行通行证（RAR UL Grant）”后，手机终于获得了向基站正式开口说话的权利。本章节详细规定了：手机该如何破解这张通行证上的物理参数，并在正确的“跑道（PUSCH）”上，把包含自己真实身份的 Msg3 数据精准无误地发射出去。

---

## 📖 协议原文拆解 (一)：发射 Msg3 的跑道锁定（Active UL BWP 配置）

> **协议原文**
> An active UL BWP with SCS configuration $\mu$, as described in clause 12 and in [4, TS 38.211], for a PUSCH transmission scheduled by a RAR UL grant is indicated by higher layers.

### 📚 术语表
* **UL BWP (Uplink Bandwidth Part)**：上行带宽部分。**一句话解释**：5G 频谱就像一条极宽的超级高速公路（比如 100MHz 宽），但手机能力有限跑不了全宽，于是基站给它划出了一条专用的“子干道”（比如 20MHz 宽），这就叫 BWP。
* **Active UL BWP**：激活的上行带宽部分。**一句话解释**：手机虽然可能被分配了好几条备用跑道，但在任意时刻，它**只能且必须**在唯一一条处于“激活态”的跑道上发数据。
* **$\mu$ (Mu)**：SCS 的配置参数。**一句话解释**：这条跑道里划分的“单根车道宽度”（比如 $\mu=1$ 代表 30kHz 的车道宽度）。

### 🚪 核心机制：物理层不背锅，听高层指挥
* **场景倒带**：手机物理层刚刚成功解析了基站发来的 RAR UL Grant（通行证），准备开足马力发射 Msg3（也就是 PUSCH 传输）。
* **物理层的困惑**：通行证里写了要在“第 X 个频率格子”上发射。但问题是，这个“格子”是相对于哪条跑道（BWP）而言的呢？而且这条跑道的车道有多宽（SCS）？物理层自己是不知道的。
* **高层一锤定音**：协议在此处给出了最高指示——**“由高层（higher layers）来指示！”** 
* **现实意义**：物理层就是一个纯粹的“打工人”，不要自己瞎猜。发射 Msg3 时，到底用哪条“激活的上行跑道（Active UL BWP）”，以及用什么“车道宽度（SCS $\mu$）”，统统由手机的 MAC/RRC 层（高层）查阅系统配置后，直接下达给物理层执行。

### 🔄 Msg3 跑道配置流水线
```text
 高层 (MAC / RRC)
    │
    ├─ 1. 查询系统广播或专属配置，确定当前处于激活状态的跑道 (Active UL BWP)
    ├─ 2. 确认这条跑道的车道宽度 (SCS configuration μ)
    │
    ├─► 下达硬性指令给物理层：“就在这条跑道上，按这个宽度给我发！”
    │
 物理层 (PHY)
    │
    ├─ 接收高层指令，锁定频段和宽度
    ├─ 叠加 RAR UL Grant 里的微调资源位置
    │
    ▼
 在精确划定的 Active UL BWP 上，点火发射 Msg3 (PUSCH)！🚀
```

## 📖 协议原文拆解 (二)：Msg3 频域资源的“基准跑道”选择

> **协议原文**
> If useInterlacePUCCH-PUSCH is not provided by BWP-UplinkCommon and BWP-UplinkDedicated, for determining the frequency domain resource allocation for the PUSCH transmission within the active UL BWP
> - if the active UL BWP and the initial UL BWP have same SCS and same CP length and the active UL BWP includes all RBs of the initial UL BWP, or the active UL BWP is the initial UL BWP, the initial UL BWP is used
> - else, the RB numbering starts from the first RB of the active UL BWP and the maximum number of RBs for frequency domain resource allocation equals the number of RBs in the initial UL BWP

### 📚 增量术语表
* **useInterlacePUCCH-PUSCH**：交织结构配置。**一句话解释**：非授权频段（如 Wi-Fi 频段）专用的一种“梳子状”频谱占用方式。如果没有提供这个配置，说明现在是在正规的授权频段上“连续发包”。
* **RB (Resource Block)**：资源块。**一句话解释**：频域资源的基本单位，就像是一条跑道上划分好的一个个“停车位/格子”。
* **initial UL BWP**：初始上行带宽。**一句话解释**：手机刚开机、最穷的时候，全网所有手机默认共用的“公共起步小跑道”。
* **active UL BWP**：激活的上行带宽。**一句话解释**：手机当前被分配并正在使用的“当前跑道”（可能比初始跑道宽得多）。

### 🚪 第一关：前置条件（排除“梳子状”发包）
* **场景前提**：`If useInterlacePUCCH-PUSCH is not provided...`，协议首先把非授权频段那种“梳子状占道”的特殊情况排除掉。我们现在讨论的是最常规的、挨个格子数资源的正常频段。

### 📏 第二关：坐标系对齐之战（Initial vs Active）
* **核心痛点**：RAR UL Grant 里给出的频率位置（比如：请在第 10 到第 20 个格子发 Msg3），由于尺寸有限，它**只能基于一个比较窄的跑道（即初始跑道 initial UL BWP）的尺子来刻度**。
* **物理层的困境**：但是，高层刚刚在上一段协议里命令手机：“你现在必须在激活跑道（active UL BWP）上发！” 如果当前的“激活跑道”和“初始跑道”不在同一个位置，或者大小不一样，物理层到底该拿哪把尺子来数“第 10 个格子”？

协议针对这个问题，给出了极其缜密的两种“套尺子”方案：

#### 方案 A：直接借用“初始跑道”的尺子（完美兼容时）
* **触发条件**：当前跑道（Active）和初始跑道（Initial）的“车道宽度（SCS）”一样、“安全车距（CP length）”也一样，并且当前跑道**完全包住了**初始跑道（或者干脆俩就是同一条）。
* **执行动作**：既然完美兼容，手机直接在**初始跑道（Initial UL BWP）**的坐标系里去数格子。比如初始跑道的第 10 个格子，就是物理层发射 Msg3 的精确位置。

#### 方案 B：重塑坐标系（不兼容时，强行平移）
* **触发条件**：当前跑道和初始跑道长得不一样（比如不在同一个位置）。
* **执行动作（强行平移）**：
  1. **起点归零**：既然套不上，物理层就以**当前跑道（Active UL BWP）的最左边（第一个 RB）作为绝对起点（也就是第 0 号格子）**，重新开始数数。
  2. **尺子长度限制**：为了防止算出来的格子超过了 RAR UL Grant 能表达的上限，协议强制规定：你在当前跑道上往右边数格子时，**总数绝不能超过初始跑道（Initial UL BWP）的总格子数**。

### 🔄 频域坐标系对齐流水线
```text
 物理层拿着 RAR UL Grant，准备在当前跑道 (Active UL BWP) 上划定发包区域
    │
    ├─ 问自己：当前跑道和初始跑道 (Initial UL BWP) “八字合不合”？
    │  (SCS一致吗？CP一样吗？有没有完全包容？)
    │
    ├──► 【完全合拍 (方案 A)】
    │         │
    │         └─► 动作: 直接把初始跑道当基准，按初始跑道的格子来数。
    │             (在兼容区域内完美发射)
    │
    └──► 【不合拍 (方案 B)】
              │
              ├─► 动作 1 (平移起点): 直接跑到当前跑道的最左边缘，把这里当成“起点 0”！
              └─► 动作 2 (截断长度): 拿着 RAR 的数字开始往右数，但最大允许的格子数，
                                     被死死限制在初始跑道的总容量之内！
```

## 📖 协议原文拆解 (三)：频域坐标位数的“削足适履” (截断与补零)

> **协议原文**
> The frequency domain resource allocation is by uplink resource allocation type 1 [6, TS 38.214]. For an initial UL BWP size of $N_{BWP}^{size}$ RBs, a UE processes the frequency domain resource assignment field as follows
> - if $N_{BWP}^{size} \le 180$, or for operation with shared spectrum channel access in FR1 or for FR2-2 when ChannelAccessMode2-r17 is provided if $N_{BWP}^{size} \le 90$
> - truncate the frequency domain resource assignment field to its $\lceil \log_2(N_{BWP}^{size} \cdot (N_{BWP}^{size}+1)/2) \rceil$ least significant bits...
> - else
> - insert $\lceil \log_2(N_{BWP}^{size} \cdot (N_{BWP}^{size}+1)/2) \rceil - 14$ most significant bits... with value set to '0' after the $N_{UL,hop}$ bits...

### 📚 增量术语表
* **Resource allocation type 1**：上行资源分配类型1。**一句话解释**：给手机分配一段“连续不断”的频域方格（比如从第 10 格开始，连着发 5 格数据）。
* **$N_{BWP}^{size}$**：初始跑道尺寸。**一句话解释**：这条跑道上一共有多少个“停车位（RB格子）”。
* **Truncate**：截断。**一句话解释**：从一串二进制数字里，把左边（高位）多余的比特硬生生砍掉，只留右边。
* **Insert '0' / Expand**：补零/扩展。**一句话解释**：位数不够用时，在数字中间强行塞入一串 0 来凑数。
* **$\lceil \log_2(N \cdot (N+1)/2) \rceil$**：寻址比特公式。**一句话解释**：物理学界的一个死公式，用来计算“在长度为 N 的跑道上任意框选一段连续区域，一共需要多少个二进制比特位才能表示得清楚”。

### 🚪 核心痛点：固定长度的“坐标纸”遇上千变万化的“大跑道”
* **痛点溯源**：在前面的 8.2 章节里，RAR UL Grant 里的“频域分配字段”长度是**死板固定**的（普通频段 14 个 bit，非授权频段 12 个 bit）。
* **矛盾爆发**：但是大自然（基站配置的跑道 $N_{BWP}^{size}$）是千变万化的。跑道很窄时，根本用不完 14 个 bit；但如果 5G 跑道极其宽阔（大于 180 个格子），14 个 bit 就彻底不够用了，根本无法指明跑道最右边的区域。
* **物理层动作**：必须对这 14 个 bit 进行“削足适履”的变形手术！

### ✂️ 分支 A：跑道比较小（位数太多 ──► 截断！）
* **触发条件**：跑道较窄，总格子数 $\le 180$（非授权频段 $\le 90$）。
* **执行动作（Truncate）**：既然跑道小，其实只需要很少的 bit 就能指明位置（比如只需要 10 个 bit）。物理层就会极其冷酷地**把那固定的 14 bit 坐标纸的左边 4 个头直接砍掉（截断）**，只保留右边最核心的 10 个最低有效位（LSB）。

### ➕ 分支 B：跑道非常大（位数不够 ──► 强行补零！）
* **触发条件**：超宽跑道，总格子数 $> 180$。
* **执行动作（Insert '0'）**：14 个 bit 已经不够用了，公式算出来可能需要 16 个 bit 才能覆盖整条大跑道。怎么办？
  * 物理层会在跳频标志位（$N_{UL,hop}$）的后面，**强行塞入一串 0（比如塞入 2 个 0）**，硬生生把 14 bit 的坐标纸拉长到 16 bit！
* **深层物理意义（极其绝妙的代价）**：既然高位是你自己强行补的 `0`，这意味着什么？意味着无论低位的 14 个 bit 怎么变化，最终算出来的发包位置**绝对不可能落在跑道的远端**（因为高位永远是 0）。这也是协议在做资源妥协：**当跑道极大时，为了节省控制信道的开销，随机接入的 Msg3 只能被限制在跑道前半段的一个局部区域内发射。**

### 🔄 频域位数的“削足适履”流水线
```text
 物理层拿着固定的 14-bit 频域坐标纸，看向眼前的跑道尺寸 (N_BWP_size)
    │
    ├─ 计算公式：这条跑道到底需要多少位 bit 才能寻址？(假设需要 X 位)
    │
    ├──► 【跑道较小，X < 14 (比如 X=10)】
    │         │
    │         └─► 动作: 截断 (Truncate)！
    │             [ 砍掉的4位 ] [ 剩下的核心 10 位 ]
    │             (完美对齐，去发射)
    │
    └──► 【跑道极大，X > 14 (比如 X=16)】
              │
              └─► 动作: 补零拉伸 (Expand)！
                  [跳频位] + [ 强行塞入 2个 0 ] + [ 原来的 14 位 ]
                  (虽然总长度够了，但因为高位被0锁死，Msg3 只能在跑道前半截发射)
```

## 📖 协议原文拆解 (四)：非授权频段的“梳子状”占道法则 (交织资源分配)

> **协议原文**
> If useInterlacePUCCH-PUSCH is provided by BWP-UplinkCommon or BWP-UplinkDedicated, the frequency domain resource allocation is by uplink resource allocation type 2 [6, TS 38.214]. A UE processes the frequency domain resource assignment field as follows
> - truncate the frequency domain resource assignment field to the $X=6$ LSBs if $\mu=0$, or to the $X=5$ LSBs if $\mu=1$
> - for interlace allocation of a PUSCH transmission...
> - for RB set allocation of a PUSCH transmission, the RB set of the active UL BWP is the RB set of the PRACH transmission associated with the RAR UL grant...

### 📚 增量术语表
* **useInterlacePUCCH-PUSCH**：交织结构配置。**一句话解释**：告诉手机：“现在咱们在 Wi-Fi 的地盘（非授权频段）上抢道，你发数据不能挤在一坨，必须像‘梳子’一样隔几个频率发一点，把整个大马路都占住！”
* **Resource allocation type 2**：上行资源分配类型2。**一句话解释**：与类型 1（连续方格）对应，类型 2 专门指代刚才说的“梳子状（交织）”非连续方格分配方式。
* **Interlace**：交织。**一句话解释**：梳子上的一根根“梳齿”。
* **RB set**：资源块集合。**一句话解释**：在极宽的非授权频段上，为了方便抢道，被人为切分成的一块块“大局域”。

### 🚪 第一关：启动“梳子状”占道模式
* **场景倒带**：手机所在的当前频段是“公共大马路（非授权频段 NR-U）”。为了防止路边的 Wi-Fi 趁虚而入，法规（OCB要求）强制规定：手机发信号不能只占路边一小块，必须“撒网式”地铺满整条马路。
* **执行动作**：一旦高层配置了 `useInterlace`，物理层立刻把 RAR UL Grant 里的资源分配方式，强制理解为 **Type 2（梳子状交织模式）**。不再是一格一格连续数，而是一把一把地挑“梳子”。

### ✂️ 第二关：按车道宽度（SCS）精准截断
* **痛点**：在非授权频段，15kHz（$\mu=0$）的频段被划分为 10 把梳子；30kHz（$\mu=1$）的频段被划分为 5 把梳子。手机到底该用哪几把梳子呢？RAR 里的频域字段需要被重新切割。
* **执行动作（Truncate）**：
  * 如果你在 15kHz 车道（$\mu=0$），从那串比特流里**砍得只剩最后 6 位（$X=6$ LSBs）**，这 6 位数字足够用来挑选那 10 把梳子的组合。
  * 如果你在 30kHz 车道（$\mu=1$），从比特流里**砍得只剩最后 5 位（$X=5$ LSBs）**，这 5 位数字用来挑选那 5 把梳子的组合。

### 🔗 第三关：大局域的“子承父业”（RB Set 绑定）
* **痛点**：挑好了梳子，但是由于非授权频段太宽了（比如有 4 个大局域 RB Set），这把梳子到底该插在哪个大局域里发呢？
* **铁律：原路返回（继承 PRACH）**：
  * 协议规定：**绝不瞎猜，直接沿用刚才发暗号（PRACH）时所在的那个大局域（RB set）！**
  * 刚才你是在 2 号局域发暗号被基站听到的，现在拿到通行证发 Msg3 数据时，你的梳子也必须且只能插在 2 号局域里发射。
* **防错补丁**：协议最后补了一句，如果没有特殊配置频带内部的隔离带（intraCellGuardBandsUL-List），手机就默认这个局域是完完整整的，不含缝隙。

### 🔄 非授权频段 (Type 2 交织) 频域分配流水线
```text
 物理层确认进入【非授权频段 (Type 2 梳子状交织模式)】
    │
    ├─ 问自己：我现在车道宽度 (SCS) 是多少？
    │
    ├──► 【μ = 0 (15kHz 车道)】 ──► 动作: 截断保留最后 6位 比特，去挑梳子！
    │
    └──► 【μ = 1 (30kHz 车道)】 ──► 动作: 截断保留最后 5位 比特，去挑梳子！
             │
             ▼ (挑好了具体的梳子 / Interlace)
             │
             ├─ 问自己：我这把梳子该插在哪个大局域 (RB Set) 里发射？
             │
             └─► 回忆：我刚才发 PRACH 暗号是在哪个大局域？(假设是 RB Set #2)
                 │
                 ▼
          【发射绑定】: 在 RB Set #2 里，插上刚才挑好的梳子，发射 Msg3！
          (完美实现了“大局域原路返回，小齿轮铺满占道”的 NR-U 生存法则)
```

## 📖 协议原文拆解 (五)：手机省电与信号增强的秘密（是否开启“变换预编码”）

> **协议原文**
> A UE determines whether or not to apply transform precoding as described in [6, TS 38.214].

### 📚 增量术语表
* **Transform Precoding**：变换预编码。**一句话解释**：通信界的一个“信号魔法”。开启它，手机发射的信号波形就会从“起伏不定的海浪（CP-OFDM）”变成“平稳流淌的溪流（DFT-s-OFDM）”，从而让手机发射器不用那么费劲，极其省电且能射得更远。

### 🚪 核心机制：物理层“变身”的跨协议授权
* **痛点溯源**：在发射 Msg3 数据（PUSCH）时，手机的发射功率非常宝贵。如果手机在信号边缘（离基站很远），它发射的标准 5G 信号（CP-OFDM）因为峰值太高，容易导致手机的功放（PA）硬件“破音/过载”，从而导致信号发射失败。
* **物理层魔法（开启变换预编码）**：如果开启这个魔法（也就是切成了 DFT-s-OFDM 波形），信号的高低起伏会被削平，手机就可以把音量开到最大（满功率发射），一举把 Msg3 轰给远处的基站。
* **踢皮球法则（指向 TS 38.214）**：那么，此时此刻手机到底要不要开启这个省电魔法呢？TS 38.213（本协议）极其干脆地再次当了“甩手掌柜”——**“别问我，去查 TS 38.214 协议（专门负责数据信道过程的协议），那里写了你怎么做决定！”** 

### 🔄 变换预编码决策流水线 (简图)
```text
 物理层准备发射 Msg3 (PUSCH)
    │
    ├─ 问自己：我需要开启“变换预编码 (Transform Precoding)”来省电/增强信号吗？
    │
    └─► 翻开另一本武功秘籍 [TS 38.214]：
          ├─ 如果查到 `msg3-transformPrecoder` 配置为开启 ──► 变身！(用 DFT-s-OFDM 发射)
          └─ 如果查不到或者配置为关闭 ─────────────────────► 照旧！(用标准的 CP-OFDM 发射)
```

## 📖 协议原文拆解 (六)：蛇形走位的跨度尺（跳频偏移量）

> **协议原文**
> For a PUSCH transmission with frequency hopping scheduled by RAR UL grant or for a Msg3 PUSCH retransmission, the frequency offset for the second hop [6, TS 38.214] is given in Table 8.3-1.

### 📚 增量术语表
* **Frequency hopping**：跳频。**一句话解释**：手机在发送数据时“蛇形走位”，发一半数据后立刻跳到另一个频率上接着发，用来躲避某个特定频率上的干扰（抗衰落）。
* **Second hop**：第二跳。**一句话解释**：蛇形走位的“后半段”。
* **Frequency offset**：频率偏移量。**一句话解释**：跳频的“跨度”。也就是第二跳距离第一跳隔了多少个频率格子（RB）。
* **Msg3 PUSCH retransmission**：Msg3重传。**一句话解释**：如果基站没听清你的第一遍 Msg3，让你重发一遍，这叫重传。

### 🚪 核心机制：查表找跨度
* **场景倒带**：物理层在解析 RAR UL Grant 时，发现“跳频标志位”被基站设置成了 `1`。这意味着基站下达了硬性命令：“这段路况不好，你发 Msg3 时必须开启蛇形走位！”
* **寻找跳跃点**：手机的第一跳（前半段数据）位置已经在分配的坐标纸里写清楚了。关键是发送后半段数据时，第二跳该往哪跳？
* **铁律：无脑查表**：协议在这里极其干脆：**“跨度（Frequency offset）是多少？别猜了，直接去查表 8.3-1！”** 
* **现实意义**：表 8.3-1 里规定了在不同大小的跑道（BWP）上，手机跳频的标准跨度（比如跨越四分之一跑道，或者跨越半条跑道）。不论是第一次发送 Msg3，还是因为失败被要求重传 Msg3，只要开启了跳频，第二跳的落点距离永远死死遵循这张表格的规定。

### 🔄 蛇形走位 (跳频) 发射流水线
```text
 物理层准备发射 Msg3 (PUSCH)
    │
    ├─ 检查 RAR UL Grant 里的【跳频标志位】
    │
    ├──► 【标志位 = 0】
    │         └─► 动作: 直线行驶！在分配好的频率上一直发到底。
    │
    └──► 【标志位 = 1】
              │
              ├─► 动作 1: 在起点频率上，发射【第一跳】(前半截 Msg3)
              │
              ├─► 动作 2: 去查【表 8.3-1】，找到规定的【频率偏移量 (跨度)】
              │
              └─► 动作 3: 加上这个跨度，瞬间跳跃到新频率，发射【第二跳】(后半截 Msg3)
```

## 📖 协议原文拆解 (七)：车道宽度的出处与“从一而终”的载波铁律

> **协议原文**
> A SCS for the PUSCH transmission is provided by subcarrierSpacing in BWP-UplinkCommon. A UE transmits PRACH and the PUSCH on a same uplink carrier of a same serving cell.

### 📚 增量术语表
* **BWP-UplinkCommon**：上行带宽公共配置。**一句话解释**：基站像村头大喇叭一样向全网广播的一本“公共花名册”，里面规定了这片区域大家共同遵守的跑道基础规则。
* **subcarrierSpacing**：子载波间隔配置项。**一句话解释**：花名册里的一个具体条目，明确写了车道有多宽（比如 15kHz、30kHz）。
* **uplink carrier**：上行载波。**一句话解释**：大段的物理无线电频率（比如 2.1GHz 频段，或者 3.5GHz 频段）。
* **serving cell**：服务小区。**一句话解释**：为你提供服务的那个基站本体。

### 🚪 第一关：锁定车道宽度的确切出处
* **背景倒带**：在 8.3 章节的开头（拆解一）中，我们知道高层会指示物理层使用哪个车道宽度（SCS $\mu$）去发 Msg3。但这个宽度是从哪里查到的呢？
* **明确出处**：协议在这里给出了确切的源头——物理层发射 Msg3 (PUSCH) 所需的 SCS，必须从广播的公共配置 `BWP-UplinkCommon` 里的 `subcarrierSpacing` 字段中提取。
* **现实意义**：因为在 Msg3 阶段，手机还是个刚递交申请的“半个陌生人”，基站还没来得及给你发私密的 VIP 专属配置（Dedicated 配置）。所以你的发射参数，必须老老实实遵从全网公开的公共标准。

### ⚓ 第二关：“从一而终”的空间与频率铁律
* **痛点溯源**：在 5G 极度复杂的网络里，一个基站可能同时挂着好几个频段（载波聚合 CA），甚至有的基站还有特殊的“补充上行频段（SUL）”。这就导致手机有很多条路可以走。
* **物理层防作妖机制**：如果手机在频段 A 上发了暗号（PRACH），基站听到了并给它发了回执（RAR）。这时候如果手机“左右横跳”，跑到频段 B 或者跑到隔壁基站去发后面的业务数据（Msg3），整个通信握手瞬间就会崩溃。
* **强制绑定**：协议最后一句用极其严厉的语态定下了铁律：**你在哪个基站（same serving cell）的哪个频段（same uplink carrier）上发射了第一步的暗号（PRACH），你后续的业务数据（PUSCH Msg3）也必须死死绑定在这个基站的这个频段上发射！绝不允许中途换线！**

### 🔄 载波与小区绑定流水线
```text
 手机 (UE) 准备发起随机接入
    │
    ├─ 选定目标：服务小区 (Serving Cell #1)，上行载波 (Carrier A)
    │
    ├─► 发射第一步暗号：PRACH ──► (Cell #1, Carrier A)
    │
    ├─ 成功收到基站的通行证 (RAR UL Grant)
    │
    ├─ 准备发射 Msg3 (PUSCH)：查阅公共花名册 BWP-UplinkCommon 获取车道宽度 SCS
    │
    ├─ 问自己：我能用性能更好的 Carrier B 发射 Msg3 吗？
    │
    └─► 协议底线拦截：绝对不行！必须“从一而终”！
             │
             ▼
      老老实实回到 (Cell #1, Carrier A) 上，发射 Msg3！
```

## 📖 协议原文拆解 (八)：初传数据的版本号与加密密码本

> **协议原文**
> A UE transmits a transport block in a PUSCH scheduled by a RAR UL grant in a corresponding RAR message using redundancy version number 0, if the PUSCH transmission is without repetitions. If a TC-RNTI is provided by higher layers, the scrambling initialization of the PUSCH corresponding to the RAR UL grant in clause 8.2 is by TC-RNTI. Otherwise, the scrambling initialization of the PUSCH corresponding to the RAR UL grant in clause 8.2 is by C-RNTI.

### 📚 增量术语表
* **Redundancy version number 0 (RV0)**：冗余版本 0。**一句话解释**：数据编码时的“初始完整版/无删减版”。发数据时必须先发包含所有核心信息的全本（RV0），如果失败了，后面重传时才会发只含校验补丁的版本（如 RV1, RV2 等）来拼凑。
* **Repetitions**：重复发送。**一句话解释**：为了应对极端恶劣的信号环境，把同一份数据连续发好几遍的机制。
* **Scrambling initialization**：加扰初始化。**一句话解释**：给发出去的电磁波数据进行“密码加密”的初始种子，防止别人窃听或者被其他信号串扰。
* **TC-RNTI (Temporary Cell RNTI)**：临时小区网络标识。**一句话解释**：手机刚入网时，由于身份还没最终核实（Msg4 还没走完），基站发给它的“临时访客牌”。

### 🚪 第一关：初次发包必须是“全本”（RV0 铁律）
* **场景溯源**：手机拿着通行证（RAR UL Grant），终于要在 PUSCH 上发射第一包包含真实业务数据的 Msg3 了。
* **版本控制法则**：协议强制规定，只要你不处于特殊的“连续重复发射”模式下，你发射这包数据**必须且只能使用冗余版本 0（RV0）**。
* **现实意义**：这是两人第一次正式对话，基站手里对你没有任何历史记录。所以你上来必须给出“完整无删减”的初始信息包（RV0），绝不能上来就发毫无意义的增量补丁（如 RV1/RV2），否则基站根本拼凑不出一句完整的话。

### 🔐 第二关：加密“密码本”的降维打击（TC-RNTI vs C-RNTI）
数据打包好了（RV0），接下来要在发射前进行“密码加扰”，以免信号在空中裸奔。手机该用什么密码本呢？这里体现了 5G 极具逻辑性的状态继承：
* **分支 A：有临时访客牌（初次入网）**
  * **场景**：手机是个刚开机的新手，高层查了一下，自己兜里只有一张基站在 Msg2 时临时发给自己的 **TC-RNTI（临时访客牌）**。
  * **动作**：那就老老实实地用这个 **TC-RNTI** 作为初始种子，对发出去的 Msg3 进行密码加扰。
* **分支 B：有正式 VIP 卡号（连接态老客户发起）**
  * **场景**：这就是我们之前学过的，手机明明已经连着网（拥有正式的 **C-RNTI**），因为断流等原因触发了随机接入。
  * **动作**：高层一摸兜，发现根本没有临时的 TC-RNTI，只有自己早就拿到的正式 VIP 卡号 C-RNTI。协议规定（`Otherwise...`），那就直接用你的正式 **C-RNTI** 来进行密码加扰！基站一解密，立刻秒懂：“原来是老客户发来的 Msg3 啊！”

### 🔄 Msg3 数据版本与加扰加密流水线
```text
 物理层拿到 Msg3 的数据大包裹，准备上高速 (PUSCH)
    │
    ├─ 问自己：我是不是要重复发好几遍？
    │    └──► 【不是 (without repetitions)】 ──► 动作 1: 打上 RV0 (完整版) 的版本号！
    │
    ├─ 准备给电磁波加密加扰 (Scrambling Initialization)
    │
    ├─ 问高层 (MAC)：我现在兜里揣着什么证件？
    │
    ├──► 【高层回答: 只有临时访客牌 (TC-RNTI)】
    │         │
    │         └─► 动作 2: 把 TC-RNTI 塞进加扰机，生成临时加密波形！
    │
    └──► 【高层回答: 啥临时牌？我是老客户，我有正式卡号 (C-RNTI)！】
              │
              └─► 动作 2: 直接把 C-RNTI 塞进加扰机，生成正式的 VIP 加密波形！
                      │
                      ▼
               加密完成，版本锁定，天线点火，发射 Msg3！
```

## 📖 协议原文拆解 (九)：多基站协同下的“上行时间对表” (TAG 指示)

> **协议原文**
> If a UE is provided tag2-Id, the UE transmits a transport block in a PUSCH scheduled by a RAR UL grant in a corresponding RAR message with timing advance corresponding to a TAG indicated by the RAR message.

### 📚 增量术语表
* **mTRP (Multiple Transmission/Reception Points)**：多传输接收点。**一句话解释**：5G 的高级玩法，一个手机同时连接着好几个相距甚远的天线塔（比如街头一个，街尾一个），两个塔同时为你服务。
* **tag2-Id**：这是 mTRP 技术里的一个极具特征的配置项。**一句话解释**：当你看到这个词，就说明基站告诉手机：“注意，现在不仅有正牌天线塔（1号塔）在理你，还有一个辅助天线塔（2号塔）也在管你哦！”
* **TAG (Timing Advance Group)**：时间提前量分组。**一句话解释**：由于街头和街尾的塔距离手机不一样远，电磁波跑路的时间也不同。为了让两个塔同时收到你的信号，手机发给 1 号塔的信号可能要提前 2 毫秒发，发给 2 号塔的可能要提前 5 毫秒发。这就是不同的 TAG（时间分组）。

### 🚪 核心痛点：多塔接力，时间容易劈腿
* **场景溯源**：手机目前正处于 mTRP（多天线塔协同）的豪华 VIP 环境中（被配置了 `tag2-Id`）。手机发出了暗号（PRACH），某个天线塔收到了，并发回了通行证（RAR UL Grant），命令手机发射 Msg3 (PUSCH)。
* **物理层的困境**：但是，不同天线塔的距离不一样！手机发 Msg3 时，到底该按 1 号塔的距离提前抢跑（应用 TAG1 的时间提前量），还是按 2 号塔的距离提前抢跑（应用 TAG2 的时间提前量）？如果对错表，Msg3 就会变成空中乱窜的炸弹干扰别人。

### ⏱️ 破解法则：听从 RAR 盲盒内的最高指示
* **一锤定音**：协议极其严密地堵上了这个漏洞：“当存在多个天线塔的时间体系时（provided tag2-Id），手机发射 Msg3 的时间提前量（TA），**必须绝对服从 RAR 回执里明确指定的那个 TAG 组！**”
* **现实意义**：谁给你发的回执，谁就在回执里负责告诉你：“小子，我是街尾那个离你比较远的塔，你发 Msg3 时，必须按我这组（TAG2）的规矩，提前 5 毫秒把数据给我发出来！” 物理层只需“唯命是从”，精准对表即可。

### 🔄 多塔协同 (mTRP) 时间对表流水线
```text
 物理层拿着 RAR 盲盒，准备发射 Msg3 (PUSCH)
    │
    ├─ 问高层：我现在是在普通环境，还是高级的多塔协同 (mTRP) 环境？
    │
    ├──► 【普通环境 (未配置 tag2-Id)】
    │         └─► 动作: 没啥好选的，就一个基站，按默认的时间提前量发！
    │
    └──► 【高级多塔协同环境 (配置了 tag2-Id)】
              │
              ├─ 物理层陷入迷茫：我现在有两套时间体系 (TAG1 和 TAG2)，听谁的？
              │
              ├─ 动作 1: 去翻看基站刚发来的那封 RAR 回执信 (RAR message)！
              │
              ├─ 动作 2: 在信里找到基站的最高指示：
              │          “本次 Msg3 传输，请使用 TAG2 时间组！”
              │
              └─► 动作 3: 立即调取 TAG2 的时间偏差值，拨快自己的时钟，
                          提前点火，把 Msg3 精准发射升空！
```

## 📖 协议原文拆解 (十)：失败后的“补救罚单” (Msg3 重传调度)

> **协议原文**
> Msg3 PUSCH retransmissions, if any, of the transport block, are scheduled by a DCI format 0_0 with CRC scrambled by a TC-RNTI provided in the corresponding RAR message [11, TS 38.321].

### 📚 增量术语表
* **Msg3 PUSCH retransmissions**：Msg3 重传。**一句话解释**：手机第一次发送 Msg3（带着真实身份的数据包）后，基站没听清或者解码失败了，要求手机再发一次补救包。
* **DCI format 0_0**：格式 0_0 的下行控制信息。**一句话解释**：基站发给手机的“最基础的上行发车调度单”，专门用来安排手机发数据的资源。
* **TC-RNTI**：临时小区网络标识。**一句话解释**：基站在第二步（Msg2 RAR 回执）里塞给手机的那张“临时访客牌”。

### 🚪 核心机制：拿“临时访客牌”去取“重传罚单”
* **场景倒带**：手机满怀希望地发出了 Msg3，然后开始等基站的最终宣判（Msg4）。可是左等右等没等到 Msg4，反而等来了一张新的控制小纸条（DCI）。
* **调度方式**：协议规定，基站如果想让你重传 Msg3，它不会再发一遍 RAR 回执了，而是会直接在普通控制信道里发一张常规的 **DCI format 0_0（调度单）**。这张调度单里写明了你重传时该用哪些新资源（比如换个频段、换个时间）。
* **防伪印章（加扰方式）**：大屏幕上那么多 DCI 0_0，手机怎么知道哪张是喊自己重传的呢？协议说：**用你手里的那张“临时访客牌（TC-RNTI）”去解密！** 这张牌是你当初在拆解 RAR 盲盒时得到的，现在刚好派上用场。如果你能用它解开这张 DCI 0_0，说明基站就是在叫你：“那个拿着临时牌的小子，你刚才的话我没听清，拿着这张新调度单，赶紧给我重发一遍！”

### 🔄 Msg3 重传调度流水线
```text
 手机 (UE) 发射了第一次 Msg3 之后...
    │
    ├─ 开始监听基站的下一步指示
    │
    ├─ 突然，大屏幕上亮起了一张新的调度单 (DCI format 0_0)
    │
    ├─ 动作 1: 掏出兜里在 Msg2 时得到的“临时访客牌 (TC-RNTI)”
    │
    ├─ 动作 2: 把 TC-RNTI 塞进解密机，去刮那张 DCI 的防伪涂层 (CRC 解扰)
    │
    ├──► 【解密失败】 ──► 说明这不是叫我的，继续等。
    │
    └──► 【解密成功】
              │
              └─► 动作 3: 大事不妙，这是我的“重传罚单”！
                  立刻打开罚单 (DCI 0_0)，查看新的时间、频率和版本号 (RV)
                  │
                  ▼
              准备就绪，按照新罚单的指示，点火重新发射 Msg3 (重传)！🚀
```

## 📖 协议原文拆解 (十一)：发射 Msg3 的精确倒计时公式 (时域资源分配)

> **协议原文**
> With reference to slots for a PUSCH transmission scheduled by a RAR UL grant, if a UE receives a PDSCH with a RAR message ending in slot $n$ for a corresponding PRACH transmission from the UE, the UE transmits the PUSCH in slot $n + k_2 + \Delta + 2^\mu \cdot K_{cell,offset}$, where $k_2$ and $\Delta$ are provided in [6, TS 38.214] and $K_{cell,offset}$ is provided by cellSpecificKoffset; otherwise, if not provided, $K_{cell,offset}=0$.

### 📚 增量术语表
* **Slot $n$**：时隙 $n$。**一句话解释**：手机收完 RAR 盲盒（PDSCH）的最后一刻所在的时间刻度，这是整个倒计时的“绝对零点”。
* **$k_2$ (k2)**：调度延迟指示。**一句话解释**：RAR 通行证里明确写明的一个数字（比如 2 或 3），告诉你“拿到通知后，请往后推延几个格子再发货”。
* **$\Delta$ (Delta)**：Msg3 专用额外推延量。**一句话解释**：由于发 Msg3 是随机接入这种特殊场景，手机处理起来比平时慢，所以协议非常人性化地给了一个“专属额外宽限时间”（通常是好几个时隙）。
* **$K_{cell,offset}$**：小区特定偏移量。**一句话解释**：某些特殊基站（比如覆盖超大面积的基站，或者上下行时间不对齐的 FDD 基站）为了统一步伐，强制加上的一个全网校准偏移值。

### 🚪 核心机制：时间叠加的“千层饼”计算法
* **场景倒带**：手机在时隙 $n$ 成功收到了包裹（RAR），从里面掏出了通行证（UL Grant），知道了去哪条跑道发数据。现在，它要计算出精确的“发车时间”。
* **公式拆解**：发车时间 = **起点($n$) + 基站调度指示($k_2$) + 协议宽限期($\Delta$) + 全网校准偏移($2^\mu \cdot K_{cell,offset}$)**
  1. **定起点**：一切倒计时，从接完 RAR 包裹的那个时隙 $n$ 开始算。
  2. **加调度**：从 RAR 通行证的时域字段里读出 $k_2$（比如往后推 2 个时隙）。
  3. **加宽限**：去查协议 38.214 里的表格，找到随机接入专属的额外宽限期 $\Delta$（比如再往后推 4 个时隙）。
  4. **加校准**：如果系统广播里有 `cellSpecificKoffset`，就把它算进去（还要乘上子载波间隔的比例系数 $2^\mu$ 以对齐时间标尺）。如果没有配置，这项就是 0。
* **物理层动作**：物理层的时钟滴答作响，当这四个值叠加出来的那个极其精准的“未来时隙”到来时，立刻点火发射 Msg3！

### 🔄 发射 Msg3 的时间计算流水线
```text
 物理层接完 RAR (PDSCH) 的最后一个符号，看了一眼墙上的钟：【当前是时隙 n】
    │
    ├─ 问自己：我该在哪个时隙发射 Msg3？
    │
    ├─ 动作 1: 解析通行证，提取基站的调度指示 [ + k_2 ]
    │
    ├─ 动作 2: 查阅协议，加上 Msg3 专属的系统宽限期 [ + Δ ]
    │
    ├─ 动作 3: 检查有没有全网时间校准偏移 [ + 2^μ * K_cell,offset ]
    │
    └─► 最终结果：锁定未来的【时隙 n + k_2 + Δ + 2^μ * K_cell,offset】
             │
             ▼
        (时间静静流逝...)
             │
             ▼
      当时钟走到这个计算出的时隙时，准时发射 PUSCH (Msg3)！🚀
```

## 📖 协议原文拆解 (十二)：信号极差时的“反复吟唱” (Msg3 重复发送机制)

> **协议原文**
> A UE can be provided in BWP-UplinkCommon a set of numbers of repetitions for a PUSCH transmission with PUSCH repetition Type A... If the UE requests repetitions... the UE transmits the PUSCH over $N_{PUSCH}^{repeat}$ slots, where $N_{PUSCH}^{repeat}$ is indicated by the 2 MSBs of the MCS field... The UE determines an MCS for the PUSCH transmission by the 2 LSBs of the MCS field... For unpaired spectrum operation, the UE determines the $N_{PUSCH}^{repeat}$ slots... where a repetition of the PUSCH transmission does not include a symbol indicated as downlink... or an SS/PBCH block... For paired spectrum operation...

### 📚 增量术语表
* **Repetition Type A**：重复类型A。**一句话解释**：为了增强信号，把同一份数据包（Msg3）连续发好几遍。Type A 的特点是“按时隙（Slot）为单位，每个时隙发一遍”。
* **$N_{PUSCH}^{repeat}$**：重复发送次数。**一句话解释**：基站命令手机把 Msg3 连续重发几次（比如连发 4 次）。
* **Unpaired / Paired spectrum**：非成对/成对频谱。**一句话解释**：非成对就是 TDD（一条马路，时而当上行用，时而当下行用）；成对就是 FDD（两条平行的单行道，上下行互不干扰）。

### 🚪 第一关：触发保命技能（请求重复发送）
* **场景倒带**：手机藏在深深的地下室，信号极差。在第一步发送暗号（PRACH）时，它故意选了一组“特殊暗号”，以此暗示基站：“老大，我信号太差了，求求你一会让我多发几遍 Msg3 吧（requests repetitions），不然你绝对听不见！”

### ✂️ 第二关：鸠占鹊巢的控制字段 (MCS 字段遭篡改)
* **核心痛点**：基站同意了手机的请求，准备在 RAR 通行证里告诉手机“连发 4 次”。但是，RAR 通行证的格式是早期写死的，里面根本**没有预留“重复次数”这个空位**！
* **物理层极限魔改**：既然没空位，就去“抢”别人的空位！协议极其粗暴地盯上了 **MCS 字段（用来控制语速的字段，共 4 个 bit）**：
  * **高位被抢（指示次数）**：MCS 字段最前面的 2 个比特（2 MSBs）被强行征用！这 2 位数字不再代表语速，而是代表**重复次数（$N_{PUSCH}^{repeat}$）**。手机拿着这 2 位数去查配置表（或默认的 1,2,3,4 次）。
  * **低位苟活（指示语速）**：原来 4 个 bit 的 MCS，被抢走 2 个后，只能用剩下可怜的 2 个最低位（2 LSBs）来勉强表示一个极其保守的极慢语速。

### 📅 第三关：连发时隙的“避坑法则” (TDD vs FDD)
基站让你“连发 4 次（占用 4 个时隙）”，时间起点就是我们上一关算出来的那个精准时隙。但这连续的 4 个时隙能一口气发完吗？
* **分支 A：单行道双向通行 (TDD / Unpaired spectrum)**
  * **痛点**：TDD 就像一条单行道，上一秒大家都在往下走（下行DL），下一秒才允许往上走（上行UL）。如果手机闭着眼睛连发 4 个时隙，很可能会撞上下行的高铁（比如撞上基站正在广播下行数据，或者正在发送 SSB 灯塔）。
  * **避坑法则**：手机必须长着眼睛发！从起点开始往后数 4 个时隙，**凡是遇到“被标为下行（DL）的符号”或者“正在播放灯塔（SSB）的符号”，这个时隙就作废跳过！** 一直往后顺延，直到成功发满 4 个“干净”的上行时隙为止。
* **分支 B：双车道畅通无阻 (FDD / Paired spectrum)**
  * **环境**：上下行是两条完全隔离的高速公路。
  * **躺平法则**：根本不用管什么下行碰撞。从起点开始，直接一口气连续占用 4 个时隙把 Msg3 轰出去！

### 🔄 Msg3 覆盖增强 (重复发送) 流水线
```text
 物理层拿着 RAR 盲盒，准备解析 Msg3 的发射参数
    │
    ├─ 问高层：我们刚才是不是申请了【重复发送】保命技能？
    │
    ├──► 【没有申请】 ──► 动作: 常规解析，MCS 字段全看作语速，只发 1 遍。
    │
    └──► 【申请了！】
              │
              ├─ 动作 1 (抢占字段): 把 MCS 字段劈成两半！
              │     前 2 位 ──► 翻译成重发次数 (比如连发 4 遍)
              │     后 2 位 ──► 翻译成极慢的保命语速 (MCS)
              │
              ├─ 动作 2 (找时隙):
              │     ├── 在 FDD 网络 ──► 闭眼连发 4 个时隙！
              │     └── 在 TDD 网络 ──► 睁眼发！跳过所有下行时隙和灯塔时隙，
              │                         凑齐 4 个干净的上行时隙发射！
              │
              ▼
    以极其坚韧的毅力，在深层地下室把 4 个相同的 Msg3 轰向基站！🚀
```

## 📖 协议原文拆解 (十三)：物理层的绝对时间红线 (收发转换时间)

> **协议原文**
> The UE may assume a minimum time between the last symbol of a PDSCH reception conveying a RAR message with a RAR UL grant and the first symbol of a corresponding PUSCH transmission scheduled by the RAR UL grant is equal to $N_{T,1} + N_{T,2} + 0.5$ msec, where $N_{T,1}$ is a time duration of $N_1$ symbols corresponding to a PDSCH processing time... $N_{T,2}$ is a time duration of $N_2$ symbols corresponding to a PUSCH preparation time... the UE considers that $N_1$ and $N_2$ correspond to the smaller of the SCS configurations for the PDSCH and the PUSCH...

### 📚 增量术语表
* **Minimum time**：最小时间间隔。**一句话解释**：基站排班时必须给手机留出的“最低合法喘息时间”，基站要是催得比这个时间还急，手机可以直接罢工。
* **$N_{T,1}$ (PDSCH processing time)**：下行数据处理时间。**一句话解释**：手机的“大脑”把基站发来的 RAR 包裹拆开、解密、并读懂里面的“上行通行证（UL Grant）”所需要的时间。
* **$N_{T,2}$ (PUSCH preparation time)**：上行数据准备时间。**一句话解释**：手机读懂了通行证后，去把自己的 Msg3 数据打包、加密、并塞进发射炮筒里所需要的准备时间。
* **Smaller of the SCS configurations**：较小的子载波间隔（车道宽度）。**一句话解释**：车道越窄（SCS越小），每个符号的时间跨度就越长。取较小者，就是按“最慢的那个节拍”来给手机宽限时间。

### 🚪 核心机制：挑战手机芯片的运算极限
* **场景倒带**：在之前的第十一关里，我们学过一个极其复杂的时隙计算公式 `$n + k_2 + \Delta + \dots$`，基站用这个公式给手机排了一个发车时间。
* **物理底线校验**：不管公式算出来的时间是多少，这个发车时间距离手机刚刚接完 RAR 包裹的时间差，**绝对不能小于一个物理极限值：$N_{T,1} + N_{T,2} + 0.5$ 毫秒！**
  * **大脑拆包裹（$N_{T,1}$）**：得给我时间看信。
  * **大脑写回信（$N_{T,2}$）**：得给我时间写信并打包。
  * **硬件大转身（0.5 毫秒）**：这额外的 0.5ms，是手机的射频天线从“接收模式（听）”硬切到“发射模式（说）”，以及给功率放大器（PA）预热所需的纯物理硬件耗时。
* **木桶效应（防剥削条款）**：基站发 RAR 用的车道（PDSCH SCS），和手机发 Msg3 用的车道（PUSCH SCS）可能不一样宽。协议规定：在算这个宽限时间时，必须按照两条车道里**“最慢的那条（较小的 SCS）”**来折算。绝不允许基站用快车道的标准去压榨手机的准备时间。

### 🔄 芯片处理与射频预热的时间流水线
```text
 【绝对零点】: 手机接完 RAR (PDSCH) 的最后一个符号
       │
       ├─► 阶段 1: 大脑开始高速运转，拆解 RAR 盲盒
       │   (耗时: N_T,1，即 N1 个符号的时间)
       │
       ├─► 阶段 2: 读懂了通行证，开始打包准备 Msg3 数据
       │   (耗时: N_T,2，即 N2 个符号的时间)
       │
       ├─► 阶段 3: 射频天线“咔咔”作响，从接收模式切换到发射模式，炮管预热！
       │   (耗时: 固定的 0.5 毫秒)
       │
       ▼
 【安全红线】: 到了这个时间点，手机才刚刚准备就绪！
       │
       ├─ 问自己：基站给我排的发车时间，越过这条红线了吗？
       │
       ├──► 【发车时间在红线之后】 ──► 完美！时间很充裕，到点准时发射！
       │
       └──► 【发车时间在红线之前】 ──► 霸气拒绝！(Does not expect)
                                       基站你催命呢？老子不发了，报错！
```

## 🏆 终极硬核总结：8.3 章节 (发射 Msg3) 的全局物理定律

如果说前面的章节是在教手机“如何跟基站对暗号并等回执”，那么 **8.3 章节就是一本极其硬核的《Msg3 物理层开火说明书》**。它通过极其严密的数学公式和逻辑分支，将高层的宏观意图完美具象化为底层的电磁波操作：

### 1. 频域资源的“刻度重塑”与“削足适履”
* **大局观的剥离**：物理层不负责决定跑道（Active UL BWP）和车道宽度（SCS），直接听命于高层（MAC）。
* **坐标系的强行对齐**：当 RAR 里的资源刻度（基于 Initial BWP）与当前大跑道不匹配时，物理层通过“原点平移”强制统一坐标系。
* **变长位数的截断与补零**：面对固定 14-bit 的坐标指令与千变万化的物理跑道大小，系统采用了冷酷的截长补短法则。甚至不惜在高位强行补零，将超宽跑道下的 Msg3 封锁在频谱前半段，以此换取极限状态下的信令压缩。
* **NR-U 的交织生存法则**：在免费的公共频段，手机强制启用 Type 2 分配，通过对指令的按需截断（留 5位 或 6位）精准挑选“梳子”，并原路继承 PRACH 的大局域（RB set），完成了全频段占道的生存挑战。

### 2. 时域发车的“千层饼倒计时”与“人权红线”
* **精准到时隙的点火公式**：发射时间不是一个定值，而是一个由“$起点 + 调度指示(k_2) + 随机接入专属宽限(\Delta) + 小区校准偏移$”层层叠加而成的精密公式。
* **绝对硬件红线 ($N_{T,1} + N_{T,2} + 0.5$ ms)**：这是本章最闪耀的物理之光。协议死死规定了基站的催促底线，强制保留手机拆解下行包裹、打包上行数据以及射频天线大转身所需的纯物理时间，并且按“最慢的那个车道节拍”来折算，杜绝了一切超越芯片极限的“压榨式调度”。

### 3. 多元化场景的极限救场与身份加扰
* **版本与密码本的身份隔离**：初传必须是毫无保留的完整版（RV0）。而在空中安全的加密（加扰）环节，协议用标识符进行了一秒验明正身——新客户强制用临时访客牌（TC-RNTI）加密，老客户直接用正式卡号（C-RNTI）加密。
* **mTRP 的时钟锁定**：面对多天线塔（mTRP）导致的时钟错乱，协议粗暴而有效：直接剥夺手机自主权，强制它必须按照 RAR 里指定的那个时钟组（TAG）来拨快指针。
* **极限弱网的“借尸还魂” (Repetitions)**：面对深层地下室的信号危机，协议没有新增任何信令负担，而是惊艳地“征用”了 MCS（语速）字段的高 2 位，来向手机下达连续重发 4 次的指令，并在 TDD 网络中赋予了手机自动绕开下行碰撞的智慧。

### 4. 失败重传的优雅降级
当第一次精心准备的 Msg3 依然没有被基站听清时，系统不再动用昂贵的 RAR 机制，而是直接降级回最基础的 DCI format 0_0 调度单。并且极其巧妙地让手机用“临时访客牌（TC-RNTI）”去刮开调度单的防伪涂层，实现了重传机制在公共信道上的无缝衔接。

**结语：8.3 章节是一座微观架构的丰碑。它在极度受限的控制开销下，通过字段重定义（抢占 MCS）、强制坐标映射（截断补零）、以及铁血的物理耗时红线，硬生生砸出了一条能够适配几十种极端场景（NR-U、RedCap、mTRP、TDD/FDD）的 Msg3 必发通道。在这里，没有含糊其辞，只有0和1交织出的暴力美学！**
