# 🔖 章节：10.1 UE procedure for determining physical downlink control channel assignment 
*(终端确定物理下行控制信道分配的流程)*
---

## 📖 章节导读：如何在一片漆黑中，精准锁定基站的“调度黑板”？

**背景信息**：
恭喜我们终于跨过了宏观的“算力分配计算题”，正式进入了 **第 10.1 节**！
这节的内容极度硬核，且极具实操性。它要回答一个物理层最基本的问题：**手机到底怎么在时间和频率的网格上，把那个叫 PDCCH 的“黑板”给扣出来？**

基站不会满屏幕贴纸条。它会划定几个叫做 **CORESET（控制资源集）** 的长方形区域作为黑板，并制定一份叫 **Search Space（搜索空间）** 的排班表，告诉你每天几点去哪块黑板上看。

10.1 节就是手机的“作息时间表”与“藏宝图”。它将详细规定：
*   黑板有多长、多宽？（CORESET 的时频资源属性）
*   看黑板的频率和起点怎么算？（Search Space 周期和偏移）
*   不同的“大喇叭”（比如看系统广播的、看寻呼的、看自己私人业务的）分别该去几号黑板看？

### 📚 基础术语预热 (本节核心组件)
*   **CORESET (Control Resource Set)**：控制资源集。**一句话解释**：物理意义上的“黑板”。它由一堆时频格子组成，比如频域上占几十个 RB，时域上占 1~3 个符号。
*   **Search Space**：搜索空间。**一句话解释**：看黑板的“时钟”。它规定了你每隔多久（Periodicity）去看一次，以及在周期内的哪一天（Offset）去看。
*   **CCE (Control Channel Element)**：控制信道元素。**一句话解释**：构成黑板的“马赛克砖块”。一条 DCI 通知单，可以由 1 块、2 块、4 块、8 块甚至 16 块 CCE 拼成（这就是传说中的聚合等级 Aggregation Level，块数越多字写得越大，越不容易传错）。

## 📖 协议原文拆解 (一)：满汉全席！5G 搜索空间 (Search Space) 家族族谱大盘点

> **协议原文**
> A set of PDCCH candidates for a UE to monitor is defined in terms of PDCCH search space sets. A search space set can be a CSS set or a USS set. A UE monitors PDCCH candidates in one or more of the following search spaces sets
> - a Type0-PDCCH CSS set on the primary cell of the MCG configured by
>   - pdcch-ConfigSIB1 in MIB or by searchSpaceSIB1 in PDCCH-ConfigCommon or by searchSpaceZero in PDCCH-ConfigCommon for a DCI format 1_0 with CRC scrambled by a SI-RNTI, or
>   - searchSpaceZero by providing searchSpaceID=0 for searchSpaceMCCH or searchSpaceMTCH for a DCI format 4_0 with CRC scrambled by a MCCH-RNTI or a G-RNTI for broadcast, or
>   - searchSpaceZero by providing searchSpaceID=0 for searchspaceMulticastMCCH for a DCI format 4_0 with CRC scrambled by a Multicast MCCH-RNTI, or by searchSpaceMulticastMTCH for a DCI format 4_1 with CRC scrambled by a G-RNTI for multicast in RRC_INACTIVE state
> - a Type0A-PDCCH CSS set configured by searchSpaceOtherSystemInformation in PDCCH-ConfigCommon for a DCI format 1_0 with CRC scrambled by a SI-RNTI on the primary cell of the MCG
> - a Type0B-PDCCH CSS set configured by
>   - searchSpaceMCCH and searchSpaceMTCH for a DCI format 4_0 with CRC scrambled by a MCCH-RNTI or a G-RNTI for broadcast, on the primary cell of the MCG
>   - searchspaceMulticastMCCH for a DCI format 4_0 with CRC scrambled by a Multicast MCCH-RNTI, or by searchSpaceMulticastMTCH for a DCI format 4_1 with CRC scrambled by a G-RNTI for PDCCH receptions in RRC_INACTIVE state
> - a Type1-PDCCH CSS set configured by ra-SearchSpace in PDCCH-ConfigCommon for a DCI format with CRC scrambled by a RA-RNTI, a MsgB-RNTI, or a TC-RNTI on the primary cell
> - a Type1A-PDCCH CSS set configured by sdt-SearchSpace in PDCCH-ConfigCommon for a DCI format with CRC scrambled by a C-RNTI or a CS-RNTI on the primary cell
> - a Type2-PDCCH CSS set configured by pagingSearchSpace in PDCCH-ConfigCommon for a DCI format 1_0 with CRC scrambled by a P-RNTI on the primary cell of the MCG
> - a Type2A-PDCCH CSS set configured by pei-SearchSpace in pei-ConfigBWP for a DCI format 2_7 with CRC scrambled by a PEI-RNTI on the primary cell of the MCG
> - a Type3-PDCCH CSS set configured by
>   - SearchSpace in PDCCH-Config with searchSpaceType = common for DCI formats with CRC scrambled by INT-RNTI, SFI-RNTI, TPC-PUSCH-RNTI, TPC-PUCCH-RNTI, TPC-SRS-RNTI, CI-RNTI, or cellDTRX-RNTI and, only for the primary cell, C-RNTI, MCS-C-RNTI, CS-RNTI(s), or PS-RNTI, or
>   - SearchSpace in pdcch-ConfigMulticast for DCI formats with CRC scrambled by G-RNTI, or G-CS-RNTI, or
>   - searchSpaceMCCH and searchSpaceMTCH on a secondary cell for a DCI format 4_0 with CRC scrambled by a MCCH-RNTI or a G-RNTI for broadcast, and
> - a USS set configured by
>   - SearchSpace or by SearchSpaceExt-v1800 in PDCCH-Config with searchSpaceType = ue-Specific for DCI formats with CRC scrambled by C-RNTI, MCS-C-RNTI, SP-CSI-RNTI, CS-RNTI(s), SL-RNTI, SL-CS-RNTI, SL Semi-Persistent Scheduling V-RNTI, or NCR-RNTI

### 📚 术语表
* **CSS (Common Search Space)**：公共搜索空间。**一句话解释**：全网“大喇叭”。基站在这里贴的纸条，是发给一群人甚至全小区所有人的。密码（RNTI）大家都有。
* **USS (UE-Specific Search Space)**：终端专属搜索空间。**一句话解释**：你的“私人信箱”。基站在这里贴的纸条，密码只有你一个人知道，专门用来调度你的个人数据。
* **RNTI (Radio Network Temporary Identifier)**：各种密码本。**一句话解释**：解密纸条的秘钥。
  * `SI-RNTI`：解系统广播。 `P-RNTI`：解寻呼(电话来了)。 `RA-RNTI`：解随机接入(开门暗号)。 
  * `C-RNTI`：解你的私人业务。

### 🚪 第一关：黑板上的“分区管理”
黑板（CORESET）虽然就那么几块，但基站对在什么时间、用什么密码看什么内容，做了极其精细的切分，这就叫 **Search Space Set（搜索空间集）**。
这段极其漫长的清单，就是 5G 控制信道调度的**“总族谱”**。我们来把这本族谱简化成“人话版”。

### 🎯 第二关：CSS 家族 (八大公共大喇叭)
公共大喇叭也分三六九等，名字以 Type 开头，数字越小优先级越高、越基础：

1. **Type0 (国家最高机密)**：只找 `SIB1` (系统基础规章)。手机入网必看的第一张纸条。
2. **Type0A (普通政策通知)**：找其他的系统广播（Other SI）。重要性仅次于 SIB1。
3. **Type0B (群体广播/多播)**：这是后补的高级功能（MBS），用来找小区里群发的视频流或者组播数据（MCCH/MTCH）。
4. **Type1 (入网接待处)**：找 **随机接入** 相关的回应（RAR, MsgB, Msg4）。我们在第8章已经很熟悉它了。
5. **Type1A (休眠态偷跑业务)**：R17新功能。手机在半睡半醒（Inactive）时偷偷发小包数据（SDT）用的调度空间。
6. **Type2 (电话传达室)**：找 **寻呼 (Paging)**。手机休眠时，只竖起半只耳朵听这个喇叭，一旦响了就说明有电话找你。
7. **Type2A (提前叫醒服务)**：找 **PEI (寻呼提前指示)**。为了更省电，基站在发寻呼前先响一下警报，告诉你“下一秒有寻呼，准备起床”。
8. **Type3 (居委会群发群控)**：管杂事的公共喇叭。比如群发：调整上行功率（TPC）、变更时隙格式（SFI）、告诉你哪些符号被占用了（INT）等。

### 🧩 第三关：USS 家族 (私人订制)
相对于庞杂的 CSS，USS 的定义非常纯粹：
* **USS**：纯粹的私人信箱。基站在这里用你的私人密码（`C-RNTI`, `CS-RNTI` 等）给你分配下载或上传资源。刷抖音、打视频电话的数据全靠这里来调度。

### 💡 章节硬核总结

**本段协议是 38.213 第 10 章的总索引（Grand Index）。它通过穷举的方式，将物理层所有的下行控制信道调度场景进行了严格的分类（Classification）与 Namespace 定义。5G NR 放弃了 LTE 时代将 CSS 和 USS 硬编码在频带两端的死板设计，转而采用灵活的“Search Space Set”逻辑封装。这里的每一个 Type（从 Type0 到 Type3，再到 USS）都代表了一组独立的 RRC 配置、一种特定的 RNTI 加扰序列集合、以及对应的 DCI format 解码池。这种基于“Type-based”的解耦设计，使得协议能够极其方便地在后续 Release 中插入新的调度功能（例如 R17 加入的 Type1A 用于 SDT，Type2A 用于 PEI 早醒，Type0B 用于组播 MBS），而无需推翻底层的物理映射框架。这是 5G 空口灵活性的核心体现。**
