# 🔖 章节：10.1 UE procedure for determining physical downlink control channel assignment 
*(终端确定物理下行控制信道分配的流程)*
---

## 📖 章节导读：如何在一片漆黑中，精准锁定基站的“调度黑板”？

**背景信息**：
恭喜我们终于跨过了宏观的“算力分配计算题”，正式进入了 **第 10.1 节**！
这节的内容极度硬核，且极具实操性。它要回答一个物理层最基本的问题：**手机到底怎么在时间和频率的网格上，把那个叫 PDCCH 的“黑板”给扣出来？**

基站不会满屏幕贴纸条。它会划定几个叫做 **CORESET（控制资源集）** 的长方形区域作为黑板，并制定一份叫 **Search Space（搜索空间）** 的排班表，告诉你每天几点去哪块黑板上看。

10.1 节就是手机的“作息时间表”与“藏宝图”。它将详细规定：
*   黑板有多长、多宽？（CORESET 的时频资源属性）
*   看黑板的频率和起点怎么算？（Search Space 周期和偏移）
*   不同的“大喇叭”（比如看系统广播的、看寻呼的、看自己私人业务的）分别该去几号黑板看？

### 📚 基础术语预热 (本节核心组件)
*   **CORESET (Control Resource Set)**：控制资源集。**一句话解释**：物理意义上的“黑板”。它由一堆时频格子组成，比如频域上占几十个 RB，时域上占 1~3 个符号。
*   **Search Space**：搜索空间。**一句话解释**：看黑板的“时钟”。它规定了你每隔多久（Periodicity）去看一次，以及在周期内的哪一天（Offset）去看。
*   **CCE (Control Channel Element)**：控制信道元素。**一句话解释**：构成黑板的“马赛克砖块”。一条 DCI 通知单，可以由 1 块、2 块、4 块、8 块甚至 16 块 CCE 拼成（这就是传说中的聚合等级 Aggregation Level，块数越多字写得越大，越不容易传错）。

## 📖 协议原文拆解 (一)：满汉全席！5G 搜索空间 (Search Space) 家族族谱大盘点

> **协议原文**
> A set of PDCCH candidates for a UE to monitor is defined in terms of PDCCH search space sets. A search space set can be a CSS set or a USS set. A UE monitors PDCCH candidates in one or more of the following search spaces sets
> - a Type0-PDCCH CSS set on the primary cell of the MCG configured by
>   - pdcch-ConfigSIB1 in MIB or by searchSpaceSIB1 in PDCCH-ConfigCommon or by searchSpaceZero in PDCCH-ConfigCommon for a DCI format 1_0 with CRC scrambled by a SI-RNTI, or
>   - searchSpaceZero by providing searchSpaceID=0 for searchSpaceMCCH or searchSpaceMTCH for a DCI format 4_0 with CRC scrambled by a MCCH-RNTI or a G-RNTI for broadcast, or
>   - searchSpaceZero by providing searchSpaceID=0 for searchspaceMulticastMCCH for a DCI format 4_0 with CRC scrambled by a Multicast MCCH-RNTI, or by searchSpaceMulticastMTCH for a DCI format 4_1 with CRC scrambled by a G-RNTI for multicast in RRC_INACTIVE state
> - a Type0A-PDCCH CSS set configured by searchSpaceOtherSystemInformation in PDCCH-ConfigCommon for a DCI format 1_0 with CRC scrambled by a SI-RNTI on the primary cell of the MCG
> - a Type0B-PDCCH CSS set configured by
>   - searchSpaceMCCH and searchSpaceMTCH for a DCI format 4_0 with CRC scrambled by a MCCH-RNTI or a G-RNTI for broadcast, on the primary cell of the MCG
>   - searchspaceMulticastMCCH for a DCI format 4_0 with CRC scrambled by a Multicast MCCH-RNTI, or by searchSpaceMulticastMTCH for a DCI format 4_1 with CRC scrambled by a G-RNTI for PDCCH receptions in RRC_INACTIVE state
> - a Type1-PDCCH CSS set configured by ra-SearchSpace in PDCCH-ConfigCommon for a DCI format with CRC scrambled by a RA-RNTI, a MsgB-RNTI, or a TC-RNTI on the primary cell
> - a Type1A-PDCCH CSS set configured by sdt-SearchSpace in PDCCH-ConfigCommon for a DCI format with CRC scrambled by a C-RNTI or a CS-RNTI on the primary cell
> - a Type2-PDCCH CSS set configured by pagingSearchSpace in PDCCH-ConfigCommon for a DCI format 1_0 with CRC scrambled by a P-RNTI on the primary cell of the MCG
> - a Type2A-PDCCH CSS set configured by pei-SearchSpace in pei-ConfigBWP for a DCI format 2_7 with CRC scrambled by a PEI-RNTI on the primary cell of the MCG
> - a Type3-PDCCH CSS set configured by
>   - SearchSpace in PDCCH-Config with searchSpaceType = common for DCI formats with CRC scrambled by INT-RNTI, SFI-RNTI, TPC-PUSCH-RNTI, TPC-PUCCH-RNTI, TPC-SRS-RNTI, CI-RNTI, or cellDTRX-RNTI and, only for the primary cell, C-RNTI, MCS-C-RNTI, CS-RNTI(s), or PS-RNTI, or
>   - SearchSpace in pdcch-ConfigMulticast for DCI formats with CRC scrambled by G-RNTI, or G-CS-RNTI, or
>   - searchSpaceMCCH and searchSpaceMTCH on a secondary cell for a DCI format 4_0 with CRC scrambled by a MCCH-RNTI or a G-RNTI for broadcast, and
> - a USS set configured by
>   - SearchSpace or by SearchSpaceExt-v1800 in PDCCH-Config with searchSpaceType = ue-Specific for DCI formats with CRC scrambled by C-RNTI, MCS-C-RNTI, SP-CSI-RNTI, CS-RNTI(s), SL-RNTI, SL-CS-RNTI, SL Semi-Persistent Scheduling V-RNTI, or NCR-RNTI

### 📚 术语表
* **CSS (Common Search Space)**：公共搜索空间。**一句话解释**：全网“大喇叭”。基站在这里贴的纸条，是发给一群人甚至全小区所有人的。密码（RNTI）大家都有。
* **USS (UE-Specific Search Space)**：终端专属搜索空间。**一句话解释**：你的“私人信箱”。基站在这里贴的纸条，密码只有你一个人知道，专门用来调度你的个人数据。
* **RNTI (Radio Network Temporary Identifier)**：各种密码本。**一句话解释**：解密纸条的秘钥。
  * `SI-RNTI`：解系统广播。 `P-RNTI`：解寻呼(电话来了)。 `RA-RNTI`：解随机接入(开门暗号)。 
  * `C-RNTI`：解你的私人业务。

### 🚪 第一关：黑板上的“分区管理”
黑板（CORESET）虽然就那么几块，但基站对在什么时间、用什么密码看什么内容，做了极其精细的切分，这就叫 **Search Space Set（搜索空间集）**。
这段极其漫长的清单，就是 5G 控制信道调度的**“总族谱”**。我们来把这本族谱简化成“人话版”。

### 🎯 第二关：CSS 家族 (八大公共大喇叭)
公共大喇叭也分三六九等，名字以 Type 开头，数字越小优先级越高、越基础：

1. **Type0 (国家最高机密)**：只找 `SIB1` (系统基础规章)。手机入网必看的第一张纸条。
2. **Type0A (普通政策通知)**：找其他的系统广播（Other SI）。重要性仅次于 SIB1。
3. **Type0B (群体广播/多播)**：这是后补的高级功能（MBS），用来找小区里群发的视频流或者组播数据（MCCH/MTCH）。
4. **Type1 (入网接待处)**：找 **随机接入** 相关的回应（RAR, MsgB, Msg4）。我们在第8章已经很熟悉它了。
5. **Type1A (休眠态偷跑业务)**：R17新功能。手机在半睡半醒（Inactive）时偷偷发小包数据（SDT）用的调度空间。
6. **Type2 (电话传达室)**：找 **寻呼 (Paging)**。手机休眠时，只竖起半只耳朵听这个喇叭，一旦响了就说明有电话找你。
7. **Type2A (提前叫醒服务)**：找 **PEI (寻呼提前指示)**。为了更省电，基站在发寻呼前先响一下警报，告诉你“下一秒有寻呼，准备起床”。
8. **Type3 (居委会群发群控)**：管杂事的公共喇叭。比如群发：调整上行功率（TPC）、变更时隙格式（SFI）、告诉你哪些符号被占用了（INT）等。

### 🧩 第三关：USS 家族 (私人订制)
相对于庞杂的 CSS，USS 的定义非常纯粹：
* **USS**：纯粹的私人信箱。基站在这里用你的私人密码（`C-RNTI`, `CS-RNTI` 等）给你分配下载或上传资源。刷抖音、打视频电话的数据全靠这里来调度。

### 💡 章节硬核总结

**本段协议是 38.213 第 10 章的总索引（Grand Index）。它通过穷举的方式，将物理层所有的下行控制信道调度场景进行了严格的分类（Classification）与 Namespace 定义。5G NR 放弃了 LTE 时代将 CSS 和 USS 硬编码在频带两端的死板设计，转而采用灵活的“Search Space Set”逻辑封装。这里的每一个 Type（从 Type0 到 Type3，再到 USS）都代表了一组独立的 RRC 配置、一种特定的 RNTI 加扰序列集合、以及对应的 DCI format 解码池。这种基于“Type-based”的解耦设计，使得协议能够极其方便地在后续 Release 中插入新的调度功能（例如 R17 加入的 Type1A 用于 SDT，Type2A 用于 PEI 早醒，Type0B 用于组播 MBS），而无需推翻底层的物理映射框架。这是 5G 空口灵活性的核心体现。**

## 📖 协议原文拆解 (二)：通知单的“私发”、“群发”与“全网广播” (DCI 类别统一定义)

> **协议原文**
> In the following, DCI formats with CRC scrambled by C-RNTI or CS-RNTI or MCS-C-RNTI are also referred to as unicast DCI formats, DCI formats with CRC scrambled by G-RNTI for multicast or G-CS-RNTI are also referred to as multicast DCI formats, and DCI formats with CRC scrambled by MCCH-RNTI or G-RNTI for broadcast scheduling PDSCH receptions are also referred to as broadcast DCI formats, and DCI formats with CRC scrambled by Multicast MCCH-RNTI or G-RNTI for multicast scheduling PDSCH receptions in RRC_INACTIVE state are also referred as multicast DCI formats for RRC_INACTIVE state.

### 📚 增量术语表
* **Unicast (单播)**：一对一发送。**一句话解释**：基站只把数据发给你这一个特定手机，别人解不开也收不到。
* **Multicast (组播/多播)**：一对“特定群体”发送。**一句话解释**：基站把数据发给一个“VIP会员群”，只有加了这个群的几百个手机能收到，群外的人收不到（5G 后期引入的高级特性 MBS）。
* **Broadcast (广播)**：一对所有人发送。**一句话解释**：广场上的大喇叭，只要在基站覆盖范围内，不管你是谁，都能听到。

### 🚪 第一关：为了后续行文方便的“简称大派送”
在 3GPP 协议里，如果每次提到某种通知单，都要把一大串密码（比如“用 C-RNTI、CS-RNTI 或 MCS-C-RNTI 加扰的 DCI 格式...”）全部念一遍，不仅极其啰嗦，还容易看花眼。
所以，这段话其实是协议编写者为了后续行文方便，在这里统一下达了**四个“术语简称（Aliases）”**。只要看懂了密码（RNTI），就知道这张纸条是发给谁的。

### 🎯 第二关：拆解四种“发件模式”
协议把所有的 DCI 调度纸条分成了四大类：

1. **私人信件 (Unicast DCI formats)**：
   * **特征**：用 `C-RNTI` (普通个人密码), `CS-RNTI` (包月套餐密码), 或 `MCS-C-RNTI` (高网速密码) 加密的纸条。
   * **含义**：这绝对是发给某个具体 UE 的专属快递单。

2. **微信群通知 (Multicast DCI formats)**：
   * **特征**：用 `G-RNTI for multicast` (多播群密码) 或 `G-CS-RNTI` (多播包月密码) 加密的纸条。
   * **含义**：这是发给特定一群人（比如同在一个小区看同一场高清直播演唱会的 1000 个手机）的资源调度单。

3. **广场大喇叭 (Broadcast DCI formats)**：
   * **特征**：用 `MCCH-RNTI` 或 `G-RNTI for broadcast` 加密的纸条。
   * **含义**：强制向全小区所有人播放的公共多媒体或控制流，所有人无差别接收。

4. **休眠态的午夜群发 (Multicast DCI formats for RRC_INACTIVE)**：
   * **特征**：用 `Multicast MCCH-RNTI` 或 休眠态专用的 `G-RNTI` 加密的纸条。
   * **含义**：手机虽然睡着了（Inactive状态），但依然能接收基站针对特定群组下发的组播数据。这是极其高阶的省电组播技术。

```text
 👤 私人信件 ───┼─ CS-RNTI (免调度/包月密码)      ══► 统称: Unicast DCI
```

### 💡 章节硬核总结

**本段协议是 38.213 第 10 章的“命名空间映射（Namespace Aliasing/Mapping）”声明。随着 5G Rel-17/18 MBS（Multicast and Broadcast Services）特性的深度融入，控制面的调度复杂性急剧增加。为了保持协议文本的逻辑清晰度，3GPP 在此处将基于 L1 RNTI（Radio Network Temporary Identifier）的底层加扰特征，向上抽象映射为了基于 L2/L3 业务面传输拓扑（Topology）的宏观术语：Unicast（单播）、Multicast（组播）与 Broadcast（广播）。这种“以加扰签名定义业务类型”的归类法，确立了后续所有关于搜索空间（Search Space）、HARQ 反馈以及盲检预算计算的讨论基调，使得 3GPP 能够在不列举繁杂 RNTI 的前提下，优雅地制定不同业务类型的 PDCCH 监听规则。**

## 📖 协议原文拆解 (三)：顶级公告栏的“寻址凭证”与“硬核排版” (Type0 CSS)

> **协议原文**
> For a DL BWP, if a UE is not provided searchSpaceSIB1 for Type0-PDCCH CSS set by PDCCH-ConfigCommon, the UE does not monitor PDCCH candidates for a Type0-PDCCH CSS set on the DL BWP. The Type0-PDCCH CSS set is defined by the CCE aggregation levels and the number of PDCCH candidates per CCE aggregation level given in Table 10.1-1.

### 📚 增量术语表
* **searchSpaceSIB1**：SIB1 搜索空间配置。**一句话解释**：基站发给你的一张“藏宝图”，明确指出了专门用来发布 SIB1（全网基础规章制度）的顶级黑板（Type0 CSS）到底藏在哪个时间和频率上。
* **CCE aggregation levels (AL)**：CCE 聚合等级。**一句话解释**：写通知单用的“字号大小”。等级越高（比如 AL=8 或 16），代表用了越多的 CCE 积木块来拼一张纸条。字越大，冗余度越高，在边缘信号极差的地方也越容易看清。

### 🚪 第一关：没给藏宝图，就别瞎找！
在 5G 里，手机可以工作在不同的带宽片段（DL BWP）上，就像在大楼的不同楼层办公。
协议第一句话是个**免责声明**：
如果基站在当前你所在的这层楼（DL BWP）的公共配置里，**没有**给你下发 `searchSpaceSIB1` 这张藏宝图。
那么手机的反应应该是：**直接摆烂，绝对不要在这个 BWP 上去瞎找 Type0-PDCCH 这块顶级黑板！**

**物理意义**：
Type0 (找 SIB1 的黑板) 是极其特殊的。基站并不一定在所有的 BWP（所有的频率片段）上都广播 SIB1。如果基站没配，说明这层楼根本没有设这个“国家级公告栏”，手机去盲检纯粹是浪费算力和电量。

### 🎯 第二关：国家级公告栏的“固定排版” (Table 10.1-1)
如果基站确实配了藏宝图，手机找到了这块 Type0 黑板。接下来该怎么看上面的纸条呢？
由于 Type0 管的是整个基站最致命的系统消息（SIB1），它的排版**绝对不能任性，必须极其死板和稳定**。
协议直接规定：这块黑板上，字号有几种（AL 是多少），每种字号的纸条贴了几张（PDCCH candidates 数量），**不能由基站随意动态配置，必须死死遵守 3GPP 协议书里的 `Table 10.1-1`！**

*（提前剧透：Table 10.1-1 通常规定，Type0 黑板上只有“大字号”，比如 AL=4 的纸条有 4 张，AL=8 的有 2 张，AL=16 的有 1 张。绝不允许出现 AL=1 这种小字号，因为系统广播必须保证全小区最边缘的差信号用户也能一眼看清！）*

```text
        ├─► 基站给配了 searchSpaceSIB1 (藏宝图) 吗？
        │
        ├──► 【没给】 ──► ❌ 本地没有这个公告栏！手机不监听，省电！

                    (所有厂家基站必须统一用这个排版，手机闭着眼睛按这个数量解)
```

### 💡 章节硬核总结

**本段协议在 L1 层面定义了 Type0-PDCCH CSS（RMSI 调度空间）的使能条件与搜索空间内部拓扑结构。作为 5G SA (Standalone) 组网下确立小区存在的生命线，Type0 CSS 具有高度的刚性约束。首先，协议通过 `PDCCH-ConfigCommon` 中的 `searchSpaceSIB1` 字段的存在与否，赋予了网络侧在非初始 BWP (Non-initial BWP) 上灵活开启或闭塞 RMSI 广播的能力，并同步免除了终端的无谓盲检负担。其次，为了最大化初始接入状态下控制信道的鲁棒性（Robustness），协议剥夺了网络侧动态配置 Type0 盲检候选数（Candidates distribution）的自由度，强行将其硬编码（Hardcoded）至 `Table 10.1-1`。这确保了 RMSI 的 DCI 必然以中高聚合等级（AL 4/8/16）进行编码，从而保障了小区边缘的高覆盖率。**

## 📖 协议原文拆解 (四)：给最牛的黑板贴上“0号专属标签”的条件

> **协议原文**
> If the active DL BWP and the initial DL BWP for a UE have same SCS and same CP length and the active DL BWP includes all RBs of the CORESET with index 0, or the active DL BWP is the initial DL BWP, or the active DL BWP includes all RBs of an MBS frequency resource provided by cfr-ConfigMCCH-MTCH or cfr-ConfigMCCH-MTCH-RedCap as described in clause 18, the CORESET configured for Type0-PDCCH CSS set has CORESET index 0 and the Type0-PDCCH CSS set has search space set index 0.

### 📚 增量术语表
* **CORESET 0 (CORESET with index 0)**：0 号黑板。**一句话解释**：这是整个 5G 系统里最早挂出来、最神圣的一块黑板（也就是我们在上一段讲过的 Type0 CSS 寄宿的地方）。它的位置是在手机刚开机、通过解析主同步信号（MIB）算出来的。
* **Search space set index 0**：0 号排班表。**一句话解释**：专门指导手机什么时候去看“0号黑板”的作息时间表。
* **CP length**：循环前缀长度。**一句话解释**：用来抗无线电多径回声干扰的信号前置缓冲垫。
* **MBS frequency resource**：多播广播业务频率资源。**一句话解释**：专门划出来发组播视频/数据的资源块（R17 引入）。

### 🚪 第一关：神圣的“0号”标签，不是随便贴的
基站在建网时，会挂出很多块黑板（CORESETs），也会列出很多个看黑板的时间表（Search Spaces）。为了方便管理，这些资源都会被打上编号（Index：0, 1, 2, 3...）。
但是，**数字 `0` 在这里是具有特权的！** 它代表着系统初始化的基石（Type0-PDCCH，管 SIB1 广播的）。
这段协议的意思是：即使手机现在已经切换到了一个日常工作的宽带（Active DL BWP），如果满足了一定的“兼容条件”，我们依然要给管 SIB1 广播的这块黑板和时间表，强行贴上 `0号` 的特权标签。

### 🎯 第二关：拿到“0号标签”的三个充分条件
手机当前所处的频段（Active DL BWP），只要满足以下**任意一个条件**，它里面负责 Type0 广播的黑板和排班表，就能合法继承 `0号` 标签：

1. **“我在新手村没走”**：当前所在的就是刚开机时的那个“初始带宽（Initial DL BWP）”。老家就是 0 号的诞生地，毫无争议。
2. **“我的地盘完美包裹了0号祖宅”**：当前宽带虽然变宽了，但它和初始宽带的频率节拍（SCS）一样，抗回声长度（CP length）也一样，并且最关键的是：**现在的宽带在物理上，完完全全地把当年老家的那块“0号黑板（CORESET 0 的所有 RB）”给包进去了！** 既然包进去了，那块区域就继续叫它 0号黑板。
3. **“特批的组播专区”**：如果当前宽带包含了基站特批的组播频率资源（MBS），这也是新特性赋予的直接继承 0号 标签的合法特权。

### 🔄 0号黑板(CORESET 0)的索引继承树
```text
【当前手机工作在 Active DL BWP，准备给负责 SIB1 调度的资源打标签】

        ├─ 检查条件1：当前是初始带宽吗？ ──► (是) 继承 0号 标签 ✅
        │
        ├─ 检查条件2：当前带宽包含了 MBS 组播资源吗？ ──► (是) 继承 0号 标签 ✅
        │
        └─ 检查条件3：当前带宽的物理环境 (SCS和CP) 和初始状态一致吗？
               │
               └─► 并且物理上包含了原始的 CORESET 0 的所有格子吗？ ──► (是) 继承 0号 标签 ✅

【继承成功的结果】：
物理层将该黑板的 ID 硬件死锁为：CORESET Index = 0
物理层将看该黑板的排班表 ID 死锁为：Search Space Index = 0
```

### 💡 章节硬核总结

**本段协议在 RRC 配置层面对系统生命线——CORESET 0 与 Search Space 0 的“索引继承性（Index Inheritance/Aliasing）”进行了严格的数学与拓扑定义。在 5G SA 组网中，CORESET 0 的物理时频参数由 MIB (Master Information Block) 通过极简的 8-bit `pdcch-ConfigSIB1` 开环指示生成。当终端由 Initial BWP 切换至激活态业务 BWP（Active DL BWP）时，若新 BWP 在 Numerology（SCS+CP）上与初始态对齐，并在频域（RB 粒度）上实现了对物理 CORESET 0 区域的超集覆盖（Superset envelopment），协议强制保留 `ID = 0` 的索引映射。这一“ID穿透”设计，使得基站无需在每一个业务 BWP 内重新消耗高层信令去重定义 SIB1 的搜索空间，而是通过让底层 L1 状态机复用开机捕获的 CORESET 0 参数，完美实现了极简开销下的系统广播无缝接力。**

## 📖 协议原文拆解 (五)：只要包住了“组播电视塔”，就能在大宽带里看群播 (MBS DCI 接收)

> **协议原文**
> If the active DL BWP and an MBS frequency resource provided by cfr-ConfigMCCH-MTCH or cfr-ConfigMCCH-MTCH-RedCap or determined by CORESET with index 0 when cfr-ConfigMCCH-MTCH or cfr-ConfigMCCH-MTCH-RedCap is not provided for a UE have same SCS and same CP length and the active DL BWP includes all RBs of the MBS frequency resource, and if the UE is provided searchSpaceMCCH or searchSpaceMTCH for Type0B-PDCCH CSS set on the primary cell or for Type3-PDCCH CSS set on a secondary cell, the UE monitors PDCCH for detection of broadcast DCI formats, as described in clause 18, on the active DL BWP.

### 📚 增量术语表
* **MBS frequency resource**：多播广播业务频域资源。**一句话解释**：基站划出来专门建“电视塔”发射群播信号（比如演唱会直播、赛事推送）的那片专属区域。
* **cfr-ConfigMCCH-MTCH**：组播频段配置。**一句话解释**：告诉你那座“组播电视塔”具体建在频率的哪几个格子上。
* **Type0B-PDCCH CSS set / Type3-PDCCH CSS set**：组播专属公告栏。**一句话解释**：基站用来贴“组播节目单”的黑板。在主基站上叫 Type0B（高优），在辅基站上叫 Type3（普通）。
* **Broadcast DCI formats**：广播/组播格式通知单。**一句话解释**：贴在上面的组播节目单，里面用群聊密码（MCCH-RNTI 或 G-RNTI）加密，告诉你几点几分去哪接收视频流。

### 🚪 第一关：如何在干私活（专属 BWP）时，不错过“电视直播”？
手机进入了业务状态，切换到了属于自己干私活的专属宽带（Active DL BWP）里。
但是，小区里正在广播一场高清演唱会（MBS 多播服务）。手机怎么才能在干私活的专属宽带里，合法地接收到这场公共直播的节目单（PDCCH）呢？

协议给出了严苛的**物理包裹定理**：
你干私活的这个宽带，在物理上，必须把建“组播电视塔”的那块地盘（MBS frequency resource）**完全包裹进来（includes all RBs）**！并且频率节拍（SCS和CP）还得一致。

*(如果基站没单独划“组播电视塔”的地盘，那就默认以 `CORESET 0`（系统0号黑板）所在的那些格子作为组播电视塔的地盘，同样要求你的专属宽带必须完全包住它。)*

### 🎯 第二关：条件齐备，合法开听节目单
当物理环境满足了上述“完美包裹”的条件后，只要基站给你下发了看组播节目单的排班表（`searchSpaceMCCH` 或 `searchSpaceMTCH`）：
* **主基站 (Primary cell)**：你就在 `Type0B` 公共黑板上盯着。
* **辅基站 (Secondary cell)**：你就在 `Type3` 公共黑板上盯着。

手机底层接到任务，正式在当前的专属大宽带上，开启对**组播通知单 (broadcast DCI formats)** 的盲检监听。

### 🔄 MBS 组播节目单监听的准入逻辑树
```text
【手机在当前 Active DL BWP 内试图接收群发的组播消息】

        ├─► 检查物理环境兼容性：
        │    1. 当前宽带和组播专区的频段特性(SCS/CP)一样吗？
        │    2. 当前宽带把组播专区的所有格子(RB)都包在肚子里了吗？
        │
        ├──► 【没有完全包住】 ──► ❌ 物理环境不支持。无法同时干私活+看直播。
        │
        └──► 【完美包住】 ──► ✅ 物理环境过关！往下走
                │
                └─► 检查排班配置：
                    基站发了组播排班表(searchSpaceMCCH/MTCH)吗？
                    │
                    └─► 【发了】 ──► 📱 开始盲检行动！
                        * 在主小区去 Type0B 找纸条。
                        * 在辅小区去 Type3 找纸条。
                        一旦解出纸条(用群密码解密)，就去接收组播下行数据(MTCH)。
```

### 💡 章节硬核总结

**本段协议在 BWP (Bandwidth Part) 框架内，为 Rel-17 MBS (Multicast and Broadcast Services) 业务构建了控制平面的时频共存合法性（Coexistence Legality）检验准则。在 5G 架构中，MBS Common Frequency Resource (CFR) 通常作为一个较窄的公共频域带出现。当 UE 处于 Connected 态并在一个较宽的 Dedicated Active DL BWP 内部运行时，协议不允许跨 BWP 的跳频监听。相反，它实施了“超集包容（Superset Envelopment）”原则：只有当 Active DL BWP 的物理时频网格（Grid）完全无损地囊括了 MBS CFR 的所有 RBs，且 Numerology（SCS+CP）对齐时，UE 才被允许在当前 BWP 内部复用其基带接收机，去盲检用于调度 MCCH/MTCH 的 `Type0B` (PCell) 或 `Type3` (SCell) 搜索空间。这一设计既保护了组播业务的广播属性，又彻底免除了 UE 的射频中心频点频繁重调（RF Retuning）的巨大开销。**

## 📖 协议原文拆解 (六)：副号公告栏的“寻址与死板排版” (Type0A CSS)

> **协议原文**
> For a DL BWP, if a UE is not provided searchSpaceOtherSystemInformation for Type0A-PDCCH CSS set, the UE does not monitor PDCCH for Type0A-PDCCH CSS set on the DL BWP. The CCE aggregation levels and the number of PDCCH candidates per CCE aggregation level for Type0A-PDCCH CSS set are given in Table 10.1-1.

### 📚 增量术语表
* **Type0A-PDCCH CSS set**：Type0A 公共搜索空间。**一句话解释**：次顶级的系统公告栏。如果 Type0 是发“国家一号文件 (SIB1)”的，那这个 Type0A 就是发“国家二号、三号补充文件 (Other System Information, OSI)”的。
* **searchSpaceOtherSystemInformation**：找二号文件的藏宝图。**一句话解释**：基站配置的一张时间表，告诉手机去哪找 Type0A 这个公告栏。

### 🚪 第一关：依旧是“没藏宝图，就不瞎找”
这一段的话术和我们在第（三）段拆解 Type0（顶级公告栏）时一模一样。
针对普通补充广播（Other System Information, OSI）所在的 Type0A 黑板，协议同样给出了免责条款：
如果基站在当前这层楼（DL BWP）里，没有给你下发寻找它的藏宝图（`searchSpaceOtherSystemInformation`）。
**那手机就理直气壮地不去盲检这个公告栏（does not monitor）。**

**物理意义**：基站并非时时刻刻都在发补充系统消息（很多时候是按需触发的）。如果不配藏宝图，就直接在物理层掐断手机的盲猜逻辑，节省电量。

### 🎯 第二关：国家级公告，排版绝不含糊！
一旦你拿到了藏宝图找到了 Type0A 这个副号公告栏，上面贴的纸条（DCI）该怎么解密呢？
协议规定：既然 Type0A 也是发“国家级系统消息”的，那它的重要性同样不容小觑。
**它纸条的字号大小（CCE Aggregation Levels）和盲检张数，同样不能任由基站随意设置！必须和顶级公告栏一样，死死钉在 3GPP 出厂字典的 `Table 10.1-1` 里！**

必须使用高冗余度的大字号（如 AL=4, 8, 16），确保躲在地下室或小区边缘的手机也能看清这些系统补充条款。

### 🔄 Type0A 黑板的配置与排版逻辑
```text
【手机寻找用来收“其他系统消息(OSI)”的 Type0A 公告栏】

        ├─► 基站给配了 searchSpaceOtherSystemInformation 吗？
        │
        ├──► 【没给】 ──► ❌ 本地没有这个公告栏！手机不监听，省电。
        │
        └──► 【给了】 ──► ✅ 找到公告栏！准备开始盲解。
                │
                └─► 【纸条的排版是怎么样的？】
                    和老大哥 Type0 一样死板：
                    翻开出厂写死的 Table 10.1-1：
                    只能使用规定好的中大号字体 (AL) 和对应的固定盲检次数！
```

### 💡 章节硬核总结

**本段协议定义了 Type0A-PDCCH CSS（用于调度 Other System Information, OSI）在 L1 层的使能与资源属性框架。在 5G 的系统信息分发架构中，除了必送的 SIB1 (由 Type0 CSS 调度)，其他的 SIB（如 SIB2~SIB9，包含小区重选、震灾预警等）被称为 OSI，采用 On-demand（按需）或周期性广播。协议通过 `searchSpaceOtherSystemInformation` 这个 RRC 钩子（Hook），控制着底层 L1 是否要挂载 Type0A 的盲检状态机。一旦挂载，为了保证这些涉及公共安全的广播信息（Broadcasting Payload）能够克服最差的无线电信道衰落，协议强行剥夺了 RRC 层对聚合等级（Aggregation Level）分布的重配置权限，将其与最高优先级的 Type0 共享同一个写死的硬编解码映射表（Table 10.1-1），以此提供系统级广播链路在物理层的绝对覆盖下限保障。**

## 📖 协议原文拆解 (七)：入网接待处的“客串”功能 (Type1 CSS 的临时征用)

> **协议原文**
> For a DL BWP, if a UE is not provided ra-SearchSpace for Type1-PDCCH CSS set, the UE does not monitor PDCCH for Type1-PDCCH CSS set on the DL BWP. If the UE has not been provided a Type3-PDCCH CSS set, or a Type1A-PDCCH CSS set, or a USS set and the UE has received a C-RNTI and has been provided a Type1-PDCCH CSS set, the UE monitors PDCCH candidates for DCI format 0_0 and DCI format 1_0 with CRC scrambled by the C-RNTI in the Type1-PDCCH CSS set.

### 📚 增量术语表
* **Type1-PDCCH CSS set**：Type1 公共搜索空间。**一句话解释**：这是小区的“入网接待处”。它最本职的工作是用来发送随机接入（RACH）的回执（如 RAR, Msg4）。
* **ra-SearchSpace**：随机接入藏宝图。**一句话解释**：告诉你接待处在什么时候开门的时刻表。
* **C-RNTI (Cell RNTI)**：手机的私有密码。**一句话解释**：手机入网成功后，基站给手机发的一个身份证号，以后基站和你一对一聊天（单播业务）都用它加密。

### 🚪 第一关：没配“接待处时刻表”，就不盲检
开头老规矩：如果基站连 `ra-SearchSpace` 这份告诉你接待处开门时间的时刻表都没发给你。
手机反应：直接不找（**does not monitor**）。既然基站不打算在这个带宽上接待新人，你也没必要去浪费电量瞎解密。

### 🎯 第二关：当你拿到了身份证，但还没分到“私人信箱”时
这段协议后半句极其精彩，描述了一个 5G 终端在入网过程中的**“尴尬真空期”**。
* **前提 1**：你刚入网成功，基站刚刚给你发了专属身份证密码（`C-RNTI`）。
* **前提 2**：但是！基站的建链流程比较慢，它还没来得及给你分配你的私人专属信箱（`USS set`），甚至连居委会的黑板（`Type3`）和小睡的黑板（`Type1A`）都没配给你。

**这时候，基站怎么给你发私人信息（个人业务数据）呢？**
难道基站只能等你有了私人信箱才能和你发微信吗？
协议说：**不行，不能等！直接征用“接待处（Type1 CSS）”！**

### 🧩 第三关：“接待处”的临时客串 (Fallback 机制)
协议规定：在这种真空期，虽然 `Type1` 黑板原本是用来处理大家排队入网（找 RA-RNTI 密码）的公共地盘。
但既然你现在只有这个“接待处”的大门可以走，你就要在这个 `Type1` 黑板上，**用你自己的私有密码（C-RNTI）** 去解密纸条！
并且，纸条的格式只能用最基础、最简单的保底格式：
* **`DCI format 0_0`** (发简单上行包裹用的)
* **`DCI format 1_0`** (收简单下行包裹用的)

**生活类比**：
你是来公司面试的新人。
第一天你面试通过了，HR给了你一个属于你自己的**正式工牌（C-RNTI）**。
但是行政部动作太慢，还没给你分配**私人专属工位（USS）**。
这时候老板突然要给你派活儿（个人业务）。因为你没有工位，老板只能把贴着你名字的通知单，强行贴在了一楼大堂的**“新人接待处公告栏（Type1 CSS）”**上。
所以，你拿到工牌后，每天经过一楼接待大堂时，不仅要看有没有自己的入职回执，还得顺便瞅一眼有没有老板用你的工牌号单独给你贴的派活小纸条（Format 0_0/1_0）。

### 🔄 Type1 公告栏的客串机制图
```text
【手机刚拿到 C-RNTI 时的过渡期监听逻辑】

        ├─► 手机检查：我有自己的私人信箱 (USS) 或 其他备用公共黑板吗？
        │
        ├──► 【有配置 USS 等】
        │       └─► 正常干活。去自己的私人信箱收用 C-RNTI 加密的个人派单纸条。
        │
        └──► 【没有任何其他黑板配置】 (尴尬真空期)
                └─► 触发底层 Fallback 机制！
                    去一楼接待大堂的 【Type1 公共公告栏】 蹲守！
```

### 💡 章节硬核总结

**本段协议揭示了 5G 物理层极其优雅的“控制面切换无缝回退（Control Plane Handover Fallback）”机制。在 RRC 状态机中，终端从完成 Contention Resolution（获取 C-RNTI）到真正接收全量 RRCReconfiguration 消息建立 Dedicated USS 之间，存在一段业务真窗期。如果此时 gNB 无法通过 C-RNTI 调度 UE，那么后续的 RRC 专属配置消息（如 SRB/DRB 的建立）也就无法下发，系统将陷入死锁。为此，协议巧妙地“重载（Overload）”了 Type1-PDCCH CSS 的物理意义：强制终端在缺失 USS 时，利用已有的 Type1 CSS 搜索空间，挂载其新获得的 C-RNTI 进行 DCI 0_0/1_0 的盲检。这一设计利用了 Type1 CSS 天生的高覆盖高鲁棒性，充当了安全接渡至正式业务态的物理层“摆渡船”。**

## 📖 协议原文拆解 (十)：拿了“0号黑板”特权的各种公共大喇叭，该怎么找对波束？

> **协议原文**
> If a UE is provided a zero value for searchSpaceID in PDCCH-ConfigCommon for a Type0/0A/1A/2-PDCCH CSS set, the UE determines monitoring occasions for PDCCH candidates of the Type0/0A/1A/2-PDCCH CSS set as described in clause 13, and the UE is provided a C-RNTI, the UE monitors PDCCH candidates only at monitoring occasions associated with a SS/PBCH block, where the SS/PBCH block is determined by the most recent of
> - a DCI format or a MAC CE activation command indicating a TCI state of the active BWP that includes a CORESET with index 0, as described in [6, TS 38.214], where the TCI-state includes a CSI-RS which is quasi-co-located with the SS/PBCH block, or
> - a random access procedure that is not initiated by a PDCCH order that triggers a contention-free random access procedure, or
> - configured-grant based PUSCH transmission in RRC_INACTIVE state as described in clause 19.1.

### 📚 增量术语表
* **Type0/0A/1A/2-PDCCH CSS set 获得 zero value**：0号特权“大联欢”。**一句话解释**：上一段只说了 Type2A (PEI) 能借用 0号标签，这段直接宣布：几乎所有的公共大喇叭（Type0看SIB1、Type0A看其他广播、Type1A看小数据、Type2看寻呼），只要它们分配到的 `searchSpaceID` 是 0，统统触发魔法，全盘照抄第 13 章的系统时刻表。
* **most recent of (最近的一次...)**：更新波束锚点的法则。**一句话解释**：手机虽然抄了系统时刻表，但是看黑板时天线（波束）该朝哪个方向对准呢？答案是：找离你“最近发生的那次”有效通信动作借波束！

### 🚪 第一关：0号特权扩容，且有了“身份证(C-RNTI)”后
前提条件是：
1. 这些庞杂的公共大喇叭（0, 0A, 1A, 2）都被赋予了 **ID=0** 的特权，都在借用最原始的第 13 章公式计算“什么时候去看黑板（monitoring occasions）”。
2. 手机此时已经不再是完全没户口的流浪汉了，它已经**成功获取了私有密码 (`C-RNTI`)**，说明它处于连接态（Connected）或具备某种身份了。

此时，协议加上了一个**空间波束（Spatial Beam）维度**的紧箍咒：
你不能漫无目的地乱看！你只被允许在那些**与特定“救护车（SS/PBCH block）”相关联的时刻**去盲检。
一句话：时域上到了点，空域上必须对准特定的波束，才准看黑板！

### 🎯 第二关：波束对准谁？(三种“借波束”大法，选离现在最近的)
那么，这个特定的“救护车（SSB 波束）”是哪一辆呢？
协议规定，去翻你的回忆录，找离当前时间**最近的（most recent）**一次发生过的、具有以下三种特征的事件，直接“抄袭”那个事件里的波束：

* **事件 1（来自高级调度的波束更新）**：
  最近，基站刚好通过底层的调度信令（DCI）或者 MAC 层的激活命令（MAC CE），给你当前宽带里那个神圣的“0号黑板（CORESET 0）”更新了波束参数（TCI State）。而那个参数里，刚好指明了关联某一个特定的 SSB。**那就用这个 SSB 的波束！**

* **事件 2（来自自己主动爬起来敲门）**：
  最近，你自己主动发起了一次随机接入（注意：不能是基站发 PDCCH 强迫你发起的那种免竞争接入）。在发起那次接入时，你自己选定了一个认为信号最好的 SSB 波束去敲门的。**那就直接延用这个你刚刚验证过的好波束！**

* **事件 3（来自休眠态的“免许可偷跑”）**：
  最近，你在半睡半醒（Inactive）状态下，利用基站早就批给你的资源（Configured-grant），偷偷发了一个小包数据。当时发数据你肯定用了一个锚定的波束。**那就用这个休眠态偷发时的波束！**

### 🔄 CSS “借波束”更新算法状态机
```text
【手机准备去看那些披着 0号马甲的公共大喇叭】
     (前提：手机已经有 C-RNTI 户口了)

        ├─► 算出了应该在什么时候看 (抄第13章时间表)
        │
        └─► 【该往哪个方向看呢？ (确定 QCL SSB 波束)】
               │
               └─► 回忆过往，找寻最近发生的一次 "有效更新点"：
                   事件池：
                   [ 2秒前 ] 发生过一次自己主动发起的 RACH 接入 (选了波束A)
                   [ 1秒前 ] 收到一条 DCI 把 CORESET 0 的波束指向了波束B
                   [半秒前] 休眠态下利用免授权资源发了个包，用了波束C

                   【判决】：离现在最近的是半秒前的事件！
```

### 💡 章节硬核总结

**本段协议在 L1 层面解决了一个极其关键的“波束对应更新（Beam Alignment Update）”难题。当多个 CSS 簇（Type0/0A/1A/2）通过 `searchSpaceID=0` 发生配置重载，并全面复用 CORESET 0 的时频资源时，UE 必须明确自己物理层接收机的空间接收滤波器（Spatial Rx Filter）该如何动态演进。由于终端已具备 C-RNTI，说明其不再处于盲目的初始态，网络与终端间已经具备了某种闭环交互的可能。协议通过极具操作性的 `most recent of` 时间序回溯法则，定义了一个动态的波束锚定状态机（Dynamic Beam Anchoring State Machine）。它允许 UE 从高层 MAC CE 指示、L1 自发 RACH 寻呼、或 CG-SDT 等多种异步发生的物理过程中，汲取最新鲜的信道空间特征，将其强行投影并继承到当前这些 0 号 CSS 的盲检任务中，确保在高速移动或多径恶化的环境下，终端永远使用“当下最准的一把伞”去接收全网公共控制信息。**

## 📖 协议原文拆解 (十一)：公共大喇叭没捞到“0号”特权时，该怎么找？(非零 ID 的处理)

> **协议原文**
> If a UE monitors PDCCH candidates for DCI formats with CRC scrambled by a C-RNTI and the UE is provided a non-zero value for searchSpaceID in PDCCH-ConfigCommon for a Type0/0A/1A/2-PDCCH CSS set, or monitors PDCCH candidates for DCI formats with CRC scrambled by a MCCH-RNTI or a G-RNTI for broadcast and the UE is provided a non-zero value for searchSpaceMCCH and searchSpaceMTCH in PDCCH-ConfigCommon for a Type0B-PDCCH CSS set, or monitors PDCCH candidates for DCI formats with CRC scrambled by a Multicast MCCH-RNTI or a G-RNTI for multicast in RRC_INACTIVE state and the UE is provided a non-zero value for searchSpaceMulticastMCCH and searchSpaceMulticastMTCH in PDCCH-ConfigCommon for a Type0B-PDCCH CSS set, the UE determines monitoring occasions for PDCCH candidates of the Type0/0A/1A/2-PDCCH CSS set, or of the Type0B-PDCCH CSS set, respectively, based on the search space set associated with the value of searchSpaceID.

### 📚 增量术语表
* **non-zero value for searchSpaceID**：非零的搜索空间编号。**一句话解释**：这是和上一段的 `zero value`（0号特权马甲）相对应的。基站这次没偷懒，老老实实地给这些公共公告栏分配了各自专门的新排班表，比如 1号排班表、5号排班表。

### 🚪 第一关：漫长的业务清单，只为描述一个“常规状态”
这段协议的前半部分是一串极其冗长、让人眼花缭乱的 `if` 罗列。但其实它只是在穷举各种复杂的业务场景：
1. 连上网的手机（用 `C-RNTI` 解码），要看普通的公共公告栏（Type 0/0A/1A/2）。
2. 想看电视节目的手机（用 `MCCH/G-RNTI` 解码），要看组播节目公告栏（Type 0B）。
3. 休眠态还想看电视节目的手机（用 `Multicast MCCH/G-RNTI` 解码），也要看专门的组播公告栏（Type 0B）。

不管上面是哪种花里胡哨的场景，只要它们面临的是同一个核心条件：
**它们寻找公告栏所使用的藏宝图编号（searchSpaceID 等），是一个“非零的值（non-zero value）”！**

### 🎯 第二关：没有特权，就照规矩办事 (老实按 ID 去找)
既然基站没有赋予你 `0 号` 这个能够让你无脑去抄系统底层的“特权马甲”，那么规矩就变得极其简单直接：
协议的结论是：`based on the search space set associated with the value of searchSpaceID.`

**物理意义**：
你拿到了几号藏宝图，就乖乖地去基站下发的高级 RRC 配置信令堆里，翻出那张对应编号图纸的具体参数。
图纸上写着周期是 10个时隙，偏移量是 2，你就严格按照公式算，在每 10个时隙的第 2天去看黑板。**绝不能再去抄第 13章 SIB1 的那些特殊时间表！**

**生活类比**：
（上一段 0号特权）：老板给你发了一个工牌号为 `0` 的通行证。你一看是 0号，连问都不用问，每天早上升国旗（播 SIB1）的时候直接混进队伍里去收通知就行了，那是全公司的统一法定集会时间。
（这一段 非零号）：老板给你发了一个工牌号为 `5` 的通行证。你就没资格去蹭大集会的时间了。你必须翻开厚厚的员工手册，找到第 `5` 号班表的具体考勤规定（比如一三五下午打卡），严格按这个独立的时间表去执行。

### 💡 章节硬核总结

**本段协议是针对 CSS 搜索空间索引分配的“显式回退（Explicit Fallback）”或称“常规化（Normalization）”声明。3GPP 协议设计中往往存在大量基于特殊索引（如 Index 0 赋予的 RMSI 同步复用特权）的捷径（Shortcuts）。但在更为丰富的网络拓扑中，gNB 有时需要将寻呼、OSI 或 MBS 广播信道从与 SIB1 同步的网格中剥离出来，进行错峰调度以缓解 CORESET 0 的信道拥塞。这就要求给这些 CSS 分配 `non-zero ID` 的搜索空间。本段协议以极其完备的数学罗列，覆盖了 C-RNTI、MCCH-RNTI、RRC_INACTIVE 态 Multicast 等诸多现代场景，重申了一条公理：一旦剥离了 Index 0 的魔法属性，所有控制平面业务都必须回归到基于该非零 ID 的严格 ASN.1 RRC 参数树的闭环解析上，绝不允许发生跨状态机的隐式时域参数借用。**

## 📖 协议原文拆解 (十二)：混沌初开时的“波束大一统”与频率节拍 (CORESET 0 QCL 与 SCS 假设)

> **协议原文**
> The UE may assume that the DM-RS antenna port associated with PDCCH receptions in the CORESET configured by pdcch-ConfigSIB1 in MIB, the DM-RS antenna port associated with corresponding PDSCH receptions, and the corresponding SS/PBCH block are quasi co-located with respect to average gain, quasi co-location 'typeA' and 'typeD' properties, when applicable [6, TS 38.214], if the UE is not provided a TCI state indicating quasi co-location information of the DM-RS antenna port for PDCCH reception in the CORESET. The value for the DM-RS scrambling sequence initialization is the cell ID. For operation without shared spectrum channel access in FR1, FR2-1 and FR2-NTN, a SCS is provided by subCarrierSpacingCommon in MIB. For operation with shared spectrum channel access in FR1 and for operation in FR2-2, a SCS is same as the SCS of a corresponding SS/PBCH block.

### 📚 增量术语表
* **pdcch-ConfigSIB1 in MIB**：MIB里的SIB1寻找配置。**一句话解释**：手机一开机收到的第一句话（MIB，极其短小），里面藏着一个 8-bit 的线索，指引手机去找到最初的那块“0号黑板”（也就是这个条款里的 CORESET）。
* **quasi co-location 'typeA' and 'typeD' properties**：准共址属性 A 和 D。**一句话解释**：物理层的专业黑话。`TypeA` 代表多普勒频移、多径延迟一样（说明信号走过的路长得一样）；`TypeD` 代表空间接收波束一样（说明可以用同一个角度的天线去收）。
* **DM-RS scrambling sequence initialization**：导频加扰序列初始化值。**一句话解释**：生成防窃听密码本的种子值。
* **subCarrierSpacingCommon**：公共子载波间隔。**一句话解释**：在 MIB 广播里告诉全网的统一频率节拍。

### 🚪 第一关：开机初期的“波束大一统”假设
当手机刚刚开机，通过极短的 MIB 找到了这块用于调度的“0号黑板（CORESET 0）”，它面临一个最要命的问题：
**基站还没来得及给我下发波束配置（没有 TCI state），我该怎么调整天线去收这块黑板上的信号，以及它带来的系统广播数据（PDSCH）？**

协议给出了初始态的终极奥义：**“万物同源，死死绑定 SSB！”**
只要你没有被专门配置 TCI 波束，手机就可以大胆假设（`may assume`）：
1. 0号黑板上的指令（PDCCH）
2. 指令调度送来的系统广播包裹（PDSCH）
3. 帮你找到基站的那辆救护车（SSB 同步信号块）

这三者的无线电波完全是从同一个发射点、沿着同样的空气路径、以同样的波束方向打过来的！
它们在**空间接收角度（TypeD）**和**多径延迟（TypeA）**、乃至**平均增益（Average Gain）**上，完全保持一致！
手机只需用最初锁定 SSB 的那套天线姿态，一动不动，就能完美接收后续的 PDCCH 和 PDSCH。

### 🎯 第二关：极其暴力的“密码种子” (Cell ID)
收信号的方向搞定了，那么信号解密的引子用什么呢？
普通的业务里，导频（DM-RS）的加扰密码很复杂。但在这里，为了追求绝对的健壮性和傻瓜化，协议规定：
**生成导频密码本的原始种子，直接粗暴地等于这个基站的物理小区ID (Cell ID)！** 手机连算都不用算，直接用 Cell ID 就能解出导频进行信道估计。

### 🧩 第三关：频率节拍 (SCS) 该听谁的？
最后，收信号的频率节拍（SCS）该用多大的？这里分了两大阵营：
* **阵营 A（传统买牌照的频段：FR1，FR2-1，以及卫星 FR2-NTN）**：
  直接去读取 MIB 广播里的 `subCarrierSpacingCommon` 字段。基站说了算。
* **阵营 B（免授权共享频段 NR-U，以及极高频太赫兹 FR2-2）**：
  在这类恶劣或极其宽泛的频谱里，甚至不依赖 MIB 指示。直接套用**“惯性法则”**：刚才你是用多大的节拍收到 SSB（救护车）的，你现在看黑板就继续用多大的节拍！

```text
【手机刚开机，准备破解第一块用于 SIB1 的 0号黑板】

 ├─► 【空间波束对准谁？】
 │      └─ 无脑跟随找到我的那个 SSB！(Type A/D QCL 绑定)
 │
 ├─► 【导频密码用什么解？】
 │      └─ 直接用本基站的身份证号 (Cell ID) 作为密码种子！
 │
 └─► 【频率节拍 SCS 怎么定？】
        ├── 传统频段 ──► 听 MIB 里的统一指挥 (subCarrierSpacingCommon)。
        └── 共享/极高频 ──► 不用听指挥，直接沿用刚才解 SSB 时的那个节拍！
```

### 💡 章节硬核总结

**本段协议是 5G 终端完成从“小区搜索（Cell Search）”跨越至“系统消息获取（System Information Acquisition）”的“第一推动力（First Mover）”。在 RRC 建立前的极端开环阶段，终端对物理信道的认知几乎为零。3GPP 展现了极致的工程解耦与降维设计：将 CORESET 0 的信道估计（DM-RS Scrambling Initialization = Cell ID）退化至最基本的小区常量；将最棘手的空间域多天线滤波问题（Spatial Filtering），通过强硬的 `QCL TypeA/TypeD` 假设，强绑定至早已解出的 SS/PBCH block 上。此外，针对频段特征的二分法设计，将 Shared Spectrum (NR-U) 及 FR2-2 等非常规部署环境中的 CORESET 0 SCS 参数直接通过同步信号锚定，进一步削减了 MIB 开销与 L1 状态机的依赖树深度。这构建了一个固若金汤的 Initial Access 物理层底座。**

## 📖 协议原文拆解 (十三)：波束冲突的“一票否决权”！当私人信件撞上入网回执

> **协议原文**
> For single cell operation or for operation with carrier aggregation in a same frequency band, a UE does not expect to monitor a PDCCH in a Type0/0A/0B/2/3-PDCCH CSS set or in a USS set if a DM-RS for monitoring a PDCCH in a Type1-PDCCH CSS set is not configured with same qcl-Type set to 'typeD' properties [6, TS 38.214] with a DM-RS for monitoring the PDCCH in the Type0/0A/0B/2/3-PDCCH CSS set or in the USS set, and if the PDCCH or an associated PDSCH overlaps in at least one symbol with a PDCCH the UE monitors in a Type1-PDCCH CSS set or with an associated PDSCH.

### 📚 增量术语表
* **Type1-PDCCH CSS set**：入网接待处/随机接入公告栏。**一句话解释**：专门收发随机接入（RACH）回执的地方。优先级在动态业务中极高，因为它关系到手机能不能顺利接入网络。
* **qcl-Type set to 'typeD'**：空间波束 QCL 类型 D。**一句话解释**：两者的波束方向是不是一样的？如果具备 typeD 特性，说明手机可以用同一个角度的天线把这两个信号一起收下来。

### 🚪 第一关：手机天线不能“左眼看东，右眼看西”
在单小区或者同频段载波聚合下，手机的接收射频头往往只有一个硬件波束焦点。在同一个时间点，手机的天线（面板）只能聚焦对准一个方向。
如果基站在**同一个时间点（哪怕只重叠了 1 个 OFDM 符号）**，给手机发了两个纸条：
* 一张是**找入网回执（Type1 CSS）**的纸条，波束方向在东边。
* 另一张是**其他的业务纸条（比如 USS 里的私人数据，或者 Type2 寻呼）**，波束方向在西边。

手机傻眼了：这俩波束方向不一样（`not configured with same qcl-Type set to 'typeD'`），我同时没法劈开天线看两边啊！

### 🎯 第二关：Type1 (入网回执) 享有“波束一票否决权”
当这种时域重叠且波束冲突的情况发生时，必须有人做出牺牲。
协议的判定极其霸道：**Type1 CSS（入网/随机接入回执）是祖宗，其他人统统让路！**

只要发现波束不同向，并且时间上有交叠：
手机**绝对不预期（does not expect）**去监听那个普通的业务黑板（USS）或者其他的公共黑板（Type0/0A/0B/2/3）。
也就是说，手机会瞬间把天线死死对准 Type1 的方向去解密随机接入回执，而**直接放弃**所有其他黑板上的纸条。

**生活类比**：
你在公司的工位上只有一双眼睛。
老板 A（Type1，管入职转正的 HR）在东门贴了一张对你至关重要的转正通知单。
主管 B（USS，管日常派单的 leader）在西门贴了一张日常工作单。
并且这俩人要求你必须在 10:00:00 整这一秒钟同时看完两张单子。
你的反应（协议底层逻辑）：这根本做不到！既然做不到，那转正通知（Type1 RACH）事关生死，我直接盯着东门看，西门的那张派单我直接当没看见（放弃盲检），哪怕被扣绩效也在所不惜。基站你瞎排班，责任你负（`does not expect`）。

```text
【同一时间点 (有符号重叠)，手机发现有两张纸条要看】

 ├─ 纸条 A: 属于 Type1 CSS (随机接入回执)  ──► 【波束方向：东】
 │

     冲突判定：波束方向一致吗 (QCL TypeD 相同)？
        │
        ├──► 【方向一致】 ──► ✅ 没问题！手机用同一个天线角度顺便把两张纸条一起解了。
        │
        └──► 【方向不一致】
                └─► 🚨 物理碰撞！天线无法分身！
```

### 💡 章节硬核总结

**本段协议是多波束（Multi-beam）5G 系统中，应对 Intra-band 射频资源抢占的经典 QoS（服务质量）底层硬路由法则。在毫米波或强波束赋形的场景中，UE 的模拟波束成形器（Analog Beamformer）在同一 OFDM 符号内仅能维持唯一一个空间滤波器状态（Spatial Domain Filter）。当 gNB 调度器出现策略缺陷，将承载 RAR/Msg4 的 `Type1 CSS` 与承载其他业务的搜索空间安排在时域交叠（Time-overlapping）且空间分离（Non-QCL-TypeD）的资源位时，协议强制实施了 L1 级的断言保护：Type1 CSS 享有绝对抢占权（Preemption Privelege）。UE 通过 `does not expect` 原语合法地 Drop（丢弃）掉冲突的次优 PDCCH candidate。这种将冲突解决机制固化至物理层基带的设计，确保了最关键的 RRC/MAC 层连通性控制流不会因为偶然的下行波束调度冲突而发生致命断链。**

## 📖 协议原文拆解 (十四)：“顺手牵羊”大法！公共大黑板上的私人小纸条 (CSS 借壳调度)

> **协议原文**
> If a UE is provided
> - one or more search space sets by corresponding one or more of searchSpaceZero, searchSpaceSIB1, searchSpaceOtherSystemInformation, pagingSearchSpace, ra-SearchSpace, and
> - a C-RNTI, an MCS-C-RNTI, or a CS-RNTI
> the UE monitors PDCCH candidates for DCI format 0_0 and DCI format 1_0 with CRC scrambled by the C-RNTI, the MCS-C-RNTI, or the CS-RNTI in the one or more search space sets in a slot where the UE monitors PDCCH candidates for at least a DCI format 0_0 or a DCI format 1_0 with CRC scrambled by SI-RNTI, RA-RNTI, MsgB-RNTI, or P-RNTI.

### 📚 增量术语表
* **searchSpaceZero, searchSpaceSIB1... 等**：各种公共藏宝图。**一句话解释**：代表了前文提过的各种 Type0, 0A, 1, 2 等**公共黑板（CSS）**的时间表。它们本来是用来发系统广播、寻呼和随机接入的。
* **C-RNTI / MCS-C-RNTI / CS-RNTI**：手机的私人密码。**一句话解释**：用来解密属于手机个人的专属业务数据。
* **SI-RNTI / RA-RNTI / P-RNTI**：公共密码本。**一句话解释**：用来解密广播、入网回执和寻呼信号的公用钥匙。

### 🚪 第一关：公私分明，还是公器私用？
正常情况下，私人密码（C-RNTI）应该去私人信箱（USS）里开锁；公共密码（SI-RNTI等）应该去公共黑板（CSS）上开锁。
但是，这段协议开启了一个极其聪明的**“偷懒/借壳”机制**。

* **条件 1**：手机已经有了这几个公共黑板的看榜时间表。
* **条件 2**：手机也已经有了自己的私人密码（处于连接态或有专属身份了）。

### 🎯 第二关：在公共黑板上，顺手用私人密码试一把
协议规定：
如果今天（在某个 Slot 里），按照排班表，你**本来就要去**某个公共黑板上，用公共密码（如 SI-RNTI 找广播，或 P-RNTI 找寻呼）去解密 DCI 0_0/1_0 这种基础格式的纸条。

既然你都已经大老远跑到这块公共黑板面前了，眼睛也都盯在这些纸条上了，**千万别浪费！**
你必须同时掏出你的**私人密码（C-RNTI / CS-RNTI）**，顺便对这块黑板上的那些纸条也试着解密一次。

**为什么能这么做？（物理意义）**
因为 `DCI format 0_0` 和 `1_0` 是 5G 里最基础的 Fallback（保底）格式。它们有一个极其特殊的性质：**不论是用公共密码加密，还是用私人密码加密，它们在这块黑板上占据的纸条大小（Payload size）是一模一样的！**
对于手机芯片的解码器来说，解同样大小的纸条，只是换个密码本校验一下 CRC 而已，几乎不增加额外的底层硬件负担（这种操作叫同一个 Size 的复用盲检）。

**生活类比**：
你每天早上都要去小区门口的**公共公告栏（CSS）**看看有没有停水停电通知（用 SI-RNTI 解读的广播）。
居委会大妈（基站）为了省事，如果有私人的包裹要给你，她懒得单独跑一趟去塞你的**私人信箱（USS）**，她直接把写有你名字的**私人包裹提取单（C-RNTI 纸条）**也贴在了公共公告栏上。
反正你每天早上都要去看停水通知，你就**“顺手牵羊”**，用你自己的密码核对一下那些没看懂的纸条，如果解开了，恭喜你，那是基站给你夹带的私人业务！

```text
 手机到达公共黑板，开始盲解上面的所有纸条 (Candidate)
        │
        ├─► 第一遍扫视：掏出公共密码 (如 P-RNTI, SI-RNTI)
        │     └─► 咔哒！解开了。这是一条寻呼/广播通知。

              │
              └─► 咔哒！居然用 C-RNTI 解开了！

```

### 💡 章节硬核总结

**本段协议揭示了 5G 物理层极为经典的“DCI Size Alignment（DCI 尺寸对齐）”带来的调度优化福利：CSS 借壳私人调度（CSS Fallback Scheduling）。在 3GPP 设计中，DCI format 0_0 / 1_0 被刻意设计为极其精简的 Fallback 格式，并且在同一 BWP 下，使用 Common RNTI 与 C-RNTI 加扰的 format 0_0/1_0 其 Payload Size 严格相等。由于盲检次数的消耗本质上是由 Payload Size 的种类决定的，协议在此处强制规定：只要 UE 在某个 Slot 的 CSS 中被唤醒去执行公共 RNTI 的盲检，就必须附加一次 C-RNTI 的 CRC 校验。这赋予了基站极大的动态调度弹性：在网络极度拥塞、UE 的 USS 空间排不进 DCI 时，基站可以随时随地将个人业务的调度信令（C-RNTI DCI）“塞入”全网广播的 CSS 空间中完成下发，实现了零额外盲检开销下的控制面容量扩容。**

## 📖 协议原文拆解 (十五)：一天只听一次喇叭！公共信息的“防刷屏”保护机制

> **协议原文**
> If a UE is provided
> - one or more search space sets by corresponding one or more of searchSpaceZero, searchSpaceSIB1, searchSpaceOtherSystemInformation, pagingSearchSpace, pei-SearchSpace, ra-SearchSpace, or a CSS set by PDCCH-Config, and
> - a SI-RNTI, a P-RNTI, a PEI-RNTI, a RA-RNTI, a MsgB-RNTI, a SFI-RNTI, an INT-RNTI, a TPC-PUSCH-RNTI, a TPC-PUCCH-RNTI, or a TPC-SRS-RNTI
> then, for a RNTI from any of these RNTIs, the UE does not expect to process information from more than one DCI format with CRC scrambled with the RNTI per slot.

### 📚 增量术语表
* **searchSpaceZero, pei-SearchSpace, etc.**：各种公共黑板的排班表。**一句话解释**：代表手机配置了接收系统广播、寻呼、提前叫醒、随机接入等公共信息的搜索空间。
* **SI-RNTI, P-RNTI, SFI-RNTI, etc.**：一大串公共密码本。**一句话解释**：这些 RNTI 全是用来解密那些面对大众的、全网广播或群发的指令的（比如系统消息、群组功率控制、时隙格式变更等）。

### 🚪 第一关：漫长的“排比句”，只为框定一个范围
协议的前半部分，用了两串长长的排比句。
其实它只是在严谨地界定一个集合：**“只要你在这个时隙（Slot）里，去任何一块公共黑板上，用任何一种公共密码去解密公共信息。”**

### 🎯 第二关：公共信息的“去重/防刷屏”法则
协议的核心在最后一句：`the UE does not expect to process information from more than one DCI format with CRC scrambled with the RNTI per slot.`

**物理意义**：
公共信息（比如系统广播、寻呼找人、时隙格式指示）具有**全局唯一性和宏观稳定性**。
比如基站广播“本小区当前带宽是 100M”，这个信息在一天（一个 Slot）内是不可能变来变去的。
有时候，为了让信号不好的手机也能听到，基站可能会在一个时隙内的不同时间点，把同一条系统广播（或者同一个寻呼）**连续发好几遍**。
但这对于手机的芯片来说，如果每收到一次都去把里面的信息提取出来、向上层汇报一次，会导致上层系统被“刷屏”，不仅浪费电，还容易引起逻辑错乱。

因此，协议下达了霸道的**防刷屏指令（`does not expect`）**：
对于**任何一种**公共密码（比如 `P-RNTI` 寻呼），在**同一个时隙（per slot）**内，不管基站发了几张同类的纸条，也不管你在黑板上解密成功了几次，手机**最多只预期处理一次（not more than one）**里面的具体信息！

**生活类比**：
小区的大喇叭（CSS）早上开始播报：“今天停水！今天停水！今天停水！”（一个 Slot 内用 SI-RNTI 发了三次同样的 DCI）。
你的大脑（手机底层处理逻辑）：在听到第一遍“今天停水”并记下来之后，后面的第二遍、第三遍虽然你也听到了（可能在底层做个软合并提高清晰度），但你**绝对不会**往你的备忘录上写三次“今天停水”，更不会去向你妈（高层 RRC）汇报三次。一天之内，对同一种公共警报，处理一次就够了！

### 🔄 公共 DCI 信息的“时隙级防重”逻辑
```text
【在一个 Slot (1天) 内，手机监听公共黑板 (CSS)】

 ├─ 08:00 听到用 SI-RNTI 加密的广播 DCI ──► 成功解密！提取系统信息并上报。 ✅
 │
 ├─ 10:00 又听到用 SI-RNTI 加密的广播 DCI ──► 拦截！
 │    🚨 触发 [does not expect] 保护。
 │    因为今天已经处理过 SI-RNTI 的信息了，直接忽略后续重复内容。 ❌
 │
 ├─ 11:00 听到用 P-RNTI 加密的寻呼 DCI ───► 这是另一种密码。今天还没处理过。
 │    成功解密！提取寻呼信息并上报。 ✅
 │
 └─ 15:00 又听到用 P-RNTI 加密的寻呼 DCI ───► 拦截！今天处理过 P-RNTI 了。 ❌
```

### 💡 章节硬核总结

**本段协议在 L1 物理层到 MAC/RRC 层的信令递交接口处，设立了一道针对公共控制信息（Common Control Information）的时隙级“去重（De-duplication）”与“防过载”屏障。对于 SI-RNTI, P-RNTI 等广播性质的标识，网络侧常常会利用多波束扫描（Beam Sweeping）或信道重复（Repetition）在一个 Slot 的不同 Symbol/CORESET 内多次发送相同的 DCI，以获取时间/空间分集增益。协议通过 `does not expect to process` 语义，明确解除了 UE 基带需要对同一 Slot 内多个同源公共 DCI 提取 Payload 并重复向上层传递的状态机义务。这不仅极大简化了 UE L1 状态机的复杂性（避免了针对同一个 RNTI 维护多个 Payload 缓冲），也从法理上杜绝了网络侧在一个 Slot 内利用相同公共 RNTI 下发互斥或矛盾调度指令的非法行为。**

## 📖 协议原文拆解 (十六)：一层楼最多能挂几块黑板？(CORESET 数量的物理极限)

> **协议原文**
> For each DL BWP configured to a UE in a serving cell, the UE can be provided by higher layer signalling with
> - $P \le 3$ CORESETs if coresetPoolIndex is not provided, or if a value of coresetPoolIndex is same for all CORESETs if coresetPoolIndex is provided
> - $P \le 5$ CORESETs if coresetPoolIndex is not provided for a first CORESET, or is provided and has a value 0 for a first CORESET, and is provided and has a value 1 for a second CORESET

### 📚 增量术语表
* **CORESETs (Control Resource Sets)**：控制资源集。**一句话解释**：基站划定用来贴控制指令（DCI）的小纸条的“物理黑板”。
* **coresetPoolIndex**：CORESET 池索引。**一句话解释**：我们在前文（第二十一段）讲算力折算时提过，这是多天线塔（mTRP）协作的核心标志。如果黑板被标上了不同的数字（0 和 1），说明这层楼的黑板是由两个不同的天线塔在分别管理的。

### 🚪 第一关：给“黑板”的数量设上限
在 5G 中，手机的一个下行工作频段（DL BWP）就像是公司里的一层办公楼。
基站为了调度灵活，可以在这层楼里挂好几块“黑板（CORESET）”。比如 0号黑板在走廊头，1号黑板在电梯口，2号黑板在会议室。
但是，黑板不能无限挂！挂多了手机记不住，也没那么多底层寄存器去存这些黑板的参数（如频域大小、时域长度、波束配置等）。
所以，协议给这层楼里**最多能挂几块黑板（$P$）**设定了严格的硬件上限。

### 🎯 第二关：单塔与双塔环境下的不同“配额”
这个上限 $P$ 是多少呢？取决于这层楼是由几个“老板（天线塔）”在管：

**1. 单塔环境（常规状态）**：
* **条件**：黑板上都没有写老板名字（`coresetPoolIndex is not provided`），或者所有黑板上写的老板名字都一样（`same for all CORESETs`）。这说明这层楼只有一个天线塔在发号施令。
* **上限**：**$P \le 3$**。这层楼最多只能挂 **3 块**黑板！通常是 0号黑板用来听广播，剩下两块用来做不同优先级的私人调度。

**2. 双塔协作环境（mTRP 状态）**：
* **条件**：黑板上明确出现了分工！一部分黑板归 0号老板管（`value 0`），另一部分黑板归 1号老板管（`value 1`）。这说明手机现在享受的是双塔并发传输的顶级待遇。
* **上限**：**$P \le 5$**。因为有两个老板在同时给你贴纸条，3 块黑板显然不够分了，协议把这层楼的黑板总数上限放宽到了 **5 块**！

### 🔄 BWP 内 CORESET 数量限制判定树
```text
【基站准备给手机的某个 DL BWP 配置物理黑板 (CORESET)】

        ├─► 检查：准备配置 mTRP 双塔协作吗？(是否使用了两个不同的 coresetPoolIndex)
        │
        ├──► 【否】 (单塔发信)
        │       └─► 硬件资源锁死！
        │           这个 BWP 内部，最多只能配置 3 块黑板 (P <= 3)。
        │           (若基站强行下发 4 块，手机底层 RRC 解析将报错)
        │
        └──► 【是】 (双塔发信，部分黑板 Pool=0，部分 Pool=1)
                └─► 硬件资源扩容！
                    这个 BWP 内部，最多允许配置 5 块黑板 (P <= 5)。
                    (给了双塔更多的调度维度和频域分布的灵活性)
```

### 💡 章节硬核总结

**本段协议定义了 L1 物理层中一个极其关键的内存状态机容量指标：每个 DL BWP 内支持的 CORESET 实例最大实例化数量。CORESET 作为控制信道的物理承载容器，其每一个实例在基带底层都需要占据独立的寄存器组来维护其频域 Bitmap、时域 Duration、Cce-REG mapping 规则以及专属的 TCI-State (QCL 假设)。在基础的单 TRP (Transmission and Reception Point) 模型下，协议将其硬性约束为 $P \le 3$，极大地限制了控制面参数的膨胀。然而，随着 Rel-16/17 引入多 TRP 协同（基于 `coresetPoolIndex` 的 Non-Coherent Joint Transmission, NCJT），单 BWP 必须能够同时锚定来自两个物理空间且互不相干的波束源。为了支撑这种空间解耦调度的灵活性，协议在此处为 mTRP 特性专门打上了扩容补丁，将 CORESET 预算上限提升至 $P \le 5$，为 5G 高阶多工调度奠定了物理拓扑基础。**

## 📖 协议原文拆解 (十七)：解剖“黑板”的物理结构 (CORESET 全参数解析)

> **协议原文**
> For each CORESET, the UE is provided the following by ControlResourceSet:
> - a CORESET index $p$, by controlResourceSetId or by controlResourceSetId-v1610, where
>  - $0 < p < 12$ if coresetPoolIndex is not provided, or if a value of coresetPoolIndex is same for all CORESETs if coresetPoolIndex is provided;
>  - $0 < p < 16$ if coresetPoolIndex is not provided for a first CORESET, or is provided and has a value 0 for a first CORESET, and is provided and has a value 1 for a second CORESET;
> - a DM-RS scrambling sequence initialization value by pdcch-DMRS-ScramblingID;
> - a precoder granularity for a number of REGs in the frequency domain where the UE can assume use of a same DM-RS precoder by precoderGranularity;
> - a number of consecutive symbols provided by duration;
> - a set of resource blocks provided by frequencyDomainResources;
> - CCE-to-REG mapping parameters provided by cce-REG-MappingType;
> - an antenna port quasi co-location, from a set of antenna port quasi co-locations provided by TCI-State, indicating quasi co-location information of the DM-RS antenna port for PDCCH reception;
> - an indication for a presence or absence of a transmission configuration indication (TCI) field for a DCI format, other than DCI format 1_0, that schedules PDSCH receptions or has associated HARQ-ACK information without scheduling PDSCH and is provided by a PDCCH in CORESET $p$, by tci-PresentInDCI or tci-PresentDCI-1-2.

### 📚 增量术语表
* **REG (Resource Element Group)**：资源粒子组。**一句话解释**：这是构成黑板（CORESET）的“基本砖块”。1 个 REG 在频域上占 1 块（12个子载波），在时域上占 1 个符号。若干个 REG 拼起来，就成了一个 CCE（写字的格子）。
* **Precoder granularity**：预编码粒度。**一句话解释**：告诉手机“基站的发射天线在多大范围内是用同一个角度发信号的”。如果粒度大，手机可以把多个砖块（REG）的信号捏在一起算，提高清晰度。

### 🚪 第一关：给每块黑板发一张“身份证” (Index $p$)
在上一段我们知道了一层楼最多挂 3 块或 5 块黑板。
这段协议的第一条，就是给这挂出来的每一块黑板发一个全小区唯一的“身份证号”（**CORESET Index $p$**）。
* **注意：为什么 $0 < p$？** 因为 `0号` 身份证是神圣不可侵犯的，它是留给我们在前面（第四段）拆解过的那个系统级“0号黑板”专用的。这里配置的是 1 号、2 号、3 号……这些后天新建的黑板。
* **号码池上限**：
  * 如果是单塔环境（无 `coresetPoolIndex` 区分），黑板的号码池是 **1 ~ 11**。
  * 如果是双塔协作环境（mTRP），基站的牌面更大了，黑板的号码池扩容到了 **1 ~ 15**。

*(你可能会问：既然最多只能挂 5 块黑板，要 15 个号码干嘛？因为号码是全小区通用的，你可以在这层楼挂 2 号和 5 号黑板，在另一层楼挂 3 号和 8 号黑板。但任意一层楼同时挂出来的数量不能超标。)*

### 🎯 第二关：“造黑板”的 7 项核心物理参数
基站在建一块黑板时，必须给出极其详尽的“设计图纸”（`ControlResourceSet`），这份图纸包含了以下 7 个物理层面的绝对参数：

1. **`pdcch-DMRS-ScramblingID` (导频密码本)**：解这块黑板上导频信号的种子密码。（不像0号黑板那样傻瓜式地用 Cell ID）。
2. **`precoderGranularity` (天线平滑度)**：告诉手机这块黑板的信号“材质”有多平滑。如果平滑度高，手机可以大面积跨格子进行信号降噪。
3. **`duration` (时域高度)**：这块黑板在时间上占**几个符号**（比如 1、2、3 个符号）。这是黑板的高度。
4. **`frequencyDomainResources` (频域宽度)**：这块黑板在频率上占**哪些具体的 RB 资源块**（用一串 0 和 1 的 Bitmap 表示）。这是黑板的宽度。
5. **`cce-REG-MappingType` (写字排版规则)**：组成这块黑板的“小砖块（REG）”是怎么拼成“写字格（CCE）”的？是连续排（交织前），还是像打乱的拼图一样散开排（交织后抗干扰）。
6. **`TCI-State` (波束朝向)**：这块黑板是朝着哪个物理角度（波束）打过来的。手机必须按这个角度调整接收天线。
7. **`tci-PresentInDCI` (内置波束方向盘)**：这个极其重要！这块黑板上的纸条（DCI）里，**是否自带了一个用来遥控后续数据（PDSCH）波束方向的“方向盘”字段**。如果有，手机还得额外分出算力去解这个方向盘。

### 🔄 CORESET 的全参数三维立体图
```text
【用一份 ControlResourceSet 图纸，在空气中建造一块黑板】

 ┌── 1. 挂哪？ (波束方向 TCI-State)
 │
 ├── 2. 怎么拼？(时域 Duration + 频域 resources) ──► 决定了黑板的 长×宽
 │
 ├── 3. 密码是啥？(DMRS Scrambling ID)
 │
 ├── 4. 砖块怎么砌？(CCE-to-REG 映射关系)
 │
 └── 5. 高级功能：黑板上贴的纸条，带不带用来改变下一步动作波束的 "遥控器"？
        (tci-PresentInDCI 开关)
```

### 💡 章节硬核总结

**本段协议在 RRC 信令字典（ASN.1 IE: `ControlResourceSet`）的尺度上，对物理下行控制信道的“宏观容器（Macro-container）”——CORESET 进行了全面的物理参数解构。它清晰地刻画了控制平面调度的两个正交维度：1) 物理资源映射维度，包括通过 `frequencyDomainResources` 定义频域 Bitmap，通过 `duration` 定义时域深度，以及通过 `cce-REG-MappingType` 定义从微观资源粒子（REG）到逻辑信道元素（CCE）的映射图谱。2) 空间与信道估计维度，包括定义宽带信道平滑特性的 `precoderGranularity`，定义序列初始化的加扰 ID，以及最重要的定义空间接收滤波器对齐准则的 `TCI-State`。这段参数集合是 UE 基带在执行任何具体的 PDCCH 盲检前，必须首先在底层 DSP 寄存器中完成初始化的“黑板基底（Chalkboard Canvas）”。**

## 📖 协议原文拆解 (十八)：“连片平滑”模式下的严苛施工标准 (预编码粒度限制)

> **协议原文**
> When precoderGranularity = allContiguousRBs, a UE does not expect
> - to be configured a set of resource blocks of a CORESET that includes more than four sub-sets of resource blocks that are not contiguous in frequency
> - any RE of a CORESET to overlap with any RE determined from
>  - lte-CRS-ToMatchAround or LTE-CRS-PatternList, if the UE is not provided pdcchCandidateReception-WithCRSOverlap, or
>  - a SS/PBCH block.

### 📚 增量术语表
* **precoderGranularity = allContiguousRBs**：全连续RB平滑模式。**一句话解释**：这是上一段提到的“天线平滑度”的最高级别。基站向手机承诺：“我这块黑板上，所有物理上连续相连的砖块（RBs），都是用同一个天线角度打过去的。”手机可以直接把这整片连续砖块的信号捏在一起做降噪（联合信道估计），信号清晰度极速飙升！
* **sub-sets of resource blocks that are not contiguous**：不连续的资源块子集。**一句话解释**：黑板在频率上被切成了几块碎片，中间断开了。

### 🚪 第一关：想要极致平滑，黑板就不能太“碎”
如果基站开启了 `allContiguousRBs`（全连片平滑模式），意味着它想让手机在频域上大面积联合处理信号来提升接收性能。
但是，如果基站在画这块黑板时，把它画得极其零碎（比如一会在低频画一段，一会在高频画一段，像斑马线一样），手机去跨越这些鸿沟做联合处理是非常消耗算力的，甚至算不过来。

所以协议给基站下了**第一道死命令 (`does not expect`)**：
* 如果你开了连片平滑模式，这块黑板在频率上，**最多只能断开成 4 段（不能超过 4 个不连续的子集）**。如果被切得比 4 段还碎，手机直接报错罢工！

### 🎯 第二关：连片平滑的“死敌”——物理阻隔 (钉子与救护车)
连片平滑模式最怕什么？最怕这片原本连续的信号里，**混进了别的杂质信号**。一旦手机把杂质信号也一起“平滑”进去了，解出来的结果全都是错的。

协议给出了**第二道绝对红线 (`does not expect`)**，在这块开启了连片平滑的黑板里，**哪怕一个像素点（RE）**，也绝对不允许和以下两种“杂质”撞车：

1. **4G 地钉 (LTE CRS)**：
   除非基站给你发了高级缝补技能书（`pdcchCandidateReception-WithCRSOverlap`，我们在第 19 段讲过），否则绝对不允许这块黑板压在 4G 的老钉子上！因为手机做不到一边大面积平滑信号，一边还要绕开地钉。
2. **5G 救护车 (SS/PBCH block)**：
   绝对不能压在 SSB 上！SSB 的能量极大，波束特征也跟这块普通黑板完全不同，如果把它“平滑”进去了，等于在清汤锅里倒了一盆辣椒油，整个黑板的数据瞬间报废。

### 🔄 预编码粒度的“双规”保护网

```text
【基站开启了最高清晰度模式：precoderGranularity = allContiguousRBs】

         ┌─► 检查黑板形状：断层了多少块？
         │      ├── > 4 块 ──► 🚨 [does not expect] 罢工！太碎了没法平滑。
         │      └── <= 4 块 ─► ✅ 形状合格。
         │
手机底层自检 ┤
         │
         └─► 检查黑板像素(RE)纯净度：混入杂质了吗？
                ├── 压到了 4G CRS (且没补丁技能) ──► 🚨 [does not expect] 罢工！
                ├── 压到了 5G SSB 救护车 ─────────► 🚨 [does not expect] 罢工！
                └── 绝对纯净 ─────────────────────► ✅ 开启大面积平滑计算，极致清晰解码！
```

### 💡 章节硬核总结

**本段协议是 5G 物理层中极其典型的“基于信道估计算法边界的配置限制（Configuration Restriction based on Channel Estimation Boundaries）”。当 `precoderGranularity` 被设置为 `allContiguousRBs` 时，UE 将启动宽带信道估计（Wideband Channel Estimation）算法，跨越连续的 PRB 边界进行信道平滑插值（Interpolation/Filtering）以获取极高的接收信噪比增益。然而，宽带估计的阿喀琉斯之踵在于对频域非连续性和外源干扰的极度敏感。因此，协议在此强行植入了三重 L1 保护原语 `does not expect`：1) 限制频率碎片的数量 $\le 4$，防止 UE PRB-bundling 硬件流水线断档；2) 严禁与非授权的 LTE CRS 发生 RE 交叠；3) 严禁与 SS/PBCH Block 交叠。这些强制隔离措施确保了 UE DSP 在执行大面积 PRB-bundling 时，其滤波器窗口内采集到的信号样本矩阵处于绝对“平稳同构（Stationary and Homogeneous）”的理想状态。**
