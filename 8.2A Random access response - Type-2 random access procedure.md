# 8.2A Random access response - Type-2 random access procedure (2步随机接入)

在这部分协议中，我们正式跨入了 5G Release 16 引入的极速版接入——**“2步随机接入（Type-2 RACH）”**。它将传统的 4 步流程压缩成了 2 步，专治各种对网络延迟要求极高的场景。

---

## 📚 术语表（通信小白的翻译魔芋）

* **Type-2 RACH**：2步随机接入。**一句话解释**：把传统的 4步磨叽流程，压缩成极速的 2步，专治各种低延迟要求。
* **MsgA (Message A)**：消息A。**一句话解释**：手机把原来的“招手排号（Msg1）”和“自报家门（Msg3）”打包在一起，一口气全发给基站。
* **MsgB (Message B)**：消息B（也就是标题里的 Type-2 RAR）。**一句话解释**：基站把原来的“回执（Msg2）”和“冲突解决（Msg4）”打包在一起，一口气全回给手机。
* **successRAR**：成功响应。**一句话解释**：基站完美听清了你的 MsgA，直接回复你“接入成功，可以上网了”。
* **fallbackRAR**：回退响应。**一句话解释**：基站只听到了你招手（PRACH），但没听清你后面报的名字（PUSCH），于是把你“降级”打回传统的 4 步流程。
* **MsgB-RNTI**：消息B的无线网络临时标识。**一句话解释**：专门用于呼叫 2步接入手机的“极速大喇叭频道”。
* **msgB-ResponseWindow**：MsgB 响应窗口。**一句话解释**：手机苦苦等待基站回信的“合法蹲守期限”，超过这个时间判定翻车。
* **PUSCH occasion (PO)**：PUSCH 发射时机。**一句话解释**：手机发射 MsgA 里的那个“名字和菜单数据包（PUSCH）”的具体物理时间段。
* **PRACH occasion (RO)**：PRACH 发射时机。**一句话解释**：手机发射 MsgA 里“暗号（PRACH）”的具体物理时间段。
* **$T_{TA}$ (Timing Advance)**：时间提前量。**一句话解释**：为了抵消电磁波跑路的物理延迟，系统规定的“提前抢跑/时延补偿”时间。

---

## 🍔 核心机制：餐厅点菜大升级与“盲盒”审判

### 1. 为什么要有 Type-2？
* **旧的 4 步流程（Type-1）**：
  * 你：“服务员，我要个号！” (Msg1)
  * 服务员：“你是 5 号，去那桌。” (Msg2)
  * 你走到桌前：“我叫张三，我要点宫保鸡丁。” (Msg3)
  * 服务员：“收到，张三的宫保鸡丁！” (Msg4)
  * **痛点**：一来一回交互 4 次，太慢了。
* **新的 2 步流程（Type-2）**：
  * 你冲进餐厅直接大喊：“服务员，我是张三，我要点宫保鸡丁！” (**MsgA = Msg1 + Msg3**)
  * 服务员如果听清了，直接回复：“张三，宫保鸡丁马上来！” (**MsgB = Msg2 + Msg4**)

### 2. MsgB 的“开盲盒”机制
基站收到 MsgA 后给你回的 MsgB，其实是一个**盲盒**，包含两种截然不同的命运：
* **大吉（开出 successRAR）**：基站不仅看到了你招手，还听清了名字。下发确认，**手机直接通关，接入成功**。
* **中吉（开出 fallbackRAR）**：基站只看到招手，没听清名字。下发回退指示（UL Grant），**迫使手机降级回退到 4步流程**，老老实实补发 Msg3。

### 🔄 Type-2 RACH 宏观流水线
```text
 手机 (UE)                                                 基站 (gNB)
    │                                                          │
    ├─────► 1. 极速发车！发送 MsgA (暗号 PRACH + 名字 PUSCH) ───►│ (基站开始拼命解码)
    │                                                          │
    │◄───── 2. 接收 MsgB (也就是 Type-2 的 RAR) ───────────────┤ 
    │                                                          │
    ├─ 开始拆盲盒解析 MsgB：                                     │
    │                                                          │
    ├──► 【大吉分支：包含 successRAR】                           │
    │         │                                                │
    │         └─► 完美！基站全听清了。接入直接大结局，开始上网！        │
    │                                                          │
    └──► 【中吉分支：包含 fallbackRAR】                          │
              │                                                │
              └─► 尴尬！没听清名字。被迫回退 (Fallback)              │
                  拿出 fallbackRAR 里的 UL Grant，准备补发 Msg3  │
                      │                                        │
                      ├─► 补发 Msg3 (乖乖重报名字) ───────────►│
```

## 📖 协议原文拆解 (一)：MsgB 的正常监听倒计时法则 (发了暗号和数据)

> **协议原文**
> In response to a transmission of a PRACH and a PUSCH, or to a transmission of only a PRACH if the PRACH preamble is mapped to a valid PUSCH occasion, a UE attempts to detect a DCI format 1_0 with CRC scrambled by a corresponding MsgB-RNTI during a window controlled by higher layers [11, TS 38.321]. The window starts at the first symbol of the earliest CORESET the UE is configured to receive PDCCH for Type1-PDCCH CSS set, as defined in clause 10.1, that is at least one symbol, after the last symbol of the PUSCH occasion corresponding to the PRACH transmission, where the symbol duration corresponds to the SCS for Type1-PDCCH CSS set. If $N_{TA,adj}^{UE}$ or $N_{TA,adj}^{common}$, as defined in [4, TS 38.211], is not zero, the window starts after an additional $T_{TA} + k_{mac}$ msec where $T_{TA}$ is defined in [4, TS 38.211] and $k_{mac}$ is provided by kmac or $k_{mac}=0$ if kmac is not provided. The length of the window in number of slots, based on the SCS for Type1-PDCCH CSS set, is provided by msgB-ResponseWindow.

### 🚪 第一关：谁有资格来蹲守？（触发条件）
* **核心含义**：只要手机一口气把 PRACH（暗号）和 PUSCH（数据包）都发了出去，或者使用了强制绑定 PUSCH 的特权暗号，就可以开启监听。
* **工作动作**：手机开始死死盯住大屏幕，寻找带着自己 `MsgB-RNTI` 防伪印章的控制小纸条（DCI 1_0）。

### ⏱️ 第二关：蹲守从哪一秒开始算？（窗口起点）
* **痛点**：手机刚发完数据，基站处理需要时间，立刻监听纯属浪费电。
* **起跑线法则**：手机发完 MsgA 的数据部分（PUSCH occasion 的最后一个符号）之后，**强制歇一口气（至少等待 1 个符号的时间）**。
* **寻找大屏幕**：休息完后，寻找离当前时间最近的“官方大屏幕”（Type1-PDCCH CSS 的 CORESET）。这块屏幕亮起的第一秒，就是监听窗口打开的起点。

### 🛰️ 第三关：卫星/远距离补丁（时空延时补偿）
* **核心机制**：如果在特定网络（如卫星通信）配置了 $T_{TA}$ 或不为零的补偿值，说明信号在空中飞需要很久。
* **现实意义**：为了省电，监听窗口的起点必须**再往后推延**一段时间（$T_{TA} + k_{mac}$ 毫秒），等包裹飞到家门口了再开门。

### 📏 第四关：门开多久会关？（窗口长度）
* **核心机制**：窗口的寿命由高层配置的 `msgB-ResponseWindow` 决定（比如 10 个时隙）。长度按照大屏幕所在频段的车道宽度（SCS）来度量。超过长度没听到 MsgB-RNTI，直接判定翻车。

### 🔄 MsgB 监听窗口倒计时流水线 (含数据发送)
```text
 时间轴向前推进 ──►
 
 [发送 MsgA 结束]
 (即 PUSCH occasion 最后一刻)
        │
        ├─► 1. 喘口气 (强制等待至少 1 个符号)
        │
        ├─► 2. 远距离修正？(若配置了 TA 补偿，强制再等 T_TA + k_mac)
        │
        ▼ 
 【寻找最近的营业点】找到最近的 Type1 CSS 大屏幕
        │
  ==== 窗口正式开启！开始监听 MsgB-RNTI (DCI 1_0) ====
        │
        │◄── 漫长的等待，长度由 msgB-ResponseWindow 决定 ──►│
        │
  ==== 窗口无情关闭！ ==================================================
        │
        ▼ (若关门前没监听到，宣告 Type-2 接入失败)
```

## 📖 协议原文拆解 (二)：只发暗号没发数据的“空手套白狼”场景

> **协议原文**
> In response to a transmission of a PRACH, if the PRACH preamble is not mapped to a valid PUSCH occasion, a UE attempts to detect a DCI format 1_0 with CRC scrambled by a corresponding MsgB-RNTI during a window controlled by higher layers [11, TS 38.321]. The window starts at the first symbol of the earliest CORESET the UE is configured to receive PDCCH for Type1-PDCCH CSS set, as defined in clause 10.1, that is at least one symbol, after the last symbol of the PRACH occasion corresponding to the PRACH transmission, where the symbol duration corresponds to the SCS for Type1-PDCCH CSS set. The length of the window in number of slots, based on the SCS for Type1-PDCCH CSS set, is provided by msgB-ResponseWindow.

### 🚪 第一关：特殊的触发条件（只发了 PRACH）
* **场景描述**：这是 2步接入里的一种特殊“残缺”状态。手机本该把 PRACH 和 PUSCH 一起打包成 MsgA 发出去，但它挑的这个暗号（PRACH preamble）在系统日历上**并没有绑定任何有效的数据车道（not mapped to a valid PUSCH occasion）**。
* **尴尬的现实**：导致手机只发出了“招手暗号（PRACH）”，手里攥着的“数据包裹（PUSCH）”根本没发出去。
* **硬核规定**：即便如此，协议依然允许手机去死死盯住大屏幕，监听 `MsgB-RNTI`。反正来都来了，看看基站会不会理我。

### ⏱️ 第二关：起跑线变了（以 PRACH 为基准）
* **对比上一段的核心差异**：
  * **上一段（发了数据）**：是从发完数据（PUSCH occasion）的最后一刻开始计时。
  * **这一段（没发数据）**：既然没发 PUSCH，那就只能**从发完暗号（PRACH occasion）的最后一个符号开始计时！**
* **动作法则**：手机发完 PRACH 后，照样**喘口气（休息至少 1 个符号）**，然后寻找最近亮起的“官方大屏幕（Type1 CSS 的最早 CORESET）”，作为监听窗口的起点。
* **消失的卫星补丁**：你会发现，这段话里**完全没有提 $T_{TA}$ 时延补偿**。为什么？因为手机连数据都没发，只发了个小小的暗号，基站处理起来极快，加上没有复杂的双向同步问题，协议在这里直接砍掉了那个复杂的补偿等待期，直接起跑！

### 📏 第三关：一样的关门法则（窗口长度不变）
* **规则继承**：虽然起跑线变了，但窗口能开多长时间，规矩没变，依然是由基站下发的 `msgB-ResponseWindow` 决定（比如还是等 10 个时隙），到点就无情关门。

### 🔄 MsgB 监听窗口倒计时流水线 (没发数据场景)
```text
 时间轴向前推进 ──►
 
 [仅仅发完了 PRACH 暗号]
 (也就是 PRACH occasion 的最后一刻，没有后续的 PUSCH了)
        │
        ├─► 1. 喘口气 (强制等待至少 1 个符号)
        │
        ▼ 
 【寻找最近的营业点】找到最近的 Type1 CSS 大屏幕
        │
  ==== 窗口正式开启！开始监听 MsgB-RNTI (DCI 1_0) ====
        │
        │◄── 漫长的等待，长度由 msgB-ResponseWindow 决定 ──►│
        │
  ==== 窗口无情关闭！ ==================================================
        │
        ▼ (注：这种情况下，哪怕基站回信了，也必定是 fallbackRAR，让你回去重发数据)
```

## 📖 协议原文拆解 (三)：拆开 MsgB 盲盒，宣判命运（Fallback vs Success）

> **协议原文**
> ... receives a transport block in a corresponding PUSCH within the window, the UE passes the transport block to higher layers. The higher layers indicate to the physical layer
> - an uplink grant if the RAR message(s) is for fallbackRAR and a random access preamble identity (RAPID) associated with the PRACH transmission is identified...
> - transmission of a PUCCH with HARQ-ACK information having ACK value if the RAR message(s) is for successRAR, where...

### 🚪 第一关：拆开盲盒，下达命运指令
手机在窗口内成功收到数据大包裹（Transport Block）后，立刻递交给高层（MAC层）拆快递。高层拆开后，会向物理层下达两条截然不同的命运指令：

* **命运 A：中吉（fallbackRAR 降级回退）**
  * **场景**：在包裹里找到了 `fallbackRAR` 且核对了暗号（RAPID）。说明基站只听到了暗号，没听清后面的数据。
  * **动作**：高层把包裹里的“上行通行证（UL Grant）”拍给物理层，命令手机：“**老老实实回退到传统的 4步接入流程**，把 Msg3 重发一遍！”
* **命运 B：大吉（successRAR 瞬间通关）**
  * **场景**：看到了 `successRAR`，说明基站不仅听到了暗号，还完美接收了数据。
  * **动作**：高层狂喜，命令物理层：“立刻去 PUCCH（上行控制信道）上**给基站发一个 `ACK`（收到确认签收单）**，告诉基站兄弟们对接完毕！”

### ⏱️ 第二关：发射通关确认信（PUCCH ACK）的五大铁律
如果抽到了大吉，手机要去发这封 `ACK` 确认信，但这封信必须严格遵守 successRAR 里规定的 5 个条件：

1. **在哪条车道发？（提车资源）**
   * 解析 successRAR 里 4比特 的资源指示符，从公共大车库（`pucch-ResourceCommon`）里提出指定的 PUCCH 资源来用。
2. **在几点几分发？（掐表算时隙）**
   * 解析 3比特 的时间指示符 $k$。精确的发送时隙为：`n + k + Δ + 2^μ * K_cell,offset`。（以当前收到 MsgB 的时隙 $n$ 为基准进行偏移叠加）。
3. **物理极限红线（预留脑力时间）**
   * **硬性规定**：手机发 ACK 的时间，距离收到 MsgB 的最后一刻，**绝对不能小于 $N_{T,1} + 0.5$ 毫秒**。
   * $N_{T,1}$ 是手机芯片消化大包裹所需的脑力处理时间。基站如果强行催促，手机可以合法拒绝发信（does not expect）。
4. **如果在公共马路发？（非授权频段 NR-U 探路法则）**
   * 如果在公共 Wi-Fi 频段发，必须服从 `ChannelAccess-CPext` 字段指挥。
   * **FR1（低中频）**：先听后说（LBT） + 提前扔雪糕筒占坑（CP extension）。
   * **FR2-2（毫米波）**：只听不说，探好路直接冲，无需占坑。
5. **眼神看哪里？（波束方向继承）**
   * 手机发这封 ACK 的天线方向（空间滤波器）和频段（BWP），必须和刚才发射 MsgA 数据（last PUSCH）时**一模一样**！坚决贯彻原路返回原则。

### 🔄 MsgB 盲盒拆解与动作流水线
```text
 高层 (MAC) 拆解 MsgB 盲盒
    │
    ├─ 问自己：抽到了什么签？
    │
    ├──► 【抽到 fallbackRAR (中吉)】
    │         │
    │         └─► 动作: 提取 UL Grant 下发物理层
    │             (手机含泪降级，去发射 Msg3)
    │
    └──► 【抽到 successRAR (大吉)】
              │
              ├─► 动作 1: 准备发射通关确认信 (PUCCH ACK)
              ├─► 动作 2: 去车库提车 (解析 4-bit PUCCH 资源指示)
              ├─► 动作 3: 掐表算时间 (时隙 n + k + 各类偏移量)
              ├─► 动作 4: 物理红线校验 (确保时间间隔 > NT,1 + 0.5ms)
              ├─► 动作 5: 探路准备 (如果是公共频段，执行 LBT 甚至占坑)
              └─► 动作 6: 原路瞄准 (对准发 MsgA 时的波束方向)
                        │
                        ▼
                 成功发射 ACK，2步极速接入完美落幕！🚀
```

## 📖 协议原文拆解 (四)：已连接状态下的“专属 VIP 快递”（C-RNTI 寻呼）

> **协议原文**
> If the UE detects the DCI format 1_0 with CRC scrambled by a C-RNTI and a transport block in a corresponding PDSCH within the window, the UE transmits a PUCCH with HARQ-ACK information having ACK value if the UE correctly detects the transport block or NACK value if the UE incorrectly detects the transport block and the time alignment timer is running [11, TS 38.321].

### 📚 术语表
* **C-RNTI (Cell Radio Network Temporary Identifier)**：小区无线网络临时标识。**一句话解释**：手机入网后分配的“正式实名VIP卡号”，基站用它来和你进行一对一的私密通话。
* **NACK (Negative Acknowledgement)**：否定确认。**一句话解释**：给基站发的“差评退回单”，告诉基站：“包裹收到了，但是里面东西碎了（解码失败），请重发一次！”
* **Time Alignment Timer (TAT)**：时间对齐定时器。**一句话解释**：手机内部的“上行同步保质期沙漏”。沙漏没漏完，说明你的信号能精准打到基站；沙漏一旦漏完，你发的东西就会撞车，必须闭嘴。

### 🚪 第一关：收到专属 VIP 快递（C-RNTI 加扰）
* **场景倒带**：手机本来就在上网（拥有正式的 C-RNTI），因为某些原因（比如断流了想催基站）发起了随机接入。
* **动作法则**：在监听窗口内，手机没有等到公共的 `MsgB-RNTI`，反而等到了**用自己“正式VIP卡号（C-RNTI）”加扰的 DCI 小纸条**！
* **现实意义**：基站的意思是：“老客户了，别走什么公共通道了，你的快递我直接走私密专线发给你了！” 随后手机顺着指引去 PDSCH 拿到了专属的实名制数据大包裹。

### 📦 第二关：快递签收与退货（ACK vs NACK）
这回拿到的不是盲盒，而是实打实的数据包。拆快递时会面临两种结果：
* **包裹完好（解码成功）**：手机立刻在控制信道（PUCCH）上发一个 **`ACK`（好评签收）**，告诉基站数据我吃下了，接入大圆满。
* **包裹破损（解码失败）**：手机发现里面的数据烂了，这时候它需要在控制信道上发一个 **`NACK`（差评退回）**，要求基站重新传一次。

### ⏱️ 第三关：给“差评”的先决条件（TAT 必须在运行）
这行字是极其隐蔽的红线：`and the time alignment timer is running`。
* **痛点**：如果手机想给基站发 `NACK`，它必须得开口说话（发射 PUCCH）。但如果此时手机的上行同步已经过期（TAT 沙漏漏完了），它一旦开口，发射的电磁波就会严重干扰其他正常上网的手机。
* **霸道规矩**：协议规定，**你想发 `NACK` 要求重传可以，但前提是你的“上行同步沙漏（TAT）”必须还在运行！** 如果沙漏已经漏完，你连发差评的资格都没有，只能乖乖闭嘴，老老实实重新从头走一遍完整的接入流程。

### 🔄 VIP 专线接入与签收流水线
```text
 手机 (UE) [已处于连接态，拥有 C-RNTI]                         基站 (gNB)
    │                                                          │
    ├─────► 发起随机接入请求 (PRACH / MsgA) ────────────────────►│
```

## 📖 协议原文拆解 (五)：接收 MsgB 的波束锁定法则（看哪里的问题）

> **协议原文**
> If the UE detects a DCI format 1_0 with CRC scrambled by the corresponding MsgB-RNTI and receives a transport block within the window in a corresponding PDSCH, the UE may assume same DM-RS antenna port quasi co-location properties, as described in [6, TS 38.214], as for a SS/PBCH block the UE used for PRACH association, as described in clause 8.1, regardless of whether or not the UE is provided TCI-State for the CORESET where the UE receives the PDCCH with the DCI format 1_0.

### 📚 术语表
* **QCL (Quasi Co-Location)**：准共址。**一句话解释**：通信界的“同源/同方向”认证，代表接收天线可以保持同样的姿势去接信号。
* **TCI-State (Transmission Configuration Indicator)**：传输配置指示。**一句话解释**：基站平时下发的“官方交通指南”，规定了手机看某个区域时应该朝哪个方向看。
* **SS/PBCH block (SSB)**：同步信号块。**一句话解释**：基站像灯塔一样定期发射的“广播探照灯”。

### 🚪 第一关：波束方向的绝对继承（QCL 绑定）
* **场景描述**：手机正在窗口期苦苦等待，终于，它看到了印有自己 `MsgB-RNTI` 的控制纸条，准备去 PDSCH 通道接大包裹（MsgB）。
* **核心难题**：5G 的波束极窄，手机如果在这个关键瞬间看错方向，即使包裹到了眼前也会漏掉。
* **一锤定音**：协议规定，手机在接收这个包裹的引导灯（DM-RS）时，其接收方向（QCL）**必须且只能与自己当初发射 MsgA（PRACH）时选中的那个“灯塔（SSB）”保持绝对一致！** 
* **细节差异**：注意，在之前的 4步接入（Type-1）中，协议说可以参考 SSB **或者** CSI-RS。但在极速的 2步接入（Type-2）里，为了把复杂度和出错率降到最低，协议极其干脆地**只认 SSB（基础广播灯塔）**作为空间方向的唯一老祖宗。

### 🛡️ 第二关：霸道条款（无视常规 TCI-State）
* **霸气豁免**：最后一句 `regardless of whether or not the UE is provided TCI-State...` 再次彰显了随机接入流程的“特权”。
* **现实意义**：平时在这个区域收数据，手机得乖乖听基站的 TCI-State 指南来转动天线。但在 2步接入这个节骨眼上，系统还没完全稳定。协议告诉手机：“**不管平时基站怎么规定的，现在你一律无视！你刚才信了哪束光（SSB）把暗号发出去的，现在就死死盯住那个方向接回信！**”这是一种极其粗暴却最不容易翻车的保底策略。

### 🔄 极速接入 (Type-2) 的波束锁定流水线
```text
 手机 (UE) 的天线波束动作                                     基站 (gNB)
      │                                                        │
      │◄─────── 1. 测量：寻找最亮的灯塔 ──────────────────────── (发射多个 SSB 波束)
      │                                                        │
 【锁定灯塔】: 假设锁定正前方的 SSB #0                               │
      │                                                        │
      ├──────── 2. 顺着 SSB #0 的方向，一口气发出 MsgA ─────────►│ (接收 PRACH + PUSCH)
      │                                                        │
 【死守方向】: 天线雷打不动，继续死死盯住 SSB #0 的方向 (同源 QCL)     │
      │                                                        │
      │◄─────── 3. 顺着同样的方向，接住包裹 ──────────────────── (发射 DCI + MsgB)
      │                                                        │
  (无视一切官方 TCI-State 繁文缛节，贯彻“原路返回”的保底铁律！)
```

## 📖 协议原文拆解 (六)：时空悖论防范与 Type-2 翻车重传指南

> **协议原文**
> The UE does not expect to be indicated to transmit the PUCCH with the HARQ-ACK information at a time that is prior to a time when the UE applies a TA command that is provided by the transport block. If the UE does not detect the DCI format 1_0 with CRC scrambled by the corresponding MsgB-RNTI within the window... (四大死法) ... the higher layers can indicate to the physical layer to transmit only PRACH according to Type-1 random access procedure or to transmit both PRACH and PUSCH according to Type-2 random access procedure [11, TS 38.321]. If requested by higher layers, the UE shall be ready to transmit a PRACH no later than $N_{T,1}+0.75$ msec after the last symbol of the window, or the last symbol of the PDSCH reception...

### 📚 术语表
* **TA command (Timing Advance command)**：时间提前量命令。**一句话解释**：包裹（MsgB）里夹带的一张“对表通知”，告诉你由于距离基站多远，以后发信号必须提前抢跑多少毫秒。
* **does not expect**：不期望。**一句话解释**：3GPP 协议里的黑话，意思是“基站绝对不能这么干，如果基站这么干了，手机有权直接报错或无视”。

### 🛡️ 第一关：物理学悖论防范（TA生效前严禁发信）
* **场景倒带**：手机抽到了大吉（successRAR），正准备高高兴兴去发 `ACK`。但包裹里还附带了一张极其重要的“TA对表通知”，用来纠正手机的内部时钟。
* **时空悖论**：如果基站猴急，安排手机发 `ACK` 的时间极其靠前，导致手机**还没来得及消化并应用这张“对表通知”**，就要去发 `ACK`，会发生什么？
* **严重后果**：手机就会用**错误的时间刻度（旧表）**把 `ACK` 发出去，这个信号会像一颗脱轨的导弹，在空中严重干扰其他正常手机的通信。
* **霸道红线**：所以协议硬性规定：手机绝对不接受（does not expect）这种“对表前就催着发信”的离谱调度！基站必须给足手机换算和对表的时间。

### 🚪 第二关：判定翻车的“花式四连跪”（Type-2 版）
如果在等待盲盒的过程中，发生了以下 4 种情况的任意一种，这波极速接入就算彻底翻车：
1. **死法 1（没看懂屏幕）**：窗口期内没听到带 `MsgB-RNTI` 的 DCI。
2. **死法 2（时空错乱）**：听到 DCI 了，但里面的系统帧号（SFN）和当初自己发暗号的时间对不上。
3. **死法 3（包裹损坏）**：去数据通道拿包裹，但包裹破损解不开。
4. **死法 4（里面没我的号）**：包裹拆开了，但高层发现里面全是对别人的回复，没有自己的暗号（RAPID）。

### 🔄 第三关：复活重开的“路线选择”与“冷却倒计时”
只要踩中死法，高层就会命令物理层重新发车。但在 2步接入里，重发面临一个极具策略性的路线选择：

* **路线二选一**：
  * **降级走慢车道（退回 Type-1）**：高层命令：“刚才跑太快摔了，这回稳妥点，**只发 PRACH（暗号）**，老老实实走传统的 4步流程吧！”
  * **头铁继续飙车（重试 Type-2）**：高层命令：“刚才可能只是运气不好，不要怂，**继续把 PRACH和 PUSCH 绑一起**，再走一次 2步极速流程！”
* **复活冷却倒计时（NT,1 + 0.75ms）**：
  * 重发不能立刻执行。物理层需要经历极其精确的冷却期：从“窗口结束”或“接收破包裹结束”的那一刻起，倒数 **$N_{T,1}$（脑力消化时间） + 0.75 毫秒（射频硬件预热时间）**。
  * 倒计时一结束，手机必须立刻“满血复活”，按照选定的路线再次开火！

### 🔄 Type-2 失败重传路线分流图
```text
 时间轴向前推进 ──►
 
 [MsgB 监听窗口 / 包裹解析阶段]
  ├─ 死法1：没听到 DCI
  ├─ 死法2：听到 DCI，时间对不上
  ├─ 死法3：包裹解不开
  └─ 死法4：包裹里没自己的号
         │
         ▼ (确认翻车瞬间，开始计算冷却时间)
         │
 ────────┼───────────────────► 【计算脑力 + 体力冷却时间: NT,1 + 0.75 ms】
         │
         ▼ 冷却结束！高层 (MAC) 下达重传路线指令：
 
         ├─► 【路线 A: 求稳降级】 ──► 发射 PRACH (乖乖退回 Type-1 慢速 4步流程)
         │
         └─► 【路线 B: 头铁继续】 ──► 发射 MsgA (再玩一次 Type-2 极速 2步流程)
```

## 📖 协议原文拆解 (七)：后续数据的车道宽度继承（SCS 继承法则）

> **协议原文**
> Unless the UE is configured a SCS, the UE receives subsequent PDSCH using same SCS as for the PDSCH reception providing the RAR message.

### 📚 增量术语表
* **SCS (Subcarrier Spacing)**：子载波间隔。**一句话解释**：无线频段里的“车道宽度”。车道越宽，跑得越快；车道越窄，跑得越稳。
* **subsequent PDSCH**：后续的物理下行共享信道。**一句话解释**：手机接入成功后，基站源源不断发来的正常业务数据包（比如你刚连上网开始刷的短视频数据）。

### 🚪 核心关卡：接入成功后的“车道宽度”选择
* **场景倒带**：手机历经千辛万苦，刚刚在某个特定的车道宽度（SCS）上，成功接住了基站发来的 RAR 回执包裹，终于宣告接入成功！接下来，手机准备敞开肚皮接收正式的业务数据了（后续的 PDSCH）。
* **面临难题**：5G 是个多车道混合的高速公路，有 15kHz、30kHz、120kHz 等各种宽度的车道。由于刚刚才接上头，手机接下来该去哪种宽度的车道上等数据呢？
* **铁律：无明文，则继承（默认照旧）**：
  * 协议给出了极其简洁的保底法则：**只要基站没有明确给你指定一个新的车道宽度（Unless configured），那你就“照旧”！**
  * 刚才你是用什么宽度的车道接住 RAR 回执包裹的，接下来你就继续在这个宽度的车道上接收后续的业务数据。
* **现实意义**：这是一个经典的“平滑过渡”设计。在刚入网的极短时间内，基站可能还来不及给手机下发极其复杂的高级专属配置（RRC Reconfiguration）。为了防止手机在切换状态时“迷路失联”，系统干脆让它顺着刚才走通的老路继续跑，直到收到明确的改道通知为止。

### 🔄 SCS 继承流水线
```text
 手机接收 RAR 回执 (假设当时处在 SCS = 30kHz 的车道上)
        │
        ├─ 接入成功！准备接收后续刷视频的数据 (subsequent PDSCH)
        │
        ├─ 问自己：基站给我发专属的车道配置指令了吗？
        │
        ├──► 【收到了明确配置 (Configured)】
        │         │
        │         └─► 动作: 听基站的！立刻切换到基站指定的新车道宽度去接收。
        │
        └──► 【没收到明确配置 (Unless... / 默认情况)】
                  │
                  └─► 动作: 照旧！继续死守刚才的 30kHz 车道接收后续数据！
                      (无缝衔接，平稳过渡)
```

## 📖 协议原文拆解 (八)：翻车后的“甩锅法则”（移交高层处理）

> **协议原文**
> If the UE does not detect the DCI format 1_0 with CRC scrambled by the corresponding MsgB-RNTI within the window, or if the UE detects the 1_0 with CRC scrambled by the corresponding MsgB-RNTI within the window and LSBs of a SFN field in the DCI format 1_0, if applicable, are not same as corresponding LSBs of the SFN where the UE transmitted the PRACH, or the UE does not correctly receive a corresponding transport block within the window, the UE procedure is as described in [11, TS 38.321].

### 📚 增量术语表
* **TS 38.321**：3GPP 协议的 MAC（介质访问控制）层规范。**一句话解释**：物理层（负责发波）的“直属领导”，专门负责统筹调度、重传策略和决定是不是要重新走一遍接入流程。

### 🚪 第一关：物理层的“绝望三连”（判定翻车）
* **场景倒带**：在极速接入（Type-2）的监听窗口期内，手机的物理层（“眼睛”）眼巴巴地盯着基站的方向。
* **物理层宣告失败**：如果发生了以下三种极其绝望的情况：
  1. **死法 1（音信全无）**：直到关门，连带有 `MsgB-RNTI` 的小纸条都没看到。
  2. **死法 2（时空错位）**：看到了小纸条，但核对时间（SFN 低位比特）发现，根本不是回应自己刚才那次发射的。
  3. **死法 3（外卖摔烂）**：时间对上了，也去 PDSCH 拿了包裹，但包裹破了，死活解不开（CRC 错误）。

### 🛡️ 第二关：物理层甩锅，移交高层（MAC）裁决
* **核心动作**：物理层只管收发，不负责制定“战略”。所以一旦遇到上面这三种失败，物理层就两手一摊，直接向上汇报：**“老大（MAC层），我尽力了，啥也没接到 / 接烂了。”**
* **协议联动法则**：协议在此处写下 `the UE procedure is as described in [11, TS 38.321]`，就是 5G 体系中经典的**“跨层踢皮球”**。物理层的任务到此结束，接下来手机该怎么“擦屁股”（比如：是调大发射功率重传？还是退回 4步流程？还是干脆放弃报错？），全部听 MAC 层协议（TS 38.321）的指挥。

### 🔄 失败处理：物理层到高层的上报流水线
```text
 物理层 (PHY, TS 38.213)                                 高层 (MAC, TS 38.321)
    │                                                        │
    ├─ 在 MsgB 窗口期内苦苦等待...                            │
    │                                                        │
    ├──► 遇到死法 1: DCI 没找到                              │
    ├──► 遇到死法 2: DCI 找到了，但 SFN 时间对不上             │
    └──► 遇到死法 3: 包裹 (PDSCH) 解码彻底失败                │
             │                                               │
             ▼ (物理层彻底死心)                              │
             │                                               │
             └─► 向上甩锅：“老大，这次 MsgB 接收彻底翻车了！” ─►│ (MAC 层接收失败报告)
                                                             │
                                                             ├─► MAC层翻开自己的小本本 (TS 38.321)
                                                             ├─► 增加失败计数器 (Preamble Transmission Counter)
                                                             ├─► 决定提升功率 (Power Ramping)
                                                             └─► 决定下一次重发策略 (选 Type-1 还是 Type-2)
```

## 🏆 终极硬核总结：5G 极速接入 (Type-2 RACH) 的物理层底层法则

如果说 4步接入（Type-1）是步步为营的“回合制对话”，那么 5G Release 16 引入的 2步接入（Type-2）就是一场**“要么通关、要么回退”的极限飙车**。协议 8.2A 章节就是这场飙车的“物理层交警”，它用极其严密的逻辑，规定了从发射 MsgA 到拆解 MsgB 盲盒的全套生死法则：

### 1. 掐表等信的“物理学严谨” (监听倒计时机制)
手机把“暗号 (PRACH) + 数据 (PUSCH)”打包成 MsgA 一口头发射后，绝不允许立刻开始监听。
* **起跑线平移**：必须强制喘息至少 **1 个符号**。如果发了数据，从发完数据的最后一刻算起；如果因为没资源只发了暗号，就从发完暗号的最后一刻算起。
* **空间折叠补偿**：如果在远距离（如卫星）通信中，监听起点必须向后推延 $T_{TA} + k_{mac}$ 的飞行时间，把手机电量压榨到极致。

### 2. 盲盒审判与双线命运 (MsgB 的拆解法则)
在 `msgB-ResponseWindow` 窗口内，高层（MAC）拆开物理层递交的包裹，将面临两种命运：
* **大吉（successRAR）**：基站全听清了！手机瞬间通关，并在严格遵守 $N_{T,1} + 0.5$ 毫秒的物理红线下，通过 PUCCH 向基站发射 `ACK` 确认信。
* **中吉（fallbackRAR）**：基站只听清了暗号！手机无条件认怂，提取包裹里的 `UL Grant`，**降级回退到 4步流程**，老老实实去补发 Msg3 数据。

### 3. 连接态老客户的“VIP 特权” (C-RNTI 专属流程)
如果发起 Type-2 接入的手机已经处于连接态，它等来的可能不是公共大喇叭，而是用自己真实卡号（C-RNTI）加密的专属私信。此时，流程跳出盲盒机制，直接转为标准数据接收。只有在上行同步沙漏（TAT）还在运行的绝对前提下，手机才被允许发射 `NACK` 要求重传，死死守住不干扰全网的红线。

### 4. 万无一失的空间锚点 (QCL 继承与 TCI 豁免)
在 5G 极窄的波束环境下接收回执，手机该看哪个方向？协议给出了最粗暴也最稳妥的“原路返回原则”：**无视当前区域的一切官方 TCI-State 规矩，你当初信了哪束光（SSB）发出的请求，现在就死死盯住那个方向接回信！**

### 5. 花式死法与重装上阵 (PHY 到 MAC 的跨层踢皮球)
如果窗口期结束没听到 DCI、时钟帧号对不上、或者包裹损坏解不开，物理层绝不自己做主，而是**直接把烂摊子甩给高层（MAC）**。MAC 层接管指挥权后，可以选择头铁继续走 2步（发 MsgA），也可以求稳降级走 4步（发 PRACH）。而在做出决定后，物理层依然必须老老实实度过 $N_{T,1} + 0.75$ 毫秒的硬件预热冷却期，才能再次点火升空！

**结论：8.2A 章节是一套“进可攻（瞬间通关），退可守（降级 4步），败可救（移交高层）”的完美闭环。它用极其微观的时间（毫秒/符号）、空间（QCL 波束）、以及频域（SCS 继承）刻度，兜住了 5G 极速接入的下限，让通信系统在极限飙车时，依然稳如泰山。**
