# 8.2A Random access response - Type-2 random access procedure (2步随机接入)

在这部分协议中，我们正式跨入了 5G Release 16 引入的极速版接入——**“2步随机接入（Type-2 RACH）”**。它将传统的 4 步流程压缩成了 2 步，专治各种对网络延迟要求极高的场景。

---

## 📚 术语表（通信小白的翻译魔芋）

* **Type-2 RACH**：2步随机接入。**一句话解释**：把传统的 4步磨叽流程，压缩成极速的 2步，专治各种低延迟要求。
* **MsgA (Message A)**：消息A。**一句话解释**：手机把原来的“招手排号（Msg1）”和“自报家门（Msg3）”打包在一起，一口气全发给基站。
* **MsgB (Message B)**：消息B（也就是标题里的 Type-2 RAR）。**一句话解释**：基站把原来的“回执（Msg2）”和“冲突解决（Msg4）”打包在一起，一口气全回给手机。
* **successRAR**：成功响应。**一句话解释**：基站完美听清了你的 MsgA，直接回复你“接入成功，可以上网了”。
* **fallbackRAR**：回退响应。**一句话解释**：基站只听到了你招手（PRACH），但没听清你后面报的名字（PUSCH），于是把你“降级”打回传统的 4 步流程。
* **MsgB-RNTI**：消息B的无线网络临时标识。**一句话解释**：专门用于呼叫 2步接入手机的“极速大喇叭频道”。
* **msgB-ResponseWindow**：MsgB 响应窗口。**一句话解释**：手机苦苦等待基站回信的“合法蹲守期限”，超过这个时间判定翻车。
* **PUSCH occasion (PO)**：PUSCH 发射时机。**一句话解释**：手机发射 MsgA 里的那个“名字和菜单数据包（PUSCH）”的具体物理时间段。
* **PRACH occasion (RO)**：PRACH 发射时机。**一句话解释**：手机发射 MsgA 里“暗号（PRACH）”的具体物理时间段。
* **$T_{TA}$ (Timing Advance)**：时间提前量。**一句话解释**：为了抵消电磁波跑路的物理延迟，系统规定的“提前抢跑/时延补偿”时间。

## 🍔 核心机制：餐厅点菜大升级与“盲盒”审判

### 1. 为什么要有 Type-2？
* **旧的 4 步流程（Type-1）**：
  * 你：“服务员，我要个号！” (Msg1)
  * 服务员：“你是 5 号，去那桌。” (Msg2)
  * 你走到桌前：“我叫张三，我要点宫保鸡丁。” (Msg3)
  * 服务员：“收到，张三的宫保鸡丁！” (Msg4)
  * **痛点**：一来一回交互 4 次，太慢了。
* **新的 2 步流程（Type-2）**：
  * 你冲进餐厅直接大喊：“服务员，我是张三，我要点宫保鸡丁！” (**MsgA = Msg1 + Msg3**)
  * 服务员如果听清了，直接回复：“张三，宫保鸡丁马上来！” (**MsgB = Msg2 + Msg4**)

### 2. MsgB 的“开盲盒”机制
基站收到 MsgA 后给你回的 MsgB，其实是一个**盲盒**，包含两种截然不同的命运：
* **大吉（开出 successRAR）**：基站不仅看到了你招手，还听清了名字。下发确认，**手机直接通关，接入成功**。
* **中吉（开出 fallbackRAR）**：基站只看到招手，没听清名字。下发回退指示（UL Grant），**迫使手机降级回退到 4步流程**，老老实实补发 Msg3。

### 🔄 Type-2 RACH 宏观流水线
```text
 手机 (UE)                                                 基站 (gNB)
    │                                                          │
    ├─────► 1. 极速发车！发送 MsgA (暗号 PRACH + 名字 PUSCH) ───►│ (基站开始拼命解码)
    │                                                          │
    │◄───── 2. 接收 MsgB (也就是 Type-2 的 RAR) ───────────────┤ 
    │                                                          │
    ├─ 开始拆盲盒解析 MsgB：                                     │
    │                                                          │
    ├──► 【大吉分支：包含 successRAR】                           │
    │         │                                                │
    │         └─► 完美！基站全听清了。接入直接大结局，开始上网！        │
    │                                                          │
    └──► 【中吉分支：包含 fallbackRAR】                          │
              │                                                │
              └─► 尴尬！没听清名字。被迫回退 (Fallback)              │
                  拿出 fallbackRAR 里的 UL Grant，准备补发 Msg3  │
                      │                                        │
                      ├─► 补发 Msg3 (乖乖重报名字) ───────────►│
```

## 📖 协议原文拆解 (一)：MsgB 的正常监听倒计时法则 (发了暗号和数据)

> **协议原文**
> In response to a transmission of a PRACH and a PUSCH, or to a transmission of only a PRACH if the PRACH preamble is mapped to a valid PUSCH occasion, a UE attempts to detect a DCI format 1_0 with CRC scrambled by a corresponding MsgB-RNTI during a window controlled by higher layers [11, TS 38.321]. The window starts at the first symbol of the earliest CORESET the UE is configured to receive PDCCH for Type1-PDCCH CSS set, as defined in clause 10.1, that is at least one symbol, after the last symbol of the PUSCH occasion corresponding to the PRACH transmission, where the symbol duration corresponds to the SCS for Type1-PDCCH CSS set. If $N_{TA,adj}^{UE}$ or $N_{TA,adj}^{common}$, as defined in [4, TS 38.211], is not zero, the window starts after an additional $T_{TA} + k_{mac}$ msec where $T_{TA}$ is defined in [4, TS 38.211] and $k_{mac}$ is provided by kmac or $k_{mac}=0$ if kmac is not provided. The length of the window in number of slots, based on the SCS for Type1-PDCCH CSS set, is provided by msgB-ResponseWindow.

### 🚪 第一关：谁有资格来蹲守？（触发条件）
* **核心含义**：只要手机一口气把 PRACH（暗号）和 PUSCH（数据包）都发了出去，或者使用了强制绑定 PUSCH 的特权暗号，就可以开启监听。
* **工作动作**：手机开始死死盯住大屏幕，寻找带着自己 `MsgB-RNTI` 防伪印章的控制小纸条（DCI 1_0）。

### ⏱️ 第二关：蹲守从哪一秒开始算？（窗口起点）
* **痛点**：手机刚发完数据，基站处理需要时间，立刻监听纯属浪费电。
* **起跑线法则**：手机发完 MsgA 的数据部分（PUSCH occasion 的最后一个符号）之后，**强制歇一口气（至少等待 1 个符号的时间）**。
* **寻找大屏幕**：休息完后，寻找离当前时间最近的“官方大屏幕”（Type1-PDCCH CSS 的 CORESET）。这块屏幕亮起的第一秒，就是监听窗口打开的起点。

### 🛰️ 第三关：卫星/远距离补丁（时空延时补偿）
* **核心机制**：如果在特定网络（如卫星通信）配置了 $T_{TA}$ 或不为零的补偿值，说明信号在空中飞需要很久。
* **现实意义**：为了省电，监听窗口的起点必须**再往后推延**一段时间（$T_{TA} + k_{mac}$ 毫秒），等包裹飞到家门口了再开门。

### 📏 第四关：门开多久会关？（窗口长度）
* **核心机制**：窗口的寿命由高层配置的 `msgB-ResponseWindow` 决定（比如 10 个时隙）。长度按照大屏幕所在频段的车道宽度（SCS）来度量。超过长度没听到 MsgB-RNTI，直接判定翻车。

### 🔄 MsgB 监听窗口倒计时流水线 (含数据发送)
```text
 时间轴向前推进 ──►
 
 [发送 MsgA 结束]
 (即 PUSCH occasion 最后一刻)
        │
        ├─► 1. 喘口气 (强制等待至少 1 个符号)
        │
        ├─► 2. 远距离修正？(若配置了 TA 补偿，强制再等 T_TA + k_mac)
        │
        ▼ 
 【寻找最近的营业点】找到最近的 Type1 CSS 大屏幕
        │
  ==== 窗口正式开启！开始监听 MsgB-RNTI (DCI 1_0) ====
        │
        │◄── 漫长的等待，长度由 msgB-ResponseWindow 决定 ──►│
        │
  ==== 窗口无情关闭！ ==================================================
        │
        ▼ (若关门前没监听到，宣告 Type-2 接入失败)
```

## 📖 协议原文拆解 (二)：只发暗号没发数据的“空手套白狼”场景

> **协议原文**
> In response to a transmission of a PRACH, if the PRACH preamble is not mapped to a valid PUSCH occasion, a UE attempts to detect a DCI format 1_0 with CRC scrambled by a corresponding MsgB-RNTI during a window controlled by higher layers [11, TS 38.321]. The window starts at the first symbol of the earliest CORESET the UE is configured to receive PDCCH for Type1-PDCCH CSS set, as defined in clause 10.1, that is at least one symbol, after the last symbol of the PRACH occasion corresponding to the PRACH transmission, where the symbol duration corresponds to the SCS for Type1-PDCCH CSS set. The length of the window in number of slots, based on the SCS for Type1-PDCCH CSS set, is provided by msgB-ResponseWindow.

### 🚪 第一关：特殊的触发条件（只发了 PRACH）
* **场景描述**：这是 2步接入里的一种特殊“残缺”状态。手机本该把 PRACH 和 PUSCH 一起打包成 MsgA 发出去，但它挑的这个暗号（PRACH preamble）在系统日历上**并没有绑定任何有效的数据车道（not mapped to a valid PUSCH occasion）**。
* **尴尬的现实**：导致手机只发出了“招手暗号（PRACH）”，手里攥着的“数据包裹（PUSCH）”根本没发出去。
* **硬核规定**：即便如此，协议依然允许手机去死死盯住大屏幕，监听 `MsgB-RNTI`。反正来都来了，看看基站会不会理我。

### ⏱️ 第二关：起跑线变了（以 PRACH 为基准）
* **对比上一段的核心差异**：
  * **上一段（发了数据）**：是从发完数据（PUSCH occasion）的最后一刻开始计时。
  * **这一段（没发数据）**：既然没发 PUSCH，那就只能**从发完暗号（PRACH occasion）的最后一个符号开始计时！**
* **动作法则**：手机发完 PRACH 后，照样**喘口气（休息至少 1 个符号）**，然后寻找最近亮起的“官方大屏幕（Type1 CSS 的最早 CORESET）”，作为监听窗口的起点。
* **消失的卫星补丁**：你会发现，这段话里**完全没有提 $T_{TA}$ 时延补偿**。为什么？因为手机连数据都没发，只发了个小小的暗号，基站处理起来极快，加上没有复杂的双向同步问题，协议在这里直接砍掉了那个复杂的补偿等待期，直接起跑！

### 📏 第三关：一样的关门法则（窗口长度不变）
* **规则继承**：虽然起跑线变了，但窗口能开多长时间，规矩没变，依然是由基站下发的 `msgB-ResponseWindow` 决定（比如还是等 10 个时隙），到点就无情关门。

### 🔄 MsgB 监听窗口倒计时流水线 (没发数据场景)
```text
 时间轴向前推进 ──►
 
 [仅仅发完了 PRACH 暗号]
 (也就是 PRACH occasion 的最后一刻，没有后续的 PUSCH了)
        │
        ├─► 1. 喘口气 (强制等待至少 1 个符号)
        │
        ▼ 
 【寻找最近的营业点】找到最近的 Type1 CSS 大屏幕
        │
  ==== 窗口正式开启！开始监听 MsgB-RNTI (DCI 1_0) ====
        │
        │◄── 漫长的等待，长度由 msgB-ResponseWindow 决定 ──►│
        │
  ==== 窗口无情关闭！ ==================================================
        │
        ▼ (注：这种情况下，哪怕基站回信了，也必定是 fallbackRAR，让你回去重发数据)
```

## 💡 章节硬核总结

**协议在 8.2A 中极其缜密地堵住了一个逻辑漏洞：如果手机走 2 步接入（Type-2）时，因为资源配置问题，导致它“只发出了 MsgA 的暗号（PRACH），却没发出 MsgA 的数据（PUSCH）”该怎么办？协议给出的答案是：不要慌，继续去蹲守 MsgB！只不过倒计时的起跑线，从“数据发送结束”向前平移到了“暗号发送结束”。这也意味着，系统通过这套倒计时机制，完美地把 2 步接入的各种特殊/残缺状态，全都收拢到了同一个大屏幕监听逻辑里。既然你没发数据，那蹲到的盲盒注定只能是 `fallbackRAR`（降级回传统的 4 步流程发数据），但整个系统的时序运转不会有任何卡壳！**
